# 7.1 为什么需要多 Agent：单 Agent 的天花板在哪里

> [!abstract] 本节定义
> 单个 Agent 处理复杂任务时存在三个结构性瓶颈：**上下文窗口限制**（物理边界）、**并行能力缺失**（时间边界）、**专业化不足**（质量边界）。多 Agent 系统通过分工与并发突破这三道边界，但引入多 Agent 本身也有代价，必须权衡后决策。

> [!question] 带着问题阅读
> 1. 如果你来设计功能追齐 Agent，一个 Agent 能处理 15 个仓库吗？为什么？
> 2. 什么样的任务特征暗示"是时候拆分 Agent 了"？
> 3. 多 Agent 让系统变复杂——这个代价值得付吗？

---

## 1. 单 Agent 的三个瓶颈

### 1.1 上下文窗口：物理边界

当前主流模型的上下文窗口在 128K～200K tokens 之间，听起来很大，但在真实的企业级任务中消耗极快：

| 内容 | 估算 tokens |
|------|------------|
| System Prompt（含工具定义） | ~3,000 |
| 单个仓库的目录结构 | ~500 |
| 单个关键文件内容 | ~2,000 |
| 15 个仓库（目录+关键文件） | ~37,500 |
| 历史对话轮次（10轮） | ~8,000 |
| **合计** | **~51,000** |

这还是乐观估算——现实中仓库里有复杂组件、长逻辑文件，单文件就可能超过 3,000 tokens。**功能追齐 Agent 需要同时了解 15 个仓库，这在单 Agent 下几乎不可能做到全面而准确。**

上下文溢出带来的不是报错，而是**静默劣化**：模型开始遗忘早期内容、产生幻觉、输出质量下降，且难以从外部观测。

### 1.2 并行能力：时间边界

单 Agent 是顺序执行的——探索完 repo-A，才能探索 repo-B。

假设探索一个仓库平均需要 5 次工具调用，每次调用含 LLM 推理耗时约 3 秒，那么：

- 探索 1 个仓库：~15 秒
- 探索 15 个仓库（顺序）：**~225 秒（约 4 分钟）**
- 探索 15 个仓库（5 个并行）：**~45 秒**

对于需要快速响应的研发流程，4 分钟 vs 45 秒 是截然不同的用户体验。**并行化不仅是性能优化，更是让 Agent 系统实际可用的必要条件。**

### 1.3 专业化不足：质量边界

探索仓库和生成代码修改，对 Agent 的能力要求完全不同：

- **探索阶段**：需要宽泛的搜索策略，System Prompt 应强调"尽量发现、客观记录"，工具集中在 `list_repo_files`、`read_repo_file`
- **生成阶段**：需要严格的代码质量约束，System Prompt 应强调"遵循现有风格、最小化修改范围"，工具集中在 `diff_feature`、`resolve_feature_spec`

把两者放在同一个 Agent 里，System Prompt 必然充满矛盾的指令。**专业化的 Agent 在各自领域表现更好，这是有实验支持的结论。**

## 2. 什么时候引入多 Agent

引入多 Agent 需要同时满足**至少两个**引入信号，否则单 Agent + 更好的 Context Engineering 是更优选择：

### 引入信号

**信号 A：单次任务消耗上下文 > 50%**
如果一次完整任务执行下来，上下文使用量稳定超过总窗口的 50%，说明已经在上下文边界附近运行，继续扩展任务复杂度会直接撞墙。

**信号 B：有明显可并行的子任务**
任务可以分解为多个互相独立（不存在严格顺序依赖）的子任务。例如"探索 15 个仓库"——仓库之间没有依赖关系，天然可以并行。

**信号 C：不同子任务需要显著不同的指令**
如果为了兼顾不同子任务的需求，System Prompt 里出现了大量"但如果你在做 X 的时候……""除非你是在处理 Y……"这类条件性指令，说明一个 Agent 已经在扮演多个角色，拆分是更干净的选择。

> [!tip] 原则：先单后多
> **先用单 Agent 验证可行性，再引入多 Agent 提升规模。** 多 Agent 不应该是起点——在你搞清楚任务流程之前，额外的协调复杂度只会让调试更困难。Anthropic 的工程实践也强调：大多数任务并不需要多 Agent，当且仅当单 Agent 确实触碰了边界时才引入。

---

## 3. 多 Agent 的代价

多 Agent 不是免费的，它带来三类新的成本：

### 3.1 协调复杂度

需要一个 Orchestrator 来分发任务、收集结果、处理失败。这本身是额外的工程工作量，且 Orchestrator 的错误会级联影响所有 Sub-Agent。

### 3.2 调试困难

单 Agent 出错，你有一条清晰的执行链路可以追溯。多 Agent 出错，你需要判断：是 Orchestrator 的决策错了？是 Research Agent 探索不够深入？还是 Analysis Agent 的对比逻辑有问题？**分布式系统的调试复杂度在 Agent 系统里同样适用。**

### 3.3 Token 消耗增加

每个 Agent 都有自己的 System Prompt、工具定义、对话历史。一个完整的多 Agent 任务的总 token 消耗，通常是等效单 Agent 任务的 **2～4 倍**。成本需要纳入决策。

**只有当多 Agent 带来的收益（速度、质量、规模）明确超过这些代价时，才应该引入。**

---

## 4. 追齐 Agent 引入多 Agent 的理由

回到功能追齐 Agent，它满足了所有三个引入信号：

**信号 A 满足**：单个任务（追齐一个 H5 页面的所有功能）需要读取 2-3 个仓库的多个文件，上下文消耗轻松超过 50%。如果同时追齐多个页面，单 Agent 根本无法处理。

**信号 B 满足**：15 个仓库的 Research 阶段完全独立，可以启动多个 Research Agent 并行探索，互不干扰。

**信号 C 满足**：
- Research Agent 的 System Prompt 应该是"广撒网、忠实记录"
- Analysis Agent 的 System Prompt 应该是"严格对比、列出差距"
- Execution Agent 的 System Prompt 应该是"最小改动、代码风格一致"

三者完全不同，放在一起是系统混乱的根源。

**结论**：功能追齐 Agent 应该采用层级 + 并行的混合多 Agent 架构，由 Orchestrator 协调三种专业化 Sub-Agent。

---

## 本节小结

- 单 Agent 的三个结构性瓶颈：上下文窗口（物理）、顺序执行（时间）、专业化不足（质量）
- 引入多 Agent 的三个信号：上下文 > 50%、存在可并行子任务、子任务需要显著不同指令
- 多 Agent 的代价：协调复杂度、调试困难、token 消耗翻倍
- 功能追齐 Agent 满足全部三个引入信号，适合多 Agent 架构
