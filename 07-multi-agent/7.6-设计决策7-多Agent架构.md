# 7.6 设计决策 7：功能追齐 Agent 多 Agent 完整架构

本节将第七章所有讨论的设计原则整合为完整的架构规划，形成功能追齐 Agent 的多 Agent 系统蓝图。

---

## 1. 架构总览

功能追齐 Agent 采用 **Hierarchical + Parallel 混合拓扑**，完整交互序列如下：

```
用户触发
    │
    ▼
[Orchestrator]
    │ 读取 memory/MEMORY.md
    │ 获取待追齐功能列表
    │
    ├──────────────────────────────────────────┐ 并行（最多3个）
    ▼                    ▼                     ▼
[Research Agent A]  [Research Agent B]  [Research Agent C]
 探索 h5-cart         探索 h5-checkout     探索 h5-profile
    │                    │                     │
    └──────── 写入 task-state.json ────────────┘
                         │
                         ▼
               [Orchestrator] 确认 Research 完成
                         │
                         ▼
               [Analysis Agent]
                对比 App vs H5，输出差距清单
                         │
                    写入 task-state.json
                         │
                         ▼
               [Orchestrator] 确认 Analysis 完成
                         │
                         ▼
               [Execution Agent]
                生成代码修改方案（unified diff）
                         │
                    写入 task-state.json
                         │
                         ▼
               [Orchestrator] 更新 completed.json
                         │
                         ▼
                    输出最终报告
```

**数据流向**：`MEMORY.md` → Research（写入 findings）→ Analysis（读 findings，写入 gaps）→ Execution（读 gaps，写入 diffs）→ `completed.json`

## 2. 各 Agent 的 System Prompt 要点

### Orchestrator System Prompt

```markdown
# Orchestrator — 功能追齐协调者

你负责协调功能追齐系统的整体执行流程，绝不直接执行具体业务操作。

核心约束：
1. 读取 memory/MEMORY.md 获取待追齐功能列表
2. 每次最多并行启动 3 个 Research Agent，防止 API 限流
3. Research 阶段完成后，逐功能执行 Analysis → Execution 序列
4. 每个阶段只读取 task-state.json 中的 summary 字段作为进度依据
5. 单个子任务失败时，记录到 task-state.json 并继续其他任务（降级策略）
6. 所有阶段完成后，更新 memory/completed.json 并生成汇总报告
7. 不在自己的上下文中缓存 Sub-Agent 的完整输出
```

### Research Agent System Prompt

```markdown
# Research Agent — 仓库探索专家

你的唯一职责：全面、客观地探索指定代码仓库，记录与目标功能相关的信息。

核心约束：
1. 先用 list_repo_files 了解整体目录结构（不超过 3 层深度）
2. 重点关注：路由配置、页面组件、状态管理、API 层、类型定义
3. 对每个关键文件，记录：路径、用途、主要导出、行数
4. 记录你认为"可能缺失"的功能，但明确标注这是推测而非确认
5. 输出写入 task-state.json 的 research 字段，summary 不超过 200 tokens
6. 置信度 < 0.7 时，在 warnings 字段注明原因
7. 不要尝试理解 App 侧的代码，那不是你的任务
```

### Analysis Agent System Prompt

```markdown
# Analysis Agent — 功能差距分析师

你的唯一职责：基于 Research 发现，对比 App 最新版本与 H5 当前版本，输出结构化差距清单。

核心约束：
1. 从 task-state.json 读取 research.findings，这是你的唯一输入来源
2. 使用 diff_feature 工具进行标准化对比
3. 所有差距必须有证据支撑（来自 findings 中的具体文件或代码片段）
4. 差距优先级：P0（影响核心业务流程）/ P1（重要体验功能）/ P2（体验优化）
5. 每个差距项包含：描述、优先级、App 侧代码位置、H5 侧应实现位置
6. 输出写入 task-state.json 的 analysis 字段，gaps 数组按优先级排序
7. 不要生成代码，只描述"应该做什么"，不说"怎么做"
```

### Execution Agent System Prompt

```markdown
# Execution Agent — 代码修改实施者

你的唯一职责：基于差距清单，生成最小化、可直接应用的代码修改方案。

核心约束：
1. 从 task-state.json 读取 analysis.gaps，只处理 P0 和 P1 差距
2. 用 resolve_feature_spec 获取每个功能点的完整规格说明
3. 用 read_repo_file 读取需要修改的文件当前内容
4. 最小化修改原则：只改必须改的，不做"顺手重构"或"风格统一"
5. 严格遵循现有代码风格：缩进（2空格/4空格）、命名规范、注释风格
6. 每个修改项输出标准 unified diff，包含足够的上下文行（-U 5）
7. 如果修改涉及跨仓库依赖或需要后端配合，明确标注"需要人工确认"
8. 置信度 < 0.8 的修改项，在 warnings 中注明不确定原因
```

## 3. 数据流设计

`task-state.json` 完整结构及各 Agent 读写权限：

| 字段路径 | 写入方 | 读取方 | 用途 |
|---------|--------|--------|------|
| `featureId` | Orchestrator | 所有 Agent | 任务唯一标识 |
| `stage` | Orchestrator | Orchestrator | 当前执行阶段 |
| `research.summary` | Research Agent | Orchestrator | Research 进度摘要 |
| `research.confidence` | Research Agent | Orchestrator | 质量信号（< 0.7 触发警告）|
| `research.findings` | Research Agent | Analysis Agent | 完整的仓库探索结果 |
| `research.warnings` | Research Agent | Orchestrator | 异常情况说明 |
| `analysis.summary` | Analysis Agent | Orchestrator | Analysis 进度摘要 |
| `analysis.confidence` | Analysis Agent | Orchestrator | 质量信号 |
| `analysis.gaps` | Analysis Agent | Execution Agent | 按优先级排序的差距清单 |
| `execution.summary` | Execution Agent | Orchestrator | Execution 进度摘要 |
| `execution.diffs` | Execution Agent | 最终用户 | 可直接应用的代码修改 |
| `execution.manualReviews` | Execution Agent | 最终用户 | 需要人工确认的修改项 |

## 4. 并行策略

### 并行度控制

```typescript
// parallel-orchestration.ts
const MAX_PARALLEL = 3; // 防止 API 限流（大多数 LLM API RPM 限制在 10-60）

async function runParallelResearch(features: Feature[]): Promise<void> {
  for (let i = 0; i < features.length; i += MAX_PARALLEL) {
    const batch = features.slice(i, i + MAX_PARALLEL);
    
    console.log(`[Orchestrator] 并行 Research 批次 ${Math.floor(i/MAX_PARALLEL) + 1}/${Math.ceil(features.length/MAX_PARALLEL)}`);
    
    // 使用 allSettled 确保单个失败不影响整批
    const results = await Promise.allSettled(
      batch.map(async (feature) => {
        await runResearchAgent(feature.id, feature.targetRepos);
        return feature.id;
      })
    );
    
    // 记录成功和失败
    for (const [idx, result] of results.entries()) {
      const featureId = batch[idx].id;
      if (result.status === 'fulfilled') {
        console.log(`✓ Research 完成: ${featureId}`);
      } else {
        console.error(`✗ Research 失败: ${featureId}`, result.reason);
        await updateStageStatus(featureId, 'research', 'failed', String(result.reason));
      }
    }
    
    // 批次间间隔，避免触发 API 限流
    if (i + MAX_PARALLEL < features.length) {
      await sleep(1000); // 1秒间隔
    }
  }
}

// Analysis 和 Execution 顺序执行（有数据依赖）
async function runSequentialAnalysisExecution(completedResearch: string[]): Promise<void> {
  for (const featureId of completedResearch) {
    try {
      await runAnalysisAgent(featureId);
      await runExecutionAgent(featureId);
      await markCompleted(featureId);
    } catch (error) {
      await updateStageStatus(featureId, 'current', 'failed', String(error));
      // 继续处理下一个功能
      continue;
    }
  }
}
```

### 并行安全保证

- **Research Agent 并行安全**：每个 Agent 写入独立的 `task-state/{featureId}.json`，无写冲突
- **Analysis/Execution 顺序执行**：对同一个 featureId，Analysis 必须在 Research 完成后执行，Execution 必须在 Analysis 完成后执行，通过 Orchestrator 的顺序控制保证

## 5. 单 Agent vs 多 Agent 对比

| 维度 | 单 Agent | 多 Agent（本章架构） |
|------|---------|-------------------|
| **上下文消耗** | 单次任务 > 100K tokens（可能溢出） | Orchestrator < 10K，每个 Sub-Agent < 30K |
| **执行时间（15个仓库）** | ~225 秒（顺序） | ~45-60 秒（并行） |
| **代码复杂度** | 低（单文件可实现） | 中等（需要 Orchestrator + 3 个 Agent + 通信层） |
| **可维护性** | 低（System Prompt 混乱，职责不清） | 高（每个 Agent 职责单一，独立测试） |
| **错误恢复** | 全局重试（从头开始） | 阶段级重试（只重试失败阶段） |
| **可扩展性** | 难（增加仓库直接导致上下文溢出） | 好（增加并行度即可） |
| **token 总消耗** | 1x | ~1.5-2x（有 Orchestrator 开销） |
| **适用规模** | ≤ 3 个仓库，单个简单功能 | ≥ 5 个仓库，多个功能并行追齐 |

**结论**：对于功能追齐 Agent 的实际规模（10-15 个仓库，多个功能并行），多 Agent 架构在执行时间和质量上的收益远超额外的工程复杂度。

## 6. 第七章总结

| 节次 | 核心要点 |
|------|---------|
| 7.1 | 单 Agent 的三个瓶颈（上下文/并行/专业化），引入多 Agent 的三个信号 |
| 7.2 | 五种拓扑模式（Sequential/Parallel/Hierarchical/Competitive/Collaborative），追齐 Agent 用混合拓扑 |
| 7.3 | Orchestrator 的四项职责与三条禁忌，保持 Orchestrator 精简是关键 |
| 7.4 | Sub-Agent 专业化设计（独立 System Prompt + 限定工具集 + 结构化输出），上下文完全隔离 |
| 7.5 | 共享状态优于直接传递，传"结论"不传"过程"，通信数据 < 1000 tokens |
| 7.6 | 完整架构：并行 Research + 顺序 Analysis/Execution，`task-state.json` 作为 Blackboard |
