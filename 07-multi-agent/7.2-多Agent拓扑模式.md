# 7.2 多 Agent 拓扑模式：五种架构选择

> [!abstract] 本节定义
> 多 Agent 系统的**拓扑**（Topology）描述 Agent 之间的协调关系：谁指挥谁、谁依赖谁、谁和谁可以并行。选择合适的拓扑直接影响系统的可靠性、可扩展性和调试难度。本节系统梳理五种标准拓扑，并给出选择决策框架。

---

## 1. 五种拓扑模式

### Sequential（顺序链）

Agent A 完成后，将结果传递给 Agent B，B 完成后传递给 Agent C，形成严格的线性流水线。

```
[Agent A] → [Agent B] → [Agent C] → 最终输出
```

**适用场景**：任务有严格的前后依赖，上一步的输出是下一步的必要输入。例如：先做数据清洗，再做数据分析，再生成报告。

**典型案例**：Claude 的 Prompt Chaining 模式——先让一个 Agent 做规划，再让另一个 Agent 做执行。

---

### Parallel（并行）

多个 Agent 同时处理不同的数据分片或子任务，最后由聚合层汇总结果。

```
          ┌→ [Agent B1] ─┐
[分发器] → ├→ [Agent B2] ─┼→ [聚合器] → 最终输出
          └→ [Agent B3] ─┘
```

**适用场景**：任务可以水平切分，各分片之间无依赖关系。例如：同时探索 15 个独立仓库。

**注意**：需要处理部分失败情况——某个 Agent 失败不应导致整批结果丢失，`Promise.allSettled` 优于 `Promise.all`。

---

### Hierarchical（层级）

一个 Orchestrator Agent 负责分解任务、分发给多个 Sub-Agent，收集结果后进行聚合决策。Sub-Agent 之间互相不知道对方的存在。

```
          [Orchestrator]
         /      |       \
[SubA] [SubB] [SubC]
```

**适用场景**：任务复杂度高，需要统一的协调者来维护全局状态；各 Sub-Agent 有明显的专业化分工。

**这是功能追齐 Agent 的核心拓扑。**

---

### Competitive（竞争）

多个 Agent 独立完成同一任务，最后选择最优结果（或多数投票）。

```
[Agent A] ─┐
[Agent B] ─┼→ [评判者] → 最佳结果
[Agent C] ─┘
```

**适用场景**：对输出质量要求极高，可以接受 2-3 倍的 token 消耗来换取更可靠的结果。例如：安全关键的代码审查、高风险决策。

**代价**：token 消耗是单 Agent 的 N 倍（N = 参与竞争的 Agent 数量），一般只用于高价值、低频任务。

---

### Collaborative（协作）

多个 Agent 共享同一个工作空间，可以互相看到对方的工作进展，进行讨论和协作。类似多人协作文档。

```
[Agent A] ←→ [共享工作区] ←→ [Agent B]
                   ↕
              [Agent C]
```

**适用场景**：任务需要多角色互相 review、讨论，例如：一个 Agent 写代码，另一个 Agent 做 code review，第三个 Agent 写测试。

**挑战**：需要精心设计消息协议，防止 Agent 之间互相干扰或陷入无休止的讨论循环。

---

## 2. 各模式对比

| 拓扑 | 适用场景 | 优点 | 缺点 | Token 消耗 |
|------|---------|------|------|-----------|
| Sequential | 有严格顺序依赖的流水线 | 简单、可预测、易调试 | 无并行，耗时长 | 1x |
| Parallel | 独立子任务的批量处理 | 速度快，水平扩展 | 需要聚合层，部分失败处理复杂 | 1x（分摊） |
| Hierarchical | 复杂任务的专业分工 | 模块化，职责清晰 | Orchestrator 是单点故障 | 1.2～1.5x |
| Competitive | 高质量要求的关键决策 | 输出质量高，冗余可靠 | token 消耗倍增，延迟高 | Nx |
| Collaborative | 需要多角色 review 的任务 | 产出全面，自我纠错 | 设计复杂，可能死循环 | 1.5～3x |

---

## 3. 追齐 Agent 的拓扑选择

功能追齐 Agent 采用 **Hierarchical + Parallel 的混合拓扑**：

### 层级结构（Hierarchical）

```
              [Orchestrator]
              ↙     ↓     ↘
[Research]  [Analysis]  [Execution]
```

Orchestrator 负责全局协调：读取 `MEMORY.md` 了解待追齐任务 → 分发给 Research Agent → 收到摘要后分发给 Analysis Agent → 收到差距清单后分发给 Execution Agent → 将结果写入 `completed.json`。

三个 Sub-Agent 互相隔离：Research Agent 不知道 Analysis Agent 的存在，Analysis Agent 也不知道 Execution Agent 的工作。这种隔离是刻意的——防止上下文污染，让每个 Agent 专注自己的任务。

### 并行维度（Parallel）

在 Research 阶段，如果有多个 H5 页面待追齐，可以同时启动多个 Research Agent 并行探索不同仓库：

```
[Orchestrator]
    ├→ [Research Agent: h5-cart]      并行
    ├→ [Research Agent: h5-checkout]  并行
    └→ [Research Agent: h5-profile]   并行
```

最大并行度建议控制在 3 个，防止 API 限流（大多数 LLM API 的 RPM 限制在 10-60 之间，留出余量）。

> [!info] 拓扑选择决策流程
> 1. **任务是否有严格顺序依赖？** 是 → 优先考虑 Sequential
> 2. **是否有可以独立并行的子任务？** 是 → 在顺序基础上加入 Parallel 维度
> 3. **不同子任务是否需要显著不同的指令？** 是 → 引入 Hierarchical，每个专业角色一个 Sub-Agent
> 4. **是否需要极高的输出质量且有预算？** 是 → 考虑 Competitive
> 5. **是否需要多角色互相 review？** 是 → 考虑 Collaborative，但做好防死循环设计

---

## 本节小结

- 五种拓扑：Sequential（流水线）、Parallel（并行分片）、Hierarchical（层级分工）、Competitive（竞争选优）、Collaborative（协作共创）
- 拓扑选择的核心维度：任务是否有顺序依赖、是否可并行、是否需要专业分工、质量要求和预算
- 功能追齐 Agent 使用 Hierarchical（三个专业 Sub-Agent）+ Parallel（多仓库同时探索）的混合拓扑
- 混合拓扑最常见——纯粹的单一拓扑在真实系统中很少见
