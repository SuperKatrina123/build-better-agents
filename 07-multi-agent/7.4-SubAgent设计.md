# 7.4 Sub-Agent 设计：专业化的执行单元

> [!abstract] 本节定义
> Sub-Agent 是多 Agent 系统中实际完成业务工作的执行单元。Sub-Agent 设计的核心原则是**专业化**——每个 Sub-Agent 只做一类事，并在这类事上做到极致。专业化通过三个维度体现：专注的 System Prompt、限定的工具集、结构化的输出格式。

---

## 1. 三个 Sub-Agent 的专业分工

功能追齐 Agent 包含三个职责鲜明的 Sub-Agent，每个都有独立的 System Prompt 和工具集：

### Research Agent：仓库探索专家

**职责**：深入探索目标仓库，全面收集与待追齐功能相关的信息。

**System Prompt 核心约束**：
```markdown
# Research Agent

你是代码仓库探索专家。你的任务是全面了解指定仓库的结构和与目标功能相关的代码。

工作原则：
- 广泛探索，不要遗漏重要目录
- 客观记录你发现的内容，不要做功能判断
- 优先关注：路由定义、组件结构、状态管理、API 调用
- 如果不确定某个文件是否相关，记录下来，留给 Analysis 阶段判断
- 探索深度：目录树（2-3层）+ 关键文件全文 + 相关文件片段
```

**工具集**（仅限这两个）：
- `list_repo_files(repo, path)` — 列出目录结构
- `read_repo_file(repo, filePath)` — 读取文件内容

**输出目标**：仓库结构摘要 + 相关代码片段，写入 `task-state.json` 的 `research` 字段。

---

### Analysis Agent：功能对比分析师

**职责**：基于 Research Agent 的发现，对比 App 最新版本和 H5 当前版本的功能差距，输出优先级排序的差距清单。

**System Prompt 核心约束**：
```markdown
# Analysis Agent

你是功能差距分析专家。你需要对比 App 最新版本和 H5 当前版本，找出所有需要补充的功能点。

工作原则：
- 严格基于证据（Research 阶段的发现）进行对比，不要凭印象推断
- 差距按优先级分类：P0（核心流程必须有）/ P1（重要功能）/ P2（体验优化）
- 对每个差距点，指明：缺少什么、App 代码在哪里、H5 应该在哪里实现
- 输出格式必须是结构化 JSON，便于 Execution Agent 直接使用
```

**工具集**（仅限这一个）：
- `diff_feature(appRepo, h5Repo, featureId)` — 调用结构化功能对比工具

**输入**：读取 `task-state.json` 中 `research.findings`（Research Agent 的完整发现）

**输出**：差距清单（按优先级排序），写入 `task-state.json` 的 `analysis` 字段。

---

### Execution Agent：代码修改实施者

**职责**：基于 Analysis Agent 输出的差距清单，生成具体的代码修改方案（标准 unified diff 格式）。

**System Prompt 核心约束**：
```markdown
# Execution Agent

你是代码修改专家。你的任务是基于差距分析，生成可直接应用的代码修改方案。

工作原则：
- 最小化修改：只改需要改的，不做"顺手重构"
- 严格遵循现有代码风格（缩进、命名规范、注释风格）
- 每个修改附上说明：为什么这样改、影响范围
- 输出标准 unified diff 格式，可直接用 `git apply` 应用
- 如果某个差距点超出安全修改范围，明确标注"需要人工确认"
```

**工具集**：
- `read_repo_file(repo, filePath)` — 读取需要修改的文件
- `resolve_feature_spec(featureId)` — 获取功能的完整规格说明

**输入**：读取 `task-state.json` 中 `analysis.gaps`（功能差距清单）

**输出**：包含 diff 的修改方案，写入 `task-state.json` 的 `execution` 字段。

## 2. Sub-Agent 的上下文隔离

三个 Sub-Agent 的上下文是完全隔离的——每个 Sub-Agent 只能看到：
1. 自己的 System Prompt
2. 自己的任务输入（来自 Orchestrator 的调用指令）
3. 自己的工具调用历史
4. 通过共享文件（`task-state.json`）读取的前置阶段输出

**Research Agent 不知道** Analysis Agent 存在，也不知道最终要生成 diff。这种"无知"是刻意的：如果 Research Agent 知道最终目标是生成 diff，它可能会在探索阶段就开始"猜测"哪些代码需要改，导致探索不够全面客观。

**上下文隔离防止的问题**：
- **角色混淆**：一个 Agent 试图同时扮演多个角色
- **偏见传播**：Research Agent 的某个错误推断影响后续所有阶段
- **上下文膨胀**：前一个 Agent 的完整对话历史被带入后一个 Agent

## 3. Sub-Agent 的输出格式

每个 Sub-Agent 的输出都是结构化 JSON，写入 `task-state.json`：

```typescript
// task-state.json 中 Sub-Agent 输出的结构定义
interface SubAgentOutput {
  summary: string;          // 给 Orchestrator 看的精简摘要（< 200 tokens）
  confidence: number;       // 0-1，自评置信度
  findings: unknown;        // 完整发现（供下一阶段 Sub-Agent 读取）
  next_steps?: string[];    // 可选：建议的后续行动
  warnings?: string[];      // 可选：值得注意的异常
}
```

**为什么需要 `confidence` 字段？**
Sub-Agent 在某些情况下（如仓库结构异常、关键文件缺失）可以主动降低置信度。Orchestrator 读到低置信度的摘要后，可以决定是否需要重新执行或转交人工处理，而不是盲目继续。

> [!example] 案例：Research Agent 的 JSON 输出
> Research Agent 探索 `h5-cart-page` 仓库后，写入 `task-state.json` 的输出：

## 本节小结

- 三个 Sub-Agent 的专业化分工：Research（宽泛探索）→ Analysis（严格对比）→ Execution（最小修改）
- 每个 Sub-Agent 有独立的 System Prompt、限定的工具集、结构化的输出格式
- 上下文隔离是刻意的设计——防止角色混淆、偏见传播、上下文膨胀
- 统一的输出格式：`summary`（给 Orchestrator）+ `findings`（给下一阶段 Sub-Agent）+ `confidence`（质量信号）
