# 7.5 Agent 间通信：数据如何在 Agent 之间流动

> [!abstract] 本节定义
> **Agent 间通信**是多 Agent 系统的核心工程挑战：Agent 完成工作后，结果如何传递给需要它的下一个 Agent？传递什么（全部输出还是摘要）、通过什么渠道（直接传递还是共享存储）、传递多少（token 预算是多少）——这三个问题的答案直接决定系统的性能和可靠性。

---

## 1. 两种通信模式

![[agent-communication.excalidraw]]

![多 Agent 协议](https://gw.alipayobjects.com/zos/k/g3/multi_agent_protocol.svg)

### 模式一：直接传递（Direct Passing）

Orchestrator 调用 Sub-Agent A，拿到结果后直接作为参数传给 Sub-Agent B：

```typescript
// 直接传递示例
const researchResult = await runResearchAgent(featureId);
// researchResult 直接作为参数传给下一个 Agent
const analysisResult = await runAnalysisAgent(featureId, researchResult.output);
```

**优点**：
- 实现简单，流程清晰
- 无外部存储依赖，延迟低
- 适合简单的线性流水线

**缺点**：
- 强耦合：上游 Agent 的输出格式变化会破坏下游 Agent
- 无法支持一对多：如果多个 Agent 需要同样的数据，需要重复传递
- 完整输出会占据下游 Agent 的输入上下文，造成上下文膨胀
- 无审计记录：中间状态不可追溯

---

### 模式二：共享状态（Blackboard / Shared State）

每个 Agent 完成工作后将结果写入共享存储（文件、数据库、KV store），其他 Agent 按需读取：

```typescript
// 共享状态示例
// Research Agent 写入
await writeTaskState(featureId, 'research', {
  summary: '...',
  findings: { ... }
});

// Analysis Agent 读取（只读取它需要的字段）
const researchFindings = await readTaskState(featureId, 'research.findings');
const analysisResult = await runAnalysisAgent(featureId, researchFindings);
```

**优点**：
- 解耦：各 Agent 通过共享存储间接通信，不直接依赖对方
- 支持多读多写：多个 Agent 可以读取同一份数据，无需重复传递
- 可审计：所有中间状态持久化，便于调试和重试
- 分层读取：Orchestrator 读摘要，下游 Agent 读完整发现

**缺点**：
- 需要外部存储（文件系统、Redis 等），增加基础设施依赖
- 需要定义清晰的数据结构和读写协议
- 并发写入需要考虑锁机制（虽然在 Agent 系统中通常通过顺序控制避免）

---

## 2. 追齐 Agent 选择共享状态

功能追齐 Agent 选择**共享状态模式**，以 `task-state.json` 作为 Blackboard。原因如下：

**理由 1：多方读写同一份数据**
Research Agent 写入的 `findings` 需要被 Analysis Agent 读取；Analysis Agent 写入的 `gaps` 需要被 Execution Agent 读取；Orchestrator 需要读取每个阶段的 `summary`。直接传递模式无法优雅处理这种"扇出读取"。

**理由 2：不同 Agent 读取不同层级**
Orchestrator 只需要 `summary`（~200 tokens），Analysis Agent 需要 `research.findings`（~2,000 tokens），Execution Agent 需要 `analysis.gaps`（~1,000 tokens）。通过共享存储，每个 Agent 精确读取自己需要的层级，不多也不少。

```typescript
// task-state.json 结构示例
const taskState = {
  featureId: "cart-coupon",
  stage: "execution",
  research: {
    summary: "h5-cart-page 缺少优惠券相关代码...",  // Orchestrator 读这里
    confidence: 0.87,
    findings: {                                        // Analysis Agent 读这里
      keyFiles: [...],
      presentFeatures: [...],
      suspectedMissingFeatures: [...]
    }
  },
  analysis: {
    summary: "发现 3 个 P0 差距，5 个 P1 差距",      // Orchestrator 读这里
    confidence: 0.92,
    gaps: [                                            // Execution Agent 读这里
      { id: "gap-001", priority: "P0", description: "优惠券输入框", appFile: "..." }
    ]
  },
  execution: {
    summary: "生成 4 个文件的 diff，影响范围：Cart.tsx, coupon.ts",
    confidence: 0.95,
    diffs: [...]                                       // 最终输出
  }
};
```

**理由 3：支持断点续传**
如果 Execution 阶段失败，不需要重新运行 Research 和 Analysis。Orchestrator 可以直接从 `task-state.json` 中读取已完成的阶段结果，只重新执行失败的阶段。

## 3. 通信数据的压缩

Agent 间传递"结论"而不是"过程"，这是降低 token 消耗的关键原则：

**Research Agent 的工作过程**（不传递）：
- 调用 `list_repo_files` 15 次
- 调用 `read_repo_file` 23 次
- 读取了 47,000 tokens 的文件内容
- 中间经历了 12 轮 LLM 推理

**Research Agent 传递给 Analysis Agent 的内容**（实际传递）：
```json
{
  "keyFiles": [3个核心文件路径],
  "presentFeatures": [4个已有功能],
  "suspectedMissingFeatures": [3个可能缺失功能],
  "codeSnippets": {"关键文件的关键片段"}
}
```
**约 800 tokens** — 压缩比约 60:1。

这种压缩是有损的，但 Loss 是可接受的：Analysis Agent 需要的是"已有什么、缺少什么"的结论，而不是 Research Agent 探索过程中读取的每一行代码。

> [!tip] 原则：1000 tokens 通信上限
> **Agent 间通信数据量应 < 1000 tokens**（不含必要的代码片段引用）。超过这个值，说明你在传递"过程"而不是"结论"——应该让接收方 Agent 自己决定需要读取哪些细节，而不是由发送方决定"把所有东西都传过去"。

## 4. 避免上下文污染

> [!warning] 常见错误：把前一个 Agent 的完整输出塞入下一个 Agent 的上下文
> 
> ```typescript
> // ❌ 错误做法：把完整输出作为系统消息传入
> const researchOutput = await runResearchAgent(featureId);
> const analysis = await callLLM({
>   systemPrompt: ANALYSIS_SYSTEM_PROMPT,
>   messages: [
>     { role: 'user', content: `这是 Research Agent 的完整输出：\n${JSON.stringify(researchOutput)}` }
>     //                        ^^^^ 可能有 5,000+ tokens，全部占用 Analysis 的上下文
>   ]
> });
> 
> // ✅ 正确做法：只传递 Analysis Agent 需要的字段
> const researchFindings = await readTaskState(featureId, 'research.findings');
> const analysis = await callLLM({
>   systemPrompt: ANALYSIS_SYSTEM_PROMPT,
>   messages: [
>     { role: 'user', content: `请基于以下 Research 发现进行差距分析：\n${JSON.stringify(researchFindings.suspectedMissingFeatures)}` }
>     //                        ^^^^ 只传递必要字段，约 200 tokens
>   ]
> });
> ```

上下文污染的后果：
- **质量下降**：过多无关信息干扰模型的注意力，核心任务被"稀释"
- **成本上升**：每个 token 都要付费，冗余信息直接增加成本
- **幻觉风险**：上下文中存在来自不同 Agent 的不一致信息时，模型可能尝试"调和"矛盾，产生幻觉

## 本节小结

- 两种通信模式：直接传递（简单但耦合强）vs 共享状态（解耦但需要外部存储）
- 追齐 Agent 选择共享状态（`task-state.json`），原因：多方读写、分层读取、断点续传
- 关键设计：Orchestrator 读 `summary`，下游 Sub-Agent 读完整 `findings`，不同角色读不同深度
- 通信数据压缩原则：传"结论"不传"过程"，1000 tokens 以内
- 避免上下文污染：只向 Agent 传递它真正需要的字段
