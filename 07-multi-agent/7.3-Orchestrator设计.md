# 7.3 Orchestrator 设计：协调者的责任边界

> [!abstract] 本节定义
> **Orchestrator** 是多 Agent 系统的"指挥中心"——它不直接执行具体任务，而是负责任务分解、分发、状态追踪、结果聚合和错误处理。Orchestrator 的设计质量决定了整个系统的健壮性：一个好的 Orchestrator 应该"轻量且克制"，而不是一个什么都做的"超级 Agent"。

---

## 1. Orchestrator 的职责

![[orchestrator-subagent.excalidraw]]

Orchestrator 的核心职责有且仅有以下四项：

**① 任务分解与分发**
读取任务输入（如 `MEMORY.md` 中的待追齐功能列表），将整体任务拆解为多个子任务，确定各子任务的分配对象（哪个 Sub-Agent）和执行顺序（哪些可以并行，哪些有前后依赖）。

**② 状态追踪**
维护一份全局任务状态，知道每个子任务的当前进展（待开始 / 执行中 / 已完成 / 失败）。这个状态通常以 JSON 文件形式持久化（如 `task-state.json`），而不是保存在 Orchestrator 的上下文中。

**③ 结果聚合**
收集各 Sub-Agent 返回的摘要（Summary），整合为全局进展报告，判断是否可以进入下一阶段，或者是否需要重试某个失败的子任务。

**④ 错误处理**
当某个 Sub-Agent 失败时，决定是重试、降级处理（跳过这个仓库继续其他），还是整体中止。错误处理策略应该在 Orchestrator 的 System Prompt 中明确约定。

---

## 2. Orchestrator 不应该做的事

> [!warning] 常见误区：把 Orchestrator 做成"超级 Agent"
> 很多团队在初期会把 Orchestrator 设计成既协调又执行的"超级 Agent"——它既分发任务，又自己去读仓库文件、自己生成 diff。这看起来减少了 Agent 数量，实际上集中了所有问题：上下文臃肿、System Prompt 混乱、调试困难。Orchestrator 应该是"最精简"的 Agent，而不是"最强大"的那个。

**不直接执行具体任务**
Orchestrator 不应该调用 `read_repo_file`、`diff_feature` 这类具体的业务工具。这些工具应该属于 Sub-Agent。Orchestrator 只需要知道"任务是否完成"和"结果摘要是什么"，不需要知道具体是怎么完成的。

**不维护 Sub-Agent 的内部状态**
Orchestrator 不应该存储 Research Agent 探索了哪些文件、调用了多少次工具、中间经历了什么推理步骤。这些内部状态属于 Sub-Agent 的私有上下文。Orchestrator 只需要 Sub-Agent 返回的最终摘要。

**不在上下文里保存 Sub-Agent 的完整输出**
这是最常见的性能问题。如果 Orchestrator 把每个 Sub-Agent 的完整输出（可能是 2,000～5,000 tokens）都加入自己的上下文，一个有 15 个 Sub-Agent 的任务会让 Orchestrator 的上下文膨胀到 60,000～75,000 tokens，远超必要的量。**只传递摘要，详细内容写入共享文件。**

---

## 3. Orchestrator 的上下文设计

Orchestrator 的 System Prompt 应该专注于**协调逻辑**，而不是业务逻辑：

```markdown
# Orchestrator System Prompt

你是功能追齐系统的协调者。你的职责：
1. 读取 memory/MEMORY.md 了解当前待追齐的功能列表
2. 对每个待追齐功能，依次协调 Research → Analysis → Execution 三个阶段
3. 每个阶段完成后，读取 task-state.json 中对应的 summary 字段
4. 当所有阶段完成后，更新 memory/completed.json

你绝对不要：
- 自己读取仓库文件
- 自己生成代码 diff
- 把 Sub-Agent 的完整输出存储在你的消息历史中
  
Sub-Agent 的详细输出已写入 task-state.json，你只需要读取 summary 字段。
```

**L3 上下文内容**（Orchestrator 自己维护的）：
- 当前任务阶段（研究中 / 分析中 / 执行中）
- 各子任务的完成状态（以 task_id 为 key 的简短状态表）
- 当前轮次的决策依据（为什么分发给这个 Sub-Agent）

**写入外部存储的内容**：
- Sub-Agent 的详细发现 → `task-state.json` 的 `findings` 字段
- Sub-Agent 的摘要 → `task-state.json` 的 `summary` 字段（供 Orchestrator 读取）
- 已完成的追齐记录 → `memory/completed.json`

## 4. 追齐 Agent Orchestrator 实现

以下是 Orchestrator 主循环的 TypeScript 实现，展示了核心协调逻辑：

```typescript
// orchestrator.ts
import { readMemory, writeTaskState, readTaskState } from './utils';
import { runResearchAgent } from './agents/research';
import { runAnalysisAgent } from './agents/analysis';
import { runExecutionAgent } from './agents/execution';

interface TaskStatus {
  featureId: string;
  stage: 'research' | 'analysis' | 'execution' | 'done' | 'failed';
  retryCount: number;
}

async function orchestrate(): Promise<void> {
  // 步骤 1：读取待追齐任务列表
  const memory = await readMemory('memory/MEMORY.md');
  const pendingFeatures = memory.pendingFeatures; // 从 MEMORY.md 解析

  console.log(`[Orchestrator] 待追齐功能：${pendingFeatures.length} 个`);

  // 步骤 2：并行启动 Research 阶段（最多 3 个并行）
  const PARALLEL_LIMIT = 3;
  for (let i = 0; i < pendingFeatures.length; i += PARALLEL_LIMIT) {
    const batch = pendingFeatures.slice(i, i + PARALLEL_LIMIT);

    const researchResults = await Promise.allSettled(
      batch.map(feature => runResearchAgent(feature.id, feature.repos))
    );

    for (const [idx, result] of researchResults.entries()) {
      const featureId = batch[idx].id;
      if (result.status === 'fulfilled') {
        // 只从 task-state.json 读取 summary，不在 Orchestrator 上下文中保存完整输出
        const taskState = await readTaskState(featureId);
        console.log(`[Orchestrator] Research 完成: ${featureId} — ${taskState.research.summary}`);
      } else {
        console.error(`[Orchestrator] Research 失败: ${featureId}`, result.reason);
        await writeTaskState(featureId, { stage: 'failed', error: String(result.reason) });
        // 失败的任务跳过，继续处理其他任务（降级策略）
      }
    }
  }

  // 步骤 3：顺序执行 Analysis 和 Execution（依赖 Research 结果）
  const completedResearch = await getCompletedResearchTasks();

  for (const featureId of completedResearch) {
    try {
      // Analysis 阶段：读取 Research findings，输出差距清单
      await runAnalysisAgent(featureId);
      const analysisState = await readTaskState(featureId);
      console.log(`[Orchestrator] Analysis 完成: ${featureId} — ${analysisState.analysis.summary}`);

      // Execution 阶段：读取差距清单，生成代码修改方案
      await runExecutionAgent(featureId);
      const executionState = await readTaskState(featureId);
      console.log(`[Orchestrator] Execution 完成: ${featureId} — ${executionState.execution.summary}`);

      // 更新 completed.json
      await markFeatureCompleted(featureId);

    } catch (error) {
      console.error(`[Orchestrator] 处理失败: ${featureId}`, error);
      // 记录失败，不中断其他任务
    }
  }

  console.log('[Orchestrator] 本轮追齐完成');
}

orchestrate().catch(console.error);
```

注意代码中的关键设计决策：
1. `Promise.allSettled` 而不是 `Promise.all`——单个 Research 失败不影响其他任务
2. Orchestrator 只读取 `summary` 字段，不读取 `findings` 字段（详细内容留给 Analysis Agent）
3. 并行度限制为 `PARALLEL_LIMIT = 3`，防止 API 限流
4. 每个阶段失败独立记录，采用降级策略（跳过失败任务继续）

## 本节小结

- Orchestrator 的四项职责：任务分解、状态追踪、结果聚合、错误处理
- Orchestrator 不应该做的三件事：不直接执行任务、不维护 Sub-Agent 内部状态、不在上下文里保存完整输出
- Orchestrator 的上下文设计：System Prompt 专注协调逻辑，详细输出写入外部文件
- 追齐 Agent 的 Orchestrator 实现：`Promise.allSettled` 并行 + 降级策略 + 只读取摘要
