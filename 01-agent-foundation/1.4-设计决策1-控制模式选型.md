# 1.4 设计决策 #1：给你的任务选对控制模式

## 1. 决策框架：五个问题

面对一个新任务，按顺序问这五个问题：

![[control-mode-decision-tree.excalidraw]]

---

**Q1：这个任务的执行步骤固定吗？**

- ✅ 固定 → 优先 **Prompt Chaining**（顺序执行，可预期）
- ❌ 不固定，取决于运行时状态 → 往下问

**Q2：不同的输入需要走不同的处理路径吗？**

- ✅ 是 → 在 Chaining 中加入 **Routing** 节点
- ❌ 否，所有输入走同一逻辑 → 往下问

**Q3：有子任务可以并行处理吗？**

- ✅ 有独立的子任务 → 加入 **Parallelization**（提速）
- ✅ 需要多次验证同一结果 → 加入 **Parallelization 投票法**（提可靠性）
- ❌ 没有 → 往下问

**Q4：任务太复杂、单个 Agent 上下文不够用吗？**

- ✅ 是 → 引入 **Orchestrator-Workers**
- ❌ 否，一个 Agent 能搞定 → 往下问

**Q5：有明确的质量标准，且需要反复优化吗？**

- ✅ 是 → 加入 **Evaluator-Optimizer** 环节
- ❌ 否，一次输出即可 → 单步 Agent 或 Prompt Chaining 就够了

> [!tip] 原则
> 这五个问题不是互斥的，**一个系统可以同时回答多个"是"**。
> 每个"是"对应一种模式，叠加在一起就是你的最终架构。

## 2. 常见任务的选型参考

| 任务类型 | 推荐模式组合 | 理由 |
|---|---|---|
| 每日自动报告生成 | Prompt Chaining | 步骤固定，线性执行 |
| 客服问答系统 | Routing + Chaining | 不同问题类型走不同流程 |
| 代码审查 | Parallelization（多维度）+ Evaluator-Optimizer | 多个检查维度并行，结果需要质量把关 |
| 功能追齐 | Chaining + Routing + Parallel + Evaluator-Optimizer | 四种模式嵌套 |
| 研究报告生成 | Orchestrator-Workers + Evaluator-Optimizer | 子任务多，需要协调；报告质量需迭代 |
| 自动化测试生成 | Chaining + Evaluator-Optimizer | 先生成用例，再验证覆盖率，不通过则补充 |

## 3. 功能追齐 Agent 的最终选型

本章一直用功能追齐作为实战案例，现在做最终总结：

> [!info] 方法
> **最终架构：四层模式嵌套**
>
> ```
> 外层：Prompt Chaining（调研 → 分析 → 执行 → 验收）
>
>   调研层：Routing
>     ├── 文档清晰    → Chaining 直接提取
>     ├── 文档模糊    → Agent 转 App 测试用例
>     └── 文档不完整  → 记录到待确认列表，跳过
>
>   分析层：Parallelization
>     ├── App 仓库分析（并行）
>     └── H5 仓库分析（并行）
>
>   执行层：Orchestrator-Workers（多功能点并行执行）
>     ├── Worker A：UI 追齐
>     ├── Worker B：逻辑追齐
>     └── Worker C：字段映射 / BFF 迁移
>
>   验收层：Evaluator-Optimizer
>     生成代码 → 跑测试 → 不通过则修改 → 直到全部通过
> ```

## 5. 下一步：上下文工程

控制模式决定了系统的骨架。但骨架搭好之后，Agent 能不能跑好，
取决于一个更核心的问题：

> **模型在每一步能看到什么信息？**

这就是第二章要解决的——**上下文工程**。

> [!warning] 过渡提示
> 很多人搭好控制模式之后，发现 Agent 跑起来结果很差。
> 大概率不是模式选错了，而是上下文设计出了问题。
> 第二章会解释为什么，以及怎么系统性地解决它。

## 本章总结

**第一章学了什么：**

| 节 | 核心结论 |
