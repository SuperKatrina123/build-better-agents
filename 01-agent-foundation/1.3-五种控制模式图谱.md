## 2. 逐一拆解

### 模式一：Prompt Chaining（提示链）

**结构**：任务拆成顺序步骤，每步 LLM 处理上一步的输出，中间可加代码检查点。

```
输入 → [LLM Step 1] → 检查点 → [LLM Step 2] → 检查点 → [LLM Step 3] → 输出
```

**适用场景**：流程固定、步骤明确、线性推进的任务。

> [!example] 案例：功能追齐里的调研环节
> ```
> 读飞书文档 → 提取功能清单 → 格式化为结构化 JSON → 存入 feature-spec.json
> ```
> 每步有明确输入输出，中间可以加验证（JSON 格式是否合法），是标准的 Prompt Chaining。

**优点**：可预期、可调试、每步可独立优化。
**局限**：无法应对中间步骤出现意外情况（文档模糊、格式异常）。

---

### 模式二：Routing（路由）

**结构**：对输入分类，路由到对应的专用处理流程。

```
输入 → [分类器] → 情况 A → [处理流 A]
                 → 情况 B → [处理流 B]
                 → 情况 C → [处理流 C]
```

**适用场景**：不同类型的输入需要走不同的处理逻辑。

> [!example] 案例：文档质量路由
> 读完飞书文档后，先判断文档质量：
> ```
> 文档描述清晰    → 直接提取功能点（Prompt Chaining）
> 文档描述模糊    → 转 App 测试用例补充（需要 Agent 判断）
> 文档链接到 App  → 切换仓库上下文读取
> 文档标注"待确认" → 跳过，记录到待处理列表
> ```
> 路由器本身很轻（一次 LLM 分类调用），但让后续处理效率大幅提升。

**优点**：不同路径可以独立优化，简单问题不走重型流程。
**局限**：分类准确率决定整体质量，分类错误会造成后续连锁失败。

---

### 模式三：Parallelization（并行）

**结构**：两种变体。

**分段法（Sectioning）**：把任务拆成独立子任务，并发执行。
```
任务
├── 子任务 A ──┐
├── 子任务 B ──┼→ 聚合结果
└── 子任务 C ──┘
```

**投票法（Voting）**：同一任务跑多次，取共识结果。
```
同一输入 → 运行 3 次 → 取多数一致的结果
```

> [!example] 案例：并行读取两个仓库
> 分析阶段，App 仓库和 H5 仓库可以同时读取，互不依赖：
> ```typescript
> const [appFeatures, h5Features] = await Promise.all([
>   analyzeRepo(APP_REPO, feature),
>   analyzeRepo(H5_REPO, feature),
> ]);
> const diff = compare(appFeatures, h5Features);
> ```
> 并行读取比串行快一倍，且两边分析结果互不影响。

> [!example] 案例：高风险操作用投票法
> 判断某个字段是否需要迁移 BFF，跑三次取共识：
> 两次说"需要迁移"，一次说"不需要" → 执行迁移
> 避免单次幻觉导致错误操作。

**优点**：提速（分段法）；提高可靠性（投票法）。
**局限**：Token 消耗成倍增加；聚合逻辑需要额外设计。

---

### 模式四：Orchestrator-Workers（编排器-工作者）

**结构**：中央 LLM 动态分解任务，委派给 Worker LLM，综合结果。

```
[Orchestrator]
    ├── 分析任务 → 决定拆成哪些子任务
    ├── 委派给 Worker A
    ├── 委派给 Worker B
    └── 综合结果 → 输出最终答案
```

**适用场景**：任务复杂、子任务之间需要协调、单个 Agent 上下文不够用。

> [!example] 案例：功能追齐的多 Agent 版本（第七章会详细讲）
> ```
> Orchestrator：分析要追齐哪些功能，决定任务分工
>   ├── Worker A（调研）：读飞书文档 + App 测试用例
>   ├── Worker B（分析）：读两端代码，输出 diff 报告
>   └── Worker C（执行）：生成追齐代码
> Orchestrator：检查各 Worker 产出，综合输出最终方案
> ```
> Worker 只回传摘要，搜索和调试细节留在自己的上下文里，不污染 Orchestrator。

**优点**：支持复杂任务；Worker 可以并行；主上下文保持干净。
**局限**：调试复杂；Orchestrator 分解质量决定整体效果。

---

### 模式五：Evaluator-Optimizer（评估器-优化器）

**结构**：生成器产出，评估器给反馈，循环直到达标。

```
输入 → [生成器] → 初版结果
              ↓
         [评估器] → 通过？ → 输出
              ↓ 不通过
         反馈给生成器 → 重新生成
```

**适用场景**：质量标准难以用代码精确定义，需要迭代优化的任务。

> [!example] 案例：代码质量把关
> 生成追齐代码后，用评估器检查：
> ```
> 生成器：输出追齐代码
> 评估器检查：
>   - TypeScript 编译通过？
>   - App 测试用例通过？
>   - 字段映射完整？
>   - 有没有引入新的 any 类型？
> 不通过 → 把具体错误反馈给生成器 → 重新生成
> 通过 → 输出最终代码
> ```

**优点**：自动迭代，减少人工介入；质量上限更高。
**局限**：评估器本身的质量决定上限；可能陷入无效循环。

> [!warning] 误区
> Evaluator-Optimizer 最常见的问题：评估器给的反馈太模糊（"代码质量不够好"），
> 生成器不知道怎么改，只能随机变动，陷入无效循环。
> **评估器的反馈必须具体、可操作。**

## 3. 五种模式的选型速查

| 模式 | 核心特征 | 选它的信号 |
|---|---|---|
| Prompt Chaining | 线性顺序 | 步骤固定，每步依赖上一步结果 |
| Routing | 条件分叉 | 不同输入需要走不同逻辑 |
| Parallelization | 并发执行 | 子任务互相独立，或需要多次验证 |
| Orchestrator-Workers | 动态分工 | 任务复杂，需要协调多个子任务 |
| Evaluator-Optimizer | 迭代循环 | 有明确质量标准，需要反复优化 |

> [!tip] 原则
> **大多数真实系统是多种模式的组合。**
> 先识别任务里有哪些"部件"，再给每个部件选合适的模式，而不是给整个系统套一个模式。

## 4. 功能追齐 Agent 的完整模式组合

![[feature-align-patterns.excalidraw]]

```
整体：Prompt Chaining（调研 → 分析 → 执行 → 验收，顺序固定）
  │
  ├── 调研步骤内：Routing（文档清晰 / 模糊 / 链接到外部 → 走不同处理）
  │
  ├── 分析步骤内：Parallelization（App 仓库 + H5 仓库同时读取）
  │
  ├── 执行步骤内：Orchestrator-Workers（多功能点并行追齐）
  │
  └── 验收步骤内：Evaluator-Optimizer（代码生成 → 测试 → 不通过则修改）
```

这就是一个典型的"**模式嵌套**"架构——外层 Workflow 控制骨架，内层按需选择合适的模式。

## 本节小结

- 五种模式覆盖了绝大多数 AI 系统的设计场景
- Prompt Chaining（线性）/ Routing（分叉）/ Parallelization（并行）/ Orchestrator-Workers（分工）/ Evaluator-Optimizer（迭代）
- 真实系统是多种模式的组合，识别"部件"比套"整体模式"更重要
- 功能追齐 Agent 是四种模式的嵌套：Chaining 骨架 + Routing + Parallel + Evaluator-Optimizer
