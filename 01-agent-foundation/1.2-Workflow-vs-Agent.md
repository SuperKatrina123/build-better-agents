# 1.2 Workflow vs Agent：控制权在谁手里

> [!abstract] 本节导读
> 上一节搞清楚了 Agent Loop 的结构。但"跑起来一个 Loop"不代表你需要 Agent。
> 很多问题用 Workflow 解决更稳、更便宜、更好维护。
> 本节帮你建立一个清晰的判断标准：**什么时候该用 Agent，什么时候该用 Workflow。**
> **预计时长**：核心阅读 25 分钟，练习 15 分钟。

---

> [!question] 带着问题阅读
> 如果有人告诉你"这个功能要做成 Agent"，你怎么判断他说的对不对？

---

## 1. 一句话区分

Anthropic 对两者有一个直接的判断标准：

> **执行路径由代码预先写死的是 Workflow，由 LLM 动态决定下一步的是 Agent。**

核心区别只有一个：**控制权在谁手里。**

```
Workflow：代码说"下一步做什么"
  if (step === 'A') { doB() } else { doC() }

Agent：模型说"下一步做什么"
  response = llm.think(currentContext) → "我要调用工具 X"
```

> [!abstract] 定义
> **Workflow**：控制流由代码预定义。同样的输入，永远走同一条路径。
> **Agent**：控制流由 LLM 推理决定。同样的输入，可能产生不同的执行路径。

---

## 2. 七个维度对比

![Workflow 与 Agent 对比|554](https://gw.alipayobjects.com/zos/k/dh/workflow_vs_agent.svg)

| 维度 | Workflow | Agent |
|---|---|---|
| **控制权** | 代码预定义，同输入走同路径 | LLM 动态决策，路径不固定 |
| **执行方式** | 工具顺序固定，错误走预设分支 | 工具按需选择，可尝试自我修复 |
| **状态管理** | 显式状态机，节点跳转清晰 | 隐式上下文，状态在 messages[] 累积 |
| **维护成本** | 改流程需修改代码并重新部署 | 调整系统提示即可，无需重新部署 |
| **可观测性** | 日志定位节点，延迟可预估 | 需完整执行记录，轮数不固定 |
| **人机协作** | 人在预设节点介入 | 人在任意轮次介入或接管 |
| **适用场景** | 流程固定、边界清晰 | 需要中间推理与灵活判断 |

> [!tip] 原则
> **两者没有高下之分，关键是给任务找到更合适的解法。**
> 现实中很多标着 Agent 的产品，深入看其实更接近 Workflow——这不是贬义，有时候恰恰是正确的选择。

---

## 3. 判断框架：两个问题定结论

面对一个任务，问自己两个问题：

**Q1：中间步骤是固定的吗？**
- 是 → 倾向 Workflow
- 否，步骤取决于运行时的情况 → 倾向 Agent

**Q2：出错时，处理方式是固定的吗？**
- 是（走预设的错误分支）→ 倾向 Workflow
- 否（需要模型判断怎么修复）→ 倾向 Agent

```
两个都是"是" → Workflow 更合适
两个都是"否" → Agent 更合适
一是一否       → 混合架构（Workflow 中嵌 Agent 节点）
```

> [!info] 方法
> **混合架构是最常见的正确答案。**
> 稳定的步骤用 Workflow 控制（可靠、可预估），不确定的步骤用 Agent 处理（灵活、自适应）。

---

## 4. 实战视角：功能追齐任务的拆分

我们要做的功能追齐 Agent，用这个框架来拆一下：

**整体流程（固定的）→ Workflow**
```
读飞书文档 → 读 App 仓库 → 读 H5 仓库 → 对比差异 → 输出报告
```
这个顺序每次都一样，用 Workflow 控制更稳。

**每个步骤内部（不确定的）→ Agent**
```
"读飞书文档"这一步：
  - 文档描述清晰 → 直接提取功能点
  - 文档描述模糊 → 需要判断：去看 App 测试用例？还是标记为"待确认"？
  - 文档链接到其他页面 → 需要决定：要不要跟进读那个链接？
```
这里的"下一步做什么"取决于文档内容，是典型的 Agent 场景。

所以最终设计是：

![[workflow-vs-agent-feature-align.excalidraw]]

```
Workflow 层（外层骨架，控制大步骤）
  └── Step 1: 调研 → Agent 节点（处理模糊文档）
  └── Step 2: 分析 → Agent 节点（对比差异，判断优先级）
  └── Step 3: 执行 → Agent 节点（生成代码，处理边界）
  └── Step 4: 验收 → Workflow 节点（跑测试，结果确定）
```

> [!example] 案例
> Step 4 验收是纯 Workflow：跑测试、检查 TypeScript 编译、对比 feature checklist。
> 结果只有"通过"和"不通过"，没有模糊地带，不需要 LLM 来判断。
> **能用代码验证的，永远不要交给 LLM 判断。**

---

## 5. 常见的误判模式

> [!warning] 误区 1：把所有步骤都做成 Agent
> Agent 的每一步都要调用 LLM，成本高、延迟高、结果不稳定。
> 能用代码写死的逻辑，就用代码写死。

> [!warning] 误区 2：把所有步骤都做成 Workflow
> 遇到模糊输入（比如描述不清的飞书文档），Workflow 只能走预设的失败分支，无法灵活应对。
> 结果是要么漏掉很多 case，要么写出一个巨大的 if-else 树。

> [!warning] 误区 3：用"有没有工具调用"来判断是否是 Agent
> Workflow 也可以调用工具（比如固定顺序地调用 API）。
> 判断标准是**控制权**，不是工具调用。

## 本节小结

- Workflow 和 Agent 的核心区别：控制权在代码还是在 LLM
- 判断标准：步骤固定 + 错误处理固定 → Workflow；需要中间推理 → Agent
- **混合架构是最常见的正确答案**：Workflow 控制骨架，Agent 处理不确定节点
- 功能追齐任务：大流程是 Workflow，每步内部是 Agent
