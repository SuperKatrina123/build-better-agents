# 术语表与推荐阅读

> [!abstract] 文档定位

> 2. **概念对照**：厘清容易混淆的近义词
> 3. **延伸阅读**：按主题分类的高质量外部资料，按需深入

## 核心术语

### Agent
在目标约束下，基于当前状态和可见上下文，对环境做出动作，并根据反馈更新下一步判断的运行时系统。
→ 区别于 Workflow：Agent 拥有运行时自主决策权，Workflow 执行预定义流程。

### Context Engineering（上下文工程）
对模型每一轮决策前可见信息的来源、结构、装配、更新与约束进行系统化设计。
→ 区别于 Prompt：Prompt 优化单轮表达，Context Engineering 设计整个运行时信息系统。

### Context Rot（上下文腐化）
随着对话轮次增加，上下文中积累了过多无效或过时信息，导致模型注意力被稀释、推理质量下降的现象。
→ 详见：[[02-context-engineering/2.1-Context-Rot|2.1 Context Rot]]

### ACI（Action Context Interface）
工具接口设计原则——工具的名称、参数、描述、返回值应该从"LLM 的视角"来设计，而不是从"代码实现"的视角。
→ 详见：[[03-tool-design/3.1-什么是ACI|3.1 什么是 ACI]]

### Harness
包裹 Agent Loop 的控制框架，包含输入验证、权限控制、行为约束、输出审查、错误兜底。给 Agent 装上安全围栏。
→ 详见：[[06-harness/6.1-什么是Harness|6.1 什么是 Harness]]

### Memory Consolidation（记忆巩固）
在会话结束前，主动将高价值信息从 In-Context 写入 External Storage，供后续会话复用的过程。
→ 详见：[[04-memory-system/4.2-记忆巩固|4.2 记忆巩固]]

### MEMORY.md 模式
用一个结构化 Markdown 文件作为 Agent 跨会话记忆入口的设计模式。可读、可版本控制、可手动编辑、零依赖。
→ 详见：[[04-memory-system/4.5-跨会话连续性MEMORY.md模式|4.5 MEMORY.md 模式]]

### Orchestrator
在多 Agent 系统中承担总控、任务分配和结果汇总职责的控制节点。负责协调，不直接执行具体任务。
→ 详见：[[07-multi-agent/7.3-Orchestrator设计|7.3 Orchestrator 设计]]

### LLM-as-Judge
用一个专门的 LLM 来评判另一个 LLM 输出质量的评测模式，用于处理无法用规则精确评判的主观性输出。
→ 详见：[[08-evaluation/8.4-LLM评判与自动化评测|8.4 LLM-as-Judge]]

### Prompt Caching（提示词缓存）
LLM 服务商提供的机制：稳定的上下文前缀在首次处理后被缓存，后续调用直接复用，降低延迟和费用。
→ 详见：[[02-context-engineering/2.5-Prompt-Caching|2.5 Prompt Caching]]

### ReAct
一种边推理边行动的执行模式（Reasoning + Acting），让系统在推理和工具调用之间交替推进。

### Plan-and-Execute
一种先明确计划、再沿计划执行并按需修正的执行模式，适合结构明确的长任务。
→ 详见：[[05-planning-long-tasks/5.1-任务分解与规划|5.1 任务分解与规划]]

## 概念对照表

| 概念 A | 概念 B | 关键区别 |
|--------|--------|---------|
| Prompt | Context Engineering | 前者优化当前表达，后者设计运行时信息系统 |
| Workflow | Agent | 前者执行预定义流程，后者拥有运行时决策权 |
| State（状态）| History（历史）| 前者服务当前控制，后者记录过去发生的事 |
| Memory（记忆）| Retrieval（检索）| 前者回答"该保留什么"，后者回答"现在如何找回来" |
| 评测 | 观测 | 评测回答"好不好"，观测回答"当时发生了什么" |
| Harness | Guardrail | Harness 是整体控制框架，Guardrail 是其中的规则约束层 |
| Copilot | Agent | 前者由人主控，后者拥有部分运行时控制权 |
| In-Context Memory | External Memory | 前者会话级，后者跨会话持久 |

## 推荐阅读

### 必读：本课程的核心来源

这两篇是整套知识库的主要参考来源，建议在学完一半内容后精读：

- **[你不知道的 Agent：原理、架构与工程实践](https://tw93.fun/2026-03-21/agent.html)**
  tw93 的深度综述，覆盖 Agent Loop、上下文工程、记忆系统、多 Agent 拓扑，图多易读。本课程大量图片和案例来源于此。

- **[Building Effective Agents — Anthropic](https://www.anthropic.com/research/building-effective-agents)**
  Anthropic 官方的 Agent 工程实践指南，强调"从简单开始、只在必要时增加复杂度"，是工具设计和 Harness 章节的理论基础。

---

### 上下文工程

- **[Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172)**
  实验证明 LLM 对上下文中间部分的注意力显著下降，是 Context Rot 现象的学术依据。

- **[Anthropic Prompt Caching Docs](https://docs.anthropic.com/en/docs/build-with-claude/prompt-caching)**
  Prompt Caching 的官方实现文档，含 TTL、缓存命中率、成本计算的具体说明。

---

### 工具设计与 ACI

- **[Toolformer: Language Models Can Teach Themselves to Use Tools](https://arxiv.org/abs/2302.04761)**
  工具调用的早期基础论文，帮助理解为什么工具接口的设计方式会影响 LLM 的调用质量。

- **[Model Context Protocol (MCP)](https://modelcontextprotocol.io/)**
  Anthropic 提出的工具/资源暴露协议标准，是 ACI 设计在协议层的具体实现。

---

### 记忆与状态

- **[Generative Agents: Interactive Simulacra of Human Behavior](https://arxiv.org/abs/2304.03442)**
  斯坦福的生成式 Agent 论文，提出了完整的记忆系统设计（感知、记忆、规划、行动），是记忆系统章节的重要参考。

- **[Claude Code Best Practices — Anthropic](https://www.anthropic.com/engineering/claude-code-best-practices)**
  Claude Code 的工程实践，包含 MEMORY.md 模式在 Coding Agent 中的实际应用案例。

---

### 规划与长任务

- **[ReAct: Synergizing Reasoning and Acting in Language Models](https://arxiv.org/abs/2210.03629)**
  ReAct 原论文，边推理边行动的执行范式，是动态规划策略的理论来源。

- **[Tree of Thoughts: Deliberate Problem Solving with LLMs](https://arxiv.org/abs/2305.10601)**
  ToT 论文，探索多路径推理在复杂任务规划中的应用。

---

### 评测与可靠性

- **[Judging LLM-as-a-Judge with MT-Bench and Chatbot Arena](https://arxiv.org/abs/2306.05685)**
  LLM-as-Judge 方法的基础论文，分析了这种评测方式的优势和局限性。

- **[AgentBench: Evaluating LLMs as Agents](https://arxiv.org/abs/2308.03688)**
  专门针对 Agent 能力的评测基准，可参考其评测维度设计。

---

### 多 Agent 系统

- **[AutoGen: Enabling Next-Gen LLM Applications via Multi-Agent Conversation](https://arxiv.org/abs/2308.08155)**
  微软的多 Agent 框架论文，包含 Orchestrator-SubAgent 架构的实现思路。

- **[LangGraph 文档](https://langchain-ai.github.io/langgraph/)**
  目前最成熟的多 Agent 编排框架之一，适合在第七章学完后参考其工程实现。

---
