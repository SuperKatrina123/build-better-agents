# 跨场景迁移指南

> [!abstract] 文档定位
> 这页解决一个结构性问题：知识库里的主案例是“功能追齐 Agent”，但你未必正在做 Coding Agent。
> 它的作用是帮你区分“通用原理”和“案例实现”，把同一套设计语言迁移到不同类型的 Agent 上。

---

## 先抓住这个阅读顺序

每章优先按这三个层次来读：

1. **通用原理**
   先理解这一章到底在解决什么普遍问题。
2. **场景映射**
   再看这个问题在不同 Agent 里长什么样。
3. **主案例展开**
   最后才回到功能追齐 Agent，看一套完整落地方案。

如果顺序反过来，你很容易把“案例配置”误认成“通用方法”。

---

## 四类典型 Agent

| 类型 | 核心目标 | 主要环境 | 典型动作 |
|------|----------|----------|----------|
| Coding Agent | 分析和修改代码、配置、文档 | 仓库、文件系统、CI | 读文件、改文件、跑命令 |
| Research Agent | 搜集、筛选、总结信息 | 网页、文档库、资料源 | 检索、比较、归纳、引用 |
| Support Agent | 响应用户问题并推动问题解决 | 工单、知识库、用户历史 | 检索知识、生成答复、升级工单 |
| 流程 / 业务执行 Agent | 推动一组业务步骤达成结果 | 表单、审批系统、业务 API | 读状态、触发动作、更新记录 |

---

## 哪些原则是跨场景稳定的

| 章节 | 稳定原则 | 不要误以为只属于 Coding Agent 的东西 |
|------|----------|--------------------------------------|
| 第 1 章 | 控制权、状态、反馈闭环 | `messages[]`、文件工具只是实现形态之一 |
| 第 2 章 | 模型每轮看到什么、如何装配和压缩 | 仓库摘要只是上下文的一种内容类型 |
| 第 3 章 | 工具接口必须从模型可用性出发设计 | `read_repo_file` 不是工具设计的默认原型 |
| 第 4 章 | 跨会话保留什么、何时写入、如何读取 | `MEMORY.md` 是模式，不是唯一载体 |
| 第 5 章 | 长任务需要计划、状态、恢复点 | `task-state.json` 只是状态落盘的一种方式 |
| 第 6 章 | 确定性边界、权限和输出审查 | “改代码”只是高风险动作的一种 |
| 第 7 章 | 何时拆多 Agent、如何分工通信 | Orchestrator-SubAgent 不只适用于代码场景 |
| 第 8 章 | 概率输出需要新的评测方式 | “生成 diff”不是唯一被评测的结果形态 |
| 第 9 章 | 原型到生产的差距总会出现 | 可观测性和成本控制不只属于工程 Agent |

## 同一章，在四类 Agent 里分别看什么

### 第 2 章：上下文工程

| 类型 | 本章关注对象 | 典型上下文内容 |
|------|--------------|----------------|
| Coding Agent | 当前任务、仓库结构、工具结果 | 文件摘要、功能规格、修改状态 |
| Research Agent | 研究目标、来源质量、阶段结论 | 主题、来源列表、可信度标注 |
| Support Agent | 用户问题、历史工单、知识库命中 | 用户画像、工单状态、检索答案 |
| 流程 Agent | 当前步骤、规则条件、外部系统返回 | 表单字段、审批状态、接口结果 |

### 第 5 章：规划与长任务

| 类型 | 规划单元 | 检查点示例 |
|------|----------|------------|
| Coding Agent | 功能点、文件改动步骤 | 已分析文件、已完成改动、待验证项 |
| Research Agent | 子问题、资料批次、结论合并 | 已读来源、已验证结论、待补证据 |
| Support Agent | 工单阶段、沟通轮次、升级路径 | 已确认信息、已执行动作、待回复事项 |
| 流程 Agent | 业务节点、审批阶段、异常分支 | 当前节点、已完成动作、待人工确认项 |

### 第 6 章：Harness

| 类型 | 高风险动作 | 典型保护措施 |
|------|------------|--------------|
| Coding Agent | 写文件、跑命令、改配置 | 路径白名单、预算限制、输出审查 |
| Research Agent | 引用错误来源、伪造结论 | 来源校验、引用格式约束、低可信度标注 |
| Support Agent | 错答、越权承诺、误操作工单 | 权限检查、回复模板、人工升级阈值 |
| 流程 Agent | 错触发业务动作、错改状态 | 审批校验、幂等保护、审计日志 |

## 怎么避免被主案例“带偏”

- 看到“功能追齐 Agent”时，先问：这里讲的是通用原则，还是案例参数？
- 看到具体文件名、工具名、JSON 结构时，先做一次抽象翻译：
  - 这对应的是“上下文”“状态”“工具”“记忆”“检查点”中的哪一类？
- 如果你做的不是 Coding Agent，就把每章里的案例对象替换成你自己的任务对象，再判断原理是否仍然成立。

---

## 推荐用法

- 第一次通读时：每章先看 `00-chapter-index.md` 的“通用原理 / 场景映射 / 主案例展开”。
- 第二次深入时：只跟自己相关的场景读正文。
- 做方案时：优先提炼通用决策框架，再实例化到你的 Agent，而不是直接照搬功能追齐 Agent 的配置。
