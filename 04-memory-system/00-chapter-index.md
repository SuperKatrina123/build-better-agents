# 第四章：记忆系统——上下文的时间维度

> 上下文工程解决"当前会话"的信息质量，记忆系统解决"跨会话"的知识积累。
> 没有记忆系统的 Agent，每次都是从零开始。

---

## 章节目标

完成本章后，你应该能够：
1. 识别四种记忆形式，为不同类型的信息选择合适的存储方式
2. 设计记忆巩固协议，让 Agent 在会话结束时主动保存有价值的知识
3. 为功能追齐 Agent 搭建 MEMORY.md 记忆系统（含初始化流程）
4. 添加 read_memory / write_memory 两个工具，让 Agent 自主维护记忆

---

## 通用原理

记忆系统讨论的不是“要不要上向量库”，而是所有 Agent 都必须面对的时间维度问题：

- 当前会话结束后，哪些信息值得保留
- 哪些信息必须跨会话继续生效，哪些应该只在局部短暂存在
- 记忆何时写入、如何组织、在什么时机读回

只要 Agent 不是一次性问答，而是要长期推进任务、积累知识、避免重复探索，本章的方法就成立。

## 主案例展开：功能追齐 Agent

```
工具清单（本章新增）：
  read_memory(key)              → 读取仓库结构 / 追齐结论
  write_memory(key, content)    → 写入新发现 / 更新进度

记忆目录：
  memory/MEMORY.md              → 跨会话入口（常驻注入 ~500 tokens）
  memory/completed.json         → 追齐状态（高频读写）
  memory/repos/{仓库}/          → 仓库知识（按需检索）
  memory/features/{功能}.md     → 追齐结论（按需检索）

会话协议：
  开始时：读 MEMORY.md → 跳过已完成 → 优先用已有记忆
  执行中：发现新知 → 立即 write_memory
  结束时：更新进度 → 写入追齐结论
```

这里的 `MEMORY.md`、目录结构和两个记忆工具，是主案例的一种可读、可版本化实现。更通用的原则是：

- 先定义“什么值得记住”，再选择存储介质
- 先定义读取时机，再决定记忆是否需要常驻注入
- 记忆系统服务于未来判断，不是历史材料堆积
