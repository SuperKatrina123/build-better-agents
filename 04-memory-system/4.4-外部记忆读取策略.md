# 4.4 外部记忆的读取策略：按需检索 vs 常驻注入

> [!abstract] 本节导读
> 有了好的记忆存储，还需要正确的读取策略。
> 不同类型的记忆，适合不同的读取方式——全部常驻会占满上下文，全部按需会漏掉关键信息。
> 本节给出三种模式和它们的适用场景。
> **预计时长**：核心阅读 15 分钟。

---

> [!question] 带着问题阅读
> 你的 Agent 在会话开始时，会读取哪些记忆？是全部读，还是只读一部分？
> 你有没有想过"读哪些"这个问题的答案？

---

## 1. 三种读取模式

![[memory-retrieval-modes.excalidraw]]

### 模式一：常驻注入（Always-Loaded）

**定义**：会话开始时，固定把某些记忆注入到 L1/L3 层。
**适用**：体积小（≤ 500 tokens）、每次任务都需要的记忆。

```typescript
// 会话初始化时自动注入
async function initSession() {
  const repoMap = await readMemory("repo-map.config.ts");      // 仓库映射
  const completed = await readMemory("completed.json");         // 已完成功能

  return {
    systemPromptAddendum: `
## 当前追齐进度
已完成：${completed.completed.map(f => f.feature).join(", ")}
进行中：${completed.inProgress.join(", ")}

## 仓库映射摘要（完整配置见工具）
${formatRepoMapSummary(repoMap)}
    `,
  };
}
```

**追齐 Agent 常驻的记忆**：
- `completed.json` 的摘要（已完成 + 进行中 + 待处理，约 200 tokens）
- repo-map 的简版索引（15 个仓库的名称+关键词，约 300 tokens）
- 合计约 500 tokens，在 L1 预算范围内

---

### 模式二：工具调用检索（On-Demand）

**定义**：LLM 在任务执行中，根据需要主动调用工具读取记忆。
**适用**：体积中等（500~3000 tokens）、只在特定任务中需要的记忆。

```typescript
// read_memory 工具
{
  name: "read_memory",
  description: "读取已存储的关于某个仓库或功能的记忆信息。当你需要了解某个仓库的结构、已知坑点，或某个功能的历史追齐结论时调用。",
  inputSchema: {
    type: "object",
    required: ["key"],
    properties: {
      key: {
        type: "string",
        description: "记忆键，格式：'repo/{仓库名}/structure' 或 'feature/{功能名}'"
      }
    }
  }
}
```

**LLM 调用时机（典型序列）**：
```
任务：追齐购物车收藏功能

1. LLM 先查常驻记忆 → 发现"购物车"功能中有"收藏功能"待处理
2. LLM 调用 read_memory("repo/h5-cart-page/structure")
   → 直接获得仓库目录结构，跳过 list_repo_files 探索
3. LLM 调用 read_memory("repo/app-cart-module/feature-spec")
   → 获取 App 端功能规格缓存（如果有）
4. 开始追齐，减少了 3~5 次工具调用
```

> [!tip] 原则
> **让 LLM 知道"有哪些记忆可以读"，是模式二能有效工作的前提。**
> 常驻记忆里应该包含记忆目录索引——LLM 看到索引才知道能读什么。
> 不需要全量索引，一个关键词级别的摘要就够。

---

### 模式三：向量语义检索（Semantic Search）

**定义**：把记忆内容向量化，用语义相似度检索最相关的条目。
**适用**：体量大（100+ 条记忆）、需要"根据任务内容自动匹配相关记忆"。

追齐 Agent 当前阶段（15 个仓库、几十个功能点）**不需要向量检索**。
关键词匹配 + 工具调用就能满足需求，而且更可靠、成本更低。

> [!info] 方法
> **向量检索的引入门槛**：当记忆条目超过 100 个，且 LLM 经常"不知道用哪条记忆"时，才考虑引入向量检索。
> 过早引入向量检索会增加不确定性——语义相似不等于内容相关。
> 对于结构清晰的 key-value 记忆，精确查找永远优于模糊匹配。

---

## 2. 读取模式组合设计

追齐 Agent 的推荐组合：

```
会话开始（常驻注入）
  ↓ L1 自动注入 500 tokens
  completed.json 摘要 + repo-map 索引

任务执行（按需检索）
  ↓ LLM 主动调用 read_memory
  仓库结构摘要 + 技术规格缓存 + 历史追齐结论

任务完成（记忆巩固）
  ↓ 写回 External Storage
  更新 completed.json + 写入新的功能追齐结论
```

**记忆读取的 token 消耗估算**：
- 常驻注入：~500 tokens（固定）
- 按需检索：每次 `read_memory` 约 300~600 tokens，典型任务 2~3 次
- 合计：~1100~2300 tokens（记忆层）
- 相比于"没有记忆、每次从头探索"节省：~3000~5000 tokens

---

## 3. 记忆索引：让 LLM 知道能读什么

在 completed.json 里加一个 `availableMemories` 字段，作为轻量级记忆目录：

```json
{
  "completed": [...],
  "inProgress": ["促销标签展示"],
  "pending": ["收藏功能", "批量加购"],
  "availableMemories": {
    "repos": ["h5-cart-page", "h5-profile-page", "app-cart-module"],
    "features": ["cart-batch-delete", "cart-coupon-display"],
    "hint": "调用 read_memory('repo/{名称}/structure') 读取仓库摘要，read_memory('feature/{功能}') 读取追齐结论"
  }
}
```

这 50 tokens 的索引，让 LLM 知道记忆目录里有什么，大幅提升按需检索的命中率。

---

## 本节小结

- 三种读取模式：常驻注入（≤500 tokens）/ 工具检索（按需）/ 向量检索（100+ 条时考虑）
- 追齐 Agent 推荐：常驻注入（摘要+索引）+ 工具检索（仓库结构+功能结论）
- 记忆索引（availableMemories）让 LLM 知道"能读什么"
- 有记忆 vs 无记忆：典型任务节省约 3000~5000 tokens
