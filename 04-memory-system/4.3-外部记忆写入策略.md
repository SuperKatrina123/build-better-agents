# 4.3 外部记忆的写入策略：结构化存储设计

> [!abstract] 本节导读
> 知道"要巩固什么"之后，还需要知道"怎么存"。
> 记忆的存储结构决定了读取效率——结构清晰才能按需精准读取，
> 结构混乱会让记忆系统退化成一个"垃圾堆"：什么都有，但找不到有用的。
> **预计时长**：核心阅读 20 分钟。

---

> [!question] 带着问题阅读
> 如果让你设计一个"所有仓库知识"的存储结构，你会怎么组织文件？
> 按功能？按仓库？按时间？——不同的组织方式，读取效率差异很大。

---

## 1. 写入决策框架

![[memory-write-decision.excalidraw]]

外部记忆按内容性质分两大类：

**稳定事实** → 结构化文件（JSON / Markdown）
- 仓库目录结构、技术栈约定、API 接口定义
- 特点：基本不变，每次写入覆盖旧值

**动态状态** → 状态文件（JSON）
- 任务进度、已完成功能列表、当前优先级
- 特点：频繁更新，需要追加和版本管理

---

## 2. 追齐 Agent 的记忆目录结构

```
memory/
├── repo-map.config.ts          # 仓库映射（手动维护，工具层读取）
├── completed.json              # 已完成追齐的功能列表
├── repos/
│   ├── h5-cart-page/
│   │   ├── structure.md        # 目录结构摘要（巩固后不常变）
│   │   ├── tech-stack.md       # 技术栈约定
│   │   └── known-issues.md     # 已知坑点（巩固时追加）
│   ├── h5-profile-page/
│   │   └── ...
│   └── app-cart-module/
│       └── ...
├── features/
│   ├── cart-batch-delete.md    # 批量删除追齐结论
│   ├── cart-coupon-display.md  # 优惠券展示追齐结论
│   └── ...
└── sessions/
    └── 2026-04-02-cart.json    # 今日会话摘要（可选）
```

**设计原则**：

1. **按仓库分目录**：每个仓库的知识独立存放，互不影响
2. **按功能点分文件**：每个功能追齐结论存一个文件，方便按需读取
3. **completed.json 全局维护**：追齐状态是高频读写的，单独一个文件
4. **会话摘要可选**：用于追踪历史，不需要时可以不写

## 3. 各类文件的格式设计

### repos/{仓库}/structure.md

```markdown
# h5-cart-page 仓库结构摘要

## 基本信息
- 框架：React 18 + TypeScript
- 状态管理：Redux Toolkit + RTK Query
- 样式：CSS Modules
- 最后探索：2026-03-15（commit: abc1234）

## 关键目录
- src/pages/Cart/         → 购物车页面入口
- src/components/Cart/    → 购物车专属组件（注意：不在 pages/ 下）
- src/store/cartSlice.ts  → Redux slice
- src/api/cart.ts         → 接口定义

## 快速定位
| 功能 | 路径 |
|------|------|
| 页面入口 | src/pages/Cart/index.tsx |
| 商品列表 | src/components/Cart/CartItemList.tsx |
| 价格计算 | src/components/Cart/hooks/useCartPrice.ts |
```

### features/{功能点}.md

```markdown
# 追齐结论：购物车批量删除

## 基本信息
- 追齐日期：2026-03-20
- App 版本：v2.3.1
```

## 4. 写入的原子性

> [!warning] 误区
> **不要在记忆写入过程中途失败。**
> 如果 Agent 写了一半的记忆文件，下次读到不完整的内容，会做出错误决策。

推荐的写入模式：先写临时文件，成功后原子替换：

```typescript
async function atomicWriteMemory(path: string, content: string) {
  const tmpPath = `${path}.tmp`;
  await fs.writeFile(tmpPath, content);
  await fs.rename(tmpPath, path);  // 原子操作，避免半写状态
}
```

## 本节小结

- 稳定事实 → 结构化 Markdown 文件（按仓库/功能分目录）
- 动态状态 → JSON 文件（completed.json 全局维护）
- 目录设计：repos/{仓库}/ + features/{功能}.md + completed.json
- 写入原子性：先写 .tmp 再 rename，避免半写状态
