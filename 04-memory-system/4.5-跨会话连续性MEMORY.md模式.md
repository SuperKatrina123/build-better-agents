# 4.5 跨会话连续性：MEMORY.md 模式

> [!abstract] 本节导读
> MEMORY.md 模式是 coding agent（如 Claude Code、Cursor）中被广泛验证的跨会话记忆方案。
> 它用一个结构化的 Markdown 文件作为 Agent 的"长期记忆入口"，
> 既简单（不需要数据库），又可靠（文件可读、可 diff、可版本控制）。
> **预计时长**：核心阅读 20 分钟。

---

> [!question] 带着问题阅读
> 你用过 Claude Code 或 Cursor 吗？它们都会在项目根目录或特定位置维护一个"知识文件"。
> 这就是 MEMORY.md 模式的具体体现。

---

## 1. 为什么是 Markdown 文件，而不是数据库

相比向量数据库或结构化数据库，Markdown 文件有几个实用优势：

**优势一：天然可读**
Agent 读、开发者也能读，出了问题可以直接打开文件排查。

**优势二：可版本控制**
MEMORY.md 可以 commit 进仓库，支持 diff 和回滚。
"上次追齐出问题是因为记忆里写错了什么"——直接看 git diff 就能定位。

**优势三：可直接编辑**
开发者可以手动修正错误的记忆，或者提前写入已知信息。
这种"人机协作维护记忆"的模式，比全自动维护更安全。

**优势四：零依赖**
不需要 Redis、Pinecone、SQLite……文件系统就够了。
适合 MVP 阶段快速验证，后期有需要再替换底层存储。

## 2. MEMORY.md 的结构设计

```markdown
<!-- memory/MEMORY.md -->
# 功能追齐 Agent 记忆文件

> 最后更新：2026-04-02
> 本文件由 Agent 自动维护，开发者可直接编辑修正。

---
```

## 3. Agent 与 MEMORY.md 的交互协议

在 System Prompt 里加入以下约定：

```markdown
## 记忆管理约定

**会话开始时**：
读取 memory/MEMORY.md，作为你了解项目状态的第一手资料。
如果文件不存在，说明是首次使用，先完成初始化探索再写入。

**发现新知识时**：
如果你发现了关于仓库结构、技术约定或追齐规律的重要信息，
调用 write_memory 工具更新 MEMORY.md。

**任务完成时**：
将功能的追齐状态更新到 MEMORY.md 的"追齐进度"部分。
将本次发现的仓库知识写入"仓库知识库"部分。

**发现记忆过时时**：
如果工具返回结果与 MEMORY.md 中的记录不符，
优先相信工具结果，并调用 write_memory 更新 MEMORY.md，
同时在输出中注明：「已更新 MEMORY.md：{具体内容}」。
```

---

## 4. 首次初始化：让 Agent 自建记忆

第一次启动追齐 Agent 时，MEMORY.md 还不存在。
这时让 Agent 执行一次"记忆初始化"任务：

```
初始化任务 Prompt：
"请对以下仓库进行探索，并生成初始的 MEMORY.md 文件：
- 仓库列表：[根据 repo-map.config.ts]
- 对每个仓库：列出目录结构、识别技术栈、记录关键文件路径
- 生成格式参考 MEMORY.md 模板
完成后输出 MEMORY.md 内容"
```

这个初始化任务通常需要 20~30 次工具调用，
但只需要执行一次，后续所有会话都可以复用这份基础记忆。

> [!example] 案例
> 初始化 15 个仓库的 MEMORY.md：
> - 耗时：约 8 分钟，消耗约 25,000 tokens
> - 产出：一份 3000 tokens 的 MEMORY.md
> - 后续每次任务节省：约 4000~6000 tokens（跳过仓库探索）
> - 5 次任务后，初始化成本就已经回本

---

## 5. MEMORY.md 的维护策略

| 情况 | 处理方式 |
|------|---------|
| Agent 写入了错误信息 | 开发者直接编辑文件修正，Agent 下次读取即生效 |
| 仓库发生重大重构 | 手动更新相关仓库条目，或让 Agent 重新探索该仓库 |
| MEMORY.md 变得过大（> 5000 tokens）| 把详细内容移到 repos/{仓库}/structure.md，MEMORY.md 只保留摘要+链接 |
| 想回滚到某个历史状态 | git checkout MEMORY.md |

> [!tip] 原则
> **MEMORY.md 是人机协作的文档，不是 Agent 的私有状态。**
> 开发者可以、也应该主动参与维护。
> 初期让 Agent 主导写入，遇到错误时人工修正——这个协作模式比全自动或全手动都更可靠。

---

## 本节小结

- MEMORY.md 模式：用结构化 Markdown 文件作跨会话记忆入口
- 优势：可读、可版本控制、可手动编辑、零依赖
- 协议：会话开始时读取 → 发现新知时更新 → 任务完成时巩固 → 记忆过时时纠正
- 首次初始化：让 Agent 探索所有仓库，一次性生成 MEMORY.md（5 次任务后回本）
