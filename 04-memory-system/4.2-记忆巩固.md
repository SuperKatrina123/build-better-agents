# 4.2 记忆巩固：从上下文到持久化

> [!abstract] 本节导读
> 会话结束后，有价值的信息默认会消失——这是 Agent 系统里最容易被忽视的设计缺口。
> "记忆巩固"就是在会话结束前，主动把值得保留的信息写入 External Storage。
> 本节讲清楚：什么时候写、写什么、怎么写不引入噪声。
> **预计时长**：核心阅读 20 分钟。

---

> [!question] 带着问题阅读
> 你的 Agent 上周完成了一个任务，学到了一些关于项目结构的知识。
> 这周再启动时，它还知道吗？如果不知道，这就是没有做记忆巩固。

---

## 1. 为什么需要主动巩固

没有记忆巩固的 Agent 会出现一个隐性问题：**每次任务都是从零开始的**。

它每次都要重新探索仓库结构、重新了解技术规范、重新发现上次已经总结过的规律。
这不仅消耗 token，还降低效率——而且随着使用时间增加，这个问题会越来越明显。

![记忆巩固](https://gw.alipayobjects.com/zos/k/cx/memory_consolidation_.svg)

记忆巩固的核心思路：**会话内的高价值信息，在会话结束前写入外部存储，供后续会话直接使用。**

---

## 2. 什么信息值得巩固

不是所有信息都值得写入 External Storage。用三个标准来筛选：

**标准一：跨会话复用价值**
这个信息在未来的会话里还会用到吗？

```
✅ 值得巩固：
- 仓库 A 使用 RTK Query，接口在 src/api/ 下
- 功能 X 已经追齐完成（v2.3.1 版本对应）
- 购物车页面的批量删除逻辑依赖 useCartEdit hook

❌ 不值得巩固：
- 本次任务的中间推理过程
- 已经无效的报错信息
- 临时测试的代码片段
```

**标准二：获取成本高**
这个信息下次重新获取需要多少工具调用？成本越高，越值得巩固。

```
高成本（值得巩固）：
- 探索仓库结构：需要 3~5 次 list_repo_files + read_repo_file
- 提取技术规格：需要读取多个测试文件并分析

低成本（可以不巩固）：
- 单个文件的内容：下次直接读即可，文件本身就是持久化的
```

**标准三：稳定性**
频繁变动的信息写进 External Storage，会引入"记忆错误"——比不记忆更危险。

```
高稳定（适合巩固）：
- 仓库目录结构（基本不变）
- 技术栈约定（偶尔变）
- 已完成的追齐功能（只增不减）

低稳定（不要巩固）：
- 临时分支的代码状态
- 正在开发中的功能规格
- 开发进度（每天都在变）
```

> [!tip] 原则
> **三个标准同时满足时，巩固价值最高：跨会话复用 + 获取成本高 + 内容稳定。**
> 仓库摘要是追齐 Agent 里最典型的高价值巩固对象。

## 3. 巩固的时机

### 时机一：任务完成时（主动巩固）

```typescript
// Agent 任务完成钩子
async function onTaskComplete(taskResult: AlignmentResult) {
  // 判断是否有值得巩固的信息
  const memoryUpdates = extractMemoryUpdates(taskResult);

  for (const update of memoryUpdates) {
    await memoryStore.write(update.key, update.value);
  }

  // 更新已完成功能列表
  await memoryStore.append("completed_features", taskResult.feature);
}
```

### 时机二：会话中途发现新知识（即时巩固）

```
LLM 在分析仓库时发现：
"h5-cart-page 的 CartItem 组件在 src/components/Cart/ 而不是 pages/ 下"

这是一个高价值发现，应该立即写入记忆，不等到任务结束。
```

可以给 LLM 一个 `write_memory` 工具，让它在发现重要新知识时主动调用：

```typescript
{
  name: "write_memory",
  description: "将本次发现的重要信息写入长期记忆。当你发现了关于某个仓库的重要结构信息、技术约定或追齐规律时调用。",
  inputSchema: {
    type: "object",
    required: ["key", "content", "reason"],
    properties: {
      key: { type: "string", description: "记忆的键，如 'repo/h5-cart-page/structure'" },
      content: { type: "string", description: "要记住的内容（≤500 tokens）" },
      reason: { type: "string", description: "为什么这个信息值得记忆（帮助未来判断是否仍然有效）" }
    }
  }
}
```

### 时机三：定期整理（批量巩固）

对于大量累积的历史追齐结论，可以定期（比如每周）跑一次整理脚本，
把散碎的记忆条目合并、去重、更新成结构化的长期记忆文档。

## 4. 防止"记忆错误"

写入外部存储的信息，如果过时了还继续被使用，会比没有记忆更危险。

```
危险场景：
记忆里写着："购物车页面入口是 src/pages/CartPage.tsx"
但这个文件在上个月的重构里被改名了。

Agent 看到这个记忆，直接去读 CartPage.tsx → 文件不存在 → 任务失败。
比没有这条记忆时（Agent 会自己探索）的结果更差。
```

**防范措施**：

1. **写入时记录来源版本**：
```json
{
  "key": "repo/h5-cart-page/entry_file",
  "value": "src/pages/Cart/index.tsx",
  "source": "探索于 2026-03-15，仓库 commit abc1234",
  "confidence": "high"
}
```

2. **读取时校验**：
```typescript
async function readMemoryWithValidation(key: string) {
  const memory = await store.read(key);

  // 如果记忆超过 30 天，标记为"可能过时"
  if (daysSince(memory.timestamp) > 30) {
    return { ...memory, stale: true, warning: "此记忆超过 30 天，建议验证后使用" };
  }

  return memory;
}
```

3. **LLM 发现记忆和现实不符时，主动更新**：
在 System Prompt 里加一条约定：
```
当工具返回结果与记忆中的内容不符时，调用 write_memory 更新记忆，
并在输出中注明"已更新记忆：[key]"。
```

> [!warning] 误区
> **不要让记忆系统自动信任所有写入的内容。**
> 记忆的可靠性需要主动维护——过时的记忆应该标记或删除，而不是无限期保留。

---

## 本节小结

- 记忆巩固 = 会话结束前把高价值信息写入 External Storage
- 三个筛选标准：跨会话复用 + 获取成本高 + 内容稳定
- 三个巩固时机：任务完成时 / 即时发现时 / 定期批量整理
- 防记忆错误：写入时记录版本、读取时校验新鲜度、LLM 发现过时时主动更新
