# 8.2 评测维度设计：用什么标准衡量 Agent 的好坏

> [!abstract] 核心定义
> 没有明确的评测维度，就没办法客观地说 Agent "好"还是"不好"。
> 
> 评测维度是评测体系的骨架：它决定了我们**观察什么**、**测量什么**、**用哪些数据做决策**。本节建立功能追齐 Agent 的完整评测维度体系——五个维度，每个维度对应可量化的指标。

> [!question] 带着问题阅读
> - 评测一个 Agent，应该从哪几个角度衡量？
> - 不同阶段（开发期/稳定期），哪些维度更重要？
> - 如何为这些维度建立基准值（Baseline）？

## 1. 五个核心维度

![[eval-dimensions.excalidraw]]

评测追齐 Agent，从五个维度综合衡量：

### 正确性（Correctness）

**核心问题：** 功能是否真的被追齐？H5 行为是否与 App 一致？

这是最重要的维度，也是最难自动化的维度。正确性的判断需要理解"功能语义"，而不只是比对代码文本。

追齐购物车数量功能时：
- ✅ 正确：H5 中增减按钮触发 `updateQuantity` API，与 App 行为一致
- ❌ 错误：H5 只更新了前端状态，没有调用后端 API（与 App 行为不一致）
- ❌ 错误追齐：把已有的"清空购物车"功能也改了（不在本次任务范围内）

### 效率（Efficiency）

**核心问题：** 完成任务消耗了多少资源？工具调用是否精简？

Agent 可能用 10 次工具调用完成任务，也可能用 40 次——如果结果一样，40 次就是低效。效率不只影响速度，更直接影响成本。

衡量效率的关键指标：
- 工具调用总次数
- 每类工具的调用次数（`read_repo_file` 是否有冗余？）
- 总 token 消耗（输入 + 输出）
- 完成任务的墙上时间（wall clock time）

### 鲁棒性（Robustness）

**核心问题：** 遇到不理想的输入时，Agent 是否能优雅降级？

真实世界的仓库往往不是"教科书式"的：目录结构混乱、文档缺失、代码有历史债务。鲁棒性衡量 Agent 在这些情况下的表现。

鲁棒性场景分类：
- 仓库结构不标准（组件在意料之外的目录里）
- 功能已部分实现（上次追齐做了一半）
- 依赖组件缺失（需要追齐的组件依赖一个还不存在的基础组件）
- 需求描述模糊（feature spec 写得不够清晰）

### 成本（Cost）

**核心问题：** 每次任务消耗多少 API 费用？月度成本可接受吗？

Agent 的 token 消耗可能惊人。一个功能追齐任务如果读了 50 个文件、每个文件 500 行，上下文就已经是 25,000 行代码，再加上 System Prompt 和多轮对话，很容易超过 100k tokens。

成本控制是长期运营 Agent 的关键，不能只看准确率，还要看"每次追齐的实际费用"。

### 可解释性（Explainability）

**核心问题：** Agent 能否清晰说明为什么要做这些修改？

可解释性直接影响**开发者信任**。如果 Agent 生成了一大堆 diff 但说不清楚原因，开发者在 review 时就需要自己搞清楚每一行改动的意图——这反而增加了工作量。

好的可解释性：
```
## 追齐修改说明

本次追齐"商品数量增减"功能，参考 iOS App 的 CartItemView.swift。

修改原因：
1. CartItem.tsx：App 中数量增减直接调用后端 API，H5 原实现只更新本地状态
   → 添加了 onQuantityChange 回调，在父组件中触发 PUT /api/cart/:id
   
2. useCart.ts：App 在数量为 0 时自动移除商品，H5 原来需要单独点击删除
   → 在 updateQuantity 中添加了数量为 0 时自动调用 removeItem 的逻辑
```

差的可解释性：
```
已完成功能追齐，修改了 CartItem.tsx 和 useCart.ts。
```

## 2. 追齐 Agent 评测指标

每个维度对应具体可测量的指标：

| 维度 | 指标 | 测量方法 | 目标值 |
|------|------|----------|--------|
| **正确性** | 功能追齐成功率 | 人工验证 or LLM Judge | ≥ 85% |
| | 错误追齐率（改了不该改的） | diff 分析 + 人工核查 | ≤ 5% |
| **效率** | 平均工具调用次数/功能 | 日志统计 | ≤ 20 次 |
| | 平均 token 消耗/功能 | API 统计 | ≤ 50k tokens |
| **鲁棒性** | 非标准仓库成功率 | 测试用例跑边界场景 | ≥ 70% |
| | 降级成功率（文档缺失时） | 测试对抗用例 | ≥ 60% |
| **成本** | 平均 API 费用/功能 | 成本统计 | ≤ ¥2/功能 |
| | 月度成本估算（100功能/月） | 压力测试 | ≤ ¥200/月 |
| **可解释性** | 修改理由完整度评分 | LLM Judge (1-5分) | ≥ 4分 |
| | 开发者能否独立验证 | 人工问卷 | ≥ 80% 说"能理解" |

### 指标计算方式示例

**功能追齐成功率的计算：**

```typescript
interface EvalResult {
  testCaseId: string;
  passed: boolean;          // 人工或 LLM 判断
  correctness: number;      // 1-5 分
  falsePostiive: boolean;   // 是否追了不该追的
}

function calcSuccessRate(results: EvalResult[]): number {
  const passed = results.filter(r => r.passed && !r.falsePositive);
  return passed.length / results.length;
}
```

**平均工具调用次数的计算：**

```typescript
interface ToolCallLog {
  taskId: string;
  toolName: string;
  timestamp: number;
}

function calcAvgToolCalls(logs: ToolCallLog[]): number {
  const byTask = groupBy(logs, l => l.taskId);
  const counts = Object.values(byTask).map(calls => calls.length);
  return counts.reduce((a, b) => a + b, 0) / counts.length;
}
```

## 3. 指标权重

不同阶段，各维度的重要性不同：

**早期开发阶段（0-3 个月）：**

```
正确性 (40%) > 鲁棒性 (25%) > 可解释性 (20%) > 效率 (10%) > 成本 (5%)
```

原因：这个阶段首要目标是让 Agent "能用"——能正确追齐功能，遇到问题不崩溃。此时的 token 消耗不是首要关注点，因为使用量还小。

**稳定运行阶段（3 个月以后）：**

```
成本 (30%) > 效率 (25%) > 正确性 (25%) > 鲁棒性 (15%) > 可解释性 (5%)
```

原因：Agent 已经稳定运行，正确率已经达标。这个阶段需要关注"能不能持续负担"——成本和效率变得最重要。可解释性重要性降低（团队已经熟悉 Agent 的行为模式）。

> [!info] 方法：建立 Baseline
> 在第一次完整跑评测之前，用最简单的实现（甚至手工跑一次）记录各维度的数字。
> 
> Baseline 建立步骤：
> 1. 选 5 个有代表性的测试用例（黄金路径为主）
> 2. 运行 Agent，记录：工具调用次数、token 消耗、完成时间
> 3. 人工评判正确性，记录成功率
> 4. 把这些数字写入 `eval/baseline.json`
> 
> 后续每次优化，都以 Baseline 为对比基准，确保改进方向正确。
>
> ```json
> {
>   "date": "2024-03-01",
>   "version": "v0.1.0",
>   "metrics": {
>     "successRate": 0.6,
>     "avgToolCalls": 35,
>     "avgTokens": 85000,
>     "avgCostCNY": 4.2
>   }
> }
> ```

## 本节小结

- 追齐 Agent 的评测维度：**正确性、效率、鲁棒性、成本、可解释性**
- 每个维度都有可量化的具体指标和测量方法
- 不同阶段指标权重不同：早期看正确性，稳定期看成本
- 建立 Baseline 是量化改进的前提
