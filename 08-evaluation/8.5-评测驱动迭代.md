# 8.5 评测驱动迭代：用数据指导 Agent 优化

> [!abstract] 核心定义
> 评测的目的不是"证明 Agent 好"，而是"发现 Agent 哪里不好"，然后有针对性地改进。
> 
> **评测驱动迭代**是 Agent 工程的核心工作流：用数据定位问题，用实验验证改进，用回归测试防止退步。没有这个循环，Agent 的优化就是在瞎猜。

> [!question] 带着问题阅读
> - 评测-改进的循环应该怎么执行？
> - 常见的 Agent 失败模式背后的原因是什么？
> - 如何防止"改好了这里，破坏了那里"？

## 1. 评测-改进循环

评测驱动迭代的核心是一个闭环：

```
        ┌─────────────────────────────────┐
        ↓                                 │
   运行评测                          再次评测
        ↓                                 │
   分析失败原因                      实施改进
        ↓                                 │
   确定改进方向 ─────────────────────────┘
```

**每个步骤的具体操作：**

### Step 1：运行评测

```bash
# 每周定期跑
npx ts-node eval/runner.ts \
  --agent-version $(git rev-parse --short HEAD) \
  --output-dir eval/results/$(date +%Y-%m-%d)
```

### Step 2：分析失败原因

不要只看通过率，要看**为什么失败**。

```typescript
// 失败分析脚本
interface FailureAnalysis {
  caseId: string;
  failureType: "rule-check" | "llm-judge";
  failedDimension: string;
  rootCause: string;  // 需要人工填写
  suggestedFix: string;
}

function groupFailuresByPattern(results: CaseResult[]): Map<string, CaseResult[]> {
  const failed = results.filter(r => !r.passed);
  
  // 按失败类型分组
  const groups = new Map<string, CaseResult[]>();
  
  for (const result of failed) {
    const pattern = inferFailurePattern(result);
    if (!groups.has(pattern)) groups.set(pattern, []);
    groups.get(pattern)!.push(result);
  }
  
  return groups;
}

function inferFailurePattern(result: CaseResult): string {
  // 根据失败信息推断模式
  if (result.ruleChecks.find(r => r.ruleName === "completed-json" && !r.passed)) {
    return "memory-not-updated";
  }
  if (result.judgeResult.scores.correctness < 3) {
    return "wrong-function-aligned";
  }
  if (result.toolCallCount > 30) {
    return "excessive-tool-calls";
  }
  return "other";
}
```

### Step 3：确定改进方向

失败原因 → 改进方向的映射：

| 失败模式 | 可能根因 | 改进方向 |
|----------|----------|----------|
| 正确性低 | System Prompt 不够具体 | 补充追齐规则，增加反例说明 |
| 正确性低 | 工具调用顺序错误 | 优化 Orchestrator 的步骤规划 |
| 效率低 | 重复读取同一文件 | 给 Research Agent 添加"已读文件"缓存 |
| 效率低 | 读了大量不相关文件 | 优化 list_repo_files 的过滤逻辑 |
| 鲁棒性低 | 非标准路径下工具报错 | 在工具中添加路径发现逻辑 |
| 鲁棒性低 | completed.json 格式错误 | 在 write_memory 工具中添加 schema 验证 |
| 成本高 | 上下文过大 | 启用上下文压缩，按需加载文件片段 |
| 可解释性低 | Analysis Agent 输出不规范 | 在 Analysis Agent Prompt 中要求结构化输出 |

### Step 4：实施改进

**改进时的纪律：**

```
改进原则：
1. 一次只改一个变量（不要同时改 Prompt + 工具 + 架构）
2. 改完立刻重跑受影响的测试用例，不要等周期评测
3. 如果改进导致其他用例退步，先分析原因，不要盲目回滚
4. 把每次改进和对应的评测结果记录下来（见 eval/changelog.md）
```

### Step 5：再次评测（防止回归）

```typescript
// 每次改进后的验证流程
async function verifyImprovement(
  changedComponent: string,  // 改了哪个组件
  improvementDesc: string
): Promise<void> {
  // 1. 跑所有回归用例（防止改坏了其他东西）
  const regressionResults = await runTestCases(regressionTestCases);
  
  // 2. 跑与改动相关的测试用例（验证改进有效）
  const relatedCases = getRelatedTestCases(changedComponent);
  const relatedResults = await runTestCases(relatedCases);
  
  // 3. 对比改进前后
  const before = loadPreviousResults(changedComponent);
  const comparison = compareResults(before, relatedResults);
  
  console.log(`改进效果：
    成功率：${before.successRate} → ${comparison.successRate}
    平均工具调用：${before.avgToolCalls} → ${comparison.avgToolCalls}
    回归测试：${regressionResults.passed}/${regressionResults.total} 通过
  `);
}
```

## 2. 常见失败模式与改进方向

### 失败模式 1：正确性低（最常见）

**现象：** LLM Judge 给正确性打分 < 3，理由是"功能追齐不完整"或"改了错误的文件"

**根因分析：**

```
正确性低的根因树：

追齐不完整
├── Research Agent 没有读到关键文件
│   → 改进：完善 read_repo_file 的路径推断逻辑
│
├── diff_feature 返回的差异不完整
│   → 改进：优化 diff_feature 工具的比对维度
│
└── Analysis Agent 生成的修改方案有遗漏
    → 改进：在 Analysis Agent Prompt 中增加"完整性自检"步骤

追齐错误（改了不该改的）
├── Orchestrator 任务范围界定不清
│   → 改进：在 System Prompt 中明确"本次任务范围"
│
└── Execution Agent 过度优化，改了相关代码
    → 改进：给 Execution Agent 添加"最小修改"约束
```

**改进示例（优化 Analysis Agent Prompt）：**

```typescript
const analysisAgentSystemPrompt = `
你是追齐 Agent 的分析师，负责生成修改方案。

完成分析后，执行自检：
□ 是否覆盖了 diff_feature 报告中的所有差异？
□ 修改是否仅限于目标仓库？
□ 是否有遗漏的边界情况（如数量为 0 时的行为）？

只有自检通过后，才输出最终修改方案。
`;
```

### 失败模式 2：效率低（工具调用冗余）

**现象：** 工具调用次数超过 30 次，日志中有大量重复的 `read_repo_file` 调用

**根因：** Research Agent 没有缓存已读文件，每次需要某个文件时都重新读取

**改进：添加会话级文件缓存：**

```typescript
class ResearchAgent {
  private fileCache = new Map<string, string>();
  
  async readFile(repo: string, path: string): Promise<string> {
    const cacheKey = `${repo}:${path}`;
    
    if (this.fileCache.has(cacheKey)) {
      // 缓存命中，不调用工具
      return this.fileCache.get(cacheKey)!;
    }
    
    const content = await this.tools.read_repo_file({ repo, path });
    this.fileCache.set(cacheKey, content);
    return content;
  }
}
```

### 失败模式 3：鲁棒性低（边界场景崩溃）

**现象：** 边界用例（ec-001~ec-005）通过率很低，Agent 在遇到非标准目录时报错

**根因：** `list_repo_files` 工具假设了标准目录结构，非标准结构下返回空列表，导致后续步骤无法进行

**改进：在工具中添加降级策略：**

```typescript
async function list_repo_files(
  repo: string,
  path: string = "src"
): Promise<string[]> {
  // 先尝试标准路径
  let files = await readDirectory(`${repo}/${path}`);
  
  if (files.length === 0) {
    // 降级：遍历常见的替代路径
    const fallbackPaths = ["pages", "components", "lib", "app"];
    for (const fallback of fallbackPaths) {
      files = await readDirectory(`${repo}/${fallback}`);
      if (files.length > 0) break;
    }
  }
  
  if (files.length === 0) {
    // 最终降级：返回根目录结构，让 Agent 自己判断
    files = await readDirectory(repo);
  }
  
  return files;
}
```

### 失败模式 4：成本高（上下文过大）

**现象：** 每次任务消耗 > 100k tokens，月度成本超预算

**根因：** Research Agent 读取了完整文件，但实际上 Analysis Agent 只需要文件的特定片段

**改进：按需加载文件片段：**

```typescript
// 不读整个文件，只读相关函数/组件
async function read_repo_file_section(
  repo: string,
  path: string,
  targetSymbol: string  // 要读取的函数或组件名
): Promise<string> {
  const fullContent = await readFile(`${repo}/${path}`);
  
  // 用简单的启发式方法提取目标代码块
  const section = extractCodeSection(fullContent, targetSymbol);
  
  return section || fullContent; // 找不到则返回完整内容作为兜底
}
```

## 3. 回归测试集

每次发现一个新的失败模式，把对应的用例加入回归测试集。

**回归测试集管理原则：**

```
加入回归测试集的条件：
1. Agent 在这个场景下失败过（有真实 bug）
2. 这个场景有一定普遍性（不是极端的一次性情况）
3. 有明确的"通过"标准（可以自动判断是否修复）

不加入的情况：
- 只是测试覆盖率的需要（没有真实 bug 记录）
- 场景过于特殊（只可能在该特定仓库出现）
```

**回归用例记录格式：**

```json
{
  "id": "reg-001",
  "description": "回归：修复#42，Agent 误改 ProductCard.tsx",
  "bugReport": {
    "issueId": "#42",
    "date": "2024-03-10",
    "description": "追齐购物车功能时，Agent 意外修改了 ProductCard 中的样式",
    "rootCause": "Analysis Agent 误判了商品展示组件与购物车组件的关联关系"
  },
  "fix": {
    "date": "2024-03-12",
    "component": "Analysis Agent System Prompt",
    "description": "增加了明确的'仅修改与目标功能直接相关的文件'约束"
  },
  "tags": ["regression"],
  "createdAt": "2024-03-12"
}
```

## 4. 追齐 Agent 的迭代节奏

**推荐的迭代节奏：**

```
每周（常规）：
- 跑一次完整评测（所有测试用例）
- 快速 review 失败用例（15 分钟）
- 修复明显的小问题（规则检查失败、prompt 措辞）

每两周（深度）：
- 深度分析失败模式
- 找出本阶段最需要解决的 1-2 个问题
- 设计并实施改进方案
- 验证改进效果，更新 Baseline

每月（架构层）：
- 回顾过去一个月的评测趋势（通过率走势图）
- 评估是否有系统性问题需要架构调整
- 更新评测维度和指标（随着 Agent 成熟，侧重点会变）
- 更新测试用例集（淘汰过时的，加入新场景）
```

> [!tip] 原则：优先级排序
> 修复顺序：**正确性 > 鲁棒性 > 效率 > 成本 > 可解释性**
> 
> 原因：
> - 不正确的 Agent 再快也没有价值，错误追齐比没追齐更糟糕（会引入 bug）
> - 不鲁棒的 Agent 遇到真实场景就崩溃，无法在生产环境使用
> - 效率和成本可以在正确性达标后逐步优化
> - 可解释性影响开发者体验，但不影响功能正确性，最后优化
> 
> 不要因为"效率优化容易做"就先做效率优化——先确保 Agent 做对，再做快、做便宜。

## 本节小结

- 评测驱动迭代：评测 → 分析 → 改进 → 再评测，是闭环
- 每次改进只改一个变量，改完立刻验证，防止回归
- 四类常见失败模式：正确性低、效率低、鲁棒性低、成本高，各有不同的改进方向
- 每发现新 bug，立即加入回归测试集
- 优先级：正确性 > 鲁棒性 > 效率 > 成本 > 可解释性
