# 8.3 测试用例设计：为 Agent 构建可靠的测试集

> [!abstract] 核心定义
> 好的测试集是评测体系的基础。对 Agent 来说，测试用例不只是"输入"，还包括"预期的行为模式"和"评判标准"。
> 
> 与传统测试用例不同，Agent 测试用例不定义精确的期望输出，而是定义**行为约束**：Agent 应该做哪些事、不应该做哪些事、最终结果应该满足什么质量标准。

> [!question] 带着问题阅读
> - Agent 的测试用例和传统测试用例有什么本质区别？
> - 如何设计能真正发现问题的测试用例？
> - 测试用例如何维护，避免"测试腐化"？

## 1. 测试用例的结构

追齐 Agent 的测试用例包含四个核心字段：

```typescript
interface AgentTestCase {
  id: string;                     // 唯一标识，便于追踪
  description: string;            // 人类可读的描述
  
  // 输入：给 Agent 的任务
  input: {
    feature: string;              // 目标功能描述
    targetRepo: string;           // 要修改的 H5 仓库（fixture 名）
    referenceContext?: string;    // 额外参考信息（可选）
  };
  
  // 预期行为：不是精确输出，是行为模式
  expectedBehavior: {
    shouldCallTools: string[];    // 必须调用的工具（顺序不限）
    shouldNotCallTools?: string[]; // 不应该调用的工具
    shouldModifyFiles: string[];  // 应该修改的文件（路径模式）
    shouldNotModifyFiles?: string[]; // 不应该修改的文件
    memoryUpdated: boolean;       // 是否需要更新 MEMORY.md
    completedJsonUpdated: boolean; // 是否需要更新 completed.json
  };
  
  // 评判标准：质量维度的判断标准
  evaluationCriteria: {
    correctness: string;          // 如何判断功能追齐正确
    maxToolCalls?: number;        // 工具调用上限
    maxTokens?: number;           // token 消耗上限
    requiresExplanation: boolean; // 是否需要修改说明
  };
  
  tags: TestCaseTag[];            // 分类标签
}

type TestCaseTag = 
  | "happy-path"    // 黄金路径
  | "edge-case"     // 边界用例
  | "adversarial"   // 对抗用例
  | "regression";   // 回归用例
```

### 结构设计的关键原则

**1. 约束行为，不约束输出**

```typescript
// ❌ 错误：约束精确输出
expectedOutput: `
  --- a/src/CartItem.tsx
  +++ b/src/CartItem.tsx
  @@ -12,6 +12,8 @@ export function CartItem...
`  // ← 这样写，Agent 每次输出稍有不同就判失败

// ✅ 正确：约束行为和质量
expectedBehavior: {
  shouldModifyFiles: ["src/CartItem.tsx", "src/hooks/useCart.ts"],
  shouldNotModifyFiles: ["src/api/products.ts"]  // 不相关文件不该动
}
evaluationCriteria: {
  correctness: "H5 商品数量增减后应调用 PUT /api/cart/:id，与 App 行为一致"
}
```

**2. 评判标准要具体可操作**

```typescript
// ❌ 模糊的评判标准
correctness: "代码要好" // ← LLM Judge 无法据此评判

// ✅ 具体的评判标准
correctness: `
  验证以下条件：
  1. CartItem 组件有 onQuantityChange 或等价的数量变更处理
  2. 数量变更时调用了后端 API（直接或通过 hook）
  3. 数量减为 0 时，商品被移除（与 iOS App 的 CartItemView.swift 行为一致）
  4. 没有引入新的状态管理库（保持与仓库一致的技术栈）
`
```

## 2. 测试类别设计

### 黄金路径（Happy Path）

标准仓库结构 + 清晰需求，期望 Agent 成功完成。

这类用例用于验证基本功能，建立 Baseline。

```typescript
const happyPathCase: AgentTestCase = {
  id: "hp-001",
  description: "标准仓库：追齐购物车数量增减功能",
  input: {
    feature: "cart-quantity-control",
    targetRepo: "fixture-standard-shop"  // 标准目录结构的 fixture
  },
  expectedBehavior: {
    shouldCallTools: ["read_repo_file", "diff_feature", "write_memory"],
    shouldModifyFiles: ["src/components/CartItem.tsx"],
    memoryUpdated: true,
    completedJsonUpdated: true
  },
  evaluationCriteria: {
    correctness: "CartItem 支持数量增减，数量为 0 时自动移除",
    maxToolCalls: 20,
    maxTokens: 60000,
    requiresExplanation: true
  },
  tags: ["happy-path"]
};
```

### 边界用例（Edge Cases）

仓库结构不规范、功能已部分实现、依赖组件缺失等情况。

```typescript
const edgeCases: AgentTestCase[] = [
  {
    id: "ec-001",
    description: "非标准目录：组件在 pages/ 而非 components/",
    input: {
      feature: "cart-quantity-control",
      targetRepo: "fixture-nonstandard-shop"  // 组件放在非标准路径
    },
    expectedBehavior: {
      shouldCallTools: ["list_repo_files", "read_repo_file"],  // 应该先探索目录
      shouldModifyFiles: ["pages/Cart/CartItem.tsx"],  // 找到实际路径
      memoryUpdated: true,
      completedJsonUpdated: true
    },
    evaluationCriteria: {
      correctness: "Agent 能找到非标准路径下的组件并正确修改",
      maxToolCalls: 30,  // 允许更多调用来探索目录
      requiresExplanation: true
    },
    tags: ["edge-case"]
  },
  {
    id: "ec-002", 
    description: "功能已部分实现：数量增加已有，减少缺失",
    input: {
      feature: "cart-quantity-control",
      targetRepo: "fixture-partial-impl",  // 只实现了增加，没有减少
      referenceContext: "增加功能已实现，只需追齐减少和清零功能"
    },
    expectedBehavior: {
      shouldCallTools: ["read_repo_file", "diff_feature"],
      shouldModifyFiles: ["src/components/CartItem.tsx"],
      shouldNotModifyFiles: ["src/api/cart.ts"],  // API 层已正确，不应该动
      memoryUpdated: true,
      completedJsonUpdated: true
    },
    evaluationCriteria: {
      correctness: "只追齐缺失的减少功能，不破坏已有的增加功能",
      maxToolCalls: 25,
      requiresExplanation: true
    },
    tags: ["edge-case"]
  }
];
```

### 对抗用例（Adversarial）

故意制造困难，测试 Agent 的极限。

```typescript
const adversarialCase: AgentTestCase = {
  id: "adv-001",
  description: "模糊需求：feature spec 描述不完整",
  input: {
    feature: "shopping-improvement",  // 故意模糊的功能名
    targetRepo: "fixture-standard-shop"
  },
  expectedBehavior: {
    shouldCallTools: ["resolve_feature_spec"],  // 必须先查询 spec
    memoryUpdated: false,  // 如果需求不清晰，不应该盲目写 memory
    completedJsonUpdated: false
  },
  evaluationCriteria: {
    correctness: "Agent 应该拒绝执行或请求澄清，而不是猜测并修改代码",
    requiresExplanation: true
  },
  tags: ["adversarial"]
};
```

### 回归用例（Regression）

历史上出现过 bug 的场景，用于防止问题复现。

```typescript
const regressionCase: AgentTestCase = {
  id: "reg-001",
  description: "回归：修复#42，Agent 曾经修改了不相关的 ProductCard",
  input: {
    feature: "cart-quantity-control",
    targetRepo: "fixture-standard-shop"
  },
  expectedBehavior: {
    shouldModifyFiles: ["src/components/CartItem.tsx"],
    shouldNotModifyFiles: [
      "src/components/ProductCard.tsx",  // #42 issue：曾经误改这个
      "src/pages/ProductList.tsx"
    ],
    memoryUpdated: true,
    completedJsonUpdated: true
  },
  evaluationCriteria: {
    correctness: "正确追齐购物车功能，且不触碰商品列表相关文件",
    maxToolCalls: 20,
    requiresExplanation: true
  },
  tags: ["regression"]
};
```

## 3. Fixture 仓库设计

不用真实仓库跑测试——真实仓库太大、太慢，而且会因为代码更新导致测试不稳定。

**Fixture 仓库原则：**
- 只包含测试用例所需的最小文件集
- 目录结构固定，不随真实仓库更新
- 版本管理：fixture 改动需要同步更新相关测试用例

**追齐 Agent 的三套 Fixtures：**

```
eval/fixtures/
├── fixture-standard-shop/          # 标准结构仓库
│   ├── src/
│   │   ├── components/
│   │   │   ├── CartItem.tsx        # 购物车项（未追齐版本）
│   │   │   └── ProductCard.tsx     # 商品卡片（正确实现，不应被改动）
│   │   ├── hooks/
│   │   │   └── useCart.ts          # 购物车 hook（未追齐版本）
│   │   └── api/
│   │       └── cart.ts             # API 层（已正确，不应被改动）
│   └── package.json
│
├── fixture-nonstandard-shop/       # 非标准结构仓库
│   ├── pages/
│   │   └── Cart/
│   │       └── CartItem.tsx        # 组件在 pages/ 下（非标准）
│   └── src/
│       └── utils/
│           └── cart.js             # 用 JS 而不是 TS
│
└── fixture-minimal-shop/           # 最小仓库（只有入口）
    ├── index.tsx
    └── package.json                # 组件还没创建，需要新建
```

**Fixture 文件示例（`CartItem.tsx` 未追齐版本）：**

```typescript
// fixture-standard-shop/src/components/CartItem.tsx
// 这是"未追齐"版本：数量增减只更新本地状态，没有调用 API

import React, { useState } from 'react';

interface CartItemProps {
  id: string;
  name: string;
  price: number;
  initialQuantity: number;
}

export function CartItem({ id, name, price, initialQuantity }: CartItemProps) {
  const [quantity, setQuantity] = useState(initialQuantity);

  // ❌ 只更新本地状态，App 版本会调用 API
  const handleIncrease = () => setQuantity(q => q + 1);
  const handleDecrease = () => setQuantity(q => Math.max(0, q - 1));

  return (
    <div className="cart-item">
      <span>{name}</span>
      <button onClick={handleDecrease}>-</button>
      <span>{quantity}</span>
      <button onClick={handleIncrease}>+</button>
      <span>¥{price * quantity}</span>
    </div>
  );
}
```

## 4. 测试用例的维护

测试用例会过时。当仓库结构变化、功能需求更新、评测标准调整时，测试用例需要同步更新。

**维护策略：**

```
eval/
├── test-cases/
│   ├── happy-path/
│   │   ├── hp-001.json
│   │   └── hp-002.json
│   ├── edge-cases/
│   │   ├── ec-001.json
│   │   └── ec-002.json
│   ├── adversarial/
│   │   └── adv-001.json
│   └── regression/
│       └── reg-001.json
├── fixtures/            # fixture 仓库
├── baseline.json        # 基准测试结果
└── CHANGELOG.md         # 测试用例变更记录
```

**定期 Review 机制：**
- 每月 review 一次测试用例是否仍然有效
- 当 Agent 代码有重大改动时，检查相关测试用例
- 每次发现新 bug，立即添加对应的回归用例

## 本节小结

- Agent 测试用例由四部分构成：**input、expectedBehavior、evaluationCriteria、tags**
- 四类测试用例：黄金路径、边界用例、对抗用例、回归用例
- 用精简的 Fixture 仓库替代真实仓库，保证测试的稳定性和速度
- 测试用例需要定期维护，每次发现新 bug 立即添加回归用例
