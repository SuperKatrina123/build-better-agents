# 8.4 LLM-as-Judge 与自动化评测流水线

> [!abstract] 核心定义
> Agent 的输出不能用 assert 检验，但可以用另一个 LLM 来"评判"——这就是 **LLM-as-Judge** 模式。
> 
> 评判 LLM 扮演"质检员"角色：它接收 Agent 的输入、输出和评判标准，按维度逐条打分，并给出理由。配合自动化的规则检查，构成完整的评测流水线。

> [!question] 带着问题阅读
> - LLM-as-Judge 的 Prompt 怎么设计才有效？
> - 哪些事情应该用规则检查，哪些交给 LLM 评判？
> - 自动化评测流水线的各个阶段分别做什么？

## 1. LLM-as-Judge 原理

让一个专门的 LLM（**评判模型**）来评价另一个 LLM（被测 Agent）的输出。

**为什么 LLM 能胜任这个角色？**

LLM 具备理解代码语义、比较功能行为差异、评估代码风格合规性的能力。这正是人类工程师在 Code Review 时做的事，而 LLM 可以以人类速度的数百倍完成同样的工作。

**评判模型的选择：**

```
被测 Agent：claude-3-5-sonnet（执行追齐任务，追求能力）
评判模型：claude-3-opus（评判追齐质量，追求准确性）
```

使用不同的模型做评判，可以减少"自我评价偏差"——避免被测模型既做运动员又做裁判员。

**评判的输入和输出：**

```typescript
interface JudgeInput {
  testCase: AgentTestCase;       // 测试用例（包含评判标准）
  agentOutput: AgentRunOutput;   // Agent 的完整输出
}

interface AgentRunOutput {
  toolCallLog: ToolCall[];       // 所有工具调用记录
  generatedDiff: string;         // 最终生成的代码 diff
  explanation: string;           // Agent 给出的修改说明
  memoryUpdate?: string;         // 对 MEMORY.md 的更新内容
  completedJsonUpdate?: object;  // 对 completed.json 的更新
}

interface JudgeOutput {
  scores: {
    correctness: number;        // 1-5 分
    explainability: number;     // 1-5 分
  };
  passed: boolean;              // 综合是否通过
  reasoning: string;            // 评判理由
  issues: string[];             // 发现的具体问题
}
```

## 2. 评判 Prompt 设计

评判 Prompt 是 LLM-as-Judge 的核心，设计要点：
- 提供**明确的评判标准**（不能模糊）
- 要求**逐条打分**（1-5 分制）
- 要求**引用具体代码证据**
- 要求**给出不通过的原因**

**追齐 Agent 的评判 Prompt：**

```
你是一个代码评审专家，专注于评估"H5 功能追齐"任务的完成质量。

## 任务背景
功能追齐是指：将 H5 页面的某个功能更新到与 App 最新版本一致。

## 本次评判任务

### 测试用例
- 目标功能：{feature_description}
- 评判标准：
  {correctness_criteria}

### Agent 生成的 diff
```diff
{generated_diff}
```

### Agent 的修改说明
{explanation}

## 评判维度

### 维度 1：正确性（1-5 分）
评判标准：
- 5分：功能完全追齐，行为与 App 完全一致，无多余修改
- 4分：功能基本追齐，有小瑕疵但不影响功能
- 3分：功能部分追齐，核心逻辑正确但边界情况缺失
- 2分：功能追齐方向正确但实现有明显错误
- 1分：功能未正确追齐，或追齐了错误的功能

请引用 diff 中的具体代码行来支撑你的评分。

### 维度 2：可解释性（1-5 分）
评判标准：
- 5分：修改说明清晰，能让开发者独立验证每处改动
- 4分：说明基本清晰，偶有遗漏但不影响理解
- 3分：说明存在，但不够详细，需要开发者自己分析
- 2分：说明过于简单，无法据此验证改动合理性
- 1分：没有修改说明，或说明与实际改动不符

## 输出格式
请严格按以下 JSON 格式输出：
{
  "correctness": {
    "score": <1-5>,
    "evidence": "<引用具体代码行>",
    "reasoning": "<评判理由>"
  },
  "explainability": {
    "score": <1-5>,
    "reasoning": "<评判理由>"
  },
  "passed": <true/false>,  // correctness >= 4 且 explainability >= 3
  "issues": ["<具体问题1>", "<具体问题2>"],
  "summary": "<一句话总结>"
}
```

**调用评判模型的代码：**

```typescript
async function judgeAgentOutput(
  input: JudgeInput,
  judgeModel: string = "claude-3-opus-20240229"
): Promise<JudgeOutput> {
  const prompt = buildJudgePrompt(input);
  
  const response = await anthropic.messages.create({
    model: judgeModel,
    max_tokens: 2000,
    messages: [{ role: "user", content: prompt }]
  });
  
  const rawOutput = response.content[0].text;
  // 提取 JSON 部分（LLM 可能在 JSON 前后有额外文本）
  const jsonMatch = rawOutput.match(/\{[\s\S]*\}/);
  if (!jsonMatch) throw new Error("Judge output is not valid JSON");
  
  return JSON.parse(jsonMatch[0]) as JudgeOutput;
}
```

## 3. 规则检查

不需要 LLM 评判的事情，用规则检查——快、准、便宜。

**规则检查清单：**

```typescript
interface RuleCheckResult {
  ruleName: string;
  passed: boolean;
  detail: string;
}

async function runRuleChecks(output: AgentRunOutput): Promise<RuleCheckResult[]> {
  return [
    // 规则 1：diff 格式合法
    checkDiffFormat(output.generatedDiff),
    
    // 规则 2：修改只在允许的仓库里
    checkModifiedFilesScope(output.generatedDiff, allowedRepos),
    
    // 规则 3：completed.json 格式正确
    checkCompletedJsonSchema(output.completedJsonUpdate),
    
    // 规则 4：工具调用次数在预算内
    checkToolCallBudget(output.toolCallLog, maxToolCalls = 30),
    
    // 规则 5：没有直接修改 node_modules
    checkNoNodeModulesModification(output.generatedDiff),
    
    // 规则 6：TypeScript 文件没有明显语法错误（简单检查）
    checkBasicTsSyntax(output.generatedDiff)
  ];
}

function checkDiffFormat(diff: string): RuleCheckResult {
  // unified diff 格式检查
  const validDiff = /^---\s.+\n\+\+\+\s.+\n@@/m.test(diff);
  return {
    ruleName: "diff-format",
    passed: validDiff || diff.trim() === "",  // 空 diff 也算通过（没有改动）
    detail: validDiff ? "OK" : "Diff 格式不合法，无法应用"
  };
}

function checkToolCallBudget(
  log: ToolCall[],
  maxCalls: number
): RuleCheckResult {
  const count = log.length;
  return {
    ruleName: "tool-call-budget",
    passed: count <= maxCalls,
    detail: `工具调用次数：${count}/${maxCalls}${count > maxCalls ? " ⚠️ 超出预算" : ""}`
  };
}

function checkCompletedJsonSchema(update: object | undefined): RuleCheckResult {
  if (!update) {
    return { ruleName: "completed-json", passed: true, detail: "本次任务不需要更新" };
  }
  
  // 验证 schema：必须有 featureId、completedAt、repoId 字段
  const required = ["featureId", "completedAt", "repoId"];
  const missing = required.filter(f => !(f in update));
  
  return {
    ruleName: "completed-json",
    passed: missing.length === 0,
    detail: missing.length === 0 
      ? "Schema 验证通过" 
      : `缺少必填字段：${missing.join(", ")}`
  };
}
```

## 4. 自动化流水线

![[eval-pipeline.excalidraw]]

完整的自动化评测流水线：

```typescript
interface EvalRunConfig {
  testCases: AgentTestCase[];
  agentVersion: string;
  outputDir: string;
}

interface EvalRunResult {
  summary: EvalSummary;
  caseResults: CaseResult[];
  generatedAt: string;
}

async function runEvalPipeline(config: EvalRunConfig): Promise<EvalRunResult> {
  const caseResults: CaseResult[] = [];
  
  for (const testCase of config.testCases) {
    console.log(`Running test case: ${testCase.id}`);
    
    // Step 1: 准备 Fixture 环境
    const fixture = await setupFixture(testCase.input.targetRepo);
    
    // Step 2: 运行 Agent
    const agentOutput = await runAgent({
      feature: testCase.input.feature,
      repoPath: fixture.path,
      captureToolCalls: true  // 开启工具调用日志
    });
    
    // Step 3: 规则检查（快速）
    const ruleResults = await runRuleChecks(agentOutput);
    
    // Step 4: LLM 评判（慢，但必要）
    const judgeResult = await judgeAgentOutput({
      testCase,
      agentOutput
    });
    
    // Step 5: 汇总本条用例结果
    caseResults.push({
      testCaseId: testCase.id,
      ruleChecks: ruleResults,
      judgeResult,
      passed: ruleResults.every(r => r.passed) && judgeResult.passed,
      toolCallCount: agentOutput.toolCallLog.length,
      tokenUsage: agentOutput.tokenUsage
    });
    
    // 清理 Fixture 环境
    await cleanupFixture(fixture);
  }
  
  // Step 6: 生成评测报告
  const summary = generateSummary(caseResults);
  const report: EvalRunResult = {
    summary,
    caseResults,
    generatedAt: new Date().toISOString()
  };
  
  // 写入报告文件
  await writeReport(config.outputDir, config.agentVersion, report);
  
  return report;
}
```

**流水线的并发优化：**

```typescript
// 多个测试用例可以并发跑，大幅缩短总耗时
async function runEvalPipelineConcurrent(
  config: EvalRunConfig,
  concurrency: number = 3
): Promise<EvalRunResult> {
  const chunks = chunkArray(config.testCases, concurrency);
  const allResults: CaseResult[] = [];
  
  for (const chunk of chunks) {
    const chunkResults = await Promise.all(
      chunk.map(tc => runSingleTestCase(tc))
    );
    allResults.push(...chunkResults);
  }
  
  return {
    summary: generateSummary(allResults),
    caseResults: allResults,
    generatedAt: new Date().toISOString()
  };
}
```

## 5. 评测结果的使用

评测不是为了追求高分，而是发现问题。

**评测结果的正确使用方式：**

```
每次评测后的必做事项：

1. 看总体通过率（是否达到本阶段目标？）
2. 分析失败用例（失败的原因是什么？）
3. 找规律（多个失败用例有什么共同点？）
4. 确定改进优先级（影响最大的问题先修）
5. 写改进方案（具体要改哪里）
6. 执行改进后重新评测（验证改进是否有效）
```

**失败分析示例：**

```
评测结果分析 - 2024-03-15

总体通过率：12/20 = 60%（目标 80%，未达标）

失败用例分布：
- 规则检查失败：3 个（completed.json 格式错误 2 个，diff 格式错误 1 个）
- LLM Judge 失败：5 个（正确性 < 4 分：3 个，可解释性 < 3 分：2 个）

失败规律：
- completed.json 格式错误都在边界用例（ec-001, ec-002, ec-003）
  → 可能是 Orchestrator 在非标准路径场景下没有触发 write_memory 工具
  → 改进：在 System Prompt 中强调"无论仓库结构如何，都必须更新 completed.json"

- 可解释性低的 2 个用例都是对抗场景（adv-001, adv-002）
  → Agent 面对模糊需求时，倾向于"做了再说"，解释很简略
  → 改进：在 Analysis Agent 中增加"必须说明追齐依据"的要求
```

> [!warning] 误区：过度信任 LLM 评判
> LLM-as-Judge 本身也会出错。以下情况需要人工复核：
> - 评判模型给了高分但你直觉上觉得有问题
> - 评判理由引用的代码行不存在或不相关
> - 正确性分数与规则检查结果矛盾（规则检查说失败，但 LLM 说通过）
> 
> LLM 评判适合**规模化筛选**（快速找出明显失败的用例），不适合做**最终决策**（关键功能的上线判断应该有人工参与）。

## 本节小结

- LLM-as-Judge：用评判模型评估被测 Agent 的输出质量，按维度打分
- 评判 Prompt 要提供具体标准、要求逐条打分、要求引用代码证据
- 规则检查处理可程序化的约束（格式、路径、schema、预算）
- 自动化流水线：fixture 准备 → Agent 执行 → 规则检查 → LLM 评判 → 报告生成
- LLM 评判适合规模化筛选，关键决策仍需人工
