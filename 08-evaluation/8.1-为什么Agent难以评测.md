# 8.1 为什么 Agent 难以评测：概率输出 + 过程复杂性

> [!abstract] 核心定义
> 传统软件测试有确定的预期输出；Agent 的输出是**概率性的**，同样的输入可能产生不同但都正确的输出。这让传统测试方法失效，需要新的评测框架。
> 
> 对功能追齐 Agent 来说：同一个功能追齐任务，Agent 可能生成风格不同但逻辑等价的代码，两份代码都是"正确答案"，但传统 assert 只认一个。

> [!question] 带着问题阅读
> - 为什么不能用单元测试的思路测 Agent？
> - Agent 评测的核心困难在哪里？
> - 什么样的评测框架适合概率性输出？

## 1. 传统测试 vs Agent 测试

软件工程里，测试的基本逻辑是：给定输入，断言输出。

**传统代码测试的模型：**

```typescript
// 确定性函数：相同输入永远产生相同输出
function add(a: number, b: number): number {
  return a + b;
}

// 测试：直接 assert
test("add two numbers", () => {
  expect(add(2, 3)).toBe(5); // 永远成立
});
```

这个模型之所以奏效，是因为函数是**纯粹的、确定性的**——给定 `(2, 3)`，永远返回 `5`。

**Agent 测试面临的困境：**

```typescript
// Agent 任务：追齐购物车功能
const result = await alignmentAgent.run({
  feature: "商品数量增减按钮",
  targetRepo: "h5-shop",
  referenceApp: "app-ios"
});

// 如何 assert？
expect(result.diff).toBe(???) // ← 不可能写出确定的期望值
```

Agent 生成的 diff 每次都可能不同：
- 第一次：修改 `CartItem.tsx`，添加 `onQuantityChange` prop
- 第二次：修改 `useCart.ts`，在 hook 里处理数量变化
- 两种实现都是正确的，但 diff 文本完全不同

传统的字符串 assert 完全失效。

### 不确定性的来源

Agent 的不确定性来自多个层次：

| 层次 | 来源 | 影响 |
|------|------|------|
| LLM 采样 | temperature > 0 时每次输出不同 | 相同 prompt 产生不同代码 |
| 工具调用顺序 | Agent 自主决定调用哪些工具 | 探索路径不同，但结果等价 |
| 上下文差异 | 记忆状态、已完成功能列表 | 两次运行的上下文不同 |
| 环境噪声 | 仓库文件的微小差异 | 触发不同的代码路径 |

## 2. Agent 评测的三个挑战

### 挑战一：输出不确定性

相同的任务，Agent 两次运行的结果不同，但可能都正确。

```
任务：追齐"商品收藏"功能

第一次运行结果：
- 修改了 ProductCard.tsx
- 新增了 useFavorite.ts hook
- 修改了 api/favorites.ts

第二次运行结果：
- 修改了 ProductCard.tsx
- 修改了 store/favoriteSlice.ts（使用 Redux 而非自定义 hook）
- 修改了 api/favorites.ts
```

两次结果：文件列表不同，代码风格不同，但功能都被正确追齐了。

> [!warning] 不确定性 ≠ 错误
> 不确定性本身不是问题，问题是我们的**评测方法**必须能处理这种不确定性。如果评测方法只认"唯一正确答案"，就会把正确的 Agent 判为失败。

### 挑战二：过程不透明

Agent 做了 20 次工具调用，哪些是必要的？哪些是冗余的？

```
工具调用序列（追齐购物车功能）：
1. list_repo_files("h5-shop")
2. read_repo_file("h5-shop/src/components/CartItem.tsx")
3. read_repo_file("h5-shop/src/hooks/useCart.ts")
4. read_repo_file("h5-shop/src/components/CartItem.tsx")  ← 重复读取？
5. diff_feature("cart-quantity", "h5-shop")
6. read_repo_file("h5-shop/src/api/cart.ts")
7. resolve_feature_spec("cart-quantity")
8. read_repo_file("app-ios/CartFeature.swift")  ← 为什么读 iOS 文件？
...
```

调用 4 是冗余的吗？还是因为第一次读取后 Agent 发现需要重新确认？调用 8 是多余的吗？还是 Agent 在对照原始行为？这些问题很难从外部观察确定。

### 挑战三：评判主观性

"这次功能追齐做得好不好"本身就是一个主观判断。

即使两个工程师看同一份 diff，可能给出不同评价：
- 工程师 A：代码简洁，追齐准确，+1
- 工程师 B：没有添加测试，缺少注释，-1

追齐 Agent 面临的主观判断包括：
- 代码风格是否与仓库一致？（风格有争议）
- 功能是否"完全"追齐？（完全的标准因人而异）
- 是否引入了合理的抽象？（抽象粒度因人而异）

## 3. 需要评测的是什么

明确了挑战之后，我们需要重新定义"评测什么"。

**不是：** "输出是否和预期完全一致"

**而是：** "输出是否满足质量标准"

这是一个关键的认知转变。

**功能追齐 Agent 的质量标准：**

```
质量维度：
1. 功能正确性：H5 行为是否与 App 一致？
   评判方式：人工验证 or LLM 评判
   
2. 代码合规性：风格是否与仓库一致？有无 lint error？
   评判方式：静态分析工具（eslint、tsc）
   
3. 最小修改原则：是否引入了不必要的改动？
   评判方式：diff 行数统计 + LLM 评判
   
4. 记忆一致性：completed.json 是否正确更新？
   评判方式：JSON schema 验证
```

这些标准有的可以自动检查（规则），有的需要 LLM 或人工评判（语义层面）。

## 4. 评测的两种形式

### 离线评测（本章重点）

在受控环境中，用预先设计的测试用例批量跑 Agent，然后集中评判。

```
离线评测流程：
测试用例集 → 批量运行 Agent → 采集所有输出 → 统一评判 → 生成报告
```

**优点：** 可重复、可对比、成本可控  
**缺点：** 测试用例可能与真实场景有差距

### 在线评测（生产环境监控）

Agent 在真实生产环境中运行，同时采集指标，发现问题后触发告警。

```
在线评测流程：
生产任务 → Agent 运行 → 自动规则检查 → 异常告警 → 人工 review
```

**优点：** 反映真实使用情况  
**缺点：** 问题发现滞后，无法在上线前拦截

> [!tip] 原则
> 不要追求"完全覆盖"，追求"能发现关键问题"。
> 
> Agent 评测是**风险控制**，不是**完备性证明**。10 个精心设计的测试用例，比 100 个随机生成的用例更有价值。评测的目标是在 Agent 进入生产前，发现最容易出错的场景。

## 本节小结

- Agent 评测的核心困难：**概率性输出 + 过程不透明 + 评判主观性**
- 传统 assert 失效，需要"质量标准评判"替代"期望值对比"
- 评测分为离线（测试集批量跑）和在线（生产环境监控）两种，本章聚焦离线评测
- 评测是风险控制，不是完备性证明
