# 第八章：评测体系——用数据让 Agent 变得更好

## 章节目标

学完本章，你应该能够：

1. **理解** 为什么传统软件测试方法不适用于 Agent，以及 Agent 评测的核心挑战是什么
2. **设计** 一套完整的评测维度体系，明确用什么标准衡量追齐 Agent 的好坏
3. **构建** 覆盖黄金路径、边界用例、对抗用例和回归用例的测试集
4. **实施** 基于 LLM-as-Judge + 规则检查的自动化评测流水线，并用评测数据驱动持续优化

---

## 通用原理

评测体系解决的不是“怎么断言某个输出字符串”，而是如何衡量一个具有概率性输出、过程复杂、结果可能不唯一的 Agent 系统。所有 Agent 都会遇到：

- 结果不完全确定，但仍然需要质量判断
- 过程很长，不能只看最终答案
- 规则检查和语义评判需要同时存在

因此，本章的重点是评测维度、用例设计、自动化判分和迭代闭环，而不是某一种输出格式。

## 场景映射

| Agent 类型 | 评测对象 | 典型成功信号 | 常见评测难点 |
|-----------|---------|-------------|-------------|
| Coding Agent | 代码修改、工具路径、解释质量 | 功能正确、改动合理、工具预算达标 | diff 不唯一、实现风格不同 |
| Research Agent | 结论质量、来源可信度、覆盖率 | 结论准确、引用充分、摘要清晰 | “看起来合理”但证据不足 |
| Support Agent | 回复质量、解决率、升级判断 | 答复准确、语气合规、转人工恰当 | 用户意图多样、满意度主观 |
| 流程 / 业务执行 Agent | 节点推进质量、状态正确性 | 规则执行正确、状态一致、异常可追踪 | 多系统交互、失败分支复杂 |

## 主案例展开：功能追齐 Agent

### 五个评测维度

| 维度 | 核心问题 | 关键指标 |
|------|----------|----------|
| 正确性 | 功能是否真的被追齐？ | 成功率 ≥ 85% |
| 效率 | 资源消耗是否合理？ | ≤ 20 次工具调用 |
| 鲁棒性 | 非标准场景能否处理？ | 边界用例 ≥ 70% |
| 成本 | API 费用可持续吗？ | ≤ ¥2/功能 |
| 可解释性 | 修改理由开发者能理解吗？ | Judge 评分 ≥ 4/5 |

### 测试用例结构

```typescript
{
  id: string,               // 唯一 ID
  input: {
    feature: string,        // 功能名称
    targetRepo: string      // Fixture 仓库名
  },
  expectedBehavior: {
    shouldCallTools: [],    // 必须调用的工具
    shouldModifyFiles: [],  // 应该修改的文件
    memoryUpdated: boolean  // 是否更新记忆
  },
  evaluationCriteria: {
    correctness: string,    // 具体的正确性判断标准
    maxToolCalls: number    // 工具调用上限
  },
  tags: []                  // happy-path / edge-case / adversarial / regression
}
```

### 自动化评测流水线步骤

```
1. 准备 Fixture 仓库（隔离测试环境）
2. 运行 Agent（开启工具调用日志捕获）
3. 规则检查（diff 格式 / 文件范围 / JSON schema / 工具预算）
4. LLM-as-Judge 评判（correctness + explainability）
5. 汇总结果（通过率 / 各维度得分 / 失败原因）
6. 生成报告（趋势对比 + 改进建议）
```

这些维度、Fixture 和 LLM-as-Judge 流水线，是主案例的完整评测实现。迁移到别的 Agent 时，真正稳定的是：

- 先定义质量维度，再设计用例
- 规则检查负责硬边界，LLM 评判负责语义层
- 评测的目标是控制风险和指导迭代，而不是追求唯一标准答案
