# 8.6 设计决策 8：功能追齐 Agent 完整评测方案

本节整合第八章所有内容，给出功能追齐 Agent 的完整评测方案——从维度设计到测试用例，从自动化流水线到迭代节奏，形成可以直接参考的决策文档。

## 1. 评测体系全景

五个评测维度、对应指标和测量方法的完整对照表：

| 维度 | 核心指标 | 测量方法 | 早期目标 | 稳定期目标 |
|------|----------|----------|----------|------------|
| **正确性** | 功能追齐成功率 | LLM Judge（correctness ≥ 4分）+ 人工抽查 | ≥ 70% | ≥ 85% |
| | 错误追齐率 | diff 分析（修改了不该修改的文件） | ≤ 10% | ≤ 5% |
| **效率** | 平均工具调用次数/功能 | 工具调用日志统计 | ≤ 35 次 | ≤ 20 次 |
| | 平均 token 消耗/功能 | Anthropic API usage 统计 | ≤ 80k | ≤ 50k |
| | 平均完成时间 | 任务开始到结束的时间戳差值 | ≤ 5 min | ≤ 3 min |
| **鲁棒性** | 非标准仓库成功率 | 边界用例测试集通过率 | ≥ 55% | ≥ 70% |
| | 降级成功率 | 对抗用例测试集通过率 | ≥ 40% | ≥ 60% |
| **成本** | 平均 API 费用/功能 | 成本统计（token × 单价） | ≤ ¥5 | ≤ ¥2 |
| | 月度成本（100功能） | 压力测试推算 | ≤ ¥500 | ≤ ¥200 |
| **可解释性** | 修改理由完整度 | LLM Judge（explainability ≥ 3分） | ≥ 70% | ≥ 80% |
| | 开发者理解率 | 季度问卷调查（样本 n ≥ 10） | ≥ 60% | ≥ 80% |

**综合通过标准（一次评测"通过"的定义）：**

```typescript
function isEvalPassed(summary: EvalSummary): boolean {
  // 正确性是硬门槛，其他维度是软目标
  const correctnessPassed = summary.successRate >= currentTarget.correctness;
  
  // 其他维度至少有 3 个达标
  const otherMetrics = [
    summary.avgToolCalls <= currentTarget.toolCalls,
    summary.robustnessRate >= currentTarget.robustness,
    summary.avgCost <= currentTarget.cost,
    summary.explainabilityRate >= currentTarget.explainability
  ];
  const otherPassed = otherMetrics.filter(Boolean).length >= 3;
  
  return correctnessPassed && otherPassed;
}
```

## 2. 最小可行评测集

从零开始建立评测体系时，建议先建立 **15 个测试用例**：

```
最小可行评测集（15 个）：

黄金路径（5 个）：
  hp-001：标准仓库 × 购物车数量增减（最核心功能）
  hp-002：标准仓库 × 商品收藏功能
  hp-003：标准仓库 × 搜索历史功能
  hp-004：标准仓库 × 用户地址管理
  hp-005：标准仓库 × 订单状态显示

边界用例（5 个）：
  ec-001：非标准目录结构 × 购物车功能
  ec-002：功能已部分实现 × 购物车功能（只缺一半）
  ec-003：依赖组件缺失 × 地址功能（需要先创建 AddressPicker 组件）
  ec-004：TypeScript 严格模式仓库（类型要求更高）
  ec-005：超大仓库（20+ 个组件，需要找到正确的那个）

回归用例（5 个）：
  reg-001：防止误改 ProductCard（历史 bug #42）
  reg-002：防止 completed.json 格式错误（历史 bug #51）
  reg-003：防止重复追齐（历史 bug #63：已完成的功能被再次追齐）
  reg-004：防止空 diff（历史 bug #70：代码没有实质改动但报告成功）
  reg-005：防止遗漏 memory 更新（历史 bug #78）
```

**随着使用增加的扩展策略：**

```
阶段一（0-1 个月）：15 个最小用例
阶段二（1-3 个月）：扩展到 30 个（新增对抗用例 + 更多边界场景）
阶段三（3 个月以后）：扩展到 50+（基于真实使用发现的问题）
```

## 3. 自动化评测脚本架构

完整的评测脚本结构：

```typescript
// eval/types.ts - 核心类型定义
export interface AgentTestCase {
  id: string;
  description: string;
  input: TestCaseInput;
  expectedBehavior: ExpectedBehavior;
  evaluationCriteria: EvaluationCriteria;
  tags: TestCaseTag[];
  createdAt: string;
  lastReviewedAt: string;
}

export interface TestCaseInput {
  feature: string;
  targetRepo: string;
  referenceContext?: string;
}

export interface ExpectedBehavior {
  shouldCallTools: string[];
  shouldNotCallTools?: string[];
  shouldModifyFiles: string[];
  shouldNotModifyFiles?: string[];
  memoryUpdated: boolean;
  completedJsonUpdated: boolean;
}

export interface EvaluationCriteria {
  correctness: string;
  maxToolCalls?: number;
  maxTokens?: number;
  requiresExplanation: boolean;
}

export type TestCaseTag = "happy-path" | "edge-case" | "adversarial" | "regression";

// eval/runner.ts - 测试运行器
export class EvalRunner {
  private agent: AlignmentAgent;
  private judge: LLMJudge;
  
  constructor(agentConfig: AgentConfig, judgeConfig: JudgeConfig) {
    this.agent = new AlignmentAgent(agentConfig);
    this.judge = new LLMJudge(judgeConfig);
  }
  
  async runAll(testCases: AgentTestCase[]): Promise<EvalRunResult> {
    const results = await this.runConcurrent(testCases, { concurrency: 3 });
    return this.generateReport(results);
  }
  
  private async runSingle(tc: AgentTestCase): Promise<CaseResult> {
    const fixture = await setupFixture(tc.input.targetRepo);
    
    try {
      const output = await this.agent.run(tc.input, { captureToolCalls: true });
      const ruleChecks = await runRuleChecks(output, tc);
      const judgeResult = await this.judge.evaluate({ testCase: tc, agentOutput: output });
      
      return {
        testCaseId: tc.id,
        ruleChecks,
        judgeResult,
        passed: ruleChecks.every(r => r.passed) && judgeResult.passed,
        toolCallCount: output.toolCallLog.length,
        tokenUsage: output.tokenUsage,
        durationMs: output.durationMs
      };
    } finally {
      await cleanupFixture(fixture);
    }
  }
  
  private generateReport(results: CaseResult[]): EvalRunResult {
    const passed = results.filter(r => r.passed);
    
    return {
      summary: {
        total: results.length,
        passed: passed.length,
        successRate: passed.length / results.length,
        avgToolCalls: avg(results.map(r => r.toolCallCount)),
        avgTokenUsage: avg(results.map(r => r.tokenUsage.total)),
        avgDurationMs: avg(results.map(r => r.durationMs)),
        byTag: groupByTag(results)
      },
      caseResults: results,
      generatedAt: new Date().toISOString()
    };
  }
}

// eval/judge.ts - LLM 评判器
export class LLMJudge {
  async evaluate(input: JudgeInput): Promise<JudgeOutput> {
    const prompt = this.buildPrompt(input);
    const response = await this.callJudgeModel(prompt);
    return this.parseResponse(response);
  }
  
  private buildPrompt(input: JudgeInput): string {
    return `
你是代码评审专家，评估 H5 功能追齐质量。

## 目标功能
${input.testCase.input.feature}

## 评判标准
${input.testCase.evaluationCriteria.correctness}

## Agent 生成的 diff
\`\`\`diff
${input.agentOutput.generatedDiff}
\`\`\`

## Agent 的修改说明
${input.agentOutput.explanation}

## 请按以下格式输出评分
{
  "correctness": { "score": 1-5, "evidence": "...", "reasoning": "..." },
  "explainability": { "score": 1-5, "reasoning": "..." },
  "passed": true/false,
  "issues": [],
  "summary": "..."
}
    `.trim();
  }
}

// eval/rules.ts - 规则检查
export async function runRuleChecks(
  output: AgentRunOutput,
  testCase: AgentTestCase
): Promise<RuleCheckResult[]> {
  return Promise.all([
    checkDiffFormat(output.generatedDiff),
    checkModifiedFilesScope(output.generatedDiff, testCase.expectedBehavior),
    checkToolCallBudget(output.toolCallLog, testCase.evaluationCriteria.maxToolCalls),
    checkMemoryUpdated(output, testCase.expectedBehavior.memoryUpdated),
    checkCompletedJsonSchema(output.completedJsonUpdate)
  ]);
}
```

## 5. 第八章总结

| 节 | 核心要点 |
|----|----------|
| **8.1 为什么难以评测** | Agent 输出概率性、过程不透明、评判主观，传统 assert 失效 |
| **8.2 评测维度设计** | 五维度：正确性、效率、鲁棒性、成本、可解释性；不同阶段权重不同 |
| **8.3 测试用例设计** | 四类用例：黄金路径、边界、对抗、回归；用 Fixture 仓库保证稳定性 |
| **8.4 LLM-as-Judge** | 评判 Prompt 要具体可操作；规则检查补充语义评判；构建自动化流水线 |
| **8.5 评测驱动迭代** | 闭环：评测→分析→改进→再评测；优先级：正确性>鲁棒性>效率>成本 |

## 章节关联

← **第六章** Harness 架构定义了追齐 Agent 的执行边界，这些边界也是规则检查的约束来源——"修改只能在允许的仓库内"这条规则，正是 Harness 里定义的权限边界的检验。

← **第七章** 多 Agent 架构（Orchestrator + Research + Analysis + Execution）增加了评测复杂度——每个 Agent 的工具调用都需要被追踪，评测需要能处理跨 Agent 的行为链。

→ **第九章** 评测是生产化的前提。只有评测达标（正确率 ≥ 85%，鲁棒性 ≥ 70%），才应该考虑将 Agent 部署到生产环境——评测不达标就上线，等于把用户当测试员。
