# 3.2 输入设计：让 LLM 能准确调用

> [!abstract] 本节导读
> 工具的 Schema 是 LLM 和工具之间的"合同"。
> Schema 写得好，LLM 能准确构造参数、很少调用失败；
> Schema 写得差，LLM 会猜测参数含义，产生大量无效调用，消耗上下文空间。
> 本节从命名到描述，系统化地讲输入设计。
> **预计时长**：核心阅读 20 分钟。

---

> [!question] 带着问题阅读
> 你的工具 Schema 里，`description` 字段是写给人看的，还是写给 LLM 看的？
> 两者的写法有什么不同？

---

## 1. Schema 不是注释，是提示词的一部分

很多开发者写工具 Schema 时，把 `description` 当成普通注释来写——
简短、技术性强、假设读者知道背景。

但 Schema 会被原封不动地放进 L2 工具定义层，成为 LLM 每次推理时都会读到的内容。
**Schema 的每一行都是提示词。**

```typescript
// ❌ 写给人看的 description
{
  name: "read_repo_file",
  description: "Reads a file from a repository",  // 废话，名字就能看出来
  parameters: {
    repo: { type: "string" },                      // 没说接受什么格式
    path: { type: "string" }                       // 是绝对路径还是相对路径？
  }
}

// ✅ 写给 LLM 看的 description
{
  name: "read_repo_file",
  description: "读取指定仓库中的单个文件，返回结构化摘要（而非原始内容）。" +
    "用于了解文件的导出内容、依赖关系和已实现的功能。" +
    "当你需要理解某个 H5 文件的代码结构时调用此工具。",
  parameters: {
    repo: {
      type: "string",
      description: "仓库名称，使用 repo-map 中的 h5Repo 或 appRepo 字段值，如 'h5-cart-page'"
    },
    path: {
      type: "string",
      description: "相对于仓库根目录的文件路径，如 'src/pages/Cart/index.tsx'"
    }
  }
}
```

## 2. 工具命名的四个原则

### 原则一：动词 + 对象，语义清晰

```
✅ read_repo_file      → 读取仓库文件
✅ diff_feature        → 对比功能差异
✅ resolve_feature_spec → 获取功能规格
❌ getFile             → 获取什么文件？
❌ process             → 做什么处理？
❌ run                 → 运行什么？
```

### 原则二：用业务语言，而非技术语言

```
✅ resolve_feature_spec  → 业务含义清晰
❌ fetch_json_from_endpoint → 技术描述，不传达业务意图

✅ diff_feature          → 说的是业务动作
❌ compare_ast_nodes     → LLM 需要额外推断才知道这用来干什么
```

### 原则三：区分读操作和写操作

```
✅ read_*, get_*, list_*, find_*  → 只读，安全
✅ write_*, create_*, delete_*, update_*  → 有副作用，LLM 会更谨慎
❌ process_feature  → 是读还是写？LLM 无法判断
```

### 原则四：避免重叠功能

一组工具里，每个工具的用途应该是互斥的。
LLM 在选择工具时会做比较，如果两个工具描述相似，选择会随机化。

```
❌ 有问题的工具组：
read_file(path)        → 读取文件
read_source_file(path) → 读取源码文件
get_file_content(path) → 获取文件内容

✅ 应该合并：
read_repo_file(repo, path)  → 统一的文件读取接口
```

## 3. 参数设计：减少调用错误

### 用枚举代替自由字符串

```typescript
// ❌ 开放字符串，LLM 可能传入任何值
parameters: {
  side: { type: "string" }  // LLM 可能传 "app"/"App"/"APP"/"app-side"
}

// ✅ 枚举约束，消除歧义
parameters: {
  side: {
    type: "string",
    enum: ["app", "h5"],
    description: "指定读取 App 端还是 H5 端的代码，传入 'app' 或 'h5'"
  }
}
```

### 描述说明"什么时候用这个参数"

```typescript
parameters: {
  include_dependencies: {
    type: "boolean",
    default: false,
    description: "是否在返回结果中包含文件的依赖关系图。" +
      "当你需要理解组件的依赖树时设置为 true，" +
      "只需要了解文件导出内容时保持默认 false（更快）"
  }
}
```

### 复杂参数用对象，加字段级描述

```typescript
parameters: {
  filter: {
    type: "object",
    description: "过滤条件，不传则返回所有功能点",
    properties: {
      status: {
        type: "string",
        enum: ["pending", "done", "blocked"],
        description: "只返回指定状态的功能点"
      },
      feature_name: {
        type: "string",
        description: "功能名称关键词，支持模糊匹配，如'批量'会匹配'批量删除'"
      }
    }
  }
}
```

> [!tip] 原则
> **每个参数的 description 应该回答三个问题：**
> 1. 这个参数是什么（语义）？
> 2. 填什么格式（约束）？
> 3. 什么时候填 / 不填（判断指导）？
> 回答了这三个问题，LLM 基本不会填错。

## 4. 工具级 description 的结构

一个好的工具级 description 包含四个部分：

```
[用途说明]  一句话说清楚这个工具做什么
[调用场景] 什么情况下应该调用这个工具
[返回内容] 返回值包含哪些信息（让 LLM 知道能用来做什么）
[注意事项] 什么情况下不应该调用，或者需要注意的边界
```

```typescript
// read_repo_file 的完整 description 示例
description: `
读取指定仓库的单个文件，返回结构化摘要（导出内容、依赖、已实现功能）。

【调用场景】需要了解某个具体文件的代码结构时调用。
比如：想知道购物车 H5 的 CartList 组件实现了哪些功能。

【返回内容】包含：exports（导出项）、dependencies（依赖）、
stateSlice（使用的 Redux slice）、features（已实现功能列表）、
missingVsAppSpec（与 App 规格相比缺失的功能）。

【注意】如果需要了解整个目录的文件列表，
应该先调用 list_repo_files，再根据需要调用本工具读取具体文件。
`
```

> [!warning] 误区
> **不要在 description 里写"如果 X 则用工具 A，如果 Y 则用工具 B"这类路由逻辑。**
> 这类内容应该在 System Prompt 里通过工作流描述来传达，
> 而不是在每个工具的 description 里重复。
> description 里只放"这个工具能做什么"，不放"什么情况下选这个工具 vs 那个工具"。

## 5. 功能追齐 Agent 的四个工具输入设计

```typescript
// 追齐 Agent 工具 Schema（输入部分）

const tools = [
  {
    name: "read_repo_file",
    description: "读取仓库中单个文件的结构化摘要（非原始内容）。用于了解 H5 或 App 文件的代码结构、导出和功能实现情况。",
    inputSchema: {
      type: "object",
      required: ["repo", "path"],
      properties: {
        repo: { type: "string", description: "仓库名，如 'h5-cart-page'（来自 repo-map）" },
        path: { type: "string", description: "相对路径，如 'src/pages/Cart/index.tsx'" },
        focus: {
          type: "string",
          enum: ["structure", "features", "dependencies"],
          default: "features",
          description: "摘要侧重点：structure=目录结构, features=已实现功能, dependencies=依赖关系"
        }
      }
    }
  },
  {
    name: "list_repo_files",
    description: "列出仓库中匹配 glob 模式的文件路径列表。在不确定具体文件位置时先调用此工具探索目录结构，再用 read_repo_file 读取具体文件。",
    inputSchema: {
      type: "object",
      required: ["repo"],
      properties: {
        repo: { type: "string", description: "仓库名" },
        glob: { type: "string", default: "src/**/*.tsx", description: "glob 模式，默认返回所有 TSX 文件" }
      }
    }
  },
  {
    name: "diff_feature",
    description: "对比 App 端与 H5 端某个功能点的实现差异，返回结构化差异报告。用于确定一个功能的具体追齐内容。",
    inputSchema: {
      type: "object",
      required: ["feature_name", "app_repo", "h5_repo"],
      properties: {
        feature_name: { type: "string", description: "功能点名称，如'批量删除'，用于定位相关代码" },
        app_repo: { type: "string", description: "App 端仓库名" },
        h5_repo: { type: "string", description: "H5 端仓库名" }
      }
    }
  },
  {
    name: "resolve_feature_spec",
    description: "从 App 端测试用例提取指定功能点的规格描述，返回行为规范（非代码）。在开始追齐前调用，获取'该功能应该如何工作'的权威描述。",
    inputSchema: {
      type: "object",
      required: ["feature_name", "app_repo"],
      properties: {
        feature_name: { type: "string", description: "功能点名称" },
        app_repo: { type: "string", description: "App 端仓库名" }
      }
    }
  }
];
```

## 本节小结

- Schema 是提示词的一部分，description 写给 LLM 看，不是写给人看的
- 命名原则：动词+对象、业务语言、区分读写、无重叠
- 参数设计：枚举代替字符串、描述调用时机、复杂对象加字段级描述
- 好的 description 回答：是什么 + 填什么格式 + 什么时候填
