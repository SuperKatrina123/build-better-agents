# 3.5 工具的错误处理与韧性

> [!abstract] 本节导读
> 工具失败是必然的——文件不存在、网络超时、解析出错……
> 失败时的错误信息也会进入 L5 对话流，影响 LLM 的后续决策。
> 好的错误处理让 Agent 面对失败时优雅降级；坏的错误处理让 Agent 陷入困惑或无限重试。
> **预计时长**：核心阅读 15 分钟。

---

> [!question] 带着问题阅读
> 当工具调用失败时，你的 Agent 会怎么做？

## 1. 错误信息也是上下文

一个常见的误解：错误处理只是"别让程序崩溃"。

在 Agent 系统里，错误处理的目标是：
**让 LLM 在看到错误信息后，知道下一步该做什么。**

```typescript
// ❌ 对 LLM 没有价值的错误
throw new Error("ENOENT: no such file or directory, open 'src/Cart/index.tsx'");

// ✅ 对 LLM 有价值的错误
return {
  success: false,
  error: "文件不存在",
  errorType: "FILE_NOT_FOUND",
  detail: "路径 'src/Cart/index.tsx' 在仓库 h5-cart-page 中不存在",
  suggestions: [
    "调用 list_repo_files 查看实际的文件路径",
    "购物车页面的入口文件可能在 src/pages/Cart/ 目录下"
  ]
};
```

第一种错误信息：LLM 只知道"出错了"，不知道怎么处理。
第二种错误信息：LLM 立刻知道可以采取什么行动（调用另一个工具）。

## 2. 错误分类与处理策略

不是所有错误都应该以相同的方式处理。按照错误的性质分类：

### A 类：可自动恢复（自动重试）

技术性临时错误，不需要 LLM 参与决策，代码层自动重试。

```typescript
const RETRYABLE_ERRORS = ['ETIMEDOUT', 'ECONNRESET', 'ENOTFOUND'];

async function withRetry<T>(fn: () => Promise<T>, maxRetries = 3): Promise<T> {
  for (let i = 0; i < maxRetries; i++) {
    try {
      return await fn();
    } catch (e) {
      if (RETRYABLE_ERRORS.includes(e.code) && i < maxRetries - 1) {
        await sleep(1000 * (i + 1));  // 指数退避
        continue;
      }
      throw e;
    }
  }
}
```

### B 类：可预期边界（返回结构化信号）

业务上可能发生的预期情况，不是"错误"，但需要告知 LLM 调整策略。

```typescript
// 文件不存在 → 这个功能可能未实现
if (!fileExists) {
  return {
    success: false,
    errorType: "NOT_IMPLEMENTED",
    message: "该功能在 H5 端尚未有对应文件，可能需要新建",
    nextStep: "根据 App 端规格新建文件，建议路径：src/pages/Cart/BatchDelete.tsx"
  };
}
```

### C 类：需要 LLM 判断（透明上报）

有多种可能原因，或者有多种处理方案，需要 LLM 来决策。

```typescript
// 解析结果置信度低 → 让 LLM 决定是否使用
if (analysisConfidence < 0.6) {
  return {
    success: true,
    result: analysisResult,
    warning: "分析结果置信度较低（60%），代码中存在动态生成的功能逻辑",
    suggestion: "建议人工确认以下功能点的实现方式",
    uncertainFeatures: lowConfidenceItems
  };
}
```

### D 类：不可处理（上报给 Harness）

严重错误，工具和 LLM 都无法处理，需要 Harness 介入（见第六章）。

```typescript
// 权限错误、配置错误等 → 直接抛出，让 Harness 处理
if (e.status === 403) {
  throw new FatalError("PERMISSION_DENIED", `无法访问仓库 ${repo}，请检查读取权限配置`);
}
```

## 3. 错误设计要避免的三个陷阱

> [!warning] 误区
>
> **陷阱一：错误信息触发无限重试**
> 如果错误信息是"请重试"，LLM 会真的重试——可能 10 次。
> 解决：区分可重试和不可重试错误，可重试的在代码层处理，不要暴露给 LLM。
>
> **陷阱二：错误信息暴露技术细节**
> 堆栈跟踪、内部路径、系统信息……这些内容对 LLM 没用，只会污染上下文。
> 解决：错误信息在工具层统一格式化，技术细节写进日志不进上下文。
>
> **陷阱三：所有错误都返回相同格式**
> "发生错误"是无效信息。LLM 需要知道是什么错误，才能决定怎么处理。
> 解决：用 `errorType` 枚举区分错误类别，每种类型给出不同的 `suggestions`。

## 4. 追齐 Agent 的错误处理实现

```typescript
// 统一的工具错误响应格式
interface ToolErrorResponse {
  success: false;
  errorType: "FILE_NOT_FOUND" | "PARSE_ERROR" | "NOT_IMPLEMENTED" | "LOW_CONFIDENCE" | "PERMISSION_DENIED";
  message: string;       // 对 LLM 友好的描述
  detail?: string;       // 技术细节（可选，用于调试）
  suggestions?: string[];  // LLM 可以采取的下一步行动
  nextStep?: string;     // 最推荐的下一步（有把握时填写）
}

// read_repo_file 的完整错误处理
async function read_repo_file(repo: string, path: string): Promise<FileAnalysis | ToolErrorResponse> {
  // A 类：网络错误，自动重试（代码层处理，不暴露给 LLM）
  const content = await withRetry(() => fetchFile(repo, path));

  // B 类：文件不存在，返回可操作信息
  if (!content) {
    return {
      success: false,
      errorType: "NOT_IMPLEMENTED",
      message: `文件 ${path} 在 ${repo} 中不存在`,
      suggestions: [
        `调用 list_repo_files("${repo}", "src/**/*.tsx") 查看实际文件结构`,
        `该功能可能尚未在 H5 端实现，需要新建`
      ]
    };
  }

  // C 类：解析置信度低，透明上报
  const analysis = parseFileStructure(content);
  if (analysis.confidence < 0.6) {
    return {
      ...analysis,
      warning: "代码使用了动态模式，功能识别置信度较低",
      suggestions: ["建议验证 features 字段的准确性"]
    };
  }

  return analysis;
}
```

> [!tip] 原则
> **错误处理是工具设计的"最后一公里"——做好了，Agent 面对意外情况依然优雅；做差了，一个预期之外的文件路径就能让整个任务失败。**

---

## 本节小结

- 错误信息也是上下文，设计目标是"让 LLM 知道下一步怎么做"
- 四类错误：A 类自动重试、B 类结构化信号、C 类透明上报、D 类上报 Harness
- 三个陷阱：触发无限重试 / 暴露技术细节 / 所有错误同一格式
- 统一错误响应格式：errorType + message + suggestions + nextStep
