# 3.1 什么是 ACI：模型与世界的接口层

> [!abstract] 本节导读
> 工具是 Agent 能力的边界——它能做什么，完全取决于它有哪些工具以及工具设计得多好。
> ACI（Agent-Computer Interface）是一个思维框架：像设计 API 一样设计工具，
> 把"工具层"当成独立的接口层来对待，而不是随手写的辅助函数。
> **预计时长**：核心阅读 20 分钟。

---

> [!question] 带着问题阅读
> 你写过的工具（或 function），是"给人用的 API"还是"给 LLM 用的 ACI"？
> 两者有什么不同？

---

## 1. 从 HCI 到 ACI

**HCI（Human-Computer Interface）**：人与计算机交互的界面设计。
好的 HCI 让人类用户能直觉地操作系统。

**ACI（Agent-Computer Interface）**：Agent（LLM）与计算机系统交互的接口设计。
好的 ACI 让 LLM 能准确理解工具的用途、正确构造调用参数、正确解读返回结果。

两者有本质区别：

| 维度 | HCI | ACI |
|------|-----|-----|
| 使用者 | 人类（视觉感知、直觉判断） | LLM（token 理解、结构推理） |
| 错误纠正 | 用户看到结果可以调整 | LLM 依赖返回值决定下一步 |
| 歧义处理 | 用户可以猜测 UI 意图 | LLM 依赖精确的文字描述 |
| 信息消费 | 视觉扫描，忽略不相关内容 | 所有返回内容都进上下文 |

**关键推论**：为人类设计的 API，不一定适合 LLM 直接调用。

---

## 2. ACI 的三个职责

![[aci-concept.excalidraw]]

来自 tw93 文章的 ACI 原则图：

![ACI 工具设计原则|532](https://gw.alipayobjects.com/zos/k/8c/aci_tool_design1.svg)

ACI 工具层承担三个职责：

**职责一：语义化接口（意图 → 调用）**
把自然语言意图翻译成精确的计算机操作。
这意味着工具名称和参数名称要"说人话"，让 LLM 通过名称就能理解用途，不需要看复杂的文档。

**职责二：结构化返回（原始结果 → 上下文）**
把外部世界返回的原始数据（文件内容、API 响应、命令输出），
转换成 LLM 当前决策需要的精简结构化信息。
**这一步直接决定 L5 对话流的信噪比**——是第二章上下文工程的延续。

**职责三：可操作错误处理（失败 → 下一步）**
工具执行失败时，不是简单抛出技术错误，
而是给 LLM 一个能指导它下一步行动的描述。

## 3. 为什么 ACI 是独立的工程关注点

一个常见的误解：工具只是"封装了 API 调用的函数"，不需要特别设计。

这导致工具层往往是这样写出来的：

```typescript
// 典型的"随手写"工具——没有 ACI 意识
const tools = {
  readFile: async (path: string) => {
    return fs.readFileSync(path, 'utf-8');  // 直接返回原始内容
  },
  compareFiles: async (file1: string, file2: string) => {
    // 直接返回 diff 工具的原始输出（几百行）
    return execSync(`diff ${file1} ${file2}`).toString();
  }
};
```

这类工具有几个问题：
1. 函数名 `readFile` 不传达业务语义（读取什么？为了什么？）
2. 直接返回原始内容，L5 瞬间被污染
3. 没有错误处理，任何异常都变成无法操作的技术错误信息

> [!tip] 原则
> **工具设计的质量是 Agent 性能的上限。**
> 再强的模型，面对返回几百行原始代码的工具，也会被上下文淹没。
> 相反，设计良好的工具，能让普通模型表现出超出预期的效果。

## 4. ACI 的五个设计维度

本章后续内容将逐一展开，这里先给出全景：

| 维度 | 核心问题 | 对应章节 |
|------|---------|---------|
| 语义化命名 | 工具名和参数名能否不看文档就理解？ | 3.2 |
| 输入约束 | 如何减少 LLM 传错参数的概率？ | 3.2 |
| 返回值设计 | 工具结果进入 L5 之前经过了什么处理？ | 3.3 |
| 工具粒度 | 一个工具负责多少事情才算合适？ | 3.4 |
| 错误处理 | 失败时给 LLM 的信息是否可操作？ | 3.5 |

> [!example] 案例
> 同一个"读取仓库文件"的需求，两种工具设计的对比：
>
> **设计 A（无 ACI 意识）**：`readFile(path)` → 返回完整文件内容
>
> **设计 B（有 ACI 意识）**：`read_repo_file(repo, path, purpose?)` → 返回结构化摘要
>
> 在 20 次追齐任务的测试里：
> 设计 A 的平均每次任务 token 消耗：~45,000；
> 设计 B：~8,000。成本降低 82%，且追齐准确率提升（Context Rot 减少）。

---

## 本节小结

- ACI = 专为 LLM 设计的工具接口层，不同于面向人类的 HCI
- ACI 的三个职责：语义化接口、结构化返回、可操作错误处理
- 工具设计质量是 Agent 性能的上限；工具的返回值直接影响 L5 信噪比
- 五个设计维度：命名、输入约束、返回值、粒度、错误处理
