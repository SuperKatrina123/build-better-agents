## 2. 功能追齐 Agent：四个工具的完整设计

### 工具一：read_repo_file

| 属性 | 值 |
|------|-----|
| 用途 | 读取仓库单个文件，返回代码结构摘要 |
| 粒度 | 单文件读取 + 摘要（单一职责）|
| Token 预算 | 返回值 ≤ 500 tokens |
| 缓存友好 | 同一文件内容稳定，可在工具层缓存摘要结果 |

**输入设计**：
```typescript
{
  repo: string,          // 仓库名（来自 repo-map）
  path: string,          // 相对路径
  focus?: "features" | "dependencies" | "structure"  // 默认 features
}
```

**返回值设计**：
```typescript
{
  file: string,
  exports: string[],
  implementedFeatures: string[],
  missingVsAppSpec?: string[],   // 如果 focus=features，自动计算差异
  stateSlice?: string,
  keyRelatedFiles?: string[],    // 最多 3 个
}
```

**错误设计**：FILE_NOT_FOUND → 建议调用 list_repo_files 探索

---

### 工具二：list_repo_files

| 属性 | 值 |
|------|-----|
| 用途 | 列出仓库中匹配 glob 的文件路径，用于目录探索 |
| 粒度 | 单次目录查询（不读取文件内容）|
| Token 预算 | 返回值 ≤ 300 tokens（最多返回 30 个路径）|
| 调用时机 | 不确定文件位置时；新仓库初次探索时 |

**输入设计**：
```typescript
{
  repo: string,
  glob?: string,           // 默认 "src/**/*.{ts,tsx}"
  maxResults?: number,     // 默认 30，防止列表过长
}
```

**返回值设计**：
```typescript
{
  files: string[],
  total: number,           // 匹配总数（可能超过 maxResults）
  truncated: boolean,      // 是否被截断
  suggestion?: string,     // 如果 truncated，建议如何缩小范围
}
```

---

### 工具三：diff_feature

| 属性 | 值 |
|------|-----|
| 用途 | 对比 App 和 H5 单个功能点的实现差异 |
| 粒度 | 单功能差异分析（不包含规格获取）|
| Token 预算 | 返回值 ≤ 800 tokens |
| 调用前提 | 通常在 resolve_feature_spec 之后调用 |

**返回值设计**：
```typescript
{
  feature: string,
  appStatus: "implemented" | "not_found",
  h5Status: "implemented" | "partial" | "not_implemented",
  gaps: Array<{
    description: string,
    location: string,      // 应该在哪个文件修改
    priority: "must" | "nice_to_have",
    estimatedLines: number,
  }>,
  totalComplexity: "simple" | "medium" | "complex",
}
```

---

### 工具四：resolve_feature_spec

| 属性 | 值 |
|------|-----|
| 用途 | 从 App 端测试用例提取功能规格，获取权威的行为描述 |
| 粒度 | 单功能规格查询（不包含差异分析）|
| Token 预算 | 返回值 ≤ 600 tokens |
| 调用时机 | 追齐任务开始时，在 diff_feature 之前 |

**返回值设计**：
```typescript
{
  feature: string,
  behaviors: Array<{
    scenario: string,
    expected: string,
    edgeCase: boolean,
  }>,
  appVersion: string,
  lastUpdated: string,
  confidence: "high" | "medium" | "low",  // 规格来源质量
}
```

## 3. 工具调用的典型序列

```
追齐一个功能点的标准流程：

① resolve_feature_spec("批量删除", "app-cart")
  ↓ LLM 检查规格质量（confidence 字段）
② list_repo_files("h5-cart-page", "src/**/Cart*.tsx")
  ↓ LLM 决定要读哪些文件
③ read_repo_file("h5-cart-page", "Cart/CartList.tsx", focus="features")
  ↓ LLM 判断当前实现与规格的大致差距
④ diff_feature("批量删除", "app-cart", "h5-cart-page")
  ↓ LLM 根据结构化差异生成追齐方案

4 次工具调用，覆盖一个完整的功能追齐分析
平均 token 消耗（工具返回）：~2200 tokens
```

## 4. 第三章总结

| 节次 | 核心概念 | 一句话要点 |
|------|---------|-----------|
| 3.1 | ACI 是什么 | 为 LLM 设计的接口层，不同于面向人类的 API |
| 3.2 | 输入设计 | Schema 是提示词，description 回答三个问题 |
| 3.3 | 输出设计 | 只返回决策需要的，工具层预处理保护 L5 |
| 3.4 | 工具粒度 | 一个工具一个自然操作单元，SRP 适用 |
| 3.5 | 错误处理 | 错误信息也是上下文，要可操作 |
| 3.6 | 设计决策 | 4 个工具，4 次调用，覆盖完整追齐分析 |

> [!tip] 原则
> **工具设计是上下文工程的延续——好的工具，在代码层就帮 LLM 把信噪比做好了。**
> 第二章管"什么信息进上下文"，第三章管"工具怎么不往上下文里注入噪声"。
> 两章组合，才是完整的上下文工程。
