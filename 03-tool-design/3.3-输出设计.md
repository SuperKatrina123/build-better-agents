# 3.3 输出设计：返回值决定上下文质量

> [!abstract] 本节导读
> 这是第三章最重要的一节，也是最容易被忽视的一节。
> 输入设计保证 LLM "能准确调用"，输出设计决定调用结果"对上下文的影响"。
> 坏的输出设计让 L5 对话流快速污染；好的输出设计让 Agent 越跑越稳。
> **预计时长**：核心阅读 25 分钟。

---

> [!question] 带着问题阅读
> 你的工具，有没有因为返回了太多内容，导致 LLM 在后续几轮都在"消化"这些内容？
> 这就是输出设计问题。

---

## 1. 工具返回值的两种命运

每一次工具调用的返回值，都会进入 messages[] 的 L5 对话流。
它的体积和质量，直接影响后续所有轮次的推理。

![[tool-return-quality.excalidraw]]

**命运一（坏的）**：返回原始内容 → L5 膨胀 → Context Rot → 后续推理质量下降
**命运二（好的）**：工具层预处理 → 精简结构化信息 → L5 保持干净 → 推理质量稳定

两者的差距可以有 10 倍以上的 token 消耗差距。

---

## 2. 输出设计的五个原则

### 原则一：只返回 LLM 当前决策需要的信息

工具知道要执行什么任务（因为 LLM 传了参数）。
在返回前，工具应该"想一想"：LLM 调用这个工具，是为了做什么下一步决策？
只返回对那个决策有用的信息。

```typescript
// ❌ 返回一切
return {
  fullContent: rawFileContent,        // 几百行代码
  metadata: allFileMetadata,          // 时间戳、权限等无关信息
  gitHistory: last100Commits,         // 从没被用到
};

// ✅ 返回决策所需
return {
  exports: extractedExports,          // LLM 需要知道这个文件导出什么
  features: detectedFeatures,        // LLM 需要知道实现了哪些功能
  missingVsSpec: gapAnalysis,        // LLM 需要知道缺什么
};
```

### 原则二：结构化，不要自由文本

自由文本看起来更"自然"，但结构化数据让 LLM 更容易精确提取需要的部分。

```typescript
// ❌ 自由文本
return `文件 Cart.tsx 实现了商品列表展示和价格计算功能，
但缺少批量删除功能。代码使用了 Redux Toolkit 进行状态管理。`;

// ✅ 结构化 JSON
return {
  file: "Cart.tsx",
  implementedFeatures: ["商品列表展示", "价格计算"],
  missingFeatures: ["批量删除"],
  stateManagement: "redux-toolkit",
};
```

结构化的额外好处：缓存键（Prompt Cache）更稳定，格式不随模型理解变化。

### 原则三：设定返回值的 token 预算

在工具设计阶段就确定：这个工具的返回值最多占多少 tokens？
**在工具层强制执行这个预算**，而不是让 LLM 自己"处理多余内容"。

```typescript
const MAX_RETURN_TOKENS = 500;  // 每个工具的 token 预算

async function read_repo_file(repo: string, path: string): Promise<FileAnalysis> {
  const content = await readFile(repo, path);
  const analysis = await analyzeFile(content);

  // 如果分析结果超出预算，做二次压缩
  if (estimateTokens(analysis) > MAX_RETURN_TOKENS) {
    return compressAnalysis(analysis, MAX_RETURN_TOKENS);
  }

  return analysis;
}
```

### 原则四：错误信息也是返回值，要可操作

见 3.5，这里先记住：错误信息不能只说"出错了"，要说"出错了，可以这样处理"。

### 原则五：区分"给 LLM 看的"和"给代码用的"

有些工具会同时被 LLM 调用和被其他代码调用。
如果两者的需求不同，应该分开设计，或者在返回值里区分字段。

```typescript
return {
  // LLM 需要的部分（进入上下文）
  summary: {
    features: [...],
    gaps: [...],
  },
  // 代码需要的部分（不需要进入上下文，可以存在外部状态里）
  _meta: {
    rawContent: ...,     // 用下划线前缀标识"非 LLM 消费字段"
    filePath: ...,
    lastModified: ...,
  }
};
```

## 3. 追齐 Agent 工具返回值设计

```typescript
// read_repo_file 返回值设计
interface FileAnalysis {
  file: string;
  exports: string[];
  stateSlice?: string;
  implementedFeatures: string[];
  missingVsAppSpec: string[];  // 直接帮 LLM 算好差异
  keyFiles?: string[];         // 相关联的其他文件（最多 3 个）
}

// diff_feature 返回值设计
interface FeatureDiff {
  feature: string;
  appStatus: "implemented" | "not_found";
  h5Status: "implemented" | "partial" | "not_implemented";
  gaps: Array<{
    description: string;     // 缺什么
    location: string;        // 应该加在哪个文件
    priority: "must" | "nice_to_have";
  }>;
  estimatedComplexity: "simple" | "medium" | "complex";
}

// resolve_feature_spec 返回值设计
interface FeatureSpec {
  feature: string;
  behaviors: Array<{
    scenario: string;    // 测试用例场景描述
    expected: string;    // 期望行为
  }>;
  edgeCases: string[];
  appVersion: string;    // 规格来源的 App 版本
}
```

> [!example] 案例
> `diff_feature` 的 `gaps` 字段里，直接包含了 `location`（应该改哪个文件）。
> 这意味着 LLM 不需要再调用 `list_repo_files` 来查找文件位置——工具已经帮它做了。
> 结果：每次追齐任务减少了平均 3~4 次工具调用，上下文里也少了 3~4 轮的 tool_result。

## 4. 大文件的处理策略

当工具需要处理大文件（> 200 行代码）时，不能直接返回，需要在工具层做预处理。

**策略 A：代码骨架提取（推荐）**

```typescript
function extractCodeSkeleton(content: string): string {
  // 保留：import、interface/type、函数签名、class 定义
  // 折叠：函数实现体
  const lines = content.split('\n');
  return lines
    .filter(line => {
      return line.startsWith('import') ||
             line.startsWith('export') ||
             line.startsWith('interface') ||
             line.startsWith('type ') ||
             line.match(/^(const|function|class|async function)\s+\w+/) ||
             line.startsWith('  // ');  // 保留顶层注释
    })
    .join('\n');
}
```

**策略 B：按 focus 参数定向摘要**

```typescript
// 根据调用者的 focus 参数，只提取相关部分
async function summarizeByFocus(content: string, focus: string): Promise<string> {
  switch (focus) {
    case "features":
      return await extractFeaturePatterns(content);
    case "dependencies":
      return extractImports(content);
    case "structure":
      return extractFileStructure(content);
    default:
      return extractCodeSkeleton(content);
  }
}
```

> [!tip] 原则
> **工具层做的预处理，是一次性的成本；进入上下文的冗余内容，是每轮推理都要付出的代价。**
> 在工具层多花一点计算，换取上下文的持续干净——永远是值得的。

---

## 本节小结

- 工具返回值进入 L5，体积和质量直接影响信噪比
- 五个原则：只返回决策需要的 / 结构化 / 设定 token 预算 / 错误要可操作 / 区分 LLM 字段和代码字段
- 大文件在工具层预处理（代码骨架 / 按 focus 摘要），不要原样返回
- 追齐 Agent 的 diff_feature 工具把 location 也算好，减少后续工具调用次数
