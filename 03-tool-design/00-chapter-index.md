# 第三章：工具设计——ACI 原则

> 工具设计的质量是 Agent 性能的上限。
> 再强的模型，也无法从糟糕的工具返回值中提炼出精准的决策。

---

## 章节目标

完成本章后，你应该能够：
1. 用 ACI 视角审视任何一个工具设计，找出潜在的问题
2. 写出让 LLM 能准确调用、很少出错的工具 Schema
3. 设计工具返回值，在工具层就控制好 L5 的信噪比
4. 为功能追齐 Agent 完成一份完整的工具设计文档

---

## 通用原理

工具设计不是“把已有函数暴露给模型”，而是为模型设计一层真正可用的行动接口。它要解决的是所有 Agent 的共同问题：

- 模型如何准确理解一个工具能做什么、该在什么时候用
- 工具输出怎样才能直接服务下一步决策，而不是制造上下文噪声
- 工具的粒度、错误信息和边界怎样设计，才能让模型少走弯路

只要系统允许模型调用环境能力，本章的 ACI 视角就适用。

## 场景映射

| Agent 类型 | 本章关注对象 | 典型工具形态 | 设计重点 |
|-----------|-------------|-------------|---------|
| Coding Agent | 仓库与执行环境接口 | 读文件、改文件、diff、跑命令 | 输出摘要、粒度自然、错误可恢复 |
| Research Agent | 信息获取与处理接口 | 搜索、抓取、抽取、比较 | 来源透明、引用清晰、避免原文灌入 |
| Support Agent | 知识与工单接口 | 检索知识、查工单、创建回复 | 参数语义明确、返回值可直接答复 |
| 流程 / 业务执行 Agent | 业务系统接口 | 查状态、提交流程、更新记录 | 权限边界、幂等性、结构化错误 |

## 主案例展开：功能追齐 Agent

| 工具 | 用途 | Token 预算 | 调用时机 |
|------|------|-----------|---------|
| `resolve_feature_spec` | 获取 App 端功能规格 | ≤ 600 | 追齐开始前 |
| `list_repo_files` | 探索仓库目录结构 | ≤ 300 | 不确定文件位置时 |
| `read_repo_file` | 读取单文件结构摘要 | ≤ 500 | 需要了解具体实现时 |
| `diff_feature` | 对比单功能实现差异 | ≤ 800 | 分析追齐内容时 |

这四个工具是主案例的具体工具集，不是本章的默认答案。真正要迁移的是：

- 工具名称、输入、输出都要从模型视角设计
- 工具只返回“下一步判断真正需要的信息”
- 粒度应对应一个自然操作单元，而不是代码里的任意函数边界
