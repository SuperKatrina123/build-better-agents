# 3.4 工具粒度：单一职责与自然缝隙

> [!abstract] 本节导读
> 工具太细，LLM 要编排几十步才能完成一个自然的操作；
> 工具太粗，LLM 失去了介入过程、做关键判断的机会。
> 找到工具粒度的"自然缝隙"，是工具设计从能用到好用的关键跨越。
> **预计时长**：核心阅读 20 分钟。

---

> [!question] 带着问题阅读
> 在你设计的工具里，LLM 在两次工具调用之间，需要做什么决策？
> 如果什么都不需要想，工具可能太粗了；如果每步都在做琐碎操作，工具可能太细了。

---

## 1. 粒度问题的本质

![[tool-granularity.excalidraw]]

工具粒度问题的本质是：**判断边界应该在代码里还是在 LLM 里？**

- 太细的工具：把"多步代码操作"拆成多个工具，LLM 负责编排这些步骤 → 调用次数多，L5 快速增长
- 太粗的工具：把"多个判断点"塞进一个工具，代码负责判断 → LLM 失去控制，黑盒不透明
- 合适的工具：**一个工具完成一个"自然的操作单元"**，判断点在工具与工具之间

---

## 2. 什么是"自然的操作单元"

"自然的操作单元"有一个简单的检验方法：

> **如果把这个操作告诉一个人类工程师，他会把它当成一步来完成，还是多步？**

- "读取这个文件" → 一步 ✅
- "找到购物车相关的所有文件，然后读取每一个" → 多步，应该拆成两个工具
- "分析购物车功能的追齐方案" → 包含多个判断，应该让 LLM 分步完成

```
追齐 Agent 的操作单元划分：

✅ 自然单元（合适的工具粒度）
  - 列出仓库文件（一个目录操作）
  - 读取单个文件（一个文件操作）
  - 对比单个功能（一个分析操作）
  - 获取功能规格（一个查询操作）

❌ 太细（不需要单独的工具）
  - 读取文件的 import 部分
  - 读取文件的 interface 定义
  - 读取文件的函数签名
  → 这三个应该合并进 read_repo_file，由 focus 参数控制

❌ 太粗（包含了应该由 LLM 完成的判断）
  - 完整追齐一个功能点（包含：规格查询 → 差异分析 → 方案选择 → 代码生成）
  → 这中间有多个判断点，应该拆成 4 个工具
```

---

## 3. 单一职责原则在工具设计中的应用

软件工程里的 SRP（Single Responsibility Principle）同样适用于工具设计：
**一个工具只做一件事，只因为一种原因改变。**

判断一个工具是否违反了 SRP，看它的 description 里有没有"以及"：

```
❌ 有"以及"，违反 SRP：
"读取文件，并分析其中实现了哪些功能，以及与 App 规格的差异"
（包含了：读取 + 分析 + 对比——三件事）

✅ 单一职责：
"读取文件，返回代码结构摘要"
（只做：读取 + 摘要——一件事）
```

### 什么时候合并，什么时候拆分

**合并的信号**：
- 两个工具总是成对出现（调用 A 后必调用 B）
- 两者之间不需要任何 LLM 判断
- 合并后不会掩盖重要的决策点

**拆分的信号**：
- 一个工具的 description 里有多个动词
- 工具内部有 if/else 逻辑，处理不同的业务场景
- LLM 需要在该操作的中间点做判断

> [!example] 案例
> 早期追齐 Agent 设计了一个 `analyze_and_diff` 工具，
> 内部先 `resolve_feature_spec`，再 `diff_feature`。
>
> 问题：LLM 无法在获取规格后、进行对比前，判断"规格是否足够清晰"。
> 如果规格模糊，直接进行对比会生成错误的追齐方案。
>
> 拆分后：LLM 先调用 `resolve_feature_spec` → 判断规格质量 → 如果清晰再调用 `diff_feature`。
> 追齐准确率显著提升，尤其是对于边界情况复杂的功能点。

## 4. 工具数量的实践建议

> [!info] 方法
> **工具数量控制建议：**
>
> - **目标**：5~8 个核心工具，覆盖 Agent 90% 的操作需求
> - **上限**：单个 Agent 不超过 15 个工具（超过后选择成本显著上升）
> - **测试**：给 LLM 一个任务，如果它需要连续调用 5 次以上工具才能完成一个原子操作，工具太细了
> - **测试**：如果 LLM 调用一次工具后不知道该怎么办（因为工具做了太多决策），工具太粗了

对于功能追齐 Agent，4 个工具的组合是经过验证的最佳配置：

```
工具调用典型序列（追齐购物车批量删除）：
1. resolve_feature_spec("批量删除", "app-cart")
   → 获取 App 端行为规格
   → LLM 判断：规格是否清晰？（判断点）
2. list_repo_files("h5-cart-page", "src/**/*.tsx")
   → 找到相关文件列表
   → LLM 判断：应该读哪几个文件？（判断点）
3. read_repo_file("h5-cart-page", "Cart/CartList.tsx")
   → 了解现有实现
   → LLM 判断：缺失的功能和规格对应哪些代码改动？（判断点）
4. diff_feature("批量删除", "app-cart", "h5-cart-page")
   → 获取结构化差异报告
   → LLM 判断：追齐方案是什么？优先级如何？（判断点）

4次调用，4个自然判断点，LLM 全程参与决策 ✅
```

## 本节小结

- 粒度问题的本质：判断边界应该在代码层还是 LLM 层
- "自然操作单元"检验法：告诉人类工程师会当几步来做
- SRP 在工具设计：description 里不应该出现"以及"（多个动词）
- 追齐 Agent 4 个工具 = 4 个自然操作单元 + 4 个 LLM 判断点，经过验证的最佳配置

> **3.5 工具的错误处理与韧性**
> 工具失败时，错误信息也会进入 L5 上下文。
> 设计好的错误处理，让 LLM 知道"出错了，可以这样继续"；
> 设计坏的错误处理，让 LLM 陷入混乱或无限重试。
