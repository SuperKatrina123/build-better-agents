# 5.2 长任务上下文维护：防止上下文在执行中退化

> [!abstract] 本节概要
> 长任务的核心挑战之一是**上下文退化**：随着执行推进，L5 对话流里积累了大量中间结果、工具输出、分析过程，有效信息被稀释，信噪比不断下降。这就是 Context Rot 在长任务中的具体表现。本节介绍三种主动维护上下文质量的策略：分段压缩、工作状态文件，以及上下文长度的主动监控。

---

## 1. 长任务的上下文退化

在一个追齐 15 个功能点的 Session 里，典型的上下文增长路径是这样的：

```
起点（2,000 tokens）：System Prompt + 功能点列表
↓ 追齐功能点1（+8,000 tokens）：读代码 + diff 分析 + 实现过程
↓ 追齐功能点2（+7,000 tokens）：同上
↓ 追齐功能点3（+9,000 tokens）：发现了复杂依赖，分析更多
...
到功能点5时（已经 ~52,000 tokens）：
  - 功能点1的详细分析对现在毫无用处
  - 但它还在上下文里，占据着宝贵空间
  - Agent 现在读到的每一个 token，都要"穿过"前面所有的噪音
```

这就是 **Context Rot**（上下文腐化）的长任务版本：不是因为内容本身变"坏"了，而是因为积累的历史信息让真正有用的内容越来越难以被 LLM 注意到。

研究表明，LLM 对上下文中**位置靠近首尾**的信息最敏感，对中间位置的信息容易"遗忘"（Lost in the Middle 效应）。随着任务执行，越来越多的关键信息会沉到中间位置，执行质量必然下滑。

---

## 2. 分段压缩策略

核心思路：**每完成一个子任务（功能点），对该子任务的上下文做一次结构性压缩**，只保留结论，丢弃过程。

类比于写作时的"章节总结"——你不需要在阅读第三章时记住第一章的每一个细节，但你需要记住第一章的核心结论。

**压缩的操作流程**：

```
功能点 N 执行完成
  ↓
提炼结论（由 Agent 自主完成）：
  - 追齐了什么：功能描述（1-2句）
  - 关键发现：与 App 的差异（要点列表，≤5条）
  - 实现方式：核心改动（代码片段或文件路径）
  ↓
将结论写入 MEMORY.md
  ↓
清理 L5 中该功能点的详细过程（通过新建子任务对话或压缩历史实现）
  ↓
开始功能点 N+1
```

**压缩前后的对比**：

```markdown
压缩前（占用约 3,000 tokens）：
  > 我先读取了 cart.tsx 文件，发现第 145 行有一个...
  > 然后我用 diff_feature 工具，得到了以下输出：
  > [大量 diff 输出]
  > 分析发现，App 的实现在 handleBatchDelete 函数中...
  > [更多分析过程]
  > 然后我修改了 H5 的 Cart.tsx，加入了...

压缩后（占用约 200 tokens）：
  ## 功能点：批量删除
  - ✅ 已追齐
  - 关键差异：App 支持长按进入多选模式，H5 原来没有
  - 实现：在 Cart.tsx 第 145 行加入 useLongPress hook，
    触发 setSelectionMode(true)
  - 测试基准：App 测试用例 TC-CART-012
```

通过压缩，同样的信息从 3,000 tokens 压缩到 200 tokens，压缩比约 15:1。

## 3. 工作状态文件

分段压缩解决了"历史上下文太多"的问题，但还有另一个问题：**当前执行状态如何在不占用上下文的情况下持久化？**

解决方案是使用**外部工作状态文件**：`task-state.json`。这个文件不放在上下文里，而是每次执行前读取、执行后更新。

**task-state.json 结构**：

```json
{
  "currentSession": {
    "startTime": "2024-01-15T10:00:00Z",
    "targetPage": "cart",
    "totalFeatures": 12,
    "processedCount": 5
  },
  "featureQueue": [
    "coupon-display",
    "settlement-detail",
    "empty-cart-state",
    "address-selection"
  ],
  "currentFeature": {
    "id": "coupon-display",
    "status": "in_progress",
    "startTime": "2024-01-15T10:45:00Z",
    "completedSteps": [
      "read_app_implementation",
      "run_diff_feature"
    ],
    "pendingSteps": [
      "implement_h5_change",
      "write_checkpoint"
    ]
  },
  "completedFeatures": [
    {"id": "batch-delete", "completedAt": "2024-01-15T10:12:00Z"},
    {"id": "quantity-edit", "completedAt": "2024-01-15T10:28:00Z"},
    {"id": "item-swipe-delete", "completedAt": "2024-01-15T10:38:00Z"},
    {"id": "price-display", "completedAt": "2024-01-15T10:43:00Z"},
    {"id": "select-all", "completedAt": "2024-01-15T10:45:00Z"}
  ],
  "failedFeatures": [],
  "findings": {
    "batch-delete": "App 使用长按触发多选，H5 已实现 ✅",
    "quantity-edit": "App 有防抖 300ms，H5 已同步 ✅",
    "item-swipe-delete": "iOS 和 Android 手势方向不同，H5 统一使用左滑 ✅"
  }
}
```

**使用方式**：Agent 每次开始一个新的功能点时，先读取 `task-state.json`，了解当前状态；执行过程中每完成一个步骤，更新 `currentFeature.completedSteps`；功能点完成后，将其从 `featureQueue` 移入 `completedFeatures`，并在 `findings` 里记录关键发现。

这样，即使上下文被清空（比如中途重启），Agent 也能从 `task-state.json` 快速恢复到正确的执行位置。

## 4. 上下文长度的主动监控

不要等到上下文爆满再处理。主动监控意味着在达到临界点之前就触发干预。

**两级限制策略**：

| 级别 | 触发条件 | 行为 |
|------|---------|------|
| 软限制 | 上下文达到 30,000 tokens | 触发一次分段压缩，压缩当前已完成功能点的历史 |
| 硬限制 | 上下文达到 80,000 tokens | 强制结束当前功能点执行，保存检查点，等待下次 Session 恢复 |

**实现示例**（Orchestrator 逻辑）：

```typescript
async function monitorContextAndCompress(
  currentTokenCount: number,
  completedFeatures: string[]
) {
  if (currentTokenCount > 80_000) {
    // 硬限制：强制保存检查点并终止
    await saveCheckpoint({
      reason: "context_hard_limit",
      tokenCount: currentTokenCount
    });
    throw new Error("CONTEXT_LIMIT_REACHED: Session saved, resume later");
  }

  if (currentTokenCount > 30_000 && completedFeatures.length > 0) {
    // 软限制：压缩最早完成的功能点历史
    const toCompress = completedFeatures.slice(0, 3);
    await compressFeatureHistory(toCompress);
    // 压缩后通知 Agent
    return `已压缩 ${toCompress.join(', ')} 的执行历史，继续下一个功能点。`;
  }

  return null;
}
```

> [!warning] 不要等到上下文爆满再处理
> 在上下文接近上限时才开始压缩，意味着 LLM 需要在**已经处于最差状态**时执行压缩操作（因为信噪比已经很低了）。主动在 30,000 tokens 时触发压缩，LLM 仍处于高质量状态，压缩结论更准确，代价更低。
>
> 粗略估计：被动应对的代价是主动管理的 **10 倍**——你不仅损失了压缩质量，还可能不得不重新执行某些步骤。

> [!example] 追齐购物车页面的上下文维护案例
>
> **场景**：正在追齐购物车页面，完成"批量删除"功能追齐后，准备开始"商品数量编辑"。
>
> **Agent 的操作**：
> ```
> 1. 检查当前 token 用量：已用 28,500 tokens（接近软限制）
>
> 2. 压缩"批量删除"的执行历史：
>    原始：3,200 tokens 的 diff 分析 + 实现过程
>    压缩为：
>    "批量删除：App 长按触发多选，H5 已实现 useLongPress。
>     测试基准：TC-CART-012。实现文件：Cart.tsx L145-L178。✅"
>    → 节省约 3,000 tokens
>
> 3. 将压缩结论写入 MEMORY.md
>
> 4. 更新 task-state.json：
>    completedFeatures 加入 "batch-delete"
>    currentFeature 设为 "quantity-edit"
>
> 5. 继续追齐"商品数量编辑"
>    此时上下文约 25,700 tokens，距软限制仍有余裕
> ```
>
> 这个模式每完成一个功能点执行一次，保持上下文始终处于可控范围内。

## 本节小结

- **上下文退化**是长任务的核心挑战，Lost in the Middle 效应在长任务中尤为明显
- **分段压缩**：每完成一个功能点，将详细过程压缩为结构性结论（压缩比 ~15:1）
- **工作状态文件**：用 `task-state.json` 外部化执行状态，上下文只保留当前必要信息
- **主动监控**：30,000 tokens 触发软压缩，80,000 tokens 触发强制检查点

**下一节预告**：上下文压缩本质上是为了给"重启"做准备。5.3 节将深入讲解检查点机制和断点续传协议，让长任务真正可中断、可恢复。
