# 第五章：规划与长任务——让 Agent 完成超出单次会话的工作

## 章节目标

完成本章后，你应该能够：

1. **理解长任务的本质挑战**：上下文有限，任务无限；会话可以中断，工作必须持续
2. **设计混合规划策略**：宏观预先规划功能点队列，微观动态执行每个功能点的追齐过程
3. **实现检查点与断点续传**：让 Agent 在任意时刻中断后，都能从上次完成的位置无损恢复
4. **掌握长任务错误隔离**：一个功能点的失败不影响其他功能点，最大化整体追齐成功率

---

## 通用原理

规划与长任务解决的是所有复杂 Agent 都会遇到的同一件事：单次上下文和单次执行能力有限，但目标却往往跨越多个步骤、多个阶段，甚至多个会话。

本章的核心不是某一种计划格式，而是：

- 如何把大目标拆成可执行、可恢复的单元
- 如何在执行过程中维护状态、避免迷失
- 如何在中断、失败和恢复之间保持工作连续性

只要任务不是“一问一答即结束”，本章就适用。

## 场景映射

| Agent 类型 | 规划单元 | 检查点对象 | 典型中断原因 |
|-----------|---------|-----------|-------------|
| Coding Agent | 功能点、改动步骤 | 已分析文件、已完成改动、待验证项 | 上下文过长、工具失败、循环 |
| Research Agent | 子问题、资料批次 | 已读来源、阶段结论、待验证问题 | 来源过多、证据冲突、会话中断 |
| Support Agent | 工单阶段、处理动作 | 已确认信息、已回复内容、待跟进事项 | 用户未回、升级转单、系统异常 |
| 流程 / 业务执行 Agent | 业务节点、审批阶段 | 当前节点、已执行动作、待人工确认项 | 外部系统失败、超时、规则分支变化 |

## 主案例展开：功能追齐 Agent

### task-state.json 核心结构

```json
{
  "currentSession": { "targetPage": "cart" },
  "featureQueue": ["settlement-detail", "empty-cart-state"],
  "currentFeature": {
    "id": "settlement-detail",
    "status": "in_progress",
    "completedSteps": ["read_app_implementation"],
    "pendingSteps": ["implement_h5_change", "write_checkpoint"]
  },
  "completedFeatures": [{"id": "batch-delete", "summary": "..."}],
  "failedFeatures": [],
  "lastCheckpoint": "2024-01-15T11:23:00Z",
  "status": "in_progress"
}
```

### 检查点时机

| 触发条件 | 动作 |
|---------|------|
| 功能点追齐完成（✅） | 保存检查点，继续下一个 |
| 功能点追齐失败（❌） | 记录失败原因，保存检查点，继续下一个 |
| 上下文达到 30,000 tokens | 压缩历史 + 继续 |
| 上下文达到 80,000 tokens | 保存检查点 + 终止 Session |
| 检测到循环（3 次相同调用） | 保存检查点 + 等待人工确认 |

### 错误处理策略

| 错误级别 | 处理方式 |
|---------|---------|
| 工具调用失败 | 重试 2 次，失败则升级为功能点失败 |
| 功能点失败 | 跳过 + 记录 + 继续 |
| 检测到循环 | 重置上下文 + 从检查点恢复 |
| 上下文硬限制 | 强制保存检查点 + 终止 |

这里的 `task-state.json`、功能点队列和检查点阈值，都是主案例的实例化设计。通用层面真正重要的是：

- 计划要能拆分、更新、恢复
- 状态要能回答“做到哪了、接下来做什么”
- 失败要被局部隔离，而不是把整条任务链一起拖垮
