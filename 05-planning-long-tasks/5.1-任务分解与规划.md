# 5.1 任务分解与规划：把大目标变成可执行的步骤

> [!abstract] 本节概要
> 长任务的失败，大多不是因为 LLM 不够聪明，而是因为没有规划。本节介绍为什么大任务需要在执行前进行规划，对比两种主流规划策略——**预先规划**（Plan-then-Execute）与**动态规划**（ReAct 风格）——并给出针对追齐 Agent 这类半结构化长任务的**混合策略**建议。

> [!question] 带着问题阅读
> - 如果一个 Agent 要追齐 15 个功能点，它是应该先列出所有功能点再逐个执行，还是执行完一个再决定下一个？
> - 计划应该放在 System Prompt 里，还是每次执行时动态注入？
> - 当发现原计划中有一步不可行时，Agent 应该怎么做？

---

## 1. 为什么大任务需要规划

没有规划的 Agent 在长任务中会遇到一个必然困境：**随着执行的推进，它越来越不知道自己在哪里、还剩多少工作、当前这步是否走对了方向**。

想象一个没有地图的人试图翻越一座山。每一步看起来都是在"往前走"，但实际上可能绕了很大的弯，甚至走到了错误的山谷里。

**功能追齐 Agent 没有规划会怎样？**

考虑追齐一个电商 H5 页面，其中包含 15 个功能点。如果 Agent 没有预先规划：

1. Agent 开始追齐"加入购物车"功能
2. 追齐到一半，发现需要先了解购物车的数据结构
3. 开始读购物车相关代码，发现与另一个功能"商品数量编辑"有依赖关系
4. 开始处理"商品数量编辑"
5. 上下文里已经有 20,000 tokens 的中间信息
6. Agent 回头想继续追齐"加入购物车"，但已经不记得之前做到哪里了
7. 重新开始——之前所有的工作白费

规划的核心价值是**让 Agent 在任意时刻都知道**：我在做什么、做完了什么、还需要做什么。

## 2. 两种规划策略

### 预先规划（Plan-then-Execute）

在任务开始时生成一份**完整的执行计划**，然后严格按计划执行。

```
阶段1：规划
  → 扫描所有待追齐功能点
  → 分析依赖关系
  → 生成有序的执行列表

阶段2：执行
  → 按列表逐项执行
  → 每完成一项，标记状态
  → 执行结束
```

**适合场景**：任务结构明确，功能点之间依赖关系清晰，不需要在执行中根据中间结果调整计划。

**优点**：
- 执行阶段 Agent 可以完全专注于执行，不需要同时思考"下一步做什么"
- 计划本身可以被人类审查和修改
- 容易实现检查点（完成计划的第 N 步 = 检查点 N）

**缺点**：
- 预先规划可能不准确（比如：计划里的某个功能点在 App 里根本不存在）
- 对探索性任务效果差（不知道会发现什么）

### 动态规划（ReAct 风格）

每执行一步之后，根据当前观察决定下一步做什么。这是 ReAct（Reasoning + Acting）框架的核心思想。

```
循环：
  观察当前状态
  → 推理：下一步应该做什么？
  → 执行一个动作
  → 观察结果
  → 回到循环
```

**适合场景**：探索性任务，不知道会遇到什么情况，需要根据发现动态调整。

**优点**：灵活，能适应意外情况

**缺点**：对于长任务，每次都要重新推理"下一步做什么"，消耗大量 token；而且越到后期，上下文里积累的信息越多，推理质量越难保证。

### 追齐 Agent 的推荐策略：混合规划

追齐 Agent 的任务具有一种特殊的半结构化特征：**功能点列表是已知的，但每个功能点的追齐过程是探索性的**。

这天然适合**混合策略**：

```
预先规划层（结构化）：
  → 从 completed.json 读取 pending 功能点列表
  → 按优先级排序（先追齐核心流程功能，再追齐边缘功能）
  → 生成功能点执行队列

动态执行层（探索性）：
  → 针对每个功能点，动态执行追齐流程
  → 读取 App 代码 → 对比 H5 代码 → 生成 diff → 实现追齐
  → 完成后更新状态，取下一个功能点
```

这样，**宏观层面有确定性计划，微观层面保持灵活性**。

## 3. 任务分解的粒度

![[task-decomposition.excalidraw]]

合理的任务分解是规划成功的基础。分解过细会产生大量管理开销；分解过粗则每个子任务太复杂，难以可靠执行。

> [!tip] 分解原则
> **每个执行单元（最细粒度的子任务）应当能在 10 次工具调用以内完成。** 如果一个子任务预计需要超过 10 次工具调用，说明它需要进一步拆分。这个数字不是绝对的，但它是一个好的经验值：10 次工具调用大约对应 5,000–10,000 tokens 的上下文增长，在这个范围内 LLM 仍然能保持高质量推理。

**追齐 Agent 的三层分解示例**：

```
L1 总目标（本次 Session 目标）
└── 追齐购物车页面的所有待追齐功能点

L2 子任务（功能点粒度）
├── 追齐功能点：批量删除商品
├── 追齐功能点：商品数量编辑
├── 追齐功能点：优惠券展示
└── 追齐功能点：结算金额明细

L3 执行单元（单个功能点内部步骤）
└── 追齐"批量删除商品"：
    ├── 步骤1：read_repo_file 读取 App 实现
    ├── 步骤2：diff_feature 对比 H5 与 App 差异
    ├── 步骤3：resolve_feature_spec 确认行为基准
    ├── 步骤4：实现 H5 代码修改
    └── 步骤5：write_memory 记录追齐结论
```

每个 L3 执行单元（一个功能点的追齐）包含约 5 步工具调用，完全在 10 次以内。

## 4. 计划应该在上下文的哪里

规划信息本身需要放在合适的位置，才能发挥作用。错误的放置会导致预算浪费或 Agent 看不到计划。

| 规划信息 | 放置位置 | 理由 |
|---------|---------|------|
| 任务类型约定、成功标准 | L1 System Prompt | 稳定，不随任务变化 |
| 当前任务的功能点列表 | L3 工作指令 | 本次 Session 有效，需要 Agent 全程可见 |
| 当前执行中的功能点状态 | L4 工作记忆（动态更新） | 实时变化，频繁读写 |
| 已完成的功能点记录 | External: MEMORY.md | 跨 Session 持久化 |

**关键原则**：**不要把整个计划塞入 L5 对话流**。L5 是对话历史，会不断增长。如果把功能点列表每次都追加到 L5，到后期 L5 会有大量重复的计划信息，浪费上下文预算，同时让 Agent 难以找到真正的执行状态。

> [!tip] 计划是执行的指引，不是枷锁
> 当 Agent 在执行中发现原计划有误（比如某个功能点根本不需要追齐，App 里已经删除了这个功能），它应该**更新计划**，而不是强行按原计划执行。一个好的 Agent 知道什么时候该调整方向。允许计划动态更新，是让 Agent 真正智能的关键。

---

> [!example] 追齐 Agent 的完整规划流程
>
> **第一步：扫描状态（规划准备）**
> ```typescript
> const memory = await read_memory("memory/MEMORY.md");
> const completed = await read_memory("memory/completed.json");
> // completed.json 结构：
> // { "done": [...], "pending": [...], "failed": [...] }
> ```
>
> **第二步：生成功能点队列**
> ```typescript
> const pendingFeatures = completed.pending;
> // 按优先级排序：核心流程功能 > 边缘功能
> const queue = sortByPriority(pendingFeatures);
> // 写入 task-state.json
> await write_task_state({
>   featureQueue: queue,
>   currentFeature: null,
>   completedFeatures: completed.done,
>   failedFeatures: completed.failed
> });
> ```
>
> **第三步：逐个执行**
> ```typescript
> for (const feature of queue) {
>   await update_task_state({ currentFeature: feature });
>   const result = await execute_feature_alignment(feature);
>   if (result.success) {
>     await mark_feature_done(feature);
>   } else {
>     await mark_feature_failed(feature, result.error);
>   }
>   await save_checkpoint(); // 每个功能点完成后保存检查点
> }
> ```

## 本节小结

- 规划让 Agent 在长任务中始终知道自己在哪里、还剩什么工作
- 预先规划适合结构化任务，动态规划适合探索性任务，追齐 Agent 适合混合策略
- 任务分解的粒度目标：每个执行单元 ≤ 10 次工具调用
- 计划信息的正确位置：功能点列表在 L3，执行状态在 L4，完成记录在外部文件

**下一节预告**：计划制定好了，执行过程中上下文会不断增长。5.2 节将讲述如何在长任务执行中主动维护上下文质量，防止信噪比下降。
