# 5.5 规划内容的上下文位置：计划住在哪里

> [!abstract] 本节概要
> 规划信息本身也需要被精心管理——**放错位置要么浪费上下文预算，要么让 Agent 根本看不到计划**。本节从上下文分层的视角，梳理不同类型的规划信息应该放在哪一层，以及计划的完整生命周期：从生成、执行、巩固到归档。

---

## 1. 各层规划信息分配

回顾上下文的五层结构（L1-L5 + External），不同类型的规划信息有各自合适的位置：

### L1 System Prompt — 稳定的任务约定

L1 放置的是**跨所有 Session 都不会改变**的规划约定：

```markdown
## 任务类型
你是一个功能追齐 Agent，负责将 H5 页面的功能对齐到 App 最新版本。

## 成功标准
- 每个功能点的行为与 App 测试用例完全一致
- 飞书文档仅供参考，App 测试用例才是行为基准
- 追齐完成的标志：diff_feature 返回无差异，或差异已被标记为"预期差异"

## 规划约定
- 每个功能点完成后必须保存检查点
- 遇到无法确定的行为时，标记为"需要人工确认"，不要猜测
- 连续 3 次工具调用相同时，停止并报告可能陷入循环
```

这些约定是永久有效的规则，放在 L1 最合适。

### L3 工作指令 — 当前任务的具体计划

L3 放置的是**本次 Session 才有效**的具体执行计划：

```markdown
## 本次任务计划

目标页面：购物车页面（`apps/cart-page`）

待追齐功能点列表（按优先级排序）：
1. batch-delete — 批量删除商品
2. quantity-edit — 商品数量编辑
3. coupon-display — 优惠券折叠展示
4. settlement-detail — 结算金额明细
5. empty-cart-state — 空购物车状态

执行策略：
- 按列表顺序逐个执行
- 每个功能点完成后更新 task-state.json
- 发现跨功能点的关键知识，及时写入 MEMORY.md
```

L3 在 Session 开始时注入，全程可见，但不会随着执行进程而膨胀（它是静态的）。

### L4 工作记忆 — 动态更新的执行状态

L4 放置的是**实时变化的执行状态**，随着任务推进而更新：

```markdown
## 当前执行状态（动态更新）

已完成（3/5）：
- ✅ batch-delete — 已追齐，App 长按触发多选
- ✅ quantity-edit — 已追齐，防抖 300ms
- ✅ coupon-display — 已追齐，新增折叠逻辑

当前执行：
- 🔄 settlement-detail（步骤 2/5：正在分析 App 实现）

待执行：
- ⏳ empty-cart-state
```

L4 是 Agent 的"工作面板"，它需要频繁更新。实现上，可以通过在 L5 中不断追加更新来模拟 L4 的动态特性，或者通过工具调用显式管理工作状态文件。

### External 外部文件 — 跨 Session 的规划进度

```
memory/MEMORY.md     → 已发现的关键知识，影响后续功能点的执行策略
memory/completed.json → 已完成/待追齐/失败的功能点列表
task-state.json      → 当前 Session 的执行状态快照
```

外部文件不占用上下文预算，但需要工具调用才能读取，因此适合**不需要在每步都看到**的信息。

## 2. 计划的生命周期

规划信息不是静态的，它有一个完整的生命周期：

```
生成阶段（Session 开始）
  ↓ Agent 读取 completed.json，扫描待追齐功能点
  ↓ 生成功能点执行队列，注入 L3
  ↓ 初始化 task-state.json

执行阶段（逐步推进）
  ↓ 每完成一步，更新 L4 工作记忆
  ↓ 关键发现随时写入 MEMORY.md（通过 write_memory 工具）
  ↓ 每完成一个功能点，更新 task-state.json + completed.json

巩固阶段（功能点完成后）
  ↓ 压缩该功能点的执行历史（从 L5 提炼结论）
  ↓ 结论写入 MEMORY.md
  ↓ 更新 completed.json 的 done 数组

归档阶段（Session 结束或任务完成）
  ↓ 最终状态保存到 completed.json
  ↓ MEMORY.md 更新最终摘要
  ↓ task-state.json 标记为 "completed" 或 "interrupted"
```

理解这个生命周期有助于在设计时为每个阶段选择合适的工具和存储位置。

## 3. 动态更新计划

静态计划无法应对执行中的意外情况。追齐 Agent 需要能够在运行时**修改计划**。

**需要更新计划的典型场景**：

1. **发现功能点已不存在**：App 最新版本删除了某个功能
   ```
   原计划包含：share-cart（分享购物车）
   执行时发现：App 最新版本已下线该功能
   → 更新计划：将 share-cart 从 pending 移入 done，标注"App 已下线，H5 同步移除"
   ```

2. **发现新的功能点**：扫描代码时发现原计划遗漏了某个功能
   ```
   执行 coupon-display 时发现：App 还有一个"叠加优惠说明"弹窗
   → 更新计划：在队列中插入 coupon-stack-info
   ```

3. **发现功能间依赖**：某个功能点必须在另一个之前完成
   ```
   发现 settlement-detail 依赖 coupon-display 的数据结构
   → 调整队列：将 coupon-display 优先级提前
   ```

**实现动态更新的两种方式**：

**方式一：给 Agent `update_plan` 工具**
```typescript
// Agent 可以直接调用这个工具来修改计划
async function update_plan(action: {
  type: "add" | "remove" | "reorder" | "mark_done";
  featureId: string;
  reason: string;
  newPosition?: number;
}) {
  const state = await readTaskState();
  // 根据 action 修改 featureQueue
  const updated = applyPlanUpdate(state, action);
  await writeTaskState(updated);
  await appendToMemory(`计划更新：${action.type} ${action.featureId}，原因：${action.reason}`);
  return `计划已更新`;
}
```

**方式二：通过 System Prompt 约定**
```
当你在执行中发现原计划有误或需要调整时：
1. 使用 write_memory 工具更新 task-state.json 中的 featureQueue
2. 在 MEMORY.md 中记录计划变更的原因
3. 继续执行更新后的计划
```

> [!tip] 计划不是常量，是 Agent 对任务的当前最佳理解
> 允许 Agent 修改计划，是让它真正智能的关键。一个只能死板执行固定计划的 Agent，在面对复杂现实时必然会产生大量错误。
>
> 但计划的修改需要有**记录**（写入 MEMORY.md），而不是静默修改。这样人类可以在事后审查计划变更的原因，也能在下次 Session 时了解为什么某些功能点不在原始计划里。

## 本节小结

- **L1**：稳定的任务类型约定和成功标准（永不改变）
- **L3**：当前 Session 的功能点计划（Session 开始时注入，静态）
- **L4**：实时执行状态（动态更新，是 Agent 的工作面板）
- **External**：跨 Session 的规划进度（completed.json、MEMORY.md）
- 计划的生命周期：生成 → 执行（L4 更新）→ 巩固（写入 MEMORY.md）→ 归档
- 允许动态更新计划，但必须记录变更原因
