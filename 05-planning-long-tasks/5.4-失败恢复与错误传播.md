# 5.4 失败恢复与错误传播控制

> [!abstract] 本节概要
> 长任务中的错误不可避免，关键是如何**控制错误的影响范围**。错误有两大类：**可恢复错误**（工具调用失败、单个功能点追齐失败）和**不可恢复错误**（上下文损坏、记忆污染）。前者可以跳过继续；后者需要重置并从检查点恢复。本节介绍错误分类框架、错误不传播原则、错误记录格式，以及如何检测和处理上下文陷入循环的情况。

---

## 1. 错误的分类

不同层次的错误需要不同的处理策略。一刀切的"遇到错误就停止"或"遇到错误就重试"都是低效的。

### L1 工具错误：单次工具调用失败

**特征**：单个工具调用返回错误，但任务状态本身没有损坏。

**常见原因**：
- 网络超时（`read_repo_file` 请求超时）
- 文件不存在（功能点对应的文件路径有误）
- API 限流（短暂的请求频率超限）

**处理策略**：
```typescript
async function callToolWithRetry<T>(
  toolFn: () => Promise<T>,
  maxRetries: number = 2
): Promise<T | null> {
  for (let i = 0; i <= maxRetries; i++) {
    try {
      return await toolFn();
    } catch (error) {
      if (i === maxRetries) {
        // 超过重试次数，返回 null，由上层决定是否跳过
        console.error(`Tool failed after ${maxRetries} retries:`, error);
        return null;
      }
      // 指数退避：1s, 2s
      await sleep(1000 * Math.pow(2, i));
    }
  }
  return null;
}
```

### L2 子任务错误：某个功能点追齐失败

**特征**：在追齐某个功能点的过程中，无法完成追齐（代码读取成功，但无法确定正确的实现方式）。

**常见原因**：
- App 代码逻辑复杂，无法确定行为基准
- 飞书文档描述与 App 实际行为矛盾，无法判断哪个是真相
- 功能点涉及需要人工决策的业务逻辑

**处理策略**：跳过 + 记录 + 继续下一个

### L3 上下文错误：LLM 陷入错误的推理循环

**特征**：Agent 连续多次执行相同的工具调用，或者在两个状态之间来回切换，没有实质性进展。

**常见原因**：
- 上下文中有相互矛盾的信息，LLM 无法决策
- 之前的错误结果污染了推理方向
- 上下文过长，LLM 对关键信息产生"遗忘"

**处理策略**：重置上下文，从最近的检查点恢复

### L4 系统错误：环境级别的故障

**特征**：LLM API 超时、内存溢出、进程崩溃等非 Agent 本身的错误。

**处理策略**：保存检查点（如果还有机会） + 重启

## 2. 错误不传播原则

**一个功能点的失败，绝对不应该影响其他功能点的追齐。**

这是功能追齐 Agent 设计中最重要的错误处理原则。15 个功能点相互独立，某个功能点追齐失败，通常是因为该功能点本身的特殊情况，与其他功能点无关。

**正确的隔离实现**：

```typescript
async function executeFeatureQueue(features: string[]): Promise<ExecutionReport> {
  const results: FeatureResult[] = [];

  for (const featureId of features) {
    // 更新当前执行状态
    await updateTaskState({ currentFeature: featureId, status: "in_progress" });

    try {
      // 执行单个功能点追齐（可能抛出任何错误）
      const result = await alignFeature(featureId);
      
      results.push({ featureId, status: "success", summary: result.summary });
      await markFeatureDone(featureId, result.summary);
      
    } catch (error) {
      // 捕获所有错误，防止传播
      const errorMessage = error instanceof Error ? error.message : String(error);
      
      results.push({
        featureId,
        status: "failed",
        error: errorMessage,
        timestamp: new Date().toISOString()
      });
      
      // 记录失败，但继续执行下一个
      await markFeatureFailed(featureId, errorMessage);
      console.log(`Feature ${featureId} failed: ${errorMessage}. Continuing...`);
    }

    // 每个功能点结束后保存检查点（无论成功还是失败）
    await saveCheckpoint();
  }

  return {
    total: features.length,
    succeeded: results.filter(r => r.status === "success").length,
    failed: results.filter(r => r.status === "failed").length,
    results
  };
}
```

注意：`catch` 块捕获了**所有**错误，包括 L1、L2 级别的。这确保了无论发生什么，循环都会继续到下一个功能点。

## 3. 错误记录的格式

错误信息需要被保存，以便下次启动时 Agent 可以了解哪些功能点失败了、为什么失败、是否值得重试。

**`completed.json` 中的 `failed` 数组**：

```json
{
  "page": "cart",
  "lastUpdated": "2024-01-15T14:30:00Z",
  "done": ["batch-delete", "quantity-edit", "coupon-display"],
  "pending": ["share-cart"],
  "failed": [
    {
      "id": "settlement-detail",
      "failedAt": "2024-01-15T12:15:00Z",
      "error": "App 结算逻辑位于 native 模块，H5 无法直接读取实现代码",
      "retryCount": 1,
      "recommendation": "需要人工查看 App 对应的测试用例 TC-SETTLE-*"
    },
    {
      "id": "empty-cart-state",
      "failedAt": "2024-01-15T13:45:00Z",
      "error": "diff_feature 工具返回超时，已重试 2 次",
      "retryCount": 2,
      "recommendation": "可能是该工具的临时问题，建议下次重试"
    }
  ]
}
```

**下次启动时的错误展示**：

Agent 启动时，恢复提示中会包含失败记录：

```
## 上次执行失败的功能点（2 个）

1. settlement-detail
   失败原因：App 结算逻辑位于 native 模块，H5 无法直接读取
   建议：需要人工查看 TC-SETTLE-* 测试用例后再重试

2. empty-cart-state
   失败原因：diff_feature 工具超时（已重试 2 次）
   建议：可重试，可能是临时问题

请决定：
a) 重试以上失败的功能点
b) 跳过失败的功能点，继续追齐剩余的待追齐项
c) 仅重试工具超时类的失败（empty-cart-state）
```

这让人类（或 Agent 自主判断）可以基于失败原因做出明智的决策。

## 4. 上下文重置与恢复

当检测到 LLM 陷入循环时，需要执行上下文重置。

**循环检测逻辑**：

```typescript
class LoopDetector {
  private recentToolCalls: string[] = [];
  private readonly windowSize = 6;
  private readonly threshold = 3;

  recordToolCall(toolName: string, args: string): boolean {
    const callSignature = `${toolName}:${args}`;
    this.recentToolCalls.push(callSignature);

    // 保持滑动窗口
    if (this.recentToolCalls.length > this.windowSize) {
      this.recentToolCalls.shift();
    }

    // 检测最近 windowSize 次调用中是否有相同调用出现 threshold 次
    const counts = new Map<string, number>();
    for (const call of this.recentToolCalls) {
      counts.set(call, (counts.get(call) || 0) + 1);
    }

    return Array.from(counts.values()).some(count => count >= this.threshold);
  }
}

// 使用
const detector = new LoopDetector();

async function executeWithLoopDetection(toolCall: ToolCall) {
  const isLooping = detector.recordToolCall(toolCall.name, JSON.stringify(toolCall.args));
  
  if (isLooping) {
    // 触发上下文重置
    await saveCheckpoint({ reason: "loop_detected" });
    throw new Error("LOOP_DETECTED: Context reset required");
  }
  
  return await executeTool(toolCall);
}
```

**重置后的恢复流程**：

```
检测到循环
  ↓
保存当前检查点（记录：当前功能点执行到一半，需要重新执行）
  ↓
清空 L5 对话历史（或新建会话）
  ↓
重新注入：System Prompt + 工作记忆（L3+L4）+ 检查点信息
  ↓
从当前功能点的开始重新执行（不是从整个任务的开始）
```

> [!warning] 错误日志不要进上下文
> 错误处理逻辑和错误日志应该写到外部文件（比如 `error-log.json`），**不要让错误详情在 L5 对话流里占据大量空间**。
>
> 上下文里只需要保留决策相关的信息："功能点 X 已失败，已跳过，继续执行功能点 Y"。
>
> 完整的错误详情（堆栈、原始错误信息）写到外部文件，需要时再读取。这可以将上下文中的错误信息占用从数百 token 压缩到十几个 token。

> [!example] 追齐 15 个功能点，2 个失败的完整流程
>
> **场景**：功能追齐 Agent 追齐某购物车页面的 15 个功能点
>
> **执行结果**：
> - ✅ 13 个功能点成功追齐
> - ❌ 2 个功能点失败：
>   - `settlement-detail`：App 结算逻辑在 native 模块，无法读取代码
>   - `address-selection`：飞书文档描述与 App 测试用例矛盾，无法确定行为基准
>
> **Agent 的处理**：
> ```
> 功能点 settlement-detail 追齐失败：
>   → 工具 read_repo_file 返回 "native module, source not available"
>   → 重试 1 次，同样失败
>   → 写入 failed 数组，记录失败原因和建议
>   → 继续执行 address-selection
>
> 功能点 address-selection 追齐失败：
>   → diff_feature 成功，发现差异
>   → 但 resolve_feature_spec 返回飞书文档与测试用例矛盾的两个版本
>   → Agent 判断：无法自主决策，需要人工介入
>   → 写入 failed 数组，记录"人工介入"标记
>   → 继续执行剩余功能点
>
> 最终报告：
>   "已追齐 13/15 个功能点。
>    2 个功能点需要人工介入：
>    1. settlement-detail：需要查看 native 模块实现
>    2. address-selection：飞书文档与测试用例有矛盾，需要产品确认"
> ```

## 本节小结

- 错误分 4 级：工具错误（重试）、子任务错误（跳过）、上下文错误（重置）、系统错误（重启）
- **错误不传播**：用 try-catch 包裹每个功能点，失败后写入记录，继续执行下一个
- 错误记录保存在 `completed.json` 的 `failed` 数组，下次启动时展示给 Agent
- 循环检测：连续 3 次相同工具调用 → 重置上下文，从检查点恢复
- 错误日志写外部文件，上下文只保留决策信息
