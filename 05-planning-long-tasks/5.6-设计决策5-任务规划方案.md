# 5.6 设计决策 5：功能追齐 Agent 长任务规划完整方案

本节是第五章的设计决策汇总。前五节介绍了任务分解、上下文维护、检查点、错误处理、规划位置的理论与原则，这里将所有内容整合为功能追齐 Agent 的**可落地参考架构**。

---

## 1. 规划策略选择

**决策：混合规划策略**

功能追齐任务具有"结构已知，过程探索"的特征：功能点列表已知（来自 `completed.json`），但每个功能点的追齐过程需要动态探索。因此采用混合策略：

```
宏观层：预先规划（Plan-then-Execute）
  → 从 completed.json 读取 pending 功能点列表
  → 按优先级排序（核心流程 > 边缘功能）
  → 生成有序的执行队列，写入 task-state.json
  → 全程按队列执行，不随意调整顺序（除非有明确理由）

微观层：动态规划（ReAct 风格）
  → 每个功能点内部，动态探索实现方式
  → 读取 App 代码 → 分析差异 → 实现追齐 → 验证
  → 根据发现动态调整该功能点的追齐步骤
```

type FeatureStatus = "pending" | "in_progress" | "done" | "failed" | "skipped";

// 功能点状态机

## 3. 检查点策略

**检查点粒度**：一个功能点完成（无论成功还是失败）

**检查点触发时机**：

```typescript
async function saveCheckpoint(reason: "feature_done" | "feature_failed" | "soft_limit" | "hard_limit" | "manual") {
  const timestamp = new Date().toISOString();

  // 1. 更新 task-state.json（执行状态）
  await write_memory("task-state.json", {
    ...currentState,
    lastCheckpoint: timestamp,
    status: reason === "hard_limit" ? "interrupted" : "in_progress"
  });

  // 2. 更新 completed.json（功能点状态）
  await write_memory("memory/completed.json", {
    done: completedFeatures,
    pending: featureQueue,
    failed: failedFeatures,
    lastUpdated: timestamp
  });

  // 3. 更新 MEMORY.md（关键知识）
  if (currentFeature?.findings) {
    await write_memory("memory/MEMORY.md",
      appendSection(`\n### ${currentFeature.id}\n${currentFeature.findings}`)
    );
  }

  console.log(`Checkpoint saved: ${reason} at ${timestamp}`);
}
```

**检查点完整性验证**：

```typescript
async function verifyCheckpointIntegrity(): Promise<boolean> {
  try {
    const taskState = await read_memory("task-state.json");
    const completed = await read_memory("memory/completed.json");
    const memory = await read_memory("memory/MEMORY.md");

    // 验证三个文件都存在且可读
    if (!taskState || !completed || !memory) return false;

    // 验证 completed.json 中的 done 列表与 task-state.json 一致
    const stateDone = new Set(taskState.completedFeatures.map(f => f.id));
    const fileDone = new Set(completed.done);
    if (!setsEqual(stateDone, fileDone)) return false;

    return true;
  } catch {
    return false;
  }
}
```

## 4. 错误处理流程

**完整的错误处理决策树**：

```
工具调用失败
  ├── 是否可重试错误（网络超时、限流）？
  │   ├── 是 → 重试，最多 2 次，指数退避（1s, 2s）
  │   │        重试成功 → 继续
  │   │        重试失败 → 升级为功能点失败
  │   └── 否（文件不存在、权限不足）→ 直接升级为功能点失败
  │
功能点追齐失败
  ├── 记录到 failed 数组（含失败原因和建议）
  ├── 更新 task-state.json
  ├── 保存检查点
  └── 继续执行队列中下一个功能点
  │
检测到循环（连续 3 次相同工具调用）
  ├── 保存当前检查点（标记当前功能点需要重新执行）
  ├── 在 MEMORY.md 记录"循环原因分析"
  └── 强制终止当前 Session，等待人工确认后重启
  │
上下文达到软限制（30,000 tokens）
  ├── 压缩最近完成的 3 个功能点历史
  └── 继续执行
  │
上下文达到硬限制（80,000 tokens）
  ├── 保存检查点（reason: "hard_limit"）
  └── 终止当前 Session，等待下次恢复
```

**代码实现（核心循环）**：

```typescript
async function runAlignmentSession(): Promise<SessionReport> {
  // 初始化或恢复
  const context = await agentStartup();
  if (context.status === "all_done") {
    return { status: "completed", message: "所有功能点已追齐" };
  }

  const loopDetector = new LoopDetector();
  let tokenCount = estimateCurrentTokens();

  // 主执行循环
  while (context.featureQueue.length > 0) {
    const featureId = context.featureQueue[0];

    // 上下文监控
    tokenCount = estimateCurrentTokens();
    if (tokenCount > 80_000) {
      await saveCheckpoint("hard_limit");
      return { status: "interrupted", reason: "context_hard_limit" };
    }
    if (tokenCount > 30_000) {
      await compressRecentHistory(context.completedFeatures.slice(-3));
    }

    // 执行功能点
    try {
      await updateTaskState({ currentFeature: featureId, status: "in_progress" });
      
      const result = await alignFeatureWithLoopDetection(featureId, loopDetector);
      
      await markFeatureDone(featureId, result.summary);
      await saveCheckpoint("feature_done");
      
    } catch (error) {
      if (error.message === "LOOP_DETECTED") {
        await saveCheckpoint("manual");
        return { status: "interrupted", reason: "loop_detected" };
      }
      
      await markFeatureFailed(featureId, error.message);
      await saveCheckpoint("feature_failed");
      // 继续执行下一个
    }

    // 从队列移除已处理的功能点
    context.featureQueue.shift();
  }

  return {
    status: "completed",
    completed: context.completedFeatures.length,
    failed: context.failedFeatures.length
  };
}
```

## 5. 第五章总结

| 节次 | 核心主题 | 关键要点 |
|------|---------|---------|
| 5.1 | 任务分解与规划 | 混合策略：宏观预先规划，微观动态执行；分解粒度 ≤ 10 次工具调用 |
| 5.2 | 上下文维护 | 分段压缩（15:1 压缩比）；task-state.json 外部化状态；30K/80K 两级监控 |
| 5.3 | 检查点与断点续传 | 检查点粒度 = 一个功能点；三文件存储；幂等性保证安全重复执行 |
| 5.4 | 失败恢复与错误传播 | 4 级错误分类；错误不传播原则；循环检测与上下文重置 |
| 5.5 | 规划的上下文位置 | L1 稳定约定，L3 当前计划，L4 执行状态，External 跨 Session 进度 |
