# 5.3 检查点与断点续传：让长任务可中断、可恢复

> [!abstract] 本节概要
> 追齐 15 个功能点可能需要 2 小时，中途可能因为 API 超时、token 耗尽、或人为中断而失败。如果没有检查点机制，所有已完成的工作在下次启动时都需要重来——代价极高。**检查点（Checkpoint）**让这种中断可以从上次完成的位置继续，而不是从零开始。本节介绍检查点的设计原则、保存内容、以及断点续传的实现协议。

---

## 1. 什么是检查点

![[checkpoint-resume.excalidraw]]

检查点是**任务在某个安全状态时保存的完整快照**。"安全状态"指的是：任务处于一个自然完成点，而不是某个操作进行到一半的中间状态。

类比数据库的事务提交：检查点 = 一次成功的 COMMIT。你不会在事务执行到一半时保存检查点，因为那是一个不一致的状态。

**对追齐 Agent 来说**，自然完成点就是**一个功能点追齐完成之后**：

```
执行流程中的检查点时机：

  [开始] → 追齐功能点1 → [CP1 ✅] → 追齐功能点2 → [CP2 ✅]
         → 追齐功能点3 → [CP3 ✅] → 追齐功能点4...

  如果在追齐功能点4时中断：
  → 下次从 CP3 之后恢复
  → 功能点1、2、3 不需要重新执行
  → 只需要重新执行功能点4
```

**检查点的完整定义**：

一个检查点包含足够的信息，让 Agent 在任意时刻启动时能够准确知道：
1. **已经做了什么**（已完成的功能点列表）
2. **下一步要做什么**（待执行的功能点队列）
3. **到目前为止发现了什么**（关键知识，影响后续执行的决策依据）

## 2. 检查点应该保存什么

检查点信息分布在三个文件中：

### `memory/completed.json` — 已完成功能点列表

```json
{
  "page": "cart",
  "lastUpdated": "2024-01-15T11:23:00Z",
  "done": [
    {
      "id": "batch-delete",
      "completedAt": "2024-01-15T10:12:00Z",
      "summary": "App 长按触发多选，H5 已实现 useLongPress"
    },
    {
      "id": "quantity-edit",
      "completedAt": "2024-01-15T10:28:00Z",
      "summary": "防抖 300ms 已同步"
    },
    {
      "id": "coupon-display",
      "completedAt": "2024-01-15T11:23:00Z",
      "summary": "新增优惠券折叠展示逻辑"
    }
  ],
  "pending": [
    "settlement-detail",
    "empty-cart-state",
    "address-selection",
    "share-cart"
  ],
  "failed": []
}
```

### `task-state.json` — 当前进行中的功能点状态

```json
{
  "currentSession": {
    "startTime": "2024-01-15T10:00:00Z",
    "targetPage": "cart"
  },
  "currentFeature": null,
  "featureQueue": [
    "settlement-detail",
    "empty-cart-state",
    "address-selection",
    "share-cart"
  ],
  "lastCheckpoint": "2024-01-15T11:23:00Z",
  "status": "ready_to_resume"
}
```

注意：当一个功能点完成后，`currentFeature` 会被置为 `null`，表示处于检查点状态，可以安全中断。

### `memory/MEMORY.md` — 已发现的关键知识

```markdown
## 购物车页面 - 跨功能点关键发现

### App 架构特征
- App 购物车使用 Redux 管理状态，H5 使用 Zustand，两者逻辑等价但 API 不同
- 价格计算逻辑在 `utils/priceCalc.ts`，两端共用同一套算法，无需追齐

### 行为基准
- 多选模式：测试用例集 TC-CART-010 ~ TC-CART-020
- 数量编辑：TC-CART-030，注意最大数量限制为 99

### 已知差异（已追齐）
- 批量删除：✅ 已追齐
- 数量编辑防抖：✅ 已追齐
- 优惠券折叠：✅ 已追齐
```

**不需要保存的内容**：
- 中间推理过程（"我先想到了 A 方案，但后来发现 B 方案更好…"）
- 工具调用历史（原始 diff 输出、中间代码读取结果）
- 已经被压缩过的分析过程

检查点的目标是**最小化但充分**：恢复执行所需的所有信息，不多也不少。

## 3. 断点续传协议

Agent 启动时的第一件事，应该是检查是否有未完成的任务：

**断点续传启动流程**：

```typescript
async function agentStartup(): Promise<AgentContext> {
  // 第一步：读取任务状态
  const taskState = await read_memory("task-state.json");
  const memory = await read_memory("memory/MEMORY.md");

  if (!taskState || taskState.status === "completed") {
    // 全新启动
    return initializeNewSession();
  }

  // 第二步：检测是否有未完成任务
  const completed = await read_memory("memory/completed.json");

  if (completed.pending.length === 0) {
    return { status: "all_done", message: "所有功能点已追齐完成" };
  }

  // 第三步：生成恢复提示
  const resumePrompt = buildResumePrompt(taskState, completed, memory);

  return {
    status: "resuming",
    context: resumePrompt,
    nextFeature: completed.pending[0]
  };
}

function buildResumePrompt(
  taskState: TaskState,
  completed: CompletedState,
  memory: string
): string {
  return `
## 任务恢复

上次中断时间：${taskState.lastCheckpoint}
已完成功能点（${completed.done.length} 个）：
${completed.done.map(f => `  - ${f.id}: ${f.summary}`).join('\n')}

待追齐功能点（${completed.pending.length} 个）：
${completed.pending.map((f, i) => `  ${i + 1}. ${f}`).join('\n')}

跨功能点关键知识：
${memory}

---
请从第一个待追齐功能点开始继续执行。
  `.trim();
}
```

当 Agent 收到这个恢复提示时，它会：
1. 了解已完成的工作（不会重复执行）
2. 知道从哪里继续
3. 获得之前积累的关键知识（不需要重新探索）

## 4. 幂等性设计

断点续传有一个前提假设：如果 Agent 不小心**重复执行**了一个已完成的功能点，不会产生副作用。这就是**幂等性**。

**幂等性的定义**：对同一操作执行多次，结果与执行一次相同。

追齐操作**天然具备幂等性**：

```
假设 "批量删除" 功能已经追齐完成（H5 已有正确实现）。
如果 Agent 再次执行 "追齐批量删除"：
  → read_repo_file 读取 H5 代码 → 发现已经有 useLongPress
  → diff_feature 对比 App → 发现无差异
  → 结论：已追齐，无需修改
  → 不会产生重复代码或错误修改
```

这个特性让检查点机制非常安全：即使检查点保存不精确（比如保存时某个功能点刚好完成了一半），重新执行最多是多做一些冗余工作，不会造成数据损坏。

**不是天然幂等的操作需要特殊处理**：

```typescript
// 危险：直接追加，不幂等
async function appendToFile(path: string, content: string) {
  await fs.appendFile(path, content); // 重复调用会重复追加！
}

// 安全：先检查再写入，幂等
async function writeWithIdempotency(
  path: string,
  featureId: string,
  content: string
) {
  const existing = await fs.readFile(path, 'utf-8').catch(() => '');
  if (existing.includes(`<!-- feature:${featureId} -->`)) {
    return; // 已存在，跳过
  }
  await fs.writeFile(path, existing + `<!-- feature:${featureId} -->\n${content}`);
}
```

> [!tip] 检查点粒度原则
> **检查点粒度 = 一个自然完成单元（一个功能点追齐完成）**。
>
> - 太细（每步工具调用后保存）：频繁 I/O 开销，状态文件复杂度爆炸
> - 太粗（整个页面完成后才保存）：中断时损失全部工作
>
> 一个功能点是最自然的粒度——它足够小（5-10 次工具调用），又足够完整（有明确的完成标准）。

## 本节小结

- **检查点** = 任务在安全状态（功能点完成后）保存的最小充分快照
- 检查点分布在三个文件：`completed.json`（完成记录）、`task-state.json`（执行状态）、`MEMORY.md`（关键知识）
- **断点续传协议**：启动时检测未完成任务 → 生成恢复提示 → 跳过已完成部分继续执行
- 追齐操作天然幂等，重复执行安全无副作用
- 推荐检查点粒度：一个功能点

**下一节预告**：检查点解决了"中断后如何恢复"的问题，但中断的原因往往是某种错误。5.4 节将深入讨论长任务中的失败恢复与错误传播控制。
