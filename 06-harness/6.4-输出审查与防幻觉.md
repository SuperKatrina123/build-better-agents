# 6.4 输出审查与防幻觉：在生效前验证 LLM 的输出

> [!abstract] 核心认知
> LLM 会幻觉，这是事实，不是缺陷。在 Agent 系统里，幻觉不只是"说了不对的话"——它可能导致"做了不该做的事"。输出审查是在 LLM 的不确定性与真实世界的确定性后果之间建立的最后一道防线。

## 1. Agent 系统中的幻觉风险

普通 LLM 对话中，幻觉的代价是"用户需要纠正一个错误答案"。但在 Agent 系统中，幻觉可能触发工具调用，产生真实的副作用：

**追齐 Agent 的典型幻觉场景**：

**场景一：功能存在性幻觉**

LLM 声称"App 在购物车页面有批量分享功能"，但实际上 App 测试用例中没有这个功能描述。Agent 基于这个幻觉判断 H5 "缺少批量分享功能"，在 `memory/MEMORY.md` 中记录了一个虚假任务，后续开发团队为此花费了两天时间分析——最终发现该功能根本不存在。

**场景二：代码实现幻觉**

LLM 声称"H5 的 `h5-cart` 仓库中缺少 `BatchDeleteButton` 组件"，但实际上该组件存在于 `src/components/Cart/BatchActions.tsx` 中，只是命名不同。Agent 触发了不必要的 `diff_feature` 调用，生成了一个"新增组件"的 diff，浪费了一次代码审查资源。

**场景三：文件路径幻觉**

LLM 在引用代码时声称"根据 `src/pages/Cart/index.tsx` 第 87 行的逻辑"，但该文件实际只有 60 行。后续代码审查人员无法复现 LLM 的分析依据，降低了整个分析报告的可信度。

这三个场景揭示了同一个根本问题：**LLM 的记忆不可信，LLM 的"观察"需要工具验证**。

## 2. 输出类型与验证策略

不同类型的 LLM 输出对应不同的验证策略：

### 事实性输出

**类型**："这个组件存在吗"、"这个功能在 App 端有对应的测试用例吗"

**验证策略**：工具二次确认。不信任 LLM 的陈述，调用工具重新获取事实：

```typescript
// LLM 声称组件存在，用工具验证
async function verifyComponentExists(
  repo: string,
  componentName: string
): Promise<{ exists: boolean; actualPath?: string }> {
  const files = await tools.list_repo_files({ repo, dir: 'src/components' });
  const match = files.find(f =>
    f.toLowerCase().includes(componentName.toLowerCase().replace(/[^a-z0-9]/g, ''))
  );
  return {
    exists: !!match,
    actualPath: match,
  };
}
```

### 代码修改方案

**类型**：`diff_feature` 生成的代码修改建议

**验证策略**：格式校验 + 语法校验

```typescript
// Step 1: diff 格式校验
function validateDiffFormat(diffOutput: string): ValidationResult {
  const lines = diffOutput.split('
');
  const hasHeader = lines.some(l => l.startsWith('---') || l.startsWith('+++'));
  const hasHunk = lines.some(l => l.startsWith('@@'));
  const hasChanges = lines.some(l => l.startsWith('+') || l.startsWith('-'));

  if (!hasHeader || !hasHunk || !hasChanges) {
    return { valid: false, error: '输出不是有效的 unified diff 格式' };
  }
  return { valid: true };
}

// Step 2: 提取新增代码并做 TypeScript 语法检查
async function validateTypeScriptSyntax(code: string): Promise<ValidationResult> {
  const { exec } = await import('child_process');
  const tmpFile = `validate_${Date.now()}.ts`;
  await fs.writeFile(tmpFile, code, 'utf-8');
  return new Promise((resolve) => {
    exec(`npx tsc --noEmit --strict ${tmpFile}`, (error, _stdout, stderr) => {
      fs.unlink(tmpFile).catch(() => {});
      if (error) {
        resolve({ valid: false, error: stderr.slice(0, 500) });
      } else {
        resolve({ valid: true });
      }
    });
  });
}
```

### 进度更新输出

**类型**：`write_memory` 写入 `memory/completed.json` 的格式

**验证策略**：JSON schema 校验

```typescript
import Ajv from 'ajv';

const completedSchema = {
  type: 'object',
  properties: {
    version: { type: 'string' },
    lastUpdated: { type: 'string', format: 'date-time' },
    completed: {
      type: 'array',
      items: {
        type: 'object',
        required: ['featureId', 'repo', 'completedAt'],
        properties: {
          featureId: { type: 'string' },
          repo: { type: 'string' },
          completedAt: { type: 'string', format: 'date-time' },
          diffPath: { type: 'string' },
        },
      },
    },
  },
  required: ['version', 'lastUpdated', 'completed'],
};

const ajv = new Ajv({ allErrors: true });
const validateCompleted = ajv.compile(completedSchema);

function validateCompletedJson(data: unknown): ValidationResult {
  const valid = validateCompleted(data);
  if (!valid) {
    return { valid: false, error: ajv.errorsText(validateCompleted.errors) };
  }
  return { valid: true };
}
```

## 3. 结构化输出作为防幻觉手段

要求 LLM 以结构化 JSON 输出，而不是自然语言叙述，是降低幻觉影响范围的有效手段。

**原理**：结构化输出迫使 LLM 把判断拆分为可以独立验证的字段。每个字段的值都可以单独校验，幻觉被"限制"在单个字段内，不会扩散到整个判断链。

```typescript
// 不好的输出格式：自然语言，难以验证
// "经过分析，我认为 h5-cart 仓库中缺少批量删除功能，
//  该功能在 App 端的购物车测试用例第 23-45 行有详细描述，
//  建议在 src/components/Cart/BatchDelete.tsx 中实现..."

// 好的输出格式：结构化，每个字段可独立验证
interface FeatureGapAnalysis {
  featureId: string;           // 可与 feature spec 文件交叉验证
  appTestCasePath: string;     // 可用 read_repo_file 验证文件存在性
  appTestCaseLines: string;    // 可验证行号在文件范围内
  h5Status: 'missing' | 'partial' | 'implemented';
  h5ExistingImpl?: string;     // 若 partial/implemented，引用具体路径，可验证
  confidence: 'high' | 'medium' | 'low';  // LLM 自评置信度
  suggestedFiles: string[];    // 建议修改的文件列表，可验证文件存在性
}
```

## 4. 降低幻觉率的 Prompt 技巧

输出审查是事后补救，降低幻觉率才是主动防御。在 System Prompt 中明确约定以下规则：

### 要求引用具体位置

```
规则：在分析代码时，所有结论必须引用具体的文件路径和行号。
格式：[结论] 依据：[repo]/[path]:[lineStart]-[lineEnd]
示例：H5 缺少批量删除按钮。依据：h5-cart/src/pages/Cart/index.tsx:1-85（全文无 BatchDelete 相关代码）
```

### 禁止模糊表述

```
禁止使用以下表述（在输出中发现时视为分析无效）：
- "可能"、"也许"、"大概"、"据我所知"
- "通常情况下"、"一般来说"
- "应该"（在描述代码现状时）
替代方式：如果不确定，明确说明"需要验证"，并给出验证方法
```

### 要求置信度自评

要求 LLM 对每个关键判断进行置信度自评，低置信度的判断自动触发工具验证流程，而不是直接进入执行阶段。

> [!tip] 核心原则
> **不信任 LLM 的记忆，信任工具的返回结果。**
>
> 当 LLM 说"文件存在"，用工具验证；当 LLM 说"这行代码是这样写的"，用工具读取那一行。任何影响后续决策的关键事实，都应该通过工具重新获取，而不是依赖 LLM 的回忆。这不是对 LLM 不信任，这是架构上的健壮性设计。

---

**本节小结**：
- Agent 系统中的幻觉风险：功能存在性幻觉、代码实现幻觉、文件路径幻觉——每种都可能触发错误的工具调用
- 不同输出类型对应不同验证策略：事实性输出用工具二次确认，代码修改用格式+语法校验，进度更新用 JSON schema 验证
- 结构化输出是防幻觉的主动手段，把模糊判断拆分为可独立验证的字段
- Prompt 层要求引用具体位置、禁止模糊表述、要求置信度自评

**下一节预告**：输出审查解决了"输出正确性"问题，但 Agent 系统还有另一个重要问题：如果 Agent 不停地运行、不停地消耗资源，如何防止失控？
