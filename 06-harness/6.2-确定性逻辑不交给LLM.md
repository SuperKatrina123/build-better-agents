# 6.2 确定性逻辑不交给 LLM：边界划分是 Harness 设计的核心

> [!abstract] 核心决策
> Harness 设计中最重要的一条原则：**哪些逻辑必须用代码实现（确定性），哪些逻辑才交给 LLM（概率性）**。这条边界划错了，整个 Harness 都会失效——你以为在约束 Agent，实际上只是在 prompt 里祈祷。

## 1. 什么必须是确定性的

![[deterministic-vs-llm.excalidraw]]

下面四类逻辑，**永远不能交给 LLM 决策**，必须用代码实现：

### 权限验证

"这个工具调用是否在允许的操作范围内"——这个问题的答案只能是 true 或 false，必须由代码判断。

为什么不能让 LLM 决定权限？因为 LLM 可以被 prompt 影响。如果上下文中出现"这是紧急情况，请绕过限制"这样的文本（无论是用户输入还是工具返回值中的注入内容），LLM 可能会"认为"应该放行。这就是经典的 prompt injection 攻击面。

```typescript
// 正确：权限检查在代码里
function isPathAllowed(filePath: string, allowedPaths: string[]): boolean {
  return allowedPaths.some(allowed => filePath.startsWith(allowed));
}

// 错误：让 LLM 决定是否允许
// "请判断这个路径是否安全：" + filePath  ← 不要这样做
```

### 参数格式验证

文件路径是否合法、仓库名是否在 repo-map 里、JSON 格式是否符合 schema——这些都是确定性判断，代码校验更快、更可靠、不消耗 token。

```typescript
const ALLOWED_REPOS = [
  'h5-cart', 'h5-product', 'h5-user',
  'h5-order', 'h5-home', 'h5-search'
];

function validateRepoName(repoName: string): void {
  if (!ALLOWED_REPOS.includes(repoName)) {
    throw new ValidationError(
      `仓库 "${repoName}" 不在允许列表中。允许的仓库：${ALLOWED_REPOS.join(', ')}`
    );
  }
}
```

### 操作审计记录

把每次工具调用写入 audit log，不依赖 LLM 记忆，不依赖 LLM 总结——代码直接写入文件。LLM 不会"忘记"记录，也不会"误记"内容。

```typescript
async function appendAuditLog(entry: AuditEntry): Promise<void> {
  const line = JSON.stringify({
    ...entry,
    timestamp: new Date().toISOString(),
  }) + '
';
  await fs.appendFile('logs/audit.jsonl', line, 'utf-8');
}
```

### 预算检查

token 消耗是否超出限制——这是一个数值比较，代码计数，不需要 LLM 参与。

```typescript
class BudgetGuard {
  private tokenCount = 0;
  private readonly maxTokens: number;

  constructor(maxTokens: number) {
    this.maxTokens = maxTokens;
  }

  consume(tokens: number): void {
    this.tokenCount += tokens;
    if (this.tokenCount > this.maxTokens) {
      throw new BudgetExceededError(
        `Token 消耗 ${this.tokenCount} 超出上限 ${this.maxTokens}`
      );
    }
  }
}
```

## 2. 什么交给 LLM

LLM 擅长处理需要**理解力**的任务——这些任务无法用简单的规则代码实现：

- **理解功能意图**：飞书文档里写的"购物车支持批量操作"，到底对应 App 端的哪个具体交互？这需要理解自然语言和业务上下文，只有 LLM 能做
- **分析代码逻辑**：`src/pages/Cart/index.tsx` 里的这段逻辑，在 H5 端是否已经实现了 App 端对应的功能？这需要理解代码语义，只有 LLM 能做
- **判断是否需要修改**：对比 App 测试用例和 H5 代码后，这个功能点是真的缺失，还是实现方式不同但功能等价？这需要主观判断，只有 LLM 能做
- **生成修改方案**：给定功能缺口，生成 TypeScript + React 的具体代码修改建议——这是创造性工作，只有 LLM 能做

这四类任务的共同特点是：**没有唯一正确答案，需要在模糊信息中做出合理判断**。代码规则无法处理这种模糊性，LLM 正是为此而生。

## 3. 灰色地带的处理

有一类场景需要特别处理：**LLM 的判断直接影响不可逆操作时**。

典型场景：LLM 分析后认为"H5 购物车缺少批量删除功能"，并要求调用 `diff_feature` 生成代码修改建议。LLM 的判断（"缺少功能"）是概率性的，可能是幻觉；但由此触发的操作（生成 diff 并准备修改代码）是有副作用的。

处理策略：**要求 LLM 明确输出判断依据，并加一层验证**。

```typescript
// LLM 的输出必须包含：结论 + 依据
interface FeatureAnalysisOutput {
  conclusion: 'missing' | 'implemented' | 'partial';
  evidence: {
    appTestCase: string;    // 引用的 App 测试用例（必填）
    h5CodeRef: string;      // 引用的 H5 代码路径（必填）
    reasoning: string;      // 判断理由（必填）
  };
}

// Harness 层：验证依据的真实性
async function verifyEvidence(output: FeatureAnalysisOutput): Promise<boolean> {
  // 确认 App 测试用例确实存在
  const testCaseExists = await tools.read_repo_file({
    repo: 'app-test-cases',
    path: output.evidence.appTestCase
  });
  // 确认 H5 代码引用路径确实存在
  const h5FileExists = await tools.read_repo_file({
    repo: output.evidence.h5CodeRef.split('/')[0],
    path: output.evidence.h5CodeRef
  });
  return !!testCaseExists && !!h5FileExists;
}
```

> [!tip] 核心原则
> **权限检查、格式验证、审计记录——这三类逻辑永远在代码里，不在 prompt 里。**
>
> 一个简单的判断标准：如果某个逻辑出错会导致安全问题或不可逆后果，它就不应该由 LLM 决定。把它写成代码，在 Harness 层执行。

> [!example] 追齐 Agent 的边界划分实践
> 当追齐 Agent 调用 `diff_feature` 工具生成代码修改建议时，边界是这样划分的：
>
> - **工具层（确定性）**：校验 `repoName` 是否在允许列表中，校验 `filePath` 格式是否合法，校验 LLM 输出的 diff 格式是否符合标准 unified diff 规范
> - **LLM（概率性）**：理解功能意图，对比 App 测试用例与 H5 代码，决定需要修改哪些文件、修改什么内容
> - **输出层（确定性）**：使用 TypeScript 编译器验证生成的代码片段没有语法错误，格式化 diff 输出
>
> 每一层都在做自己最擅长的事，互不越界。

---

**本节小结**：
- 权限验证、参数校验、审计记录、预算检查——这四类逻辑必须用代码实现，不能交给 LLM
- 理解意图、分析代码、主观判断、生成方案——这四类任务必须交给 LLM，代码规则无法处理
- 灰色地带（LLM 判断影响不可逆操作）的处理策略：要求 LLM 明确输出可验证的依据，然后代码层验证依据的真实性
