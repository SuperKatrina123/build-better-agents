# 6.5 成本与循环控制：防止 Agent 失控消耗资源

> [!abstract] 核心问题
> 没有约束的 Agent 在生产环境会失控——无限重试、确认循环、探索爆炸、token 瀑布。成本控制不是 Harness 的附加功能，它是防止 Agent 从"自动化助手"变成"资源黑洞"的核心保障。

## 1. Agent 失控的常见模式

在功能追齐 Agent 的实际运行中，以下几种失控模式最为常见：

### 模式一：无限重试循环

```
工具调用失败
  → LLM 判断"需要重试"
  → 工具调用再次失败（同样的原因）
  → LLM 判断"需要重试"
  → ... (循环直到 token 耗尽)
```

触发条件：网络超时、文件不存在、格式错误——任何工具失败。LLM 的默认倾向是"再试一次"，没有退出条件时会无限循环。

### 模式二：确认循环

```
Agent 完成分析 → 询问"是否继续？"
用户/系统没有明确响应
  → Agent 再次询问"请确认是否继续"
  → 再次等待
  → ... (循环直到超时)
```

触发条件：Interactive Mode 下的确认节点没有超时和默认值设计。

### 模式三：探索爆炸

```
Agent 需要了解某个仓库的结构
  → list_repo_files(repo, 'src')
  → 发现 50 个子目录，逐一 list_repo_files
  → 每个子目录又有子目录
  → ... (指数级增长，直到 token 耗尽)
```

触发条件：大型仓库 + 没有深度限制 + LLM 认为"需要完整了解结构"。

### 模式四：Token 瀑布

```
读取了一个超大文件（10,000 行）
  → 文件内容填满上下文
  → 其他工具调用的返回值也被截断
  → LLM 在不完整信息下做出错误判断
  → 触发更多工具调用试图补全信息
  → 上下文越来越大，token 消耗指数增长
```

## 2. 五种限制手段

### 手段一：最大工具调用次数

每个子任务设置工具调用上限，超出时强制中断：

```typescript
class ToolCallBudget {
  private count = 0;
  constructor(private readonly max: number) {}

  tick(toolName: string): void {
    this.count++;
    if (this.count > this.max) {
      throw new BudgetExceededError(
        `工具调用次数 ${this.count} 超出每子任务上限 ${this.max}。` +
        `最后调用：${toolName}`
      );
    }
  }

  remaining(): number {
    return Math.max(0, this.max - this.count);
  }
}

// 追齐 Agent：每个功能点的分析最多 30 次工具调用
const budget = new ToolCallBudget(30);
```

### 手段二：最大 Token 消耗

追踪累计 token 消耗，超出会话预算时中断：

```typescript
class TokenBudget {
  private used = 0;
  constructor(private readonly max: number) {}

  consume(inputTokens: number, outputTokens: number): void {
    this.used += inputTokens + outputTokens;
    if (this.used > this.max) {
      throw new TokenBudgetExceededError(
        `Token 消耗 ${this.used} 超出会话上限 ${this.max}`
      );
    }
  }

  utilizationRate(): number {
    return this.used / this.max;
  }
}

// 追齐 Agent：每次会话最多 80,000 tokens
const tokenBudget = new TokenBudget(80_000);
```

### 手段三：循环检测

检测连续相同的工具调用，防止无意义的重试循环：

```typescript
class LoopDetector {
  private recentCalls: string[] = [];
  private readonly windowSize = 6;
  private readonly maxRepeat = 3;

  record(toolName: string, paramsHash: string): void {
    const key = `${toolName}:${paramsHash}`;
    this.recentCalls.push(key);
    if (this.recentCalls.length > this.windowSize) {
      this.recentCalls.shift();
    }
  }

  isLooping(): { looping: boolean; pattern?: string } {
    if (this.recentCalls.length < this.maxRepeat) return { looping: false };
    const recent = this.recentCalls.slice(-this.maxRepeat);
    const allSame = recent.every(c => c === recent[0]);
    return allSame
      ? { looping: true, pattern: recent[0] }
      : { looping: false };
  }
}

// 在 Agent Loop 中使用
const detector = new LoopDetector();

async function runAgentStep(toolCall: ToolCall): Promise<void> {
  const hash = hashParams(toolCall.params);
  detector.record(toolCall.name, hash);

  const loopStatus = detector.isLooping();
  if (loopStatus.looping) {
    throw new LoopDetectedError(
      `检测到重复调用模式：${loopStatus.pattern}。` +
      `Agent 可能陷入循环，自动中断。`
    );
  }
  // ... 正常执行工具调用
}
```

### 手段四：单次工具调用超时

防止工具调用因网络问题或服务异常而无限等待：

```typescript
async function executeWithTimeout<T>(
  fn: () => Promise<T>,
  timeoutMs: number,
  toolName: string
): Promise<T> {
  const timeoutPromise = new Promise<never>((_, reject) =>
    setTimeout(() => reject(new TimeoutError(`${toolName} 超时 (${timeoutMs}ms)`)), timeoutMs)
  );
  return Promise.race([fn(), timeoutPromise]);
}

// 追齐 Agent：单次工具调用最多等待 30 秒
const result = await executeWithTimeout(
  () => tools.read_repo_file(params),
  30_000,
  'read_repo_file'
);
```

### 手段五：费用上限

追踪累计 API 费用，超出任务预算时中断：

```typescript
class CostBudget {
  private costUsd = 0;
  // 基于 Claude claude-sonnet-4.5 定价（仅供示例，请以实际为准）
  private readonly pricing = {
    inputPerMToken: 3.0,
    outputPerMToken: 15.0,
  };

  consume(inputTokens: number, outputTokens: number): void {
    const cost =
      (inputTokens / 1_000_000) * this.pricing.inputPerMToken +
      (outputTokens / 1_000_000) * this.pricing.outputPerMToken;
    this.costUsd += cost;
    if (this.costUsd > this.maxCostUsd) {
      throw new CostExceededError(
        `累计费用 $${this.costUsd.toFixed(4)} 超出任务上限 $${this.maxCostUsd}`
      );
    }
  }

  constructor(private readonly maxCostUsd: number) {}
}

// 追齐 Agent：每次任务最多 $3
const costBudget = new CostBudget(3.0);
```

## 3. 优雅降级而非硬中断

触发任意一个限制时，Agent 不应该直接抛出错误退出——那样会丢失已完成的工作。正确的做法是**优雅降级**：

```typescript
async function runAgentWithGracefulDegradation(
  task: AgentTask
): Promise<AgentResult> {
  const checkpoint = new CheckpointManager('memory/checkpoint.json');

  try {
    return await runAgentLoop(task, {
      toolBudget: new ToolCallBudget(30),
      tokenBudget: new TokenBudget(80_000),
      loopDetector: new LoopDetector(),
    });

  } catch (err) {
    if (err instanceof BudgetExceededError || err instanceof LoopDetectedError) {
      // Step 1: 保存当前进度
      await checkpoint.save({
        completedFeatures: task.completedSoFar,
        pendingFeatures: task.remainingFeatures,
        interruptReason: err.message,
        interruptedAt: new Date().toISOString(),
      });

      // Step 2: 生成部分完成报告
      return {
        status: 'partial',
        completed: task.completedSoFar,
        pending: task.remainingFeatures,
        summary: `任务因资源限制中断。已完成 ${task.completedSoFar.length} 个功能点，` +
                 `剩余 ${task.remainingFeatures.length} 个待处理。` +
                 `中断原因：${err.message}`,
        checkpointPath: 'memory/checkpoint.json',
      };
    }
    throw err; // 其他错误正常抛出
  }
}
```

## 4. 监控与报警

每次 Agent 运行后，生成标准化的 metrics 报告：

```typescript
interface AgentRunMetrics {
  sessionId: string;
  startTime: string;
  endTime: string;
  durationMs: number;
  toolCallCount: number;
  tokenConsumed: number;
  costUsd: number;
  featuresCompleted: number;
  featuresFailed: number;
  exitReason: 'success' | 'budget_exceeded' | 'loop_detected' | 'error';
}

// 超出正常范围时发送告警
function checkAndAlert(metrics: AgentRunMetrics): void {
  const alerts: string[] = [];
  if (metrics.toolCallCount > 25) alerts.push(`工具调用次数偏高：${metrics.toolCallCount}`);
  if (metrics.tokenConsumed > 60_000) alerts.push(`Token 消耗偏高：${metrics.tokenConsumed}`);
  if (metrics.costUsd > 2.0) alerts.push(`费用偏高：$${metrics.costUsd.toFixed(4)}`);
  if (metrics.exitReason !== 'success') alerts.push(`非正常退出：${metrics.exitReason}`);

  if (alerts.length > 0) {
    console.warn('[Agent Alert]', alerts.join(' | '));
    // 写入告警日志，可接入飞书消息推送
    appendAuditLog({ tool: 'metrics_alert', params: { alerts, metrics }, result: 'success' });
  }
}
```

> [!example] 真实案例：探索爆炸的循环检测
> 追齐 Agent 在分析 `h5-product` 仓库时，LLM 判断需要"了解完整的目录结构"，开始逐层遍历：`list_repo_files(h5-product, 'src')` → `list_repo_files(h5-product, 'src/components')` → `list_repo_files(h5-product, 'src/components/Product')` → ...
>
> 在第 18 次调用时，循环检测器注意到 `list_repo_files:h5-product` 在最近 6 次调用中出现了 3 次（参数在变化，但工具名+仓库名的组合重复超阈值），自动触发中断。
>
> Agent 保存当前状态，输出报告："仓库探索未完成，已遍历 src/ 下 3 层目录，建议手动指定目标路径后重启任务。"开发人员看到报告后，在任务配置中指定了目标路径 `src/pages/ProductDetail`，下次运行直接聚焦，一次成功。

**本节小结**：
- Agent 失控的四种模式：无限重试、确认循环、探索爆炸、Token 瀑布
- 五种限制手段：工具调用次数上限、Token 预算、循环检测、单次超时、费用上限
- 触发限制时优雅降级：保存检查点 → 生成部分报告 → 退出（而不是硬中断丢失进度）
- 每次运行后生成 metrics，超阈值触发告警
