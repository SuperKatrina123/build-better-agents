# 6.6 设计决策 6：功能追齐 Agent Harness 完整规划

本节是第六章的收束节点。我们将把前五节的所有 Harness 设计决策汇总为追齐 Agent 的完整控制规划，形成可以直接落地实施的参考文档。

## 1. Harness 全景

追齐 Agent 的四层 Harness 控制结构在实际系统中的具体落地：

### 环境层（基础设施保障）

| 控制项 | 实现方式 | 保护目标 |
|--------|----------|----------|
| 文件系统隔离 | Agent 进程以受限用户身份运行，只有 `./h5-repos/`、`./memory/`、`./logs/` 的读写权限 | 防止越权访问系统文件或 App 源码 |
| 网络访问控制 | 只允许访问 GitHub API、内部 repo API 和 LLM API，其他域名拒绝 | 防止数据外泄和意外的第三方 API 调用 |
| 资源上限 | 单进程 CPU 时间 300 秒，内存 2GB | 防止失控任务耗尽系统资源 |

### 工具层（代码权限边界）

工具层是 Harness 的核心实施层，每个工具的权限精确匹配其功能需求（详见第 2 节权限矩阵）。

### 协议层（System Prompt 约定）

以下约定写入 System Prompt，作为行为引导（不作为安全边界）：

```
核心约定：
1. 所有关于代码现状的结论，必须引用具体文件路径和行号
2. 不使用"可能"、"大概"、"据我所知"等模糊表述；若不确定，明确说明"需要工具验证"
3. 修改建议以标准 unified diff 格式输出，包含完整的文件路径标识
4. 在更新 memory/MEMORY.md 前，必须先验证引用的功能点确实存在于 App 测试用例中
5. 每次子任务完成后，将结果写入 memory/completed.json，格式符合 schema 要求
```

### 审查层（输出生效前的最后验证）

| 输出类型 | 验证方式 | 失败处理 |
|----------|----------|----------|
| 功能差异分析 | 工具二次确认事实性声明 | 标记为"待验证"，不进入执行队列 |
| 代码修改建议 | diff 格式校验 + TS 语法检查 | 返回错误，要求 LLM 重新生成 |
| 记忆写入 | JSON schema 校验 | 拒绝写入，记录 audit log |
| 任务完成标记 | 确认 diff 文件路径存在 | 不允许标记完成 |

## 2. 工具权限矩阵

| 工具名 | 读路径范围 | 写路径范围 | 需要人工确认 | 记录 Audit Log |
|--------|-----------|-----------|-------------|----------------|
| `read_repo_file` | 所有 H5 仓库（`h5-*`），禁止读 App 仓库 | 无 | 否 | 否（只读，免记录） |
| `list_repo_files` | 所有 H5 仓库，最大深度 3 层 | 无 | 否 | 否 |
| `read_memory` | `memory/` 目录下所有文件 | 无 | 否 | 否 |
| `diff_feature` | 所有 H5 仓库（用于分析） | 只输出 diff 字符串，不写文件 | Interactive: 是；Automated: 否 | 是（记录分析结果摘要） |
| `resolve_feature_spec` | `memory/specs/` | `memory/specs/` | 否 | 是 |
| `write_memory` | 无 | 仅 `memory/` 目录（含子目录） | Interactive: 高风险操作是；Automated: 否 | 是（每次写入全记录） |

**明确禁止的操作（代码层硬拒绝，无法通过 prompt 绕过）**：
- 写入任何 `h5-*` 仓库目录（`diff_feature` 只返回建议，不写文件）
- 读取或写入 App 测试用例仓库（该仓库是行为基准，只读也不行）
- 执行任何 shell 命令（包括 git）
- 写入 `logs/audit.jsonl` 之外的日志文件（audit log 只能通过 Harness 中间件写入）

## 3. 确定性控制清单

### 代码层实现（不可绕过）

- [ ] 仓库名白名单校验：`ALLOWED_H5_REPOS` 常量，`validateRepoName()` 函数
- [ ] 文件路径安全校验：防路径遍历攻击，`isPathAllowed()` 函数
- [ ] 写路径白名单：`WRITABLE_PATHS` 映射，`enforceWritePath()` 函数
- [ ] 工具调用次数计数：`ToolCallBudget` 类，每子任务上限 30 次
- [ ] Token 消耗追踪：`TokenBudget` 类，每会话上限 80,000 tokens
- [ ] 循环检测：`LoopDetector` 类，连续 3 次相同调用触发中断
- [ ] 单次工具超时：`executeWithTimeout()` 包装，30 秒
- [ ] 费用上限：`CostBudget` 类，每任务上限 $3
- [ ] Audit log 写入：`appendAuditLog()` 函数，写入 `logs/audit.jsonl`
- [ ] 优雅降级：`CheckpointManager` + `runAgentWithGracefulDegradation()`

### System Prompt 层约定（软性引导，不作为安全边界）

- 引用规范：所有代码结论必须附带文件路径和行号
- 禁用词列表：不允许使用模糊表述
- 输出格式：diff 使用 unified format，进度更新使用 JSON
- 验证先行：在记录功能缺失前，先用工具确认 App 测试用例的存在

## 4. 预算与限制参数

以下为追齐 Agent 的具体限制参数，基于实际运行数据校准：

| 参数 | 值 | 说明 |
|------|-----|------|
| 每功能点工具调用上限 | 30 次 | 分析一个功能点的合理探索次数，超出说明陷入循环 |
| 每会话 Token 上限 | 80,000 tokens | 覆盖 5-8 个中等复杂度功能点的完整分析 |
| 每任务费用上限 | $3.00 | 基于 claude-sonnet-4.5 定价，对应约 200K tokens 消耗 |
| 单次工具调用超时 | 30 秒 | 网络请求合理超时，远低于 Agent Loop 的感知阈值 |
| 循环检测窗口 | 最近 6 次调用中重复 3 次 | 平衡灵敏度与误报率 |
| 文件读取大小上限 | 100 KB | 防止大文件填满上下文窗口 |
| 目录遍历最大深度 | 3 层 | 足以覆盖 H5 仓库的典型目录结构 |
| 并发子任务数 | 3 个 | Orchestrator 同时分发给 3 个 Analysis Agent |

## 5. 第六章总结

| 章节 | 核心内容 | 关键决策 |
|------|----------|----------|
| 6.1 什么是 Harness | Harness 的定义、四层结构、追齐 Agent 的需求 | Harness 是让 Agent 更大胆执行的保障，不是枷锁 |
| 6.2 确定性逻辑不交给 LLM | 权限/校验/审计用代码；意图/分析/生成用 LLM | 边界划分错误会让整个 Harness 失效 |
| 6.3 工具权限设计 | 最小权限原则、三层写入控制、中间件实现 | 权限控制在代码层，System Prompt 不是安全边界 |
| 6.4 输出审查与防幻觉 | 三类验证策略、结构化输出、Prompt 约定 | 不信任 LLM 记忆，信任工具返回结果 |
| 6.5 成本与循环控制 | 五种限制机制、优雅降级、监控报警 | 触发限制时保存进度优雅退出，而非硬中断 |
