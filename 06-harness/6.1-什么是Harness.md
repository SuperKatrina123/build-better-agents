# 6.1 什么是 Harness：给 Agent 装上安全围栏

> [!abstract] 核心定义
> Harness 是 Agent 系统的"安全底座"——它定义了 Agent 能做什么、不能做什么，以及出错时如何兜底。Harness 不是限制 Agent 的枷锁，而是让 Agent 可以更大胆地执行的保障体系。有了 Harness，Agent 不必在每一步都自我怀疑，它知道越界行为会被拦截。

> [!question] 带着问题阅读
> 读这一节之前，先想三个问题：
> 1. 如果 Agent 误删了一个关键文件，你有办法快速恢复或追溯吗？
> 2. 你能在事后审查 Agent 昨天做了哪些具体操作吗？
> 3. 当 Agent 陷入无限循环、不断调用同一个工具时，系统会自动中止吗？
>
> 如果有任何一个答案是"不确定"，那么你的 Agent 系统就缺少 Harness。

## 1. Harness 的定义

Harness 这个词来自赛马和攀岩：它是套在身上的安全装具，让骑手或攀岩者可以全力发挥，同时在意外发生时提供保护。在 Agent 系统中，Harness 扮演同样的角色——它包裹在 Agent Loop 外部，提供系统级的安全保障。

**Harness = 包裹 Agent Loop 的控制框架**，包含五个核心组件：

1. **输入验证**：在 Agent 处理任何任务之前，检查输入的合法性（格式、权限、范围是否在允许列表内）
2. **权限控制**：限制 Agent 可以调用哪些工具、操作哪些资源路径
3. **行为约束**：通过 System Prompt 与代码层的双重机制约束 Agent 的行为边界
4. **输出审查**：在 Agent 的操作真正生效之前，对输出进行格式验证或人工确认
5. **错误兜底**：当 Agent 出现异常时，执行预定义的降级逻辑与状态保存

### Harness 与 Agent Loop 的关系

```
┌─────────────────────── Harness ───────────────────────┐
│                                                        │
│   输入验证                               输出审查      │
│      ↓                                     ↑          │
│  ┌──────────────── Agent Loop ───────────────┐        │
│  │  感知 → 规划 → 工具调用 → 观察 → 再规划  │        │
│  └───────────────────────────────────────────┘        │
│      ↑                                     ↓          │
│   权限控制                               错误兜底      │
│                   行为约束（贯穿全程）                  │
└────────────────────────────────────────────────────────┘
```

理解 Harness 的关键洞察是：**Harness 不是让 Agent 更弱，而是让 Agent 可以更大胆地执行**。当系统存在越界拦截机制时，Agent 可以减少不必要的犹豫和反复确认，专注于执行任务本身。

## 2. 为什么需要 Harness

没有 Harness 的 Agent 在生产环境面临三个核心风险：

### 风险一：操作不可逆

Agent 可以执行文件写入、数据库更新、外部 API 调用等操作，这些操作一旦完成，往往难以回滚。如果 Agent 因路径解析错误删除了源代码目录，损失可能是永久性的。

- **没有 Harness**：Agent 执行"清理旧版本文件"时，错误地删除了仍在使用的组件目录——操作已发生，无法撤销
- **有 Harness**：路径白名单检查在执行前拦截了这次调用，操作从未发生，错误写入 audit log 并触发告警

### 风险二：行为不可预测

LLM 是概率性模型。同样的 prompt 在不同的运行时可能因为温度参数、上下文长度、模型版本差异而产生不同的输出。测试环境运行良好的 Agent，在生产环境可能因为微小的上下文差异做出意外决策。这不是 LLM 的 bug，而是其本质特性。Harness 在这种不确定性的外围加一层确定性护栏。

### 风险三：成本失控

无约束的 Agent 可能进入无限循环：工具调用失败后不断重试，或在大型代码库中无限制地遍历文件列表。每次工具调用消耗 token，每次 API 请求产生费用。没有预算控制的 Agent 在极端情况下可以在几分钟内产生数百美元的费用。

### 追齐 Agent 的具体风险

在功能追齐 Agent 的场景中，这三个风险都真实存在：

- **操作不可逆**：`diff_feature` 工具生成的代码修改建议如果被直接应用，错误的修改可能破坏 H5 页面的现有功能，影响线上用户
- **行为不可预测**：Agent 可能把 H5 某个仓库中的一个组件误认为"App 独有功能尚未实现"，从而错误地在待办列表里生成一个虚假任务，浪费开发资源
- **成本失控**：追齐 Agent 覆盖 10-15 个代码仓库，如果没有限制，可能对每个仓库都进行无限深度的文件探索，产生大量不必要的 token 消耗

## 3. Harness 的四个层次

![[harness-layers.excalidraw]]

Harness 由外到内分为四个层次，每层有不同的职责和实现方式：

### 环境层（最外层）

环境层是最底层的物理约束，由操作系统和基础设施提供，**不可绕过**：

- **文件系统权限**：Agent 进程只有指定目录的读写权限，即使业务代码出现漏洞，也无法访问系统文件或其他用户数据
- **网络访问控制**：通过防火墙规则限制 Agent 可以访问的外部 API 和域名，防止数据泄露
- **资源上限**：CPU、内存、并发连接数的硬性限制，防止单个 Agent 任务耗尽系统资源

### 工具层

工具层在代码中实现每个工具的权限边界，是 Harness 设计的主要战场（详见第 6.3 节）：
- 工具的输入参数验证（路径格式校验、仓库名白名单检查）
- 工具的操作范围限制（`write_memory` 只能写 `memory/` 目录，不能写代码仓库）
- 工具副作用的审计记录（每次写操作追加到 `logs/audit.jsonl`）

### 协议层

协议层通过 System Prompt 约定 Agent 的行为规范，是软性约束：
- 哪些操作需要先说明理由（引用具体文件和行号）
- 输出格式的要求（结构化 JSON、标准 diff 格式）
- 禁止使用的模糊表述（"可能"、"大概"、"据我所知"）

### 审查层（最内层）

审查层在操作生效前做最后验证：
- **Interactive Mode**：生成 diff 供人工确认后再应用
- **Automated Mode**：自动校验输出格式（TypeScript 语法检查、JSON schema 验证）

_高风险高价值的操作（如代码修改）需要完整的四层 Harness 保护；低风险操作（如读取文件）只需工具层的基本校验。_

## 4. 追齐 Agent 的 Harness 需求

功能追齐 Agent 的核心操作是分析代码差异并生成修改建议，这是一个涉及多仓库读写的复杂任务。根据操作风险高低，Harness 需求如下：

**低风险只读操作（工具层基本校验即可）**：
- `read_repo_file`：读取仓库文件，无副作用，只需验证路径格式和仓库名是否在允许列表中
- `list_repo_files`：列出文件列表，无副作用，但需要防止无限递归（最大深度限制）
- `read_memory`：读取记忆文件，只需验证路径在 `memory/` 目录下

**中高风险写入操作（需要三层控制）**：
- `diff_feature`：生成代码修改方案，输出需要 diff 格式校验，Interactive Mode 下需要人工确认
- `write_memory`：写入进度记忆，只允许写 `memory/` 目录，格式需要 JSON schema 验证
- `resolve_feature_spec`：解析功能规格，输出需要结构化格式校验

**Harness 明确禁止的操作（代码层硬性拒绝）**：
- 直接修改 H5 代码仓库的源文件（只能生成 diff 建议，不能直接写入）
- 执行 git commit 或 git push 操作
- 修改 App 测试用例文件（这是行为基准，不能由 Agent 修改）

这些禁止操作不依赖 prompt 约定，而是在工具层和环境层以代码方式硬性实现——即使 LLM 强烈要求执行这些操作，系统也会拒绝。

---

**本节小结**：
- Harness 是包裹 Agent Loop 的安全控制框架，包含输入验证、权限控制、行为约束、输出审查、错误兜底五个组件
- 没有 Harness 的 Agent 在生产环境面临不可逆操作、不可预测行为、成本失控三大风险
- Harness 分四个层次：环境层（物理约束）→ 工具层（代码权限）→ 协议层（行为约定）→ 审查层（输出验证）

**下一节预告**：确定了 Harness 的结构后，接下来要回答一个更具体的设计问题：哪些逻辑必须是确定性的（用代码实现），哪些才应该交给 LLM？这条边界的划定是 Harness 设计的核心决策。
