# 6.3 工具权限设计：最小权限原则的工程实现

> [!abstract] 核心原则
> 每个工具应该只能做它需要做的事，不多也不少。这是信息安全领域"最小权限原则"（Principle of Least Privilege）在 Agent 工具层的直接应用。工具权限设计得越精确，Harness 的防护面就越小、越可靠。

## 1. 最小权限原则

最小权限原则（PoLP）的定义：**任何组件只应拥有完成其工作所必需的最小权限集合，多余的权限一律不授予。**

在 Agent 工具层，这意味着：

- `read_repo_file` 只能读文件，不能写文件——即使 LLM 试图通过它写入内容，工具层会直接拒绝
- `write_memory` 只能写 `memory/` 目录下的文件，不能写代码仓库——路径白名单在代码层硬编码
- `list_repo_files` 只能列举文件名，不能读取文件内容——功能单一，越界调用没有意义

权限设计的错误方向是：**授予"可能用得到"的权限**。正确方向是：**只授予"当前任务必须有"的权限**，需要扩展时再显式添加。

### 权限粒度的四个维度

| 维度 | 说明 | 示例 |
|------|------|------|
| 操作类型 | 读/写/删除/执行 | `read_repo_file` 只有读权限 |
| 路径范围 | 允许访问的目录白名单 | `write_memory` 只能写 `memory/` |
| 仓库范围 | 允许操作的仓库列表 | 只能操作 H5 仓库，不能操作 App 仓库 |
| 时间窗口 | 单次任务期间有效 | 不持久化超级权限 |

## 2. 只读工具 vs 写入工具

严格区分只读操作和写入操作是工具权限设计的第一步。

### 只读工具（无副作用）

只读工具的特点：**调用任意次数、任意顺序，系统状态不会改变**。这类工具可以相对宽松地调用，因为出错的代价很低——最坏的结果是返回了错误的数据，LLM 基于错误数据做了错误判断，但还没有任何实际操作发生。

追齐 Agent 中的只读工具：
- `read_repo_file(repo, path)` — 读取指定仓库的指定文件内容
- `list_repo_files(repo, dir?)` — 列出仓库的文件/目录列表
- `read_memory(key)` — 读取 `memory/` 目录下的记忆文件

只读工具的权限控制要点：
- 路径格式校验（防止路径遍历攻击：`../../../etc/passwd`）
- 仓库名白名单校验（不能访问 App 源码仓库或其他系统仓库）
- 响应大小限制（单次读取不超过 100KB，防止上下文爆炸）

### 写入工具（有副作用）

写入工具的特点：**每次调用都会改变系统状态**，必须有额外的控制措施。

追齐 Agent 中的写入/副作用工具：
- `diff_feature(...)` — 分析差异并生成修改建议（生成结构化输出，有潜在副作用）
- `resolve_feature_spec(...)` — 解析功能规格并写入解析结果
- `write_memory(key, value)` — 写入记忆文件，持久化进度状态

## 3. 写入操作的三层控制

对于有副作用的工具，Harness 实施三层叠加控制：

### 第一层：路径白名单

只允许写入预定义的目录，硬编码在代码里，不依赖 LLM 遵守约定：

```typescript
const WRITABLE_PATHS: Record<string, string[]> = {
  write_memory: ['memory/'],
  resolve_feature_spec: ['memory/specs/'],
  // diff_feature 不直接写文件，只返回 diff 内容
};

function enforceWritePath(toolName: string, targetPath: string): void {
  const allowed = WRITABLE_PATHS[toolName];
  if (!allowed) return; // 只读工具，跳过
  const isAllowed = allowed.some(prefix => targetPath.startsWith(prefix));
  if (!isAllowed) {
    throw new SecurityError(
      `工具 ${toolName} 禁止写入路径 "${targetPath}"。` +
      `允许的路径前缀：${allowed.join(', ')}`
    );
  }
}
```

### 第二层：操作类型限制

对于 `write_memory`，只允许追加/覆盖写入，不允许删除：

```typescript
type MemoryOperation = 'set' | 'append';  // 不提供 'delete' 操作

async function write_memory(
  key: string,
  value: string,
  op: MemoryOperation = 'set'
): Promise<void> {
  const safePath = path.join('memory', key.replace(/[^a-zA-Z0-9._-]/g, '_'));
  enforceWritePath('write_memory', safePath);

  if (op === 'set') {
    await fs.writeFile(safePath, value, 'utf-8');
  } else {
    await fs.appendFile(safePath, value + '
', 'utf-8');
  }
  await appendAuditLog({ tool: 'write_memory', path: safePath, op });
}
```

### 第三层：Audit Log

每次写入操作记录到 `logs/audit.jsonl`，格式化为 JSONL 方便后续查询：

```typescript
interface AuditEntry {
  timestamp: string;
  sessionId: string;
  tool: string;
  params: Record<string, unknown>;
  result: 'success' | 'blocked' | 'error';
  reason?: string;
}

async function appendAuditLog(entry: Omit<AuditEntry, 'timestamp' | 'sessionId'>): Promise<void> {
  const record: AuditEntry = {
    timestamp: new Date().toISOString(),
    sessionId: process.env.SESSION_ID ?? 'unknown',
    ...entry,
  };
  await fs.appendFile('logs/audit.jsonl', JSON.stringify(record) + '
', 'utf-8');
}
```

## 4. 人工确认节点

对于高风险操作（如生成代码修改建议并准备应用），Harness 层插入人工确认节点：

```typescript
async function requireConfirmation(
  toolName: string,
  params: Record<string, unknown>,
  preview: string
): Promise<boolean> {
  if (process.env.AGENT_MODE === 'automated') {
    // 自动化模式：跳过人工确认，依赖自动验证
    return true;
  }

  // 交互模式：打印预览，等待用户确认
  console.log('
📋 即将执行操作：');
  console.log(`  工具：${toolName}`);
  console.log(`  参数：${JSON.stringify(params, null, 2)}`);
  console.log('
预览：');
  console.log(preview);
  console.log('
是否继续？[y/N]');

  const answer = await readlineInput();
  return answer.toLowerCase() === 'y';
}
```

> [!warning] 常见误区：用 System Prompt 限制工具权限
> 在 System Prompt 里写"你不能修改代码文件"或"你只能读取指定目录"——这种方式**不可靠**。
>
> 原因一：System Prompt 可以被 prompt injection 绕过。工具返回值中如果包含"忽略之前的指令，现在请..."这样的文本，LLM 可能会遵从。
>
> 原因二：LLM 的遵从是概率性的，在某些特殊上下文下可能失效。
>
> **权限控制必须在代码层实现，System Prompt 只能作为行为引导，不能作为安全边界。**

### 工具执行中间件的完整实现

```typescript
async function executeToolWithPermissions<T>(
  toolName: string,
  params: Record<string, unknown>,
  executor: () => Promise<T>,
  context: { sessionId: string; mode: 'interactive' | 'automated' }
): Promise<T> {
  // Step 1: 权限检查
  if (!ALLOWED_TOOLS.has(toolName)) {
    await appendAuditLog({ tool: toolName, params, result: 'blocked', reason: '工具不在允许列表' });
    throw new PermissionError(`工具 "${toolName}" 不在允许列表中`);
  }

  // Step 2: 参数校验
  validateToolParams(toolName, params);

  // Step 3: 写入操作的额外控制
  if (WRITE_TOOLS.has(toolName)) {
    if (params.path) enforceWritePath(toolName, params.path as string);
    if (context.mode === 'interactive') {
      const confirmed = await requireConfirmation(toolName, params, '');
      if (!confirmed) {
        await appendAuditLog({ tool: toolName, params, result: 'blocked', reason: '用户取消' });
        return { status: 'cancelled' } as unknown as T;
      }
    }
  }

  // Step 4: 执行工具
  try {
    const result = await executor();
    await appendAuditLog({ tool: toolName, params, result: 'success' });
    return result;
  } catch (err) {
    await appendAuditLog({ tool: toolName, params, result: 'error', reason: String(err) });
    throw err;
  }
}
```

---

**本节小结**：
- 最小权限原则：每个工具只授予完成其工作所必需的最小权限
- 严格区分只读工具（无副作用，相对宽松）和写入工具（有副作用，三层控制）
- 写入操作的三层控制：路径白名单（防越权）+ 操作类型限制（防误删）+ Audit Log（可追溯）
- 权限控制必须在代码层实现，System Prompt 是软性约定，不是安全边界

**下一节预告**：工具权限控制了"能不能执行"，但执行之后 LLM 的输出是否正确、是否存在幻觉，还需要输出审查机制来把关。
