# 第六章：Harness 与确定性控制——给 Agent 装上安全围栏

## 章节目标

读完本章，你应该能够：

1. **理解 Harness 的本质**：Harness 不是限制 Agent 能力的枷锁，而是让 Agent 可以更大胆执行的安全底座——因为有了兜底，才敢全力往前冲
2. **划清确定性边界**：明确哪些逻辑必须用代码实现（权限校验、审计记录、预算检查），哪些才应该交给 LLM（意图理解、代码分析、方案生成）——边界划错，整个 Harness 就形同虚设
3. **设计工具权限矩阵**：基于最小权限原则，为 Agent 的每个工具设计精确的读/写范围、操作限制和审计策略，并以代码而非 prompt 来实施
4. **构建生产级防护体系**：将输出审查、防幻觉验证、成本控制、循环检测整合为一套完整的 Harness 实施框架，覆盖 Agent 的整个生命周期

## 追齐 Agent 速查

### Harness 四层结构

| 层次 | 实现方式 | 核心作用 | 能否绕过 |
|------|----------|----------|----------|
| 环境层 | OS 权限 + 网络规则 | 物理边界，兜底保障 | 不能 |
| 工具层 | TypeScript 代码 | 权限精确匹配，主战场 | 不能 |
| 协议层 | System Prompt | 行为引导，软性约束 | 理论上可被绕过 |
| 审查层 | 输出验证代码 | 生效前最后把关 | 不能 |

### 工具权限矩阵（摘要）

| 工具 | 类型 | 写路径限制 | Audit Log |
|------|------|-----------|-----------|
| read_repo_file | 只读 | N/A | 否 |
| list_repo_files | 只读 | N/A | 否 |
| read_memory | 只读 | N/A | 否 |
| diff_feature | 分析（不写文件） | 只返回字符串 | 是 |
| resolve_feature_spec | 写 | `memory/specs/` | 是 |
| write_memory | 写 | `memory/` | 是 |

### 预算参数

| 参数 | 值 |
|------|----|
| 每功能点工具调用上限 | 30 次 |
| 每会话 Token 上限 | 80,000 |
| 每任务费用上限 | $3.00 |
| 单次工具超时 | 30 秒 |
| 循环检测阈值 | 最近 6 次中重复 3 次 |
| 文件读取大小上限 | 100 KB |
