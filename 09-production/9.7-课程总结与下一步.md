# 9.7 课程总结：从 Agent 认知到生产化——你现在掌握了什么

> [!abstract] 本节摘要
> 你从一个了解 Agent 基础知识的开发者，学完了整个《Agent 设计与上下文工程》课程。这节做一个系统性的知识梳理，帮你连接所有章节，理清它们之间的依赖关系，并指出下一步的学习方向。知识变成能力，需要整体视角。

## 1. 九章知识地图

整个课程的知识体系，从基础概念到生产化实践，形成一条完整的链路：

```
第一章：Agent 认知基础
  └─ 什么是 Agent？工具调用循环、三种典型架构
  └─ 为什么需要上下文工程？

第二章：上下文工程
  └─ 上下文预算设计
  └─ 工具定义优化（ACI 接口设计基础）
  └─ Prompt Caching / 压缩 / 按需加载
      ↓ 上下文管理是所有章节的基础设施

第三章：工具设计（ACI）
  └─ Agent-Computer Interface 设计原则
  └─ 工具的 inputSchema 设计
  └─ 工具的错误反馈设计
      ↓ 工具是 Agent 与世界交互的唯一接口

第四章：记忆系统
  └─ MEMORY.md + completed.json 的设计
  └─ 精准注入 vs 全量注入
  └─ 跨任务知识积累
      ↓ 记忆让 Agent 越用越聪明

第五章：任务规划与执行
  └─ 任务分解（Decompose → Plan → Execute）
  └─ 检查点和断点续传
  └─ 进度追踪
      ↓ 规划让 Agent 能处理复杂长任务

第六章：Harness 安全控制
  └─ 权限控制（只读/审批/禁止）
  └─ Audit Log
  └─ 速率限制 + 预算控制
      ↓ Harness 是 Agent 进入生产的安全门

第七章：多 Agent 架构
  └─ Orchestrator + Sub-Agent 模式
  └─ 并行执行与结果汇总
  └─ Agent 间通信协议
      ↓ 多 Agent 解决规模问题

第八章：评测体系
  └─ 评测集设计（用例类型、覆盖度）
  └─ 自动化评测流水线
  └─ 评测驱动迭代
      ↓ 评测是每次升级的安全网

第九章：生产化
  └─ 三阶段生产化路径
  └─ 可观测性（日志 + 追踪 + 指标）
  └─ 成本优化 + 错误处理 + 版本管理
```

**核心依赖关系**：上下文工程（第二章）是技术基础；记忆（第四章）和工具设计（第三章）是 Agent 核心能力；Harness（第六章）和评测（第八章）是安全保障；多 Agent（第七章）和生产化（第九章）是规模化手段。

## 2. 功能追齐 Agent 完整演进

贯穿整个课程的实战案例——功能追齐 Agent，随着每章的学习不断演进：

| 章节 | 新增能力 | Agent 状态 |
|------|---------|-----------|
| 第一章 | 理解 Agent 基本结构 | 概念阶段，还没有代码 |
| 第二章 | 上下文预算 + Prompt Caching | 能运行，但每次从零开始，没有记忆 |
| 第三章 | `read_repo_file` 等工具有了清晰的 ACI 接口 | 工具可靠，LLM 调用工具的错误率降低 |
| 第四章 | `MEMORY.md` + `completed.json` | 有记忆了，不会重复追齐，越用越快 |
| 第五章 | 检查点 + 断点续传 | 能处理长任务，中途失败可以恢复 |
| 第六章 | Harness 层（权限 + 审计） | 可以给别人用了，有安全控制 |
| 第七章 | Orchestrator + Research/Analysis/Execution Sub-Agents | 能并行处理多个仓库，效率大幅提升 |
| 第八章 | 50+ 用例的评测集 + 自动化流水线 | 每次升级都有安全网 |
| 第九章 | 可观测性 + 成本控制 + 版本管理 | 生产就绪，可持续运行 |

从"一个能用的脚本"到"一个生产就绪的 Agent 系统"，这是整个课程走完的路。

## 3. 核心设计原则汇总

整个课程中反复强调的最重要原则：

1. **上下文是有限资源，要精心分配**（第二章）
   → 每一个 token 都有成本，把有限的上下文窗口用在最重要的信息上

2. **工具接口设计给 LLM 看，不是给人看**（第三章）
   → description 要具体、参数要完整、错误反馈要指向性强

3. **记忆要精准注入，不要全量注入**（第四章）
   → 注入全量记忆可能反而稀释重要信息，按任务上下文过滤

4. **先规划再执行，不要边想边做**（第五章）
   → 复杂任务先分解成可验证的步骤，每步执行前确认计划合理

5. **检查点保护状态，允许失败存在**（第五章）
   → Agent 不可能永远不失败，重要的是失败后能从上次进度恢复

6. **Harness 是 Agent 的安全操作系统**（第六章）
   → 权限控制、审计日志、预算控制，不能让 Agent 在生产中无约束运行

7. **多 Agent 分工要有明确边界**（第七章）
   → 每个 Sub-Agent 只负责自己的职责，Orchestrator 负责协调和判断

8. **评测驱动迭代，不凭感觉升级**（第八章）
   → 每次改动前后都要跑评测，用数据说话，不用直觉说话

9. **先确保 80% 通过率，再考虑生产化**（第九章）
   → 把不稳定的 Agent 推进生产只会让问题更难调试

10. **小步快跑，每步可验证**（贯穿全课程）
    → 不要一次性改造所有东西，每次只做一个改进，每次改进后验证

## 4. 你现在能构建什么

学完本课程，你具备了独立完成以下任务的能力：

**上下文工程（第二章）**
- 设计完整的上下文预算方案（System Prompt / 记忆 / 工具 / 对话的占比）
- 配置 Prompt Caching 减少稳定内容的 token 消耗
- 实现工具返回内容的压缩策略

**工具设计（第三章）**
- 设计符合 ACI 原则的工具接口（inputSchema + 描述 + 错误反馈）
- 为追齐 Agent 的 `read_repo_file`、`diff_feature` 等工具实现完整的 ACI 层
- 诊断和修复 LLM 频繁调用工具失败的问题

**记忆系统（第四章）**
- 从零搭建 `MEMORY.md` + `completed.json` 的文件记忆系统
- 实现精准记忆注入（按仓库 / 功能类型过滤）
- 设计跨任务知识积累的机制

**任务规划（第五章）**
- 设计复杂任务的分解方案（Decompose → Plan → Execute）
- 实现检查点保存和断点续传
- 设计进度追踪和恢复策略

**Harness 安全层（第六章）**
- 实现工具权限控制（只读 / 需审批 / 禁止）
- 实现 Audit Log（记录所有工具调用）
- 设置速率限制和每日预算上限

**多 Agent 架构（第七章）**
- 设计 Orchestrator + Sub-Agent 的分工方案
- 实现并行执行多个 Sub-Agent 并汇总结果
- 设计 Agent 间的通信格式和错误传递

**评测体系（第八章）**
- 设计覆盖核心场景的评测用例集
- 实现自动化评测脚本（运行 + 评分 + 报告）
- 搭建 CI/CD 集成的评测流水线

**生产化（第九章）**
- 按三阶段路径推进 Agent 从原型到生产
- 实现结构化 JSONL 日志 + SimpleTracer 追踪
- 设计成本监控和预算控制
- 实现优雅降级和完成摘要报告
- 管理 Prompt 版本和记忆格式迁移

## 5. 下一步学习方向

本课程聚焦于 Agent 设计和上下文工程的核心能力，以下方向值得深入探索：

### Fine-tuning：当通用 LLM 不够用时

本课程使用的是通用 LLM（claude-sonnet 等）通过 Prompt Engineering 来完成任务。当你发现：
- 某类任务需要大量 few-shot 示例才能稳定
- 通用模型对你的特定领域术语理解不准
- 需要极低延迟（fine-tuned 小模型通常比通用大模型快 10 倍以上）

这时候 fine-tuning 就值得考虑了。推荐从 OpenAI 的 fine-tuning 文档入手，理解数据集构建和训练成本的权衡。

### RAG 高级技术：向量检索的工程实践

本课程的记忆系统用文件（MEMORY.md）存储知识，适合小规模场景。当知识库增长到几千条记录，需要考虑：
- 向量化存储（Embedding + 向量数据库）
- 语义检索（不再是精确匹配，而是"语义相似"）
- 混合检索（向量 + BM25 关键词）
- Reranking（二次排序，提高召回精度）

### Agent 安全：更复杂的 Prompt Injection 防御

本课程在 Harness 层（第六章）介绍了基础的安全控制。真实生产中还需要考虑：
- Prompt Injection 攻击（恶意内容通过工具返回值注入 LLM）
- Indirect Prompt Injection（攻击者通过文件内容控制 Agent 行为）
- Agent 操作的不可逆性检查
- 多 Agent 系统中的信任边界

### Agent 框架：LangGraph、CrewAI 等框架的适用场景

本课程选择"从零实现"，这让你理解了底层原理。但在真实项目中，框架可以节省大量工程时间：
- **LangGraph**：适合复杂的有状态工作流，内置检查点和并发
- **CrewAI**：适合多 Agent 协作场景，有更高级的 Agent 角色系统
- **原则**：先理解底层（本课程已完成），再选择框架；不要在没有理解基础的情况下引入框架

> [!tip] 结束语：Agent 工程是软件工程的延伸
> Agent 工程不是魔法，是软件工程的延伸。好的 Agent 系统和好的软件系统一样，需要：
>
> - **清晰的边界**：每个工具、每个 Agent、每个 Prompt 都有明确的职责
> - **可靠的测试**：评测集是你对 Agent 行为的规约，不能缺失
> - **可观测的运行状态**：你无法改进你无法观察的东西
> - **持续的迭代**：没有一个 Agent 是"做完了"的，都在持续改进
>
> 你现在拥有了这些工具。接下来，去构建真正有用的东西。
