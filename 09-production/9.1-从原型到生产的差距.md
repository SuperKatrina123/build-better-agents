# 9.1 从原型到生产的差距：什么在实验室里有效，在生产中会失败

> [!abstract] 本节摘要
> Agent 原型很容易做出来，但生产化是另一回事。实验室里能工作的东西，在生产环境中往往会在你没有预料到的地方失败。本节梳理原型和生产之间最常见的五个差距，帮你在动手改造之前对"坑"有全局认识，并了解各自的应对策略。

> [!question] 带着问题阅读
> - 你的 Agent 现在处于哪个阶段？有哪几个差距已经感受到了？
> - 你愿意在什么条件下把 Agent 推向生产？评测通过率达到多少算"够了"？
> - 如果只能先解决一个差距，你会选哪个？为什么？

---

## 1. 五个最常见的差距

原型和生产系统之间的差距，不是量的差异，而是质的差异。以功能追齐 Agent 为例，逐一拆解：

### 可靠性差距

**实验室**：你手动触发脚本，如果跑到一半出错，你看着日志重新调整再跑。整个过程有人类在回路中。

**生产**：脚本定时触发，无人值守，失败了没有人发现。一个未捕获的异常可能让整个 pipeline 静默停止，而你在第二天才发现昨晚什么都没有跑。

可靠性差距是最根本的差距。解决方案是：完整的错误处理 + 检查点恢复 + 状态通知（见 9.4 节）。

### 规模差距

**实验室**：选了 3 个仓库做测试，功能点清晰，边界情况少，成功率 90%。

**生产**：15 个仓库，每个仓库有独特的代码风格、组件命名方式、业务逻辑边界，边界情况不是多了 5 倍，而是多了 10-20 倍。

规模差距的核心是**边界情况爆炸**。解决方案是：扩充评测集，覆盖尽量多的仓库特性；记忆系统积累仓库级别的知识（见第四章）。

### 成本差距

**实验室**：不在乎 token 消耗，反正是自己的 API Key，跑一次几毛钱无所谓。

**生产**：15 个仓库 × 每个仓库 5 个待追齐功能 × 每个功能 $3 = 每次运行 $225，每月运行 4 次 = $900/月。这已经是一笔需要向老板解释的预算。

成本差距往往在规模化后才暴露。解决方案是：Prompt Caching + 记忆系统 + 模型分层（见 9.3 节）。

### 可观测性差距

**实验室**：打开终端，看 `console.log`，哪里出错一目了然。

**生产**：多个 Agent 并发运行，日志混在一起，没有 session_id 就无法追踪某一次任务；没有结构化日志就无法统计成功率；没有 Dashboard 就不知道系统整体健康状况。

可观测性差距是最容易被低估的。解决方案是：结构化 JSONL 日志 + 追踪链路 + Dashboard（见 9.2 节）。

### 维护差距

**实验室**：你一个人用，Prompt 硬编码在代码里，改哪里自己知道，没有文档也没关系。

**生产**：三个月后你换了项目，新同事接手时一脸懵逼。或者你自己六个月后再看，也忘了当时的设计决策。

维护差距是长期成本最高的差距。解决方案是：Prompt 版本化 + Changelog + 部署文档（见 9.5 节）。

## 2. 追齐 Agent 的生产化路径

![[production-roadmap.excalidraw]]

这张路线图把生产化过程分成三个阶段，每个阶段有明确的目标和交付物：

**阶段一：MVP 验证（第 1-2 周）**

目标是验证 Agent 在受控环境里能稳定完成任务。

交付物：
- 可以稳定完成单个功能追齐的脚本
- 基础 JSONL 日志（至少记录工具调用和 LLM 调用）
- `memory/MEMORY.md` 开始积累
- 10 个核心评测用例，通过率 ≥ 80%

**阶段二：稳定化（第 3-4 周）**

目标是让 Agent 可以无人值守运行。

交付物：
- Harness 层（权限控制 + Audit Log）
- 完整评测体系（50+ 用例，覆盖边界情况）
- 成本监控（每日报告）
- 检查点和断点续传机制

**阶段三：规模化（第 2-3 个月）**

目标是高效处理所有 15 个仓库。

交付物：
- 多 Agent 架构（Research 并行）
- 自动化评测流水线
- 模型分层策略
- 可视化 Dashboard

## 3. 不要过早生产化

有一个常见的错误：在 Agent 还不稳定的时候就开始生产化改造。这会导致：

- 你在一个不稳定的基础上加了很多工程包装，看起来很健壮，其实核心问题没解决
- 生产化改造本身引入了新的复杂性，让 debug 变得更难
- 花了大量时间做运维工程，但 Agent 的核心能力没有提升

**硬性标准**：在受控评测集上，通过率 < 80% 时，不应该开始生产化。

这个 80% 不是拍脑袋的数字。它意味着：
- Agent 在已知的测试用例上有足够好的表现
- 主要失败模式已经被你理解和记录
- 你有信心 Agent 在边界情况下的降级行为是可控的

如果还没到 80%，先把时间花在改进 Agent 的核心能力上，而不是生产化。

> [!tip] 核心原则
> 每次只做一个生产化改进，每次改进后验证没有引入新问题。不要一次性把可观测性、错误处理、成本优化全部加上去——那会导致你不知道是哪个改动让系统变得不稳定。**小步快跑，每步可验证。**
