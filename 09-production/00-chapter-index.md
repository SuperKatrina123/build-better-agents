# 第九章：生产化——让 Agent 在真实环境中可靠运行

> 这是《Agent 设计与上下文工程》的最终章。本章把所有章节的技术积累整合为一套完整的生产化方案，帮你把实验室里的 Agent 变成真正可靠运行的系统。

---

## 章节目标

学完本章，你将能够：

1. **识别并跨越**原型和生产之间的五个关键差距（可靠性、规模、成本、可观测性、维护）
2. **实现三层可观测性**——结构化 JSONL 日志、追踪链路、每日指标 Dashboard
3. **运用五种降本手段**——Prompt Caching、精准记忆注入、模型分层、工具压缩、按需加载
4. **按三阶段路径推进**生产化，避免过早生产化和一次性大改造的两种常见错误

---

## 通用原理

生产化章节讨论的不是“某个 demo 怎么上线”，而是所有 Agent 从原型走向真实环境时都会遇到的共同差距：

- 可靠性差距：失败是否可控、可恢复、可通知
- 可观测性差距：出了问题能不能知道发生了什么
- 成本差距：系统规模放大后是否还能持续运行
- 维护差距：版本演进、迁移和协作是否可控

因此，本章的方法适用于任何准备从实验室走向真实环境的 Agent 系统。

## 场景映射

| Agent 类型 | 生产化关注点 | 典型监控对象 | 常见上线门槛 |
|-----------|-------------|-------------|-------------|
| Coding Agent | 修改安全、任务成功率、成本控制 | 工具调用链、diff 质量、失败恢复 | 评测通过率、权限控制、审计日志 |
| Research Agent | 来源稳定性、结论可信度、延迟 | 资料命中率、引用质量、报告时延 | 来源校验、质量抽检、成本预算 |
| Support Agent | 响应稳定性、用户体验、升级机制 | 首响时间、解决率、误答率 | 回复审查、转人工阈值、工单可追踪 |
| 流程 / 业务执行 Agent | 状态一致性、幂等、异常回滚 | 业务节点、执行回执、失败告警 | 权限边界、回滚策略、审计链路 |

## 主案例展开：功能追齐 Agent

### 三阶段路径一览

| 阶段 | 时间 | 关键组件 | 交付物 |
|------|------|---------|--------|
| 一：MVP 验证 | 1-2 周 | 单 Agent + JSONL 日志 + 基础记忆 | 稳定完成单功能追齐的脚本 |
| 二：稳定化 | 1 个月 | Harness + 完整评测 + 成本监控 + 检查点 | 可无人值守运行的 Agent |
| 三：规模化 | 2-3 月 | 多 Agent + 自动评测流水线 + Dashboard | 覆盖 15 仓库的生产系统 |

### 生产化检查清单（关键项）

**阶段一必须完成**
- [ ] 评测集通过率 ≥ 80%
- [ ] JSONL 结构化日志（session_id + event_type + success）
- [ ] 无未捕获异常
- [ ] MEMORY.md + completed.json 初始化

**阶段二必须完成**
- [ ] Harness 层（权限 + Audit Log）
- [ ] 成本预算上限（$10/天）
- [ ] 检查点 + 断点续传
- [ ] 优雅降级报告
- [ ] Prompt 版本化（存文件，不硬编码）

**阶段三必须完成**
- [ ] 多 Agent 架构（Orchestrator + Sub-Agents）
- [ ] 自动化评测流水线（CI/CD）
- [ ] 模型分层（Orchestrator 用 Haiku，Research/Execution 用 Sonnet）
- [ ] 可视化 Dashboard

这里的三阶段路径、检查清单和组件组合，是功能追齐 Agent 的生产化主案例。更通用的迁移原则是：

- 先把核心能力跑稳，再做生产化包装
- 每次只补一层工程能力，并验证没有引入新问题
- 生产化不是“加更多组件”，而是让系统在真实环境里可观察、可控制、可维护
