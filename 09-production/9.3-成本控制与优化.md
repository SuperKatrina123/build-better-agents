# 9.3 成本控制与优化：让 Agent 在经济上可持续

> [!abstract] 本节摘要
> Token 消耗即成本。没有成本意识的 Agent 在规模化后会产生难以承受的 API 费用。本节系统梳理成本优化的五种手段，并给出追齐 Agent 的实际配置方案，让你在不损失能力的前提下把成本控制在合理范围内。

> [!question] 带着问题阅读
> - 你的 Agent 每次运行消耗多少 token？你知道这个数字吗？
> - 如果每次运行费用 $3，规模化到 15 个仓库后的月度成本是多少？你能接受吗？
> - 哪种降本手段对你的场景最有效？

---

## 1. 成本来自哪里

要优化成本，首先要知道 token 消耗的来源。对追齐 Agent 的一次典型任务执行，消耗分布大致如下：

```
Token 消耗构成（以追齐 1 个功能点为例，总计约 8000 tokens）
─────────────────────────────────────────────────────────
System Prompt + 工具定义        ████████████  ~30%  (2400 tokens)
记忆注入 (MEMORY.md)            ████████      ~20%  (1600 tokens)
工具调用结果 (读文件/diff等)    ████████████  ~30%  (2400 tokens)
LLM 推理输出                    ████████      ~20%  (1600 tokens)
─────────────────────────────────────────────────────────
```

关键洞察：
- **System Prompt + 工具定义**占 30%，是最稳定的部分，最适合用 Prompt Caching
- **工具调用结果**占 30%，是波动最大的部分，需要通过压缩策略控制
- **记忆注入**占 20%，随着记忆系统的积累会增长，需要精准注入而非全量注入
- **LLM 推理输出**相对可控，模型选择会影响质量但不直接影响输入消耗

## 2. 五种降本手段

### 手段一：Prompt Caching

把稳定内容（System Prompt + 工具定义）设计为可缓存前缀。Anthropic 的 Prompt Caching 对缓存命中的 token 只收取约 10% 的费用。

使用 `cache_control: {"type": "ephemeral"}` 标记可缓存部分（见第二章 §2.3）：

```typescript
const messages = [
  {
    role: 'user',
    content: [
      {
        type: 'text',
        text: systemPrompt,
        cache_control: { type: 'ephemeral' }, // 缓存 System Prompt
      },
      {
        type: 'text',
        text: toolDefinitions,
        cache_control: { type: 'ephemeral' }, // 缓存工具定义
      },
      {
        type: 'text',
        text: currentTaskContext, // 每次任务不同，不缓存
      },
    ],
  },
];
```

**效果**：System Prompt + 工具定义（占 30%）缓存命中后成本降低 90%，整体降本约 27%。

### 手段二：记忆系统减少重复探索

有记忆的 Agent 不需要在每次任务开始时重新探索代码库。实验数据：有 `MEMORY.md` 的追齐 Agent 比没有记忆的版本少约 30% 的工具调用（见第四章）。

关键是**精准注入**而非**全量注入**——只注入与当前任务相关的记忆片段：

```typescript
// 不好：每次注入完整的 MEMORY.md（可能 3000+ tokens）
const memory = readFullMemory();

// 好：只注入与当前仓库/功能相关的记忆
const memory = readMemoryForContext({
  repo: currentRepo,
  featureType: currentFeatureType,
});
```

**效果**：减少 30% 工具调用，直接降低工具结果的 token 消耗。

### 手段三：Skills 按需加载

不用的工具不放进上下文。工具定义会占用 System Prompt 的一部分空间，15 个工具的定义可能消耗 500-800 tokens（见第二章 §2.2）。

```typescript
// 不好：所有工具都加载
const tools = getAllTools(); // 15 个工具，~800 tokens

// 好：按任务阶段加载
const researchTools = getToolsForPhase('research');   // 4 个工具，~200 tokens
const analysisTools = getToolsForPhase('analysis');   // 3 个工具，~150 tokens
const executionTools = getToolsForPhase('execution'); // 5 个工具，~300 tokens
```

**效果**：每次调用减少约 300-600 tokens 的工具定义开销。

### 手段四：工具返回压缩

工具调用结果是 token 消耗最大的变量。一次 `read_repo_file` 返回一个大文件可能就是 2000-4000 tokens。

压缩策略（见第二章 §2.4）：
- 只返回相关部分，而不是整个文件
- 对代码文件只返回函数签名 + 注释（跳过实现细节）
- 对 diff 结果截断到最关键的变化

```typescript
// 不好：返回整个文件
const content = await readRepoFile({ repo, path }); // 可能 4000 tokens

// 好：只返回相关行
const content = await readRepoFile({
  repo,
  path,
  maxLines: 100,    // 限制行数
  focusOn: featureName, // 只返回与功能相关的代码段
});
```

### 手段五：模型分层

不是所有任务都需要最强的模型。对简单任务用小模型，对复杂任务用大模型：

| 任务类型 | 复杂度 | 推荐模型 | 理由 |
|---------|--------|---------|------|
| 格式校验 | 低 | claude-3-haiku | 简单规则匹配 |
| 任务分解协调 | 中 | claude-3-5-haiku | 逻辑清晰即可 |
| 代码理解分析 | 高 | claude-sonnet | 需要强推理 |
| 代码生成 | 高 | claude-sonnet | 需要高质量输出 |
| 复杂架构决策 | 极高 | claude-opus | 最复杂的判断 |

## 3. 模型分层的实现

追齐 Agent 的模型选择策略：

```typescript
const MODEL_CONFIG = {
  orchestrator: 'claude-haiku-4-5',    // 任务协调，快且便宜
  research: 'claude-sonnet-4-5',       // 代码理解，需要强理解力
  analysis: 'claude-sonnet-4-5',       // 功能对比分析
  execution: 'claude-sonnet-4-5',      // 代码生成
  validation: 'claude-haiku-4-5',      // 输出格式校验
} as const;

async function callAgent(
  agentType: keyof typeof MODEL_CONFIG,
  messages: Message[]
): Promise<string> {
  const model = MODEL_CONFIG[agentType];
  return callClaude({ model, messages });
}
```

**成本对比**（以每百万 tokens 计）：
- claude-haiku：输入 $0.25 / 输出 $1.25
- claude-sonnet：输入 $3 / 输出 $15
- claude-opus：输入 $15 / 输出 $75

Orchestrator 使用 Haiku 而不是 Sonnet，在有 50 次任务执行时节省约 $10/月。

## 4. 成本监控与预算

### 成本追踪实现

```typescript
class CostTracker {
  private dailyCost = 0;
  private monthlyCost = 0;
  private readonly dailyLimit: number;
  private readonly monthlyLimit: number;

  // 价格表（每百万 tokens，美元）
  private readonly pricing: Record<string, { input: number; output: number }> = {
    'claude-haiku-4-5':  { input: 0.25,  output: 1.25 },
    'claude-sonnet-4-5': { input: 3.0,   output: 15.0 },
    'claude-opus-4-5':   { input: 15.0,  output: 75.0 },
  };

  constructor(dailyLimit: number, monthlyLimit: number) {
    this.dailyLimit = dailyLimit;
    this.monthlyLimit = monthlyLimit;
  }

  recordUsage(model: string, inputTokens: number, outputTokens: number): void {
    const price = this.pricing[model];
    if (!price) return;

    const cost = (inputTokens / 1_000_000) * price.input +
                 (outputTokens / 1_000_000) * price.output;

    this.dailyCost += cost;
    this.monthlyCost += cost;

    if (this.dailyCost > this.dailyLimit) {
      throw new Error(`Daily cost limit exceeded: $${this.dailyCost.toFixed(2)} > $${this.dailyLimit}`);
    }
    if (this.monthlyCost > this.monthlyLimit) {
      throw new Error(`Monthly cost limit exceeded: $${this.monthlyCost.toFixed(2)} > $${this.monthlyLimit}`);
    }
  }

  report(): string {
    return `Cost today: $${this.dailyCost.toFixed(2)} / $${this.dailyLimit} | Month: $${this.monthlyCost.toFixed(2)} / $${this.monthlyLimit}`;
  }
}
```

### 预算配置

```typescript
const costTracker = new CostTracker(
  10,   // $10/day 上限
  200,  // $200/month 上限
);
```

超预算时，抛出异常 → Harness 捕获 → 发送告警 → 暂停 Agent 直到人工确认。

> [!example] 成本优化前后对比
> **场景**：追齐 1 个功能点（如"深色模式开关"）
>
> **优化前**（无任何优化措施）：
> - System Prompt + 工具：2400 tokens（全量，无缓存）
> - 记忆注入：3200 tokens（全量 MEMORY.md）
> - 工具调用结果：4100 tokens（完整文件内容）
> - LLM 输出：1800 tokens
> - **合计：11500 tokens ≈ $0.052（Sonnet）**
>
> **优化后**（Caching + 精准记忆 + 压缩 + 模型分层）：
> - System Prompt + 工具：2400 tokens，但缓存命中，实际计费 240 tokens
> - 记忆注入：800 tokens（精准注入）
> - 工具调用结果：2200 tokens（压缩后）
> - LLM 输出：1600 tokens
> - Orchestrator 用 Haiku，Research/Analysis/Execution 用 Sonnet
> - **合计：约 4840 计费 tokens ≈ $0.021**
>
> **节省约 60%**。在 15 个仓库 × 5 功能/仓库的规模下，月度成本从 ~$470 降至 ~$190。

## 本节小结

五种降本手段按性价比排序：

1. **Prompt Caching**（实现成本低，效果稳定，27% 降幅）
2. **记忆系统减少重复探索**（已有记忆系统则几乎免费，30% 工具调用减少）
3. **模型分层**（需要配置，效果取决于任务分布）
4. **工具返回压缩**（需要改造工具，但效果明显）
5. **Skills 按需加载**（改造工作量较大，收益相对有限）

先做前两项，可以在不改变 Agent 行为的前提下实现约 50% 的成本节省。
