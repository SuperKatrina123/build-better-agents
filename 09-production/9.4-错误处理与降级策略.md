# 9.4 生产级错误处理：失败不是终点

> [!abstract] 本节摘要
> 生产环境的错误处理要比开发环境复杂得多——不仅要捕获错误，还要优雅降级、保护状态、通知用户、提供诊断信息。一个好的生产级 Agent，失败时的行为和成功时一样重要。本节系统梳理追齐 Agent 的错误处理设计。

> [!question] 带着问题阅读
> - 你的 Agent 现在遇到 API 超时会怎样？会重试？会保存进度？还是直接崩溃？
> - 你的用户（或你自己）在 Agent 失败后能看到有用的信息吗？还是只是一条报错堆栈？

## 1. 生产环境的错误矩阵

不同类型的错误需要不同的处理策略：

| 错误类型 | 触发场景 | 处理策略 | 说明 |
|---------|---------|---------|------|
| API 超时 | Anthropic API 响应超时 | 指数退避重试（最多 3 次） | 网络问题，通常可恢复 |
| API 限流（429） | 超出速率限制 | 等待后重试 | 等待 `retry-after` 头部指定的时间 |
| LLM 内容过滤 | 输出被安全过滤拦截 | 降级 + 记录 | 记录触发场景，不重试 |
| 工具调用失败 | `read_repo_file` 返回 404 | 降级报告 + 继续 | 跳过该功能点，继续其他 |
| 上下文长度超限 | 超过模型 context window | 压缩后重试 | 触发压缩策略（见第二章） |
| 记忆格式损坏 | MEMORY.md 解析失败 | 备份 + 重建 | 不能因记忆问题阻塞任务 |
| 系统错误（OOM 等） | 进程崩溃 | 检查点恢复 | 下次启动时从检查点恢复 |

### 实现：统一错误处理器

```typescript
type RetryStrategy = 'exponential_backoff' | 'fixed_delay' | 'no_retry';

interface ErrorHandlerConfig {
  maxRetries: number;
  strategy: RetryStrategy;
  baseDelayMs: number;
}

const ERROR_CONFIGS: Record<string, ErrorHandlerConfig> = {
  'APITimeoutError':     { maxRetries: 3, strategy: 'exponential_backoff', baseDelayMs: 1000 },
  'RateLimitError':      { maxRetries: 5, strategy: 'fixed_delay',         baseDelayMs: 60000 },
  'ContentFilterError':  { maxRetries: 0, strategy: 'no_retry',            baseDelayMs: 0 },
  'ToolNotFoundError':   { maxRetries: 0, strategy: 'no_retry',            baseDelayMs: 0 },
  'ContextLengthError':  { maxRetries: 1, strategy: 'no_retry',            baseDelayMs: 0 },
};

async function withRetry<T>(
  fn: () => Promise<T>,
  errorType: string,
  logger: AgentLogger,
  sessionId: string
): Promise<T> {
  const config = ERROR_CONFIGS[errorType] ?? { maxRetries: 1, strategy: 'exponential_backoff', baseDelayMs: 500 };
  let attempt = 0;

  while (true) {
    try {
      return await fn();
    } catch (err) {
      attempt++;
      if (attempt > config.maxRetries) {
        logger.log({
          session_id: sessionId,
          agent_name: 'system',
          event_type: 'error',
          duration_ms: 0,
          success: false,
          error: `${errorType} exhausted retries after ${attempt} attempts: ${err}`,
        });
        throw err;
      }

      const delay = config.strategy === 'exponential_backoff'
        ? config.baseDelayMs * Math.pow(2, attempt - 1)
        : config.baseDelayMs;

      await sleep(delay);
    }
  }
}
```

## 2. 优雅降级而非崩溃

当 Agent 遇到无法处理的情况时，正确的做法不是抛出未捕获异常，而是输出一份**降级报告**：

- 已完成哪些步骤
- 哪里失败了、失败原因
- 建议的下一步行动

### 降级报告结构

```typescript
interface DegradedResult {
  status: 'partial' | 'failed';
  sessionId: string;
  completedSteps: string[];
  failedAt: string;
  failureReason: string;
  partialOutput?: string;
  suggestedNextSteps: string[];
  canResume: boolean;         // 是否可以从检查点恢复
  checkpointPath?: string;    // 如果可以恢复，检查点路径
}

function buildDegradedResult(
  sessionId: string,
  completedSteps: string[],
  failedAt: string,
  error: Error,
  checkpointPath?: string
): DegradedResult {
  return {
    status: completedSteps.length > 0 ? 'partial' : 'failed',
    sessionId,
    completedSteps,
    failedAt,
    failureReason: error.message,
    suggestedNextSteps: getSuggestedSteps(error),
    canResume: !!checkpointPath,
    checkpointPath,
  };
}

function getSuggestedSteps(error: Error): string[] {
  if (error.message.includes('timeout')) {
    return [
      '等待 5 分钟后重试（API 可能临时不可用）',
      '检查网络连接',
      '如果持续超时，联系 Anthropic 支持',
    ];
  }
  if (error.message.includes('rate limit')) {
    return [
      `等待 ${error.message.match(/\d+/)?.[0] ?? 60} 秒后重新运行`,
      '考虑降低并发任务数',
    ];
  }
  return ['查看日志了解详情', '联系技术支持'];
}
```

### 使用示例

```typescript
async function runAlignTask(feature: FeatureSpec): Promise<AlignResult | DegradedResult> {
  const completedSteps: string[] = [];
  const sessionId = generateSessionId();

  try {
    await tracer.trace('read_memory', () => readMemory());
    completedSteps.push('read_memory');

    await tracer.trace('research_phase', () => runResearchAgent(feature));
    completedSteps.push('research_phase');

    await tracer.trace('analysis_phase', () => runAnalysisAgent(feature));
    completedSteps.push('analysis_phase');

    const result = await tracer.trace('execution_phase', () => runExecutionAgent(feature));
    completedSteps.push('execution_phase');

    return result;
  } catch (err) {
    const checkpointPath = await saveCheckpoint(sessionId, completedSteps, feature);
    return buildDegradedResult(
      sessionId,
      completedSteps,
      completedSteps[completedSteps.length - 1] ?? 'init',
      err as Error,
      checkpointPath
    );
  }
}
```

## 3. 状态保护机制

### 检查点实现

在每个高风险操作前保存检查点（见第五章）。生产中额外加入**状态校验**：

```typescript
interface Checkpoint {
  sessionId: string;
  feature: FeatureSpec;
  completedSteps: string[];
  partialResults: Record<string, unknown>;
  savedAt: string;
}

async function saveCheckpoint(
  sessionId: string,
  completedSteps: string[],
  feature: FeatureSpec,
  partialResults: Record<string, unknown> = {}
): Promise<string> {
  const checkpoint: Checkpoint = {
    sessionId,
    feature,
    completedSteps,
    partialResults,
    savedAt: new Date().toISOString(),
  };

  const path = `./checkpoints/${sessionId}.json`;
  await fs.promises.writeFile(path, JSON.stringify(checkpoint, null, 2));
  return path;
}

async function loadCheckpoint(sessionId: string): Promise<Checkpoint | null> {
  const path = `./checkpoints/${sessionId}.json`;
  try {
    const data = await fs.promises.readFile(path, 'utf-8');
    return JSON.parse(data) as Checkpoint;
  } catch {
    return null;
  }
}
```

### 启动时状态校验

每次 Agent 启动时，验证 `MEMORY.md` 和 `completed.json` 的格式完整性：

```typescript
async function validateState(): Promise<{ valid: boolean; errors: string[] }> {
  const errors: string[] = [];

  // 校验 MEMORY.md
  try {
    const memory = await fs.promises.readFile('./memory/MEMORY.md', 'utf-8');
    if (!memory.includes('## Repository Knowledge')) {
      errors.push('MEMORY.md: missing "Repository Knowledge" section');
    }
    if (!memory.includes('## Global Patterns')) {
      errors.push('MEMORY.md: missing "Global Patterns" section');
    }
  } catch {
    errors.push('MEMORY.md: file not found or unreadable');
  }

  // 校验 completed.json
  try {
    const raw = await fs.promises.readFile('./memory/completed.json', 'utf-8');
    const data = JSON.parse(raw);
    if (!Array.isArray(data.completedFeatures)) {
      errors.push('completed.json: "completedFeatures" must be an array');
    }
    if (typeof data.version !== 'string') {
      errors.push('completed.json: "version" field missing');
    }
  } catch (err) {
    errors.push(`completed.json: parse error - ${err}`);
  }

  return { valid: errors.length === 0, errors };
}

// 启动时调用
const validation = await validateState();
if (!validation.valid) {
  console.error('State validation failed:');
  validation.errors.forEach(e => console.error(`  - ${e}`));
  process.exit(1);
}
```

## 4. 用户通知设计

### 完成摘要报告

追齐 Agent 完成后，生成人类可读的摘要，而不只是机器日志：

```typescript
function generateCompletionReport(result: AlignResult): string {
  return `
# 功能追齐完成报告
生成时间：${new Date().toLocaleString('zh-CN')}
Session ID：${result.sessionId}

## 完成情况
✅ 成功追齐：${result.successCount} 个功能点
⏭️ 跳过（已存在）：${result.skippedCount} 个功能点
❌ 失败：${result.failedCount} 个功能点

## 资源消耗
- Token 消耗：${result.totalTokens.toLocaleString()} tokens
- 估算费用：$${result.estimatedCost.toFixed(3)}
- 耗时：${Math.round(result.durationMs / 1000)} 秒

## 追齐明细
${result.items.map(item => `- ${item.success ? '✅' : '❌'} ${item.featureName} (${item.repo})`).join('\n')}

${result.failedCount > 0 ? `## 需要人工处理的问题\n${result.failures.map(f => `- ${f.featureName}: ${f.reason}`).join('\n')}` : ''}

## 下一步
${result.failedCount > 0
  ? `请检查失败原因并手动处理 ${result.failedCount} 个功能点，或重新运行 Agent`
  : '所有功能点已成功追齐，无需额外操作'}
`.trim();
}
```

> [!example] 案例：API 超时的完整处理流程
> **场景**：追齐 Agent 在 Analysis 阶段调用 Anthropic API 时遇到超时（30s 无响应）
>
> **处理流程**：
>
> 1. **捕获超时** → `withRetry` 检测到 `APITimeoutError`，记录日志
> 2. **第一次重试** → 等待 1000ms（指数退避），重新调用 API
> 3. **第二次重试** → 等待 2000ms，再次尝试
> 4. **第三次重试** → 等待 4000ms，再次尝试
> 5. **重试耗尽** → `saveCheckpoint` 保存当前进度（已完成 research_phase）
> 6. **发送通知** → Slack/邮件：`Session align-abc123 失败：API 超时，已保存检查点，可恢复`
> 7. **返回降级报告** → `{ status: 'partial', completedSteps: ['read_memory', 'research_phase'], canResume: true, checkpointPath: './checkpoints/align-abc123.json' }`
> 8. **等待重启** → 运维人员或自动化脚本看到通知后，调用 `resumeFromCheckpoint('align-abc123')`，从 analysis_phase 继续执行

## 本节小结

生产级错误处理的四个层次：

1. **错误矩阵**：不同错误类型有不同处理策略，不能一律重试或一律放弃
2. **优雅降级**：失败时返回降级报告而非未捕获异常
3. **状态保护**：检查点 + 启动时状态校验，确保 Agent 总是从干净的状态开始
4. **用户通知**：完成报告人类可读，失败时给出具体的下一步建议
