# 9.5 版本管理与迭代：让 Agent 可以安全升级

> [!abstract] 本节摘要
> Agent 系统和普通软件一样需要版本管理，但有额外的挑战：Prompt 的修改可能导致行为变化，记忆格式的升级需要迁移脚本，工具接口的变更需要兼容性处理。本节系统梳理追齐 Agent 的版本管理策略，让每次升级都是可控的、可回滚的。

> [!question] 带着问题阅读
> - 你上次改了 System Prompt，有验证改动没有引入退步吗？
> - 如果你升级了记忆格式，旧的 MEMORY.md 文件怎么办？

---

## 1. Agent 系统的版本组成

一个 Agent 系统有三个独立的版本维度，每个维度的变更影响不同、迁移成本不同：

### 维度一：Prompt 版本

包括：
- System Prompt（Agent 的角色定义、行为约束）
- 工具描述（工具的 description 字段，影响 LLM 选择工具的方式）
- Few-shot 示例（如果有的话）

**变更影响**：Prompt 的微小措辞改变可能导致完全不同的行为。这是最不直觉的版本变更。

### 维度二：记忆格式版本

包括：
- `MEMORY.md` 的章节结构和字段
- `completed.json` 的 schema（字段名、数据类型、嵌套结构）

**变更影响**：格式不兼容会导致 Agent 无法正确读取历史积累的知识。

### 维度三：工具接口版本

包括：
- 工具的 `inputSchema`（参数名、类型、是否必填）
- 工具的返回格式
- 工具的语义（同名工具做了不同的事）

**变更影响**：LLM 可能按旧格式调用工具，导致调用失败或产生错误结果。

## 2. Prompt 版本管理

### 把 Prompt 存在文件里，不要硬编码

```
prompts/
  v1.0/
    orchestrator.md
    research-agent.md
    analysis-agent.md
    execution-agent.md
  v1.1/
    orchestrator.md
    ...
  current -> v1.1/   (符号链接或配置指向当前版本)
```

版本化的 System Prompt 文件示例（`prompts/v1.1/research-agent.md`）：

```markdown
---
version: "1.1"
changed: "2024-01-15"
changes: "增加了对 React Hook 的特殊处理逻辑"
---

你是一个专门分析 React + TypeScript 代码仓库的 Research Agent...
```

### 加载 Prompt 的代码

```typescript
import * as fs from 'fs';
import * as path from 'path';

function loadPrompt(agentName: string, version = 'current'): string {
  const promptPath = path.join('./prompts', version, `${agentName}.md`);
  const content = fs.readFileSync(promptPath, 'utf-8');
  // 去掉 frontmatter，只返回正文
  return content.replace(/^---[\s\S]*?---\n/, '').trim();
}

// 使用
const researchPrompt = loadPrompt('research-agent'); // 使用当前版本
const oldPrompt = loadPrompt('research-agent', 'v1.0'); // 使用旧版本（用于对比）
```

### Prompt 修改的评测要求

每次修改 Prompt，必须：

1. 跑现有评测集，对比修改前后的通过率
2. 如果任何测试用例从"通过"变成"失败"，需要理解原因
3. 重大改动（影响核心行为的修改）应该做 A/B 测试

```typescript
async function comparePromptVersions(
  oldVersion: string,
  newVersion: string,
  testCases: TestCase[]
): Promise<ComparisonReport> {
  const oldResults = await runEvaluation(oldVersion, testCases);
  const newResults = await runEvaluation(newVersion, testCases);

  const regressions = testCases.filter((tc, i) =>
    oldResults[i].passed && !newResults[i].passed
  );
  const improvements = testCases.filter((tc, i) =>
    !oldResults[i].passed && newResults[i].passed
  );

  return {
    oldPassRate: oldResults.filter(r => r.passed).length / testCases.length,
    newPassRate: newResults.filter(r => r.passed).length / testCases.length,
    regressions: regressions.map(tc => tc.id),
    improvements: improvements.map(tc => tc.id),
    safe: regressions.length === 0,
  };
}
```

## 3. 记忆迁移策略

### 版本化记忆格式

在 `completed.json` 中显式标记版本号：

```json
// v1 格式（旧）
{
  "version": "1",
  "completed": ["dark-mode", "notification-badge"]
}

// v2 格式（新，增加了完成时间和 repo 信息）
{
  "version": "2",
  "completedFeatures": [
    {
      "id": "dark-mode",
      "repo": "app-home",
      "completedAt": "2024-01-10T14:23:00Z",
      "tokenCost": 7823
    },
    {
      "id": "notification-badge",
      "repo": "app-profile",
      "completedAt": "2024-01-12T09:11:00Z",
      "tokenCost": 6542
    }
  ]
}
```

### 迁移脚本：v1 → v2

```typescript
import * as fs from 'fs';

interface CompletedV1 {
  version: '1';
  completed: string[];
}

interface CompletedFeatureV2 {
  id: string;
  repo: string;
  completedAt: string;
  tokenCost: number;
}

interface CompletedV2 {
  version: '2';
  completedFeatures: CompletedFeatureV2[];
}

async function migrateCompletedV1ToV2(filePath: string): Promise<void> {
  const raw = fs.readFileSync(filePath, 'utf-8');
  const v1Data = JSON.parse(raw) as CompletedV1;

  if (v1Data.version !== '1') {
    console.log(`Already at version ${v1Data.version}, skipping`);
    return;
  }

  // 备份旧文件
  const backupPath = filePath.replace('.json', '.v1.backup.json');
  fs.copyFileSync(filePath, backupPath);
  console.log(`Backed up to ${backupPath}`);

  // 迁移数据
  const v2Data: CompletedV2 = {
    version: '2',
    completedFeatures: v1Data.completed.map(id => ({
      id,
      repo: 'unknown', // v1 没有记录 repo，用 unknown 占位
      completedAt: new Date().toISOString(), // 迁移时间作为 completedAt
      tokenCost: 0,    // v1 没有记录 token 消耗
    })),
  };

  fs.writeFileSync(filePath, JSON.stringify(v2Data, null, 2));
  console.log(`Migrated ${v1Data.completed.length} features to v2 format`);
}

// 运行迁移
await migrateCompletedV1ToV2('./memory/completed.json');
```

### 启动时自动迁移

```typescript
async function ensureMemoryVersion(): Promise<void> {
  const raw = fs.readFileSync('./memory/completed.json', 'utf-8');
  const data = JSON.parse(raw);

  if (data.version === '1') {
    console.log('Migrating completed.json from v1 to v2...');
    await migrateCompletedV1ToV2('./memory/completed.json');
  }
  // 未来版本升级时继续在这里添加
}
```

## 4. 灰度发布

### 灰度策略

新版本 Agent 上线时，按以下顺序推进：

1. **内部测试**：在评测集上运行，确保通过率不低于旧版本
2. **小范围灰度**：选 1-2 个"低风险"仓库（代码相对稳定、评测用例最多）先上线
3. **观察期**：运行 3 天，对比新旧版本的成功率、成本、耗时
4. **扩大范围**：如果指标正常，推到所有仓库

```typescript
const CANARY_REPOS = ['app-home', 'app-profile']; // 灰度仓库

async function shouldUseNewVersion(repo: string): Promise<boolean> {
  const config = await loadFeatureFlags();
  if (config.newAgentVersion === 'rollout_all') return true;
  if (config.newAgentVersion === 'canary') return CANARY_REPOS.includes(repo);
  return false; // 默认用旧版本
}
```

### A/B 测试评判标准

当新旧版本在同一批仓库上都运行时，用以下标准评判：

| 指标 | 新版本必须满足 |
|------|--------------|
| 追齐成功率 | ≥ 旧版本成功率 |
| 平均 token 消耗 | ≤ 旧版本 × 1.1（允许 10% 增长） |
| P95 耗时 | ≤ 旧版本 × 1.2（允许 20% 增长） |
| 评测集通过率 | ≥ 旧版本通过率 |

> [!info] Agent 升级 Checklist
> 每次 Agent 升级，按以下步骤执行：
>
> 1. ✅ **评测通过**：新版本在评测集上通过率 ≥ 旧版本，无回归
> 2. ✅ **更新 Changelog**：在 `CHANGELOG.md` 记录改动原因、影响范围
> 3. ✅ **迁移记忆格式**（如需要）：运行迁移脚本，验证格式校验通过
> 4. ✅ **灰度发布**：先在 1-2 个仓库运行，观察日志无异常
> 5. ✅ **监控 3 天**：对比灰度仓库的指标，与旧版本基线对比
> 6. ✅ **全量发布**：指标正常后推到所有仓库，更新 `current` 版本链接

## 本节小结

Agent 版本管理的三个维度：

- **Prompt 版本**：存文件 + 评测对比，每次改 Prompt 必须跑评测
- **记忆格式版本**：显式版本号 + 迁移脚本，备份后再迁移
- **工具接口版本**：变更前评估兼容性，必要时做版本适配层
