# 9.6 设计决策 9：功能追齐 Agent 生产化完整路径

> [!abstract] 本节摘要
> 本节是第九章的设计决策节，也是功能追齐 Agent 贯穿全课程演进历程的最终收束。我们把所有生产化内容整合成完整的三阶段路径和可操作的检查清单，帮你把理论转化为可执行的行动计划。

---

## 1. 三阶段生产化路径（详细版）

### 阶段一：MVP 验证（第 1-2 周）

**目标**：验证 Agent 在受控环境里能稳定完成单个功能追齐任务。

**技术栈配置**：
- 单 Agent（不分多 Sub-Agent）
- 手动触发（`npx ts-node align.ts --feature=dark-mode --repo=app-home`）
- 基础 JSONL 日志（`logs/agent-YYYY-MM-DD.jsonl`）
- `memory/MEMORY.md` + `memory/completed.json`（手动维护）
- 核心评测集 10 个用例，覆盖最常见的功能类型

**关键验证标准**：
- 评测通过率 ≥ 80%
- 能稳定处理 `app-home` 和 `app-profile` 两个最熟悉的仓库
- 失败时有清晰的错误信息（不是未捕获异常）

**交付物**：
- `align.ts`：可运行的追齐脚本
- `memory/MEMORY.md`：初始化了至少 2 个仓库的知识
- `evals/core-10.json`：10 个评测用例 + 基线结果
- `README.md`：如何运行、有哪些工具、记忆格式说明

---

### 阶段二：稳定化（第 3-4 周）

**目标**：让 Agent 可以无人值守运行，失败时能自我保护和通知。

**新增组件**：

1. **Harness 层**（见第六章）
   - 权限控制：`read_repo_file` 只读，禁止写生产代码
   - Audit Log：所有工具调用记录到 `audit/audit.log`
   - 速率限制：每小时最多 30 次 LLM 调用

2. **完整评测体系**（见第八章）
   - 扩充到 50+ 用例，覆盖所有 15 个仓库的典型功能
   - 自动化评测脚本：`npm run eval`
   - 评测报告：通过率 + 失败用例列表 + 与上次对比

3. **成本监控**（见 9.3 节）
   - 每日 token 消耗报告
   - 预算限制：$10/天，$200/月
   - 超限时发送邮件告警

4. **检查点和断点续传**（见 9.4 节）
   - 每个阶段完成后保存检查点
   - 失败时返回降级报告
   - 支持 `--resume=SESSION_ID` 从检查点继续

**验证标准**：
- 在不被人工监督的情况下连续运行 5 次，无未捕获异常
- API 超时时能自动重试并保存检查点
- 成本报告每天自动生成

**交付物**：
- `harness.ts`：Harness 包装层
- `evals/full-50.json`：完整评测集
- `scripts/daily-report.ts`：每日摘要脚本
- `CHANGELOG.md`：版本历史

---

### 阶段三：规模化（第 2-3 个月）

**目标**：高效处理所有 15 个仓库，成本可控，可持续运行。

**新增组件**：

1. **多 Agent 架构**（见第七章）
   - Orchestrator 负责任务分解和结果汇总
   - Research Agent × 3（并行研究 3 个仓库）
   - Analysis Agent（功能对比分析）
   - Execution Agent（代码生成）

2. **自动化评测流水线**
   - PR 触发：每次 Agent 代码变更自动跑评测
   - 定期回归：每周运行完整评测集
   - 告警：通过率下降 > 5% 时发 Slack 告警

3. **模型分层**（见 9.3 节）
   - Orchestrator → claude-haiku（协调任务）
   - Research → claude-sonnet（代码理解）
   - Analysis → claude-sonnet（功能对比）
   - Execution → claude-sonnet（代码生成）

4. **Dashboard**（见 9.2 节）
   - 每日追齐功能数
   - 成功率趋势图
   - 成本趋势图
   - 告警规则

**验证标准**：
- 15 个仓库全覆盖，评测通过率 ≥ 85%
- 月度成本 ≤ $200
- 有 Dashboard 可以看实时状态

**交付物**：
- 完整的多 Agent 架构代码
- 自动化 CI/CD 评测流水线
- 可视化 Dashboard（Grafana 或简单的 HTML 报告）
- 完整的运维文档

## 2. 生产化检查清单

### 阶段一检查项（MVP 验证）

**技术要求**
- [ ] 评测集建立，通过率 ≥ 80%
- [ ] JSONL 结构化日志（包含 session_id、event_type、duration_ms、success）
- [ ] 错误捕获（无未捕获异常）
- [ ] 基础记忆系统（MEMORY.md + completed.json）
- [ ] 状态校验（启动时验证记忆格式）

**非技术要求**
- [ ] README：如何运行、环境变量配置
- [ ] 评测集文档：每个用例的预期行为说明

### 阶段二检查项（稳定化）

**技术要求**
- [ ] Harness 层（权限控制 + Audit Log）
- [ ] 成本追踪（CostTracker + 每日预算）
- [ ] 检查点机制（每阶段保存，支持断点续传）
- [ ] 优雅降级（失败返回降级报告，不是异常）
- [ ] 完成摘要报告（人类可读）
- [ ] 告警通知（超预算 / 失败 → 邮件/Slack）
- [ ] Prompt 版本化（存文件，不硬编码）

**非技术要求**
- [ ] CHANGELOG.md（记录每次改动）
- [ ] 运维文档（如何处理常见失败场景）
- [ ] 完整评测集（50+ 用例）

### 阶段三检查项（规模化）

**技术要求**
- [ ] 多 Agent 架构（Orchestrator + Sub-Agents）
- [ ] 模型分层配置（按角色选择模型）
- [ ] 自动化评测流水线（CI/CD 集成）
- [ ] 可视化 Dashboard（关键指标可视化）
- [ ] 灰度发布机制（新版本先 Canary 后全量）
- [ ] 记忆格式迁移脚本（当格式升级时）
- [ ] A/B 测试框架（新旧版本对比）

**非技术要求**
- [ ] 交接文档（新同事可以独立运维）
- [ ] 成本报告（定期向stakeholders汇报）
