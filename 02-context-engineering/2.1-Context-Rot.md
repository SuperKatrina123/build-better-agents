# 2.1 Context Rot：为什么上下文决定 Agent 的稳定性

> [!abstract] 本节导读
> 很多开发者遇到 Agent 变差时，第一反应是"换个更强的模型"或"改 Prompt"。
> 但大多数情况下，问题根源不在这里——**问题在上下文。**
> 本节解释 Context Rot 是什么、为什么发生、以及怎么识别它。
> 这是第二章所有内容的起点。
> **预计时长**：核心阅读 25 分钟。

---

> [!question] 带着问题阅读
> 你有没有遇到过：Agent 在任务前期表现很好，越到后面越乱？
> 这通常不是模型退化，而是 Context Rot 的症状。

---

## 1. 什么是 Context Rot

Transformer 的注意力机制有一个数学特性：**复杂度是 O(n²)**。
这意味着上下文越长，每个 token 对彼此的注意力权重计算量就越大，
而真正重要的信号，会被越来越多的噪声稀释。

![[context-rot-chart.excalidraw]]

> [!abstract] 定义
> **Context Rot（上下文腐化）**：随着对话轮次增加，无关内容在上下文中占比越来越高，导致模型对关键信号的注意力下降，决策质量系统性劣化的现象。

关键点在于：**有效信号的总量没有变，但它在整个上下文中的比例在缩小。**

信噪比 = 有效信号 / 全部上下文

当这个比值持续下降，Agent 就开始"遗忘"前期的关键决策，或者做出看似合理但实际偏差的判断。

---

## 2. Context Rot 的四种典型症状

你不需要等到 Agent 完全失控才能发现 Context Rot。它有四种早期信号：

### 症状一：答非所问，越答越偏

Agent 在第 10 轮时给出的回答，开始和第 1 轮的任务目标产生微妙偏差。
模型没有"忘记"目标，但目标被淹没在大量工具输出里，权重降低了。

### 症状二：反复确认已有信息

Agent 开始重复调用已经调用过的工具，或者询问已经在上下文里存在的信息。
这是注意力稀释的直接体现——模型"看不清"之前的结果了。

### 症状三：遇到矛盾时随机选择

当上下文里存在两段互相矛盾的内容（比如文档说 A，代码里是 B），
正常情况下 Agent 应该识别并处理矛盾；
Context Rot 状态下，它可能随机采用其中一个，而不报告矛盾。

### 症状四：长任务里"过早结束"

Agent 认为任务完成了，实际上还有子任务没处理。
这是因为任务清单被后续的工具输出"推离"了注意力焦点。

> [!warning] 误区
> **Context Rot 的症状看起来和"模型能力不足"几乎一模一样。**
> 最常见的错误是：不断换模型、改 Prompt，但根本没有动上下文结构。
> 结果当然是换了模型问题依然存在。

---

## 3. Context Rot 的三个根因

理解根因才能对症下药。Context Rot 通常由以下三件事叠加导致：

**根因一：什么都往上下文里塞**

最常见的做法是把所有可能用到的信息全部塞进 System Prompt：
规则、示例、背景知识、历史经验……结果是 System Prompt 越写越长，
真正关键的约束反而被淹没。

**根因二：工具输出原样保留**

每次工具调用的返回结果都完整保留在 messages[] 里。
一个读取代码文件的工具可能返回几百行内容，
十次调用下来，上下文里有几千行代码，但模型真正需要的可能只是其中的几个字段。

**根因三：稳定内容和动态内容混在一起**

"这个项目用 React 18"是稳定事实，永远不会变。
"当前任务进度是第 3 步"是动态状态，每轮都在更新。
把两者混在一起，每次更新动态状态时，都在重复传递那些稳定信息，消耗宝贵的注意力。

---

## 4. 实战视角：功能追齐 Agent 最容易发生 Context Rot 的地方

用我们的功能追齐 Agent 来具体看：

```
场景：追齐购物车功能，涉及 3 个仓库（App / H5 / BFF）

危险操作（会引发 Context Rot）：
  ❌ 一次性把三个仓库的所有文件全部读入上下文
  ❌ 把 App 仓库里 200 行的测试文件原样保留在 messages[]
  ❌ 每轮都把"项目背景：这是一个跨端电商项目……"的描述重复传给模型
  ❌ 把第一个功能点的追齐代码和第二个功能点的分析过程混在同一个 messages[]

健康操作：
  ✅ 只读取当前功能点需要的文件
  ✅ 工具返回的大文件做摘要后再写入 messages[]
  ✅ 项目背景放进常驻层，不重复传递
  ✅ 每个功能点用独立的 session，完成后只保留结论
```

> [!example] 案例
> 实际测试中，同一个追齐任务：
> - **无上下文管理**：第 8 轮开始出现字段映射错误，第 12 轮把已完成的功能重新标记为"待追齐"
> - **有上下文管理**：跑完 20 轮，关键约束始终清晰，没有出现遗忘或重复

## 5. 上下文工程的核心命题

Context Rot 揭示了一个更本质的问题：

**上下文不是"越多越好"，而是"越精准越好"。**

真正的上下文工程，不是把所有信息塞进去，而是在每一个时刻，
让模型只看到它**当前决策真正需要的信息**。

这需要系统性的设计，包括：
- 如何分层管理不同类型的信息（→ 2.2 五层上下文模型）
- 如何按需加载 Skills 和领域知识（→ 2.3 Skills 按需加载）
- 如何在上下文增长时主动压缩（→ 2.4 三种压缩策略）
- 如何利用缓存降低重复传递的成本（→ 2.5 Prompt Caching）

> [!tip] 原则
> **遇到 Agent 变差时，先检查上下文，再考虑换模型。**
> 上下文问题占 Agent 失效原因的比例，远超大多数人的预期。

## 诊断清单

> [!info] 方法
> 当 Agent 表现开始下滑时，用这个清单快速定位是否是 Context Rot：
>
> - [ ] messages[] 里有没有超过 500 行的工具返回内容？
> - [ ] System Prompt 超过 2000 tokens？且大部分内容不是每轮都需要的？
> - [ ] 同一个任务已经跑了超过 15 轮？
> - [ ] 上下文里有没有和当前任务无关的历史工具调用结果？
> - [ ] Agent 有没有重复调用已经调用过的工具？
>
> 如果超过 2 项是"是"，Context Rot 的概率很高。
> 下一步不是改 Prompt，而是清理上下文。

## 本节小结

- Context Rot = 上下文增长导致信噪比下降，Agent 决策质量系统性劣化
- 根本原因：什么都往上下文塞 + 工具输出不处理 + 稳定内容和动态内容混在一起
- 症状：答非所问、反复确认、遇到矛盾随机选择、过早结束
- **遇到问题先查上下文，再考虑换模型**
- 解决方案不是一个技巧，而是一套系统——从下一节开始展开
