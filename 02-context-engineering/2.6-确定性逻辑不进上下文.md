# 2.6 确定性逻辑不进上下文：让代码做代码该做的事

> [!abstract] 本节导读
> 上下文工程的最后一条核心原则，也是最容易被忽视的一条：
> **能用代码确定性实现的逻辑，就不应该用 Prompt 来描述，更不应该让 LLM 来执行。**
> 本节梳理哪些逻辑该进上下文、哪些不该，以及如何在追齐 Agent 里划清边界。
> **预计时长**：核心阅读 15 分钟。

---

> [!question] 带着问题阅读
> 你的 System Prompt 里有没有这样的描述：
> "如果功能已追齐，输出 status: done；如果未追齐，输出 status: pending"？
> 这个逻辑，真的需要 LLM 来判断吗？

---

## 1. 两类逻辑的本质区别

所有 Agent 需要执行的逻辑，可以分成两类：

**确定性逻辑**：给定相同的输入，永远输出相同的结果。
规则清晰，不需要"理解"，只需要执行。

**语义性判断**：需要理解语境、权衡多种可能性、处理模糊输入。
这才是 LLM 真正擅长的领域。

> [!abstract] 定义
> **"确定性逻辑不进上下文"原则**：任何可以用 if/else、正则、JSON Schema、函数来精确描述的逻辑，都应该在代码层实现，而不是通过 Prompt 告诉 LLM 去"理解和执行"。

---

## 2. 常见的确定性逻辑误放场景

### 场景一：格式校验和转换

```
❌ 放进 Prompt 的写法：
"请确保输出是合法的 JSON，字段包含 feature（字符串）、
status（枚举：done/pending/blocked）、changes（数组）。
如果 status 不在枚举范围内，请自动修正为 pending。"

✅ 正确做法：用 JSON Schema 校验 + 代码修正
```

```typescript
import { z } from "zod";

const AlignmentResultSchema = z.object({
  feature: z.string(),
  status: z.enum(["done", "pending", "blocked"]).default("pending"),
  changes: z.array(z.object({
    file: z.string(),
    type: z.enum(["add", "modify", "delete"]),
    description: z.string(),
  })),
});

// LLM 输出后，在代码层校验和修正
function parseAndValidate(llmOutput: string): AlignmentResult {
  const parsed = JSON.parse(llmOutput);
  return AlignmentResultSchema.parse(parsed); // 自动应用 .default()
}
```

### 场景二：路由判断

```
❌ 放进 Prompt：
"如果用户提到了购物车，就使用购物车相关的工具；
如果提到个人中心，就使用个人中心相关的工具。"

✅ 正确做法：Skill Router 在代码层做关键词匹配（见 2.3）
```

### 场景三：进度追踪和状态机

```
❌ 放进 Prompt：
"请记住已经追齐了哪些功能，不要重复追齐已完成的功能。"

✅ 正确做法：在代码层维护 Set<string> 已完成功能列表，
每次任务开始前代码层检查，过滤掉已完成的。
```

### 场景四：数据计算

```
❌ 放进 Prompt：
"请计算本次追齐影响了多少个文件，并统计每种变更类型的数量。"

✅ 正确做法：LLM 只负责识别变更，计数和统计在代码层完成
```

## 3. 划清边界的判断标准

只问一个问题：

> **这个判断，如果换一个人来做，结果会一样吗？**

- 格式是否合法 → 对任何人都一样 → 代码层
- status 是否在枚举范围 → 对任何人都一样 → 代码层
- 这段代码实现了 App 端的功能吗 → 需要理解代码语义 → LLM
- 这个差异是否需要追齐 → 需要权衡业务影响 → LLM

> [!tip] 原则
> **LLM 负责"理解和判断"，代码负责"校验和执行"。**
> 边界划清之后，System Prompt 会变短，上下文会更干净，Agent 更稳定，成本更低。

---

## 4. 功能追齐 Agent 的边界划分

```
代码层负责（不进上下文）：
├── 输出格式校验（Zod Schema）
├── 已完成功能的去重过滤
├── 仓库名称 → Skill 文件的映射（repo-map）
├── 文件变更的统计汇总
├── 对话轮次超限时触发压缩
└── 工具返回内容超长时的预处理摘要

LLM 负责（进上下文）：
├── 判断 App 端某功能 H5 是否已实现
├── 分析功能差异的根本原因
├── 决定追齐方案（修改哪个文件、如何修改）
├── 识别潜在的兼容性风险
└── 在多种实现方案中选择最合适的
```

> [!example] 案例
> 早期版本的追齐 Agent，把"已完成功能去重"写在 System Prompt 里：
> `"注意：以下功能已追齐完成，不要重复处理：${completedFeatures.join(', ')}"。`
> 结果随着 completedFeatures 增长，这行提示越来越长，
> 最终占到 System Prompt 的 40%，而且 LLM 偶尔还是会"忘记"去重。
>
> 改为代码层维护 Set，在传入任务列表之前就过滤：
> - System Prompt 缩短 40%
> - 去重准确率从 ~90% 提升到 100%
> - 缓存命中率同步提升（System Prompt 更稳定）

## 5. 一个容易混淆的案例

**问题**：错误处理应该放在代码层还是 LLM 层？

**答案**：看错误的性质。

```
类型一：技术错误（文件不存在、网络超时）→ 代码层 try/catch
类型二：业务判断（"这个错误是否值得重试"）→ LLM 决策
类型三：可预期的边界情况（"文件不存在说明该功能还未实现"）→ 代码层规则
```

```typescript
// ✅ 代码层处理技术错误 + 可预期边界，LLM 处理真正的业务判断
async function readRepoFileWithFallback(repo: string, path: string) {
  try {
    return await readFile(repo, path);
  } catch (e) {
    if (e.code === "ENOENT") {
      // 可预期：文件不存在 = 功能未实现，直接告知 Agent
      return `[文件不存在: ${path}，该功能在 H5 端尚未实现]`;
    }
    throw e; // 非预期错误，向上抛出，由 Harness 处理
  }
}
```

## 本节小结

- 确定性逻辑（格式校验、路由、状态机、计数）→ 代码层实现
- 语义性判断（理解差异、选择方案、权衡风险）→ LLM 决策
- 判断标准：换个人来做结果会一样吗？一样 → 代码层
- 边界划清后：System Prompt 变短、稳定性提升、缓存命中率提高
