# 2.2 五层上下文模型：给信息找到它该待的位置

> [!abstract] 本节导读
> 上一节我们知道了 Context Rot 的根因之一：**什么都往上下文里塞**。
> 解决这个问题需要一个系统性的分层框架——不同类型的信息有不同的"最佳位置"。
> 本节介绍五层上下文模型，并把它直接映射到功能追齐 Agent 的实际设计上。
> **预计时长**：核心阅读 30 分钟。

---

> [!question] 带着问题阅读
> 你的 System Prompt 里有没有放一些"以防万一"的背景信息？
> 它们真的每一轮都需要出现在上下文里吗？

---

## 1. 为什么要分层

先看一张图，直观感受一下分层的思路：

![上下文分层模型|588](https://gw.alipayobjects.com/zos/k/u8/context_layers.svg)

分层的核心逻辑只有一句话：

> **按"稳定性 × 使用频率"把信息分到不同的层，每层只放它该放的东西。**

信息可以按两个维度来分类：
- **稳定性**：这段信息会不会变？（任务目标 > 技术约定 > 当前状态 > 工具结果）
- **使用频率**：这段信息每轮都需要用到吗？（角色定义 > 仓库地址 > 当前功能点 > 某次读文件的结果）

把稳定且高频的信息放在底层（常驻），把不稳定且低频的信息放在顶层（按需或滚动清理）。

---

## 2. 五个层次详解

![[five-layer-context-model.excalidraw]]

### L1：System Prompt 层（常驻）

**放什么**：Agent 的角色定义、任务目标、不可违背的核心约束、输出格式规范。

**特征**：每次推理都必须存在，内容几乎不变，是 Agent 行为的"宪法"。

**常见错误**：把大量"可能用到"的背景信息都写进 System Prompt，导致它越来越长。
正确做法是：System Prompt 只放**每一轮都需要的**最小集合。

```
功能追齐 Agent 的 L1 内容：
- 角色：你是一个功能追齐工程师，负责把 H5 页面功能对齐到 App 最新版本
- 目标：分析差异 → 生成追齐方案 → 输出可执行的代码变更
- 约束：不修改与追齐无关的代码；有疑问时先报告不要猜测
- 输出格式：JSON {feature, status, changes[]}
```

---

### L2：工具定义层（常驻，会话级）

**放什么**：工具的 Schema、调用协议、参数类型约束、返回格式说明。

**特征**：每次会话开始时加载，会话期间不变。工具定义通常由框架自动管理，不需要手动塞进 messages[]。

**注意**：工具数量不宜过多，每增加一个工具都在消耗固定的上下文预算。
不常用的工具应该从注册列表中移除，而不是"注册了但不用"。

```
功能追齐 Agent 的 L2 内容：
- read_repo_file(repo, path) → 文件内容
- list_repo_files(repo, glob) → 文件列表
- diff_feature(app_repo, h5_repo, feature) → 功能差异报告
- resolve_feature_spec(feature_name) → App 端功能规格
```

---

### L3：Skills 层（按需加载）

**放什么**：当前任务相关的领域知识——仓库 README、技术规范文档、操作手册、代码模式示例。

**特征**：不是每个任务都需要，**只在相关时加载**。这是减少 Context Rot 最有效的手段之一。

**实现方式**：
- 把 Skills 预先切割成独立的 Markdown 文件（每个文件 ≤ 2000 tokens）
- Agent 在规划阶段判断需要哪些 Skills，通过工具动态注入
- 任务切换时，卸载上一个任务的 Skills，加载下一个任务的 Skills

> [!tip] 原则
> **Skills 的粒度应该和任务的粒度匹配。**
> 追齐"购物车页面"的 Skill，不应该包含"个人中心页面"的仓库信息。
> 粒度越细，加载越精准，上下文越干净。

```
功能追齐 Agent 的 L3 示例：
- skill/cart-repo.md      → 购物车 H5 仓库摘要 + 目录结构 + 关键文件路径
- skill/cart-app-spec.md  → App 购物车功能规格（从测试用例提取）
- skill/react-patterns.md → 项目通用的 React 代码模式
```

---

### L4：工作记忆层（动态更新）

**放什么**：当前任务的执行状态——已完成的步骤、待处理项、中间结论、临时草稿。

**特征**：每轮可能更新，但应该保持紧凑。这层不是用来"记录历史"的，而是用来"维持当前任务状态"的。

**实现方式**：
- 用结构化格式（JSON / Markdown 列表）维护一个"当前状态快照"
- 每轮推理后更新快照，旧的状态不需要保留在 messages[] 里
- 快照大小控制在 500 tokens 以内

```json
// 工作记忆示例（购物车追齐任务）
{
  "task": "购物车页面功能追齐",
  "progress": "3/7",
  "completed": ["商品数量展示", "价格计算", "优惠券展示"],
  "current": "促销标签展示",
  "pending": ["批量删除", "收藏功能", "结算跳转"],
  "blockers": []
}
```

---

### L5：对话流层（滚动清理）

**放什么**：当前轮次的工具调用结果、助手回复、用户输入。

**特征**：更新最频繁，也最容易引发 Context Rot。**工具返回的大内容不能原样保留**。

**管理策略**：
- 工具返回超过 200 行的内容，先做摘要再写入 messages[]
- 超过 N 轮后，自动触发对话历史压缩（→ 2.4 三种压缩策略）
- 与当前任务无关的历史轮次，定期清理出 messages[]

> [!warning] 误区
> **不要把工具返回的文件内容原样塞进 messages[]，然后期待模型"自己找到有用的部分"。**
> 模型会努力处理，但代价是大量 token 消耗在无关内容上。
> 应该在工具层做预处理，只把"有效结论"返回给模型。

## 3. 如何应用到功能追齐 Agent

把五层模型映射到功能追齐 Agent 的完整设计：

```
┌─────────────────────────────────────────────────────┐
│  L5 对话流层（滚动）                                 │
│  当前功能点分析结果（摘要后）· 工具调用结果（摘要后） │
├─────────────────────────────────────────────────────┤
│  L4 工作记忆层（动态）                               │
│  当前追齐任务状态 JSON · 已完成功能列表               │
├─────────────────────────────────────────────────────┤
│  L3 Skills 层（按需）                                │
│  当前页面仓库摘要 · 该页面 App 功能规格               │
├─────────────────────────────────────────────────────┤
│  L2 工具定义层（常驻）                               │
│  4 个工具 Schema（read / list / diff / resolve）     │
├─────────────────────────────────────────────────────┤
│  L1 System Prompt 层（常驻）                         │
│  角色 · 追齐目标 · 约束 · 输出格式                   │
└─────────────────────────────────────────────────────┘
```

**追齐"购物车"功能时的上下文预算分配（参考值）**：

| 层次 | 内容 | Token 预算 |
|------|------|-----------|
| L1 System Prompt | 角色 + 约束 + 格式 | ~400 |
| L2 工具定义 | 4 个工具 Schema | ~600 |
| L3 Skills | 购物车仓库摘要 + 功能规格 | ~2000 |
| L4 工作记忆 | 当前任务状态 JSON | ~300 |
| L5 对话流 | 最近 8 轮（摘要后） | ~3000 |
| **总计** | | **~6300** |

对比没有分层的做法（把所有仓库全量读入），可能轻松超过 50,000 tokens，
而且大部分都是当前任务用不到的内容。

> [!example] 案例
> 不使用分层：启动追齐任务时，把 15 个仓库的 README 全部读入。
> 结果：上下文立即被仓库信息占满，第 3 轮开始 Agent 就"忘记"了追齐的输出格式要求。
>
> 使用五层模型：只加载当前页面对应的 2 个仓库摘要（从 MEMORY.md repo-map 按需读取）。
> 结果：整个追齐流程跑完 20 轮，核心约束始终清晰。

---

## 4. 分层不是一次性设计

五层模型不是在项目开始时设计好就不动了的。
随着 Agent 迭代，你会持续调整：

- **L1 膨胀时**：审查每一条，问"这真的每轮都需要吗？" → 不需要的移到 L3
- **L3 过大时**：切分 Skills 文件，提高加载粒度
- **L4 失控时**：重新评估工作记忆的格式，精简字段
- **L5 引发 Context Rot 时**：调低触发压缩的轮次阈值（→ 2.4）

> [!info] 方法
> **分层 Review 频率建议**：
> 每次 Agent 出现质量下滑，先 review 一遍各层的内容和大小。
> 通常问题出在某一层"悄悄膨胀了"。
> 每隔 2 周做一次主动 review，而不是等出问题再查。

---

## 本节小结

- 五层模型按"稳定性 × 使用频率"分层：L1（常驻）→ L5（滚动）
- L1 只放每轮必须的内容；L3 Skills 是减少 Context Rot 的核心手段
- 功能追齐 Agent 的分层实践：6300 tokens vs 未分层的 50,000+ tokens
- 分层是持续维护的工程，不是一次性设计
