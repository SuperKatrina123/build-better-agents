# 第二章：上下文工程——Agent 的神经系统

> Agent 的能力边界由模型决定，但 Agent 的稳定性由上下文决定。

---

## 章节目标

完成本章后，你应该能够：
1. 诊断 Agent 变差的根本原因是否是 Context Rot
2. 用五层模型设计新 Agent 的上下文结构
3. 为功能追齐 Agent 完成一份上下文预算规划文档
4. 实现 Skills 按需加载 + 对话流压缩 + Prompt Caching 的完整上下文管理方案

---

## 通用原理

上下文工程解决的不是“Prompt 怎么写更漂亮”，而是“模型在每一轮决策前，究竟看到了什么，以及为什么这样装配”。它面对的是所有 Agent 的共同问题：

- 哪些信息应该常驻，哪些应该按需加载
- 哪些信息应该滚动更新，哪些应该压缩或外置
- 当上下文持续增长时，如何防止注意力被噪声拖垮

只要 Agent 依赖上下文做运行时判断，本章讨论的分层、装配、压缩、缓存和边界划分就都成立。

## 场景映射

| Agent 类型 | 本章关注对象 | 高频上下文内容 | 典型风险 |
|-----------|-------------|---------------|---------|
| Coding Agent | 当前功能点、仓库结构、工具结果 | 文件摘要、功能规格、修改状态 | 工具输出过长、历史分析污染当前任务 |
| Research Agent | 研究目标、来源质量、阶段结论 | 主题、证据片段、可信度标注 | 信息堆积、无关资料挤占关键信号 |
| Support Agent | 用户问题、历史工单、知识库命中 | 用户画像、工单状态、候选答案 | 旧工单信息干扰当前回复、知识片段过载 |
| 流程 / 业务执行 Agent | 当前节点、业务规则、系统返回 | 表单字段、审批状态、接口结果 | 旧状态残留、规则与实时状态混杂 |

## 主案例展开：功能追齐 Agent

```
┌─────────────────────────────────────────────┐
│ L5 对话流（滚动，≤8轮 / ~4000 tokens）       │
│ 工具调用结果（摘要后）· 推理过程             │
├─────────────────────────────────────────────┤
│ L4 工作记忆（动态，≤300 tokens）             │
│ 任务进度 JSON · 已完成功能列表               │
├─────────────────────────────────────────────┤
│ L3 Skills（按需，≤2个文件 / ~2000 tokens）   │← 缓存断点
│ 当前页面 H5 仓库摘要 · App 功能规格          │
├─────────────────────────────────────────────┤
│ L2 工具定义（常驻，~600 tokens）             │← 缓存断点
│ 4个工具 Schema                              │
├─────────────────────────────────────────────┤
│ L1 System Prompt（常驻，~400 tokens）        │← 缓存断点
│ 角色 · 追齐目标 · 核心约束 · 输出格式        │
└─────────────────────────────────────────────┘
总预算：~10,300 tokens（含 2000 tokens 输出预算）
```

这里的五层配置、预算数字和缓存断点，都是“功能追齐 Agent”这一主案例的具体实例。真正需要迁移的是：

- 五层模型本身
- 常驻 / 按需 / 滚动三类信息的划分方式
- Context Rot 的诊断与治理思路
