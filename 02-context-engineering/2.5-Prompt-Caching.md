# 2.5 Prompt Caching：让稳定内容"免费复用"

> [!abstract] 本节导读
> 压缩解决"上下文过长"的质量问题，Caching 解决"重复传递稳定内容"的成本问题。
> 两者都很重要，但开发者容易只关注前者而忽略后者。
> 本节讲清楚 Prompt Caching 的工作原理、命中条件、以及怎么设计才能最大化缓存利用率。
> **预计时长**：核心阅读 20 分钟。

---

> [!question] 带着问题阅读
> 每次调用 Agent 时，System Prompt 和 Skills 都会被完整地发送给模型。
> 如果它们占了 3000 tokens，一天调用 100 次，就是 300,000 tokens 的重复传递。
> 有没有办法让这些内容只被处理一次？

---

## 1. Prompt Caching 的工作原理

现代 LLM API（如 Anthropic Claude）支持 Prompt Caching：
把上下文的**稳定前缀**缓存在服务器端，后续调用只需要传递**变化的部分**。

![[prompt-caching-prefix.excalidraw]]

**缓存的本质**：KV Cache（Key-Value Cache）——Transformer 计算注意力时，稳定前缀的 Key 和 Value 矩阵被缓存，后续请求直接复用，不需要重新计算。

**收益**：
- 缓存命中的 tokens **不计入输入费用**（或按极低折扣计费）
- 推理延迟也会降低（缓存部分跳过计算）

**限制**：
- 缓存前缀必须**字节完全一致**才能命中
- 缓存有 TTL（通常 5 分钟，具体看 API 文档）
- 需要在 API 调用时显式标记缓存断点

---

## 2. 如何启用（以 Anthropic API 为例）

在消息里加 `cache_control` 标记，告诉 API 在哪个位置建立缓存断点：

```typescript
const response = await anthropic.messages.create({
  model: "claude-opus-4-5",
  max_tokens: 2048,
  system: [
    {
      type: "text",
      text: systemPrompt,
      cache_control: { type: "ephemeral" }  // L1: System Prompt 缓存点
    }
  ],
  messages: [
    {
      role: "user",
      content: [
        {
          type: "text",
          text: skillContent,              // L3: Skills 内容
          cache_control: { type: "ephemeral" }  // Skills 缓存点
        },
        {
          type: "text",
          text: workingMemory             // L4: 工作记忆（不缓存）
        },
        {
          type: "text",
          text: userMessage               // L5: 当前输入（不缓存）
        }
      ]
    }
  ]
});
```

> [!info] 方法
> `cache_control: { type: "ephemeral" }` 的意思是：
> **在这个 token 位置建立一个缓存断点**。
> 断点之前的所有内容会被缓存；断点之后的内容每次都重新处理。
> 可以设置多个缓存断点，分别对应不同的稳定层。

---

## 3. 五层模型天然匹配缓存设计

这是五层模型最美妙的地方之一：

**越稳定的层放越前面 → 缓存前缀越长 → 缓存利用率越高**

```
context 顺序（从前到后）：

[L1 System Prompt] ← 缓存断点 1（最长有效期）
[L2 工具定义]      ← 缓存断点 2
[L3 Skills]        ← 缓存断点 3（任务切换时失效）
[L4 工作记忆]      （每轮更新，不缓存）
[L5 对话流]        （每轮增长，不缓存）
```

**追齐 Agent 的缓存收益估算**：

| 部分 | tokens | 缓存命中率 | 节省 |
|------|--------|-----------|------|
| System Prompt（L1） | 400 | ~95% | 380 tokens/call |
| 工具定义（L2） | 600 | ~95% | 570 tokens/call |
| Skills（L3）| 2000 | ~70% | 1400 tokens/call |
| 合计 | 3000 | — | ~2350 tokens/call |

一天 100 次调用 → 节省约 235,000 tokens 的输入计费。

---

## 4. 缓存失效的常见坑

> [!warning] 误区
> **缓存失效 = 前缀有任何改动**。以下操作都会导致全量重新计费：
>
> 1. **System Prompt 末尾加了个空格** → 缓存失效
> 2. **Skills 文件内容更新了** → 从该断点往后的缓存失效
> 3. **两次调用之间超过 TTL（5 分钟空闲）** → 缓存失效
> 4. **动态拼接导致顺序不稳定** → 每次前缀不同，从不命中
>
> **最常见的坑**：把时间戳或随机 ID 拼进 System Prompt。
> 比如 `"当前时间：${new Date().toISOString()}"` 这类内容，
> 只要放在缓存断点之前，就会导致缓存永远无法命中。

## 5. 功能追齐 Agent 的缓存策略

```typescript
// 工具追齐 Agent 缓存设计要点

// ❌ 错误：动态内容混入稳定层
const badSystemPrompt = `
  你是一个功能追齐工程师。
  当前时间：${new Date().toISOString()}  // 🚫 破坏缓存
  任务开始于：${taskStartTime}           // 🚫 破坏缓存
`;

// ✅ 正确：动态内容放在 L4/L5，不进缓存前缀
const goodSystemPrompt = `
  你是一个功能追齐工程师，负责把 H5 页面功能对齐到 App 最新版本。
  目标：分析差异 → 生成追齐方案 → 输出可执行的代码变更。
  约束：不修改与追齐无关的代码；有疑问时先报告不要猜测。
`;  // 纯稳定内容 → 高缓存命中率

// 时间、任务ID等动态内容放在工作记忆（L4）里传递
const workingMemory = JSON.stringify({
  taskId: currentTaskId,
  startTime: taskStartTime,
  currentFeature: "批量删除",
  progress: "2/7"
});
```

> [!example] 案例
> 实践数据（基于真实 Claude API 调用）：
> 开启 Prompt Caching 前，追齐一个完整页面（约 50 次 API 调用）的 token 消耗约 180,000。
> 开启后，同样的任务消耗约 65,000 tokens（节省 64%）。
> 对应的 API 费用也同比下降。

---

## 本节小结

- Prompt Caching = 稳定前缀缓存，命中时前缀部分不计费
- 五层模型天然匹配缓存：稳定层（L1/L2/L3）在前，动态层（L4/L5）在后
- 最大的坑：动态内容（时间戳、任务 ID）混入缓存前缀，导致永远不命中
- 追齐 Agent 实测：开启 Caching 后 token 消耗降低约 64%
