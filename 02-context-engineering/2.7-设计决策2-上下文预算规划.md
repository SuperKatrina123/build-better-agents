# 2.7 设计决策 2：上下文预算规划

> [!abstract] 本节导读
> 第二章的六个概念学完了。
> 现在把它们整合成一个**动手之前必做的动作**：上下文预算规划。
> 这是本章的收束节，也是第三章（工具设计）的前置。
> **预计时长**：核心阅读 20 分钟。

---

## 1. 为什么要做预算规划

没有做过预算规划的 Agent，通常会在以下阶段出问题：

- **MVP 阶段**：上下文太小，Agent 频繁丢失关键信息
- **压力测试阶段**：长任务跑到 20 轮后质量骤降
- **规模化阶段**：token 成本远超预期，但不知道哪里在浪费

上下文预算规划的意义在于：**在写第一行代码之前，就把上下文窗口的分配方案想清楚。**

---

## 2. 上下文预算规划模板

针对任何一个 Agent，回答以下 7 个问题：

### Q1：目标模型的上下文窗口是多少？
（Claude Opus 4：200K tokens；Claude Sonnet 4.5：200K tokens；GPT-4o：128K tokens）

### Q2：L1 System Prompt 分配多少？
原则：≤ 800 tokens。超过这个数说明有内容应该移到 L3 Skills。

### Q3：L2 工具定义分配多少？
注册几个工具？每个工具 Schema 大约多少 tokens？
原则：工具总数 ≤ 10 个，总 tokens ≤ 1500。

### Q4：L3 Skills 单次最多加载多少？
结合任务粒度，一次任务最多加载几个 Skill 文件？每个文件多大？

### Q5：L4 工作记忆上限是多少？
工作记忆的 JSON 结构设计好了吗？最多占多少 tokens？

### Q6：L5 对话流分配多少？保留最近几轮？
触发压缩的阈值是什么？压缩策略是截断还是摘要？

### Q7：输出 token 预算是多少？
模型输出也消耗 token（且通常更贵）。每次调用的预期输出有多长？

---

## 3. 功能追齐 Agent 的预算规划

| 层次 | 分配预算 | 说明 |
|------|---------|------|
| L1 System Prompt | 400 tokens | 角色 + 约束 + 格式，不含项目背景 |
| L2 工具定义 | 600 tokens | 4 个工具，每个 ~150 tokens |
| L3 Skills | 2000 tokens | 最多 2 个 Skill 文件，每个 ~1000 tokens |
| L4 工作记忆 | 300 tokens | 任务状态 JSON，始终 ≤ 300 tokens |
| L5 对话流 | 4000 tokens | 保留最近 8 轮（每轮 ~500 tokens） |
| 输出预算 | 2000 tokens | 追齐方案 JSON + 说明 |
| **安全缓冲** | **1000 tokens** | 预留给工具返回的摘要 |
| **总计** | **~10,300 tokens** | 仅用 200K 窗口的 5%，空间充裕 |

**触发机制**：
- 对话流 > 8 轮 → 截断，保留最近 8 轮
- 对话流 > 20 轮 → 对 8~20 轮做摘要，保留最近 8 轮原文
- 单次工具返回 > 200 行 → 代码摘要预处理
- Skills 切换 → 旧 Skills 从 messages[] 移除

**Caching 策略**：
- L1 + L2 + L3 作为缓存前缀（按 L1 → L2 → L3 顺序排列）
- 绝不在 L1/L2/L3 里放动态内容（时间戳、任务 ID 等）

---

## 4. 设计决策清单

> [!info] 方法
> 在开始写 Agent 代码之前，完成这份清单：
>
> **上下文结构**
> - [ ] System Prompt ≤ 800 tokens，且每条内容都是每轮必需的
> - [ ] 工具总数 ≤ 10 个，移除不常用工具
> - [ ] Skills 文件已按功能模块切割，每个 ≤ 2000 tokens
> - [ ] repo-map 已配置，任务→Skills 的映射清晰
>
> **压缩策略**
> - [ ] 定义了 L5 对话流的轮次上限和触发截断的阈值
> - [ ] 超长工具返回（> 200 行）有预处理摘要逻辑
> - [ ] L4 工作记忆格式已设计，大小有上限
>
> **Caching 配置**
> - [ ] L1/L2/L3 里没有动态内容（时间戳、ID 等）
> - [ ] API 调用时正确设置了 cache_control 断点
>
> **边界划分**
> - [ ] 格式校验用 Zod Schema，不在 Prompt 里描述格式规则
> - [ ] 状态机和路由逻辑用代码实现，不用 Prompt 描述

## 第二章总结

| 节次 | 核心概念 | 一句话要点 |
|------|---------|-----------|
| 2.1 | Context Rot | 信噪比下降是 Agent 变差的根本原因 |
| 2.2 | 五层上下文模型 | 按稳定性分层，每层只放它该放的东西 |
| 2.3 | Skills 按需加载 | 领域知识模块化，运行时按任务按需注入 |
| 2.4 | 三种压缩策略 | 截断 / 摘要 / 选择性保留，按任务长度选择 |
| 2.5 | Prompt Caching | 稳定前缀免费复用，五层结构天然匹配缓存 |
| 2.6 | 确定性逻辑不进上下文 | 代码做校验和执行，LLM 做理解和判断 |
| 2.7 | 上下文预算规划 | 动手写代码之前先把窗口分配想清楚 |

> [!tip] 原则
> **上下文不是越多越好，而是越精准越好。**
> 好的上下文工程，是在每一个时刻让模型只看到它当前决策真正需要的信息。
