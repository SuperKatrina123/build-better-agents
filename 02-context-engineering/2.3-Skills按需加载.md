## 1. 什么是 Skills

Skills 是把领域知识预先整理成的**可独立加载的 Markdown 文件**。

它不是 System Prompt，不是工具定义，而是介于两者之间的东西：
比 System Prompt 更具体、更换可拆卸，比工具定义更贴近业务语义。

典型的 Skill 内容包括：
- 某个仓库的目录结构和关键文件路径
- 某个功能模块的技术规范
- 某种代码模式的使用示例
- 某个业务流程的操作手册

Skills 的核心优势：**任务切换时，可以卸载旧 Skills，加载新 Skills**，而不用重启整个会话。

## 2. Skill 文件的结构规范

一个 Skill 文件应该满足三个条件：

**条件一：独立可读**
不依赖其他 Skill 就能理解和使用，每个文件解决一个明确的问题域。

**条件二：粒度适中**
太粗（整个仓库的所有信息）→ 浪费上下文；
太细（只有一个函数的文档）→ 加载判断成本太高。
推荐：每个 Skill 文件 **1000~2000 tokens**，覆盖一个自然的功能模块。

**条件三：结构清晰**
用固定的模板，让模型快速定位关键信息，不要写成流水账。

```markdown
<!-- skill/cart-repo.md 示例 -->
# 购物车 H5 仓库摘要

## 仓库信息
- 仓库名: h5-cart-page
- 框架: React 18 + TypeScript
- 入口: src/pages/Cart/index.tsx

## 目录结构（关键路径）
src/
├── pages/Cart/
│   ├── index.tsx          # 页面入口
│   ├── components/        # 购物车专属组件
│   └── hooks/             # 业务 hooks
├── store/cart/            # Redux slice
└── api/cart.ts            # 接口定义

## 代码约定
- 状态管理：Redux Toolkit + RTK Query
- 样式方案：CSS Modules（文件名 *.module.css）
- 组件命名：大驼峰，文件和组件名一致

## 快速定位
| 功能点 | 文件路径 |
|--------|---------|
| 商品列表 | components/CartItemList.tsx |
| 价格计算 | hooks/useCartPrice.ts |
| 提交订单 | pages/Cart/index.tsx → handleCheckout |
```

## 3. Skills 的组织方式

![[skills-loading-flow.excalidraw]]

### repo-map.config.ts：Skills 的索引

Skills 本身是文件，但 Agent 需要知道"哪个任务对应哪些 Skills"。
这个映射关系存在 `repo-map.config.ts` 里：

```typescript
// repo-map.config.ts
export const repoMap: RepoMapConfig = {
  pages: [
    {
      name: "购物车",
      keywords: ["cart", "购物车", "shopping"],
      skills: [
        "skill/cart-repo.md",      // H5 仓库摘要
        "skill/cart-app-spec.md",  // App 端功能规格
      ],
      h5Repo: "h5-cart-page",
      appRepo: "app-cart-module",
    },
    {
      name: "个人中心",
      keywords: ["profile", "个人中心", "user"],
      skills: [
        "skill/profile-repo.md",
        "skill/profile-app-spec.md",
      ],
      h5Repo: "h5-profile-page",
      appRepo: "app-profile-module",
    },
    // ... 其余 13 个页面
  ],
};
```

### Skill Router：匹配任务到 Skills

Agent 在规划阶段会调用一个简单的工具来查 repo-map：

```typescript
// tool: resolve_skills
// 输入：任务描述
// 输出：需要加载的 Skill 文件列表
async function resolveSkills(taskDescription: string): Promise<string[]> {
  const matched = repoMap.pages.filter(page =>
    page.keywords.some(kw => taskDescription.includes(kw))
  );
  return matched.flatMap(page => page.skills);
}
```

> [!tip] 原则
> **Skill Router 越简单越好。**
> 用关键词匹配而不是语义搜索，减少不确定性。
> 如果模糊匹配率高，宁可多加几个关键词，也不要引入向量检索。
> 只有当仓库超过 50 个时，才值得考虑向量化 Skills 索引。

## 4. 运行时加载流程

一个完整的 Skill 加载生命周期：

```
1. 用户输入任务："追齐购物车的批量删除功能"

2. Orchestrator 调用 resolveSkills("追齐购物车的批量删除功能")
   → 返回: ["skill/cart-repo.md", "skill/cart-app-spec.md"]

3. Orchestrator 读取两个文件内容，注入到当前轮次的上下文（L3 层）

4. 子 Agent 执行任务，全程可以引用这两个 Skill 的内容

5. 任务完成（或切换到新功能点）
   → 旧 Skill 内容不再保留在 messages[] 里
   → 下一个任务重新走步骤 1-4
```

> [!warning] 误区
> **不要在 Agent 启动时一次性把所有 Skills 都加载进来"备用"。**
> 15 个页面 × 2 个 Skill 文件 = 30 个文件 × ~1500 tokens = 45,000 tokens 的固定开销。
> 而实际上每次任务只需要其中的 2 个。

## 5. App 功能规格 Skill：从测试用例提取

在功能追齐 Agent 里，一个特别重要的 Skill 类型是 **App 端功能规格**。

飞书文档的描述往往不精确，但 App 端的 E2E 测试用例通常很可靠。
可以写一个预处理脚本，把测试用例转换成结构化的 Skill 文件：

```markdown
<!-- skill/cart-app-spec.md（由测试用例生成） -->
# 购物车 App 端功能规格（v2.3.1）

## 核心功能点

### F1：商品列表展示
- 展示商品图片、名称、规格、单价、数量
- 库存不足时显示"库存不足"标签，并禁用数量加减
- 已下架商品显示"商品已下架"，不可勾选

### F2：批量删除
- 点击"编辑"进入编辑模式
- 支持单选 / 全选
- 已选商品数量 > 0 时"删除"按钮高亮可点
- 删除确认弹窗文案："确认删除选中的 N 件商品？"

### F3：价格计算
- 勾选商品实时更新"合计"金额
- 优惠券抵扣后显示"已优惠 ¥X"
- 不含运费，运费在结算页展示
```

> [!example] 案例
> 实践中发现：基于测试用例生成的 App Spec Skill，
> 比飞书文档手写的需求描述，追齐准确率高出约 30%。
> 原因很简单：测试用例描述的是**实际行为**，不是意图。

## 本节小结

- Skills = 预整理的领域知识，1000~2000 tokens/文件，结构清晰可独立读懂
- repo-map.config.ts 维护任务→Skills 的映射，Skill Router 运行时匹配
- 加载策略：只加载当前任务相关的 Skills，任务切换时卸载
- App 端 Skill 优先从测试用例提取，比需求文档更可靠
