# 设计模式库

> [!abstract] 文档定位
> 把整套课程的方法论收束成一组可复用的设计模式和反模式。
> 不是让你"省去思考"，而是让你在新项目里不用每次从白纸开始重建判断。
> **前置建议**：至少读完第 1-4 章后再来查这页效果最好。

---

> [!question] 使用方式
> 开始设计一个新 Agent 时，先扫描这页找到最接近你场景的模式；
> 遇到系统问题时，先对照反模式诊断原因；
> 团队对齐时，用这页的术语建立共同语言。

---

## 一、上下文管理模式

### 模式 1：五层预算模型

**解决问题**：上下文资源有限，如何把最重要的信息放在最有效的位置？

**结构**：
```
L1 System Prompt（稳定，Prompt Caching 对象）
L2 工具定义（稳定，Prompt Caching 对象）
L3 工作指令（当前任务，每次任务更新）
L4 工作记忆（动态摘要，滚动压缩）
L5 对话流（工具调用历史，随时压缩）
```

**适用**：所有 Agent，是最基础的上下文分层模板
**来源**：[2.2 五层上下文模型](../02-context-engineering/2.2-五层上下文模型.md)

---

### 模式 2：Skills 按需加载

**解决问题**：System Prompt 越来越长，LLM 被无关指令干扰

**结构**：把工具的详细使用说明从 System Prompt 里拆出来 → 只在需要用该工具时注入 L3

**判断标准**：
- 工具是否只在特定任务类型中使用？→ 按需加载
- 工具是否每次任务都会用？→ 留在 System Prompt

**适用**：工具数量 > 5 个的 Agent
**来源**：[2.3 Skills 按需加载](../02-context-engineering/2.3-Skills按需加载.md)

---

### 模式 3：确定性逻辑不进上下文

**解决问题**：把可以用代码判断的逻辑交给 LLM，引入不必要的不确定性

**结构**：
```
✅ 进上下文：意图理解、代码分析、方案生成（需要 LLM 的理解力）
❌ 不进上下文：权限检查、格式验证、计数、路径匹配（代码更可靠）
```

**适用**：所有 Agent，特别是有安全要求的场景
**来源**：[2.6](../02-context-engineering/2.6-确定性逻辑不进上下文.md)、[6.2](../06-harness/6.2-确定性逻辑不交给LLM.md)

---

## 二、工具设计模式

### 模式 4：ACI 优先设计

**解决问题**：工具设计从"代码实现"视角出发，LLM 经常调用错误或传错参数

**核心原则**：
1. 名称：动词 + 对象（`read_repo_file`，不是 `getFile`）
2. 描述：说明"何时调用"，不只说"做什么"
3. 参数：最少必要参数；用 enum 限制合法值；用 description 说明期望格式
4. 返回值：结构化 > 纯文本；失败时返回可自我纠正的错误信息

**适用**：所有工具接口设计
**来源**：[3.1 ACI](../03-tool-design/3.1-什么是ACI.md)、[3.2](../03-tool-design/3.2-输入设计.md)、[3.3](../03-tool-design/3.3-输出设计.md)

---

### 模式 5：工具粒度三问

**解决问题**：工具做得太细（调用次数爆炸）或太粗（LLM 无法精确控制）

**判断框架**：
1. LLM 需要独立控制这个操作的时机吗？→ 需要 = 拆细
2. 这两个操作总是一起发生吗？→ 总是 = 合并
3. 一次调用返回的信息量会撑爆上下文吗？→ 会 = 拆细并加分页

**适用**：工具设计评审
**来源**：[3.4 工具粒度](../03-tool-design/3.4-工具粒度.md)

---

## 三、记忆系统模式

### 模式 6：MEMORY.md 跨会话记忆

**解决问题**：每次会话都要重新探索，浪费时间和 token

**结构**：
```
memory/MEMORY.md          → 跨会话入口（常驻注入 L1，≤500 tokens）
memory/repos/{仓库}/      → 仓库知识（工具检索，按需加载）
memory/features/{功能}.md → 历史结论（工具检索，按需加载）
memory/completed.json     → 追齐状态（常驻注入摘要）
```

**适用**：需要跨会话积累知识的 Coding Agent / 研究 Agent
**来源**：[4.5 MEMORY.md 模式](../04-memory-system/4.5-跨会话连续性MEMORY.md模式.md)

---

### 模式 7：记忆巩固三条件

**解决问题**：什么信息值得写入 External Storage？写多了是噪声，写少了失效快

**判断公式**：同时满足以下三条才值得巩固：
1. 跨会话复用价值：未来的会话还会用到吗？
2. 获取成本高：重新获取需要 3 次以上工具调用吗？
3. 内容稳定：30 天内不会发生根本性变化吗？

**适用**：记忆巩固决策
**来源**：[4.2 记忆巩固](../04-memory-system/4.2-记忆巩固.md)

---

## 四、规划与执行模式

### 模式 8：混合规划

**解决问题**：预先规划太死板（计划经常需要修改），纯动态规划太低效（每步都要重新决策）

**结构**：
- 预先规划：任务开始时生成任务列表（高层，粒度 = 一个可独立完成的子任务）
- 动态执行：每个子任务内部动态决策，不强制预定义步骤

**适用**：有多个独立子任务的长任务（如追齐 15 个功能点）
**来源**：[5.1 任务分解与规划](../05-planning-long-tasks/5.1-任务分解与规划.md)

---

### 模式 9：检查点三文件

**解决问题**：长任务中途失败，从头重来

**结构**：
```
MEMORY.md          → 跨会话知识（任务无关）
completed.json     → 已完成单元（只增不减）
task-state.json    → 当前执行状态（任务级，完成后清空）
```

**恢复协议**：启动时检查 task-state.json → 存在则从断点恢复 → 不存在则新建

**适用**：执行时间 > 10 分钟的长任务
**来源**：[5.3 检查点与断点续传](../05-planning-long-tasks/5.3-检查点与断点续传.md)

## 五、安全控制模式

### 模式 10：Harness 四层

**解决问题**：Agent 做出不可逆操作 / 权限越界 / 成本失控

**结构**：
```
环境层：文件系统权限、网络访问控制（操作系统 / 容器级）
工具层：路径白名单、操作类型限制、Audit Log（代码实现）
协议层：行为约定、输出格式要求（System Prompt）
审查层：输出验证、高风险操作人工确认（代码 + 流程）
```

**关键原则**：权限控制在代码层，不在 System Prompt 层（System Prompt 可被 injection 绕过）

**适用**：有写入操作的生产级 Agent
**来源**：[6.1](../06-harness/6.1-什么是Harness.md)、[6.3](../06-harness/6.3-工具权限设计.md)

---

## 六、多 Agent 模式

### 模式 11：Orchestrator-SubAgent 层级

**解决问题**：单 Agent 上下文窗口不足 / 任务需要并行 / 不同子任务需要不同专业化

**结构**：
```
Orchestrator（协调，上下文精简）
  ├── SubAgent A（专业化，独立上下文）
  ├── SubAgent B（专业化，独立上下文）
  └── SubAgent C（专业化，独立上下文）
共享状态：task-state.json（Agent 间通信，不直接传递全量输出）
```

**引入门槛**：单次任务消耗上下文 > 50%，或有明显可并行的子任务

**适用**：复杂长任务、多仓库并行探索
**来源**：[7.3](../07-multi-agent/7.3-Orchestrator设计.md)、[7.5](../07-multi-agent/7.5-Agent间通信.md)

## 反模式警告

> [!warning] 这些做法看似合理，但经常导致系统性问题

| 反模式 | 症状 | 正确做法 |
|--------|------|---------|
| 把权限检查写在 System Prompt 里 | 被 prompt injection 绕过 | 在代码层实现权限验证 |
| 把所有工具都常驻 System Prompt | System Prompt 越来越长，效果下降 | Skills 按需加载 |
| 不做记忆巩固，每次从零开始 | 重复探索，成本高，效率低 | MEMORY.md + 记忆巩固协议 |
| 一遇到复杂任务就上多 Agent | 协调成本 > 收益，调试困难 | 先用单 Agent 验证，再按需引入多 Agent |
| 把 Agent 完整输出直接传给下一个 Agent | 下一个 Agent 上下文爆炸 | 提取结论摘要（< 1000 tokens）再传递 |
| 让 LLM 决定是否执行高风险操作 | 偶发幻觉导致不可逆损失 | 高风险操作强制人工确认或代码验证 |
| 不做评测就上生产 | 问题无法发现，更无法改进 | 先建最小评测集（15 个用例），再上线 |

---

## 模式选择速查

```
新 Agent 起步 → 模式 1（五层预算）+ 模式 4（ACI 设计）
工具多且复杂 → 模式 2（按需加载）+ 模式 5（粒度三问）
需要跨会话 → 模式 6（MEMORY.md）+ 模式 7（巩固三条件）
长任务 → 模式 8（混合规划）+ 模式 9（检查点三文件）
生产上线 → 模式 10（Harness 四层）
多仓库/并行 → 模式 11（Orchestrator-SubAgent）
```
