# 模板工作台

> [!abstract] 文档定位
> 把整套课程的设计决策整合成可直接填写的模板。
> 做新 Agent 时，复制对应模板，按提示填写，不需要从空白开始。
> 包含：Agent 设计文档、上下文预算表、工具设计卡、记忆系统方案、评测方案、生产化清单。

## 模板五：评测方案

> 适用于：第 8 章学完后，为 Agent 建立最小评测体系

```markdown
# [Agent 名称] 评测方案

## 评测维度（选 2-3 个主要维度）
| 维度 | 指标 | 测量方法 | 目标值 |
|------|------|---------|--------|
| 正确性 | 任务成功率 | 人工验证 | ≥ 80% |
| 效率 | 平均工具调用次数 | 日志计数 | ≤    次 |
| 成本 | 平均 token/任务 | 费用追踪 | ≤    tokens |
| 鲁棒性 | 边界用例通过率 | 自动评测 | ≥ 70% |
| 可解释性 | 输出理由可理解 | LLM-as-Judge | ≥ 4/5 分 |

## 测试用例集（最小可行：15 个）
| 类型 | 数量 | 说明 |
|------|------|------|
| 黄金路径 | 5 | 标准场景，期望成功 |
| 边界用例 | 5 | 非标准输入，测试降级 |
| 对抗用例 | 3 | 故意模糊/错误输入 |
| 回归用例 | 2 | 历史 bug 场景 |

## 评测流水线
- [ ] Fixture 仓库/数据准备
- [ ] Agent 执行脚本
- [ ] 规则检查（格式/权限/预算）
- [ ] LLM-as-Judge（主观质量）
- [ ] 报告生成

## LLM 评判 Prompt 核心要素
- 评判维度：
- 评分标准（1-5 分）：
- 要求引用证据：[ ] 是  [ ] 否

## 评测节奏
- 开发期：每次改动后跑
- 稳定期：每周一次
- 深度分析：每两周一次
```

## 模板六：生产化检查清单

> 适用于：第 9 章学完后，Agent 上线前的最终检查

```markdown
# [Agent 名称] 生产化检查清单

## 阶段一：MVP（上线前必须完成）
- [ ] 评测通过率 ≥ 80%（正确性维度）
- [ ] Harness 基础控制（路径限制 + 循环检测）
- [ ] 结构化日志（JSONL，含 session_id）
- [ ] token 预算上限（每次任务 ≤    tokens）
- [ ] 人工确认节点（高风险操作）
- [ ] MEMORY.md 初始化完成

## 阶段二：稳定化（上线后 2-4 周内完成）
- [ ] 完整 Audit Log（每次写入操作记录）
- [ ] 完整评测体系（15 个用例 + 自动流水线）
- [ ] 成本监控（每日费用追踪 + 超预算报警）
- [ ] 检查点 + 断点续传
- [ ] 优雅降级报告（失败时输出已完成部分 + 原因）
- [ ] 用户通知设计（任务完成摘要）

## 阶段三：规模化（稳定运行 1 个月后评估）
- [ ] 多 Agent 架构（如有并行需求）
- [ ] 模型分层（简单任务用小模型）
- [ ] 自动化评测 Dashboard
- [ ] 版本管理（Prompt 版本化 + 记忆迁移脚本）
- [ ] 灰度发布流程

## 上线前最终确认
- [ ] 所有高风险操作有人工确认或代码验证
- [ ] 权限控制在代码层，不依赖 System Prompt
- [ ] 有明确的回滚方案
- [ ] 团队有人了解系统架构和紧急处理流程
```
